! The oracle cannot gate a cross-tick cache. One catchUp call never hits the reuse path, so byte-identity of the one-shot is necessary but not sufficient. The decisive gate is a cached-vs-uncached multi-call harness (rate-cache-check.mts) that advances the real world in many small steps with the cache ON vs OFF and asserts byte-identical output.
1

What the cache does

The premise of the "10×" lever.

advanceIsland recomputes the full per-segment plan (computeRates + eight boundary calcs) on every advance call. For a persistent in-memory world (client per-frame, server hot-world), if a tick lands inside the same segment the previous call ended in, that recompute is redundant: rates are constant until naturalSegEnd (all boundaries are absolute times), so the next call could reuse them — byte-identical to recomputing. Implemented as a WeakMap<IslandState> of the segment plan, keyed on a ctxSig of the rate-affecting context, with a guard that skips any island owning a conditionalBonus node (those read other islands' live state).

Root cause

  • Proximate: findNextCapEvent returns its nowMs horizon when no event is pending, so naturalSegEnd is clamped to nowMs every tick → nothing is ever cacheable (tailSet=0).
  • Fundamental — the premise was wrong: an idle player is not an idle economy. A developed save keeps producing/consuming, and bins hit caps/zero sub-second, so the piecewise integrator's segments are genuinely short (≈ tick-length or shorter) regardless of player idleness. computeRates per tick is real work, not a redundant recompute.

Verdict — abandoned (do not enable)

The cache is correct but delivers ~nothing on a realistic active save and adds complexity to the hottest function. Phase 2 cache reverted; the byte-identical planSegment refactor (Phase 1, commit 1466233) is kept as a clean structural improvement. Master untouched.

If revisited: the benefit is bounded by the fraction of islands in a quiescent steady-state (full/pinned bins, no pending event). It could pay on a maxed-out endgame save, but only after fixing the findNextCapEvent horizon so steady islands get a far naturalSegEnd — and even then it does nothing for mid-development economies. The alternative lever (server hot-world, ~38%, independent of segment length) does not have this failure mode but carries an advanceWorldSystems cadence-verification cost.
