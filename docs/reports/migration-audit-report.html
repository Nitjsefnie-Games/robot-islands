Verdict
The migration was test-green but functionally broken in its own default (REMOTE) mode and carried real anti-cheat & concurrency holes — all 28 confirmed issues are now fixed and verified.
The server ran the authoritative economy with no environment context (F-03/F-18 — biome, Network Consciousness, solar, lattice, toxicity and the player-flagged active bonus all silently disabled), and three intent handlers dropped client payload fields (F-26 crate cargo label → always iron_ore, F-20 ocean platforms, F-21 path-drawn drones, F-27 terrain modifiers). On top of that: req.ip spoofing bypassed the auth rate limit (F-01), concurrent intents lost updates with no DB lock (F-04), and a snapshot-swap orphaned every panel holding a world reference (F-10/F-15). Every finding survived a 2-adversary refutation panel (3 candidate findings were killed by it). All fixes ship with tests; the full suite is green at 3377 tests. A code review of the riskiest fixes caught one over-correction (the rebuild gate, F-11) which was then widened. Residual, documented: the local-save import endpoint still trusts the player's own snapshot content (offline window now bounded) — see SPEC Appendix C.

Auth & crypto (slice 1)

F-01 trustProxy:true + un-sanitizing nginx config lets clients spoof req.ip and fully bypass the per-IP auth rate limit CRITICAL FIXED · f1368db
What was wrong · security
The last migration commit (fbe7d1d) flipped Fastify to `trustProxy: true` so the per-IP auth rate-limit (the brute-force / enumeration / scrypt-DoS defense, max 10/min in app.ts:22-25) buckets on `req.ip`. With trustProxy:true, Fastify/proxy-addr derives `req.ip` from the LEFTMOST X-Forwarded-For entry, which is fully attacker-controlled. The prescribed deployment (server/deploy/README.md step 5) is `location /api/ { proxy_pass http://127.0.0.1:5180; }` — it does NOT replace or sanitize X-Forwarded-For, so a client header passes through (or is appended leftmost). An attacker who rotates the XFF header per request gets a unique req.ip each time, so the rate-limit bucket never accumulates and the limit is completely bypassed. This re-opens unlimited password brute-force and account-enumeration against /api/auth/login and /api/auth/signup, and enables the scrypt event-loop DoS (each call ~100ms blocking). The default @fastify/rate-limit keyGenerator is `req.ip` (node_modules/@fastify/rate-limit/index.js:29), so nothing else mitigates it. The added test (app.test.ts) actually demonstrates the spoof — it drives the bucket purely via an attacker-supplied X-Forwarded-For header and asserts that as correct behavior, cementing the vulnerable design.
Evidence
server/src/app.ts:13
app.ts:13  const app = Fastify({ logger: false, trustProxy: true });

Verified live: Fastify({trustProxy:true}) with header 'X-Forwarded-For: attacker-spoof-9.9.9.9' yields req.ip === 'attacker-spoof-9.9.9.9' (leftmost, client-controlled). Default keyGenerator: node_modules/@fastify/rate-limit/index.js:29 `const defaultKeyGenerator = (req) => req.ip`. Prescribed proxy (deploy/README.md): `location /api/ { proxy_pass http://127.0.0.1:5180; }` sets no X-Forwarded-For of its own.
Fix
trustProxy:true → trustProxy:1 (one trusted nginx hop); deploy nginx now overwrites X-Forwarded-For with $remote_addr. Rate-limit test rewritten to drive the bucket via the real connection + a regression test proving a rotating XFF can't escape it.
Verification
Adversary panel: 1 survive / 1 refute · landed in f1368db · full suite green (3377 tests).
F-02 scryptSync runs synchronously on the shared event loop — every login/signup freezes all WebSocket game traffic for ~100ms IMPORTANT FIXED · f1368db
What was wrong · race-or-async
hashPassword and verifyPassword use the blocking `scryptSync` (not the async `scrypt`), and are called directly inside the /api/auth/signup and /api/auth/login route handlers (routes.ts:36, 53, 55). The server is a single Fastify process on a single event loop (index.ts: one buildApp, no cluster/worker) that ALSO serves the authoritative game WebSocket and its periodic state-push ticks (game/ws.ts). Measured on this box, one scryptSync(N=32768,r=8,p=1) call blocks the event loop ~100ms. So each auth attempt stalls every connected player's WS ticks and intent processing for ~100ms; a burst of concurrent logins (10 = the rate-limit ceiling, and unbounded once the XFF bypass above is used) produces ~1s+ of fully frozen game traffic. This is a regression introduced by the migration choosing the sync API on the request path of a process that also carries real-time game traffic.
Evidence
server/src/crypto/password.ts:15
password.ts:15  const hash = scryptSync(password, salt, KEYLEN, { N, r, p, maxmem: MAXMEM });
password.ts:35  candidate = scryptSync(password, salt, expected.length, { N: pn, r: pr, p: pp, maxmem: MAXMEM });
routes.ts:36  const user = await createUser(pool, req.body.email, hashPassword(req.body.password));
routes.ts:53/55  ok = verifyPassword(user.password, ...) / verifyPassword(DUMMY_HASH, ...)
Measured: scryptSync(...) -> 'scrypt ms: 101' on this server.
Fix
scryptSync → promisified crypto.scrypt; hashPassword/verifyPassword async; callers await. KDF now runs on the libuv threadpool, no event-loop freeze.

F-03 Offline catch-up calls advanceIsland with no RatesContext — biome/NC/solar/toxicity/lattice all ignored, diverging from online play CRITICAL FIXED · 9b18b63
What was wrong · regression
loadAndCatchUp integrates the offline gap with `advanceIsland(state, now)` — passing NEITHER the `ctx: RatesContext` NOR `wallClockNowMs` arguments. The client's economy loop (src/main.ts:2195-2204, 2282-2303) always builds a full RatesContext per island: `modifierMul` (biome multipliers), `ncBuff`, `activeBonusMul`, `terrainAt`, `solarBoost` (Mirror Sat), `geothermalActive`, cross-island lattice pooling, cable-network brownout, and `worldSeed`. With ctx omitted, computeRates falls back to identity defaults (economy.ts:1068-1071 modifierMul=IDENTITY, ncBuff=1, activeBonusMul=1; ctx?.solarBoost??0; ctx?.geothermalActive??false) and advanceIsland's `if (ctx?.worldSeed)` guard (economy.ts:2555) skips advanceToxicityRolls entirely. Concrete effect on a FRESH world (which already ships islands with modifiers — world.ts:613 fertile, 626 mineral_rich, 665 cursed_storms): per biomes.ts:292-295 fertile = extraction ×1.50, mineral_rich = ×1.25, cursed_storms = global ×0.90. So a fertile island under-produces offline by 33%, while a cursed_storms island OVER-produces offline (logging off dodges the storm penalty — an exploit). Buildings accrue ZERO toxicity offline. Lattice chains that depend on cross-island supply stall offline though they run online. This breaks SPEC §15.5 ('weather/toxicity rolls deterministic at any catchup duration'), §2003/§2133 ('offline play and active play the same code path'), and §15.6's authoritative-state contract.

F-04 Concurrent intents from two connections of the same account silently lose updates (no DB lock / transaction / version check) CRITICAL FIXED · e2560bc
What was wrong · race-or-async
applyIntent does a non-atomic read-modify-write: loadAndCatchUp (which itself loads+saves) → handler.apply mutates the in-memory game → saveSnapshot, across three separate awaits with NO row lock, SELECT…FOR UPDATE, advisory lock, transaction, or optimistic schema_version check (grep for FOR UPDATE/pg_advisory/BEGIN/transaction in server/src/game returns nothing). ws.ts:85-90 serializes intents per CONNECTION via a promise chain and its comment claims this 'prevents two intents from racing the same saves row' — but that guarantee is per-connection only. Two browser tabs (two WS connections) for the same user have independent chains and interleave freely against the single `saves` row keyed by user_id (0002_saves.sql:3). Same hazard between an intent on one connection and the 1 Hz periodic state-push tick on another (ws.ts:100, also write-via-loadAndCatchUp). I PROVED this empirically: a test firing two place-building intents at different tiles concurrently via Promise.all both ACK ok:true, yet the stored save contains only ONE workshop (expected 2, received 1) — the later writer clobbers the earlier mutation.
Evidence
server/src/game/intent-runner.ts:41
intent-runner.ts:41-64  `const game = await loadAndCatchUp(...); ... handler.apply(...); ... await saveSnapshot(...)`  — three awaits, no lock.
Empirical: two concurrent applyIntent place-building calls → `AssertionError: expected 1 to be 2` (one workshop persisted, one lost), both acks ok:true.
ws.ts:87 comment: 'This is what prevents two intents from racing (load->apply->persist) the same row.' (only true within one connection)
Fix
Per-account load→apply→persist wrapped in a tx with pg_advisory_xact_lock(hashtext(userId)); concurrent same-account intents serialize. Race test FAILS with the lock removed, passes with it.
Verification

F-05 POST /api/game/import persists fully client-authored snapshot as authoritative — contradicts the 'client numbers are never trusted' contract; arbitrary level/inventory and unbounded offline catch-up IMPORTANT FIXED · e2560bc
What was wrong · security
The import route validates a client-supplied snapshot only for STRUCTURE (isValidSaveSnapshot checks types + that v ∈ SUPPORTED_LOAD_VERSIONS — persistence.ts:1233-1242) and that it deserializes, then stores it verbatim as the account's authoritative save. F-07 POST /api/game/import accepts an unvalidated client snapshot as authoritative state — full anti-cheat bypass IMPORTANT FIXED · e2560bc
What was wrong · security
The migration's whole purpose (SPEC Appendix C trust-surface) is to make the server authoritative so a client can never fabricate state. The import endpoint defeats this: it persists a fully client-supplied snapshot as the account's authoritative game, gated ONLY by `hasSave` (409 if a game already exists) and by STRUCTURAL validity (`isValidSaveSnapshot` + `deserializeWorld` not throwing). It performs ZERO legality validation of the CONTENT — inventory quantities, populated flags, skill points, building tiers, T5 endgame state are all taken verbatim. Any fresh account can therefore POST a hand-crafted snapshot with maxed inventory / all islands populated / all skills unlocked and the server will treat it as legitimate from then on. Every per-intent legality re-check in intents.ts (the anti-cheat work of this slice) is moot once a player can seed arbitrary starting state. The SPEC documents the mechanism (§ Appendix C, lines 2253-2257) as a 'one-time local-save migration' but does not acknowledge that it is an unauthenticated-content trust hole.
Evidence
server/src/game/routes.ts:23-42
if (await hasSave(pool, userId)) return reply.code(409).send({ error: 'game already exists' });
const snapshot = body?.snapshot;
if (!isValidSaveSnapshot(snapshot)) { return reply.code(400)... }
try { deserializeWorld(snapshot, Date.now(), Date.now()); } catch (err) { ... }
await saveSnapshot(pool, userId, snapshot);  // persisted verbatim, no content legality check
Fix
Same fix as F-04/F-05: read-only projection path; persistence only inside the locked intent tx.

F-08 WebSocket intent channel has no per-connection/per-message rate limit — DB-write amplification DoS IMPORTANT FIXED · f1368db
What was wrong · security
The @fastify/rate-limit plugin only guards HTTP routes (global 100/min in app.ts:15, auth 10/min in the encapsulated scope). The WS message handler has no throttle at all. Each inbound frame chains an `applyIntent`, and applyIntent unconditionally runs `loadAndCatchUp` (which itself calls saveSnapshot) plus, on success, another saveSnapshot — so every single client frame forces at least one Postgres write, and `pushState` after each accepted intent forces yet another load+save. A single authenticated socket can fire frames as fast as it likes; the per-connection promise chain queues them all in memory (unbounded `chain` growth) while hammering the saves row. The 64 KiB maxPayload (app.ts:37) caps frame SIZE but not frame RATE. This is introduced by the migration (ws.ts is a new file); the pre-migration client-only build had no server to flood.

F-09 Successful intent can emit a spurious second `{seq:-1, ok:false}` ack when the post-intent state push fails MINOR FIXED · f1368db
What was wrong · race-or-async
