Two behavior-identical render wins; the rest is below the noise floor

Operated the optimizer persona for 5 profile→target→change→measure→keep/revert cycles on the loaded mega-save (level 87, 746 B XP, ~3.2 k in-flight transport bots). Method: live interleaved A/B without reload — wrap renderer.render for render-ms, flip a runtime flag between configs in the same ~3 s window so scene drift cancels. Behavior gate: the 3717 client tests + screenshot/structural equivalence. Result: the per-frame route-chevron re-tessellation was the dominant render cost; instancing it cut render ~1.5 ms, and a now-applicable empty-clear gate saves another ~0.5 ms per idle frame. After those, every remaining target is below the noise floor in the reproducible scene (65.8% CPU idle) or can't be made byte-identical.

The five cycles

One optimization per cycle, biggest profiled target first. Two kept, one reverted on a measured regression, two not pursued (behavior-changing or sub-noise).

1Drop the routes-overlay render groupreverted · regression
Hypothesis
The top render layer (routes-overlay) changes every frame, so its render-group rebuild looked like pure overhead — removing it might let the content batch with siblings.
Measured
GROUP 4.50 ms vs NO-GROUP 4.67 ms — removing it regressed −0.17 ms (every run separated). The group localizes the per-frame dirty; without it the parent re-collects. Reverted.
2Instance route-flow chevrons via a shared GraphicsContextkept · b9b329c
Root cause
One ▶ chevron per in-flight bot (~3.2 k here) slides along its route every frame. paintOverlay accumulated all of them into one Graphics and clear()+re-tessellated the whole batch each frame — cost scales with bot count. A content-signature gate can't help: the geometry genuinely moves.
Change + measured
Tessellate ONE unit chevron into a shared GraphicsContext, pool a Graphics(ctx) per instance, write only per-frame position/rotation (Pixi v8 reuses the one geometry). instanced 2.07 ms vs legacy 3.53 ms → −1.47 ms, every run below legacy. Byte-identical vector output (screenshot A/B). Pool trimmed so catch-up bursts don't strand hidden children.
3Gate the now-empty overlayGfx clear()kept · 6c5e082
Opportunity unlocked by #2
With chevrons moved to pooled Graphics, overlayGfx holds only the draft preview + arrival pulses — usually nothing — yet still ran an unconditional clear() every frame, dirtying the render group for no draw (the AGENTS.md empty-clear anti-pattern, finally applicable).
Change + measured
Resolve draft/pulse eligibility up front; skip the clear+draw block when nothing draws and nothing drew last frame. Skipping a provably-empty clear can't change pixels — behavior-identical. A/B under forced-idle: always-clear 1.5 ms vs gated 1.0 ms → −0.5 ms per idle frame. Amortizes small on this constant-traffic save (~12% idle frames), ≈0.5 ms/frame on a normal quiet-route save.
4Pool the per-frame drone-dot rebuildnot pursued · sub-noise
Target
repaintDroneLayer destroy()s + recreates every drone's Container/Graphics each frame. Measured with 5–6 live drones dispatched for the test.
Measured → verdict
Full cost ~0.27 ms / 5 drones (~0.05 ms each): render 0.2 ms, alloc/GC only 0.066 ms (V8 absorbs it). The instanceable part is just the body; the trail is irreducibly per-frame-variable (many alpha-ramped dots). Sub-noise at realistic drone counts — the code's "O(small), cheap" assumption holds. Not worth the refactor risk (probe reverted).
5Per-frame JS: HUD recompute / pulse rings / 5 Hz economy ticknot pursued · behavior or scope
Evaluated
HUD throttle to 5 Hz — would change the per-frame sparkline sampling window (behavior change). Pulse-ring instancing — the ring's stroke width is constant while its radius animates, so a scaled shared context isn't byte-identical. 5 Hz snapshot computeRates (the 6.6 ms ticker-JS spike) — under the 16.6 ms budget here; optimizing the core economy solver over the mega-world is a careful separate effort, not a safe quick win.
Verdict
Each either breaks the behavior invariant or sits below the noise floor in the reproducible scene. Per the optimizer rule (preserve behavior; don't chase sub-noise), the loop stops here.
2

Final state

After #2 + #3, routes-overlay dropped from the top render layer (1.2–3.8 ms) to ~0. On the loaded scene the renderer sits at render 1.8 ms + ticker-JS 1.8 ms with 65.8% CPU idle — smooth 60 FPS. Both wins scale with transport traffic, so the gain is largest exactly in the busy state that caused the felt lag.
