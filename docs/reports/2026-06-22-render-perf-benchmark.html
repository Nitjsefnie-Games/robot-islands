Benchmark · Robot Islands client · CPU profile · 2026-06-22

The browser lag is the render loop re-tessellating Graphics every frame

A 3-second frame-timing sample on the production (minified) build shows real stutter: mean 48 FPS with ~20% of frames over 32 ms and spikes to 101 ms. An 8-second CPU profile on a non-minified, source-mapped build pins the cost: roughly 17% of main-thread time goes to Pixi re-building Graphics geometry (buildLine/triangulate/packQuadAttributes) every frame, because per-frame overlay clear()+redraw marks their GraphicsContexts dirty. Add a per-frame HUD recompute (~8%) and a 5 Hz snapshot rate-solve on the main thread (~5%), and the frame budget is spent before any real work. None of it is GPU-bound.

The median is a clean 60 FPS — calm frames are fine. The lag is the tail: roughly one frame in five blows the budget, and the worst is 6× over. That signature (good median, fat tail) points at periodic work landing on top of a per-frame baseline, not a uniformly heavy renderer.

Root causes

Verified by reading the render-layer source against the profile, not inferred from names alone.

#CauseEvidence~cost
1Per-frame Graphics clear()+redraw marks GraphicsContexts dirty routes-renderer.ts paintOverlay clears + redraws chevrons / arrival-pulse / draft every frame; the *-overlay.ts family (construction, building-alerts, antenna, satellite, route-bend) all clear() too. Each clear forces re-tessellation. triangulate+buildSimpleUvs in the trace confirm filled shapes rebuild, not just lines.~17%
2Whole scene graph re-collected every frame collectRenderablesSimple recurses with high self-time; static layers (islands, buildings, ocean) aren't cached as a render group, so Pixi walks all of them each frame even when nothing moved.~9%
3HUD recomputes unconditionally per frame hud.ts samples sparklines and rebuilds rate strings every frame; DOM writes are change-gated but the compute that feeds them runs at frame rate, not the 5 Hz the data changes at.~8%
45 Hz snapshot re-solves rates on the main thread applyRemoteSnapshot → refreshRetainedRates → computeRates. Periodic, lands on top of a render frame — the most likely source of the 101 ms tail spikes.~5%
5Interactive hit-test + hover walk the full scene Every pointermove deep-recurses hitTestMoveRecursive over all interactive children; hover then recomputes visible vision cells. Cheap at idle, scales with cursor motion.~4%

Recommended fixes

Ordered by value ÷ risk. F1 alone should recover most of the dropped frames. Pick a disposition per fix; the panel at the bottom copies the set as a prompt to kick off the work.

F1Gate overlay redraws on actual changehigh value · low risk
Change
In routes-renderer.paintOverlay and the *-overlay.ts family, skip the clear()+redraw when the inputs (chevron phase bucket, alert/antenna/satellite sets, draft key) are unchanged since last frame — the same signature-gate pattern already used for the HUD DOM writes. For genuinely animated glyphs (chevrons), animate a cached sprite's transform instead of re-tessellating Graphics.
Expected
Removes most of the ~17% geometry-rebuild cost; eliminates the per-frame dirty flag that drives collectRenderables rebuilds.
F2Cache static layers as a render groupmedium value · medium risk
Change
Mark the static world layers (islands, buildings, ocean/fog) so Pixi doesn't re-collect them every frame — a render group / cacheAsTexture that only invalidates on rebuildWorldLayers(). Also set interactiveChildren=false on layers that don't need hit-testing to cut F5's cost.
Expected
Cuts the ~9% scene-traversal bucket and shrinks the hit-test walk. Risk: cache invalidation must track every state that mutates those layers.
F3Throttle HUD recompute to the 5 Hz tickmedium value · low risk
Change
The HUD numbers only change at the economy cadence (ECONOMY_TICK_MS). Drive the HUD recompute off the same shouldTick gate the economy uses instead of every rAF frame; keep camera/cursor UI per-frame.
Expected
Drops the ~8% HUD bucket to ~⅙ of itself. Low risk — the retained-outputs pattern already decouples display from compute.
F4Keep the snapshot rate-solve off the render framemedium value · medium risk
Change
Investigate whether refreshRetainedRates → computeRates on each snapshot can reuse the server-sent rates instead of re-solving locally, or be scheduled so it doesn't coincide with a render frame. This targets the 101 ms tail specifically.
Expected
Flattens the worst-frame spikes. Risk: client/server rate parity must stay exact — needs care against the §15 trust-surface.
