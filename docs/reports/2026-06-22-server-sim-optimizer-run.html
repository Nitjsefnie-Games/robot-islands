The merge scan was re-running 480× over unchanged geometry — catch-up −36%

Operated the optimizer persona against the server's authoritative per-account catchUp (deserializeWorld + advanceWorldEconomy + advanceWorldSystems — all shared pure functions, so a win speeds the client too), benchmarked on the real mega-save (81 populated islands, 956 buildings, level 98) on an isolated CPU core against a frozen copy of the DB. Gate: a SHA-256 oracle digest of the advanced world, byte-identical across every rep. Result: catchUp min 2035 → 1309 ms (−36%), all behavior byte-identical. The win is structural — the island-merge scan was redoing an O(N²) overlap check every world-systems step over geometry that only changes on a merge.

4Memoize findNextMerge on a geometry+populated signaturekept · 9774dc7
Root cause
findNextMerge runs every world-systems step (~480× for an 8-min catch-up) and its O(N²) islandsOverlap scan dominated. The result depends only on the populated set + each island's position/geometry — which only change on a merge or a population. Memoize on an exact signature of those (O(N) vs O(N²)). Only the common null result is cached (a non-null result triggers an immediate merge → signature changes → recompute), so no stale spec reference can be served.
Measured
Per-function: findNextMerge 1881 → 15 µs/call (−99%). End-to-end: world-systems phase ~990 → ~99 ms; catchUp 2035 → 1309 ms (−36%). Oracle byte-identical across all reps; 39 merge + 7 runtime tests pass.
5Economy core (computeRates / solveFlow)investigated · stop
Findings
After the merge fix, computeRates (25%) and solveFlow (16%) are the bulk — but the cost is diffuse self-time across behavior-critical passes. computeRates is called once per integration segment (no structural redundancy like the merge path); its hot allocations (per-building Object.entries, the derivations signature, the Record building) are individually sub-noise; resolveRecipe is inventory-dependent (not memoizable); the solveFlow inputs are rebuilt per call.
Decision
Per the optimizer rule (preserve behavior; don't chase sub-noise), stop here rather than push many <5% micro-opts or a behavior-risky structural rewrite into the economy core verified only by a single-save oracle.
