Tab-background throttling — correctness audit

GREEN overall 3 yellows Scope routes/economy/drones/orbital/settlement/weather/comm Method static read Files audited main.ts, routes.ts, economy.ts, drones.ts, orbital.ts, settlement.ts, sonar-buoy.ts, weather.ts, daynight.ts, persistence.ts, construction.ts, reactor-toxicity.ts, rate-history.ts
TL;DR

Game-state advancement is robust to tab-background throttling for every player-facing subsystem. Routes, economy, drones, settlement vehicles, sat movement, repair drones, sweeper cleanup, construction, toxicity, day/night, weather, and sonar buoys all read wall-clock (performance.now()) timestamps and correctly catch up across arbitrarily large gaps. A 30-second or 4-hour alt-tab produces the same state as a continuously-foregrounded session of the same wall-clock duration (modulo the 24h offline cap that economy.ts already enforces).

Three yellow flags — all behave-as-spec'd but are frame-rate-dependent by design:

  • F-1 debris hits — one roll per tick (orbital.ts:841). 30s of throttled rAF gives ~30 rolls; foregrounded gives ~1800.
  • F-2 scanner discovery — probability sampled once per tick (orbital.ts:996); dwell accrual is correct, the sampling cadence is not.
  • F-3 comm-packet propagation — one hop per tick, no time arg (orbital.ts:492). Packets stall in throttled tabs.

A fourth observation (yellow, not a bug): F-4 route dispatch bolus — tickRoutes dispatches one batch sized for the full elapsed window on resume, arriving at a single future instant rather than smooth flow. Inventory math is preserved; only the visual / timing shape changes.

The user's Chrome flags (--disable-background-timer-throttling et al.) are a useful belt-and-braces but not load-bearing — the game is already correct without them. The user could share the build with anyone, and they'd see the same game-state-correctness, with three minor frame-rate-dependent edges in the orbital layer.

The user's Chrome flags

Chrome's defaults aggressively throttle backgrounded / occluded tabs. The user disabled these via launch flags. The audit question: what is the game's behaviour without these mitigations?

--disable-background-timer-throttling
By default, setTimeout / setInterval in backgrounded tabs are clamped to 1 second minimum. Disabling restores native scheduling.
--disable-backgrounding-occluded-windows
Stops Chrome from treating a fully covered (but technically focused) window as background. Defaults to ON for visibility-based throttling.
--disable-renderer-backgrounding
Prevents the renderer process from entering a low-priority state when its tab loses focus. Default behaviour deprioritises the renderer thread.
--disable-features=IntensiveWakeUpThrottling
After ~1 minute of being hidden, Chrome caps timer wake-ups to 1 per minute. This flag turns off that secondary tier of throttling.
--disable-features=CalculateNativeWinOcclusion
Stops Chrome from computing whether the window is occluded by other native windows (which would otherwise trigger renderer backgrounding).

Without these flags, three things happen in a backgrounded tab:

  • requestAnimationFrame drops from ~60Hz to ~1Hz (or stops altogether under "frozen" tab lifecycle).
  • setTimeout / setInterval clamp to 1Hz minimum, falling to ~1/minute after 1 min.
  • The Pixi ticker, which is rAF-driven, fires at the same throttled rate.

Audit method

Static read of every tick-driver and time-using subsystem. For each, classify the time source:

  • Wall-clock absolute timestamps (e.g. arrivalTime = nowMs + transitMs, then if (arrivalTime <= nowMs)) — GREEN. Behaviour identical regardless of how often the tick fires.
  • Wall-clock delta integrated into rate (e.g. cleared = sweepers × rate × tickDeltaMs/1000) — GREEN. Same total work over any fixed wall-clock window.
  • Fixed-step or per-tick rolls (e.g. one RNG draw per ticker fire, regardless of elapsed) — YELLOW. Behaviour scales with frame rate.
  • Per-frame counters with no time anchor — would be RED. None found.

The tick loop

Single driver: app.ticker.add(...) in main.ts:1575. PixiJS's Application.ticker is rAF-backed; no maxFPS / minFPS is configured, so it fires at whatever cadence the browser's rAF provides.

src/main.ts:1573–1608 (truncated)let lastFrameMs = performance.now(); app.ticker.add(() => { ... const now = performance.now(); const nowWall = Date.now(); const prevFrameMs = lastFrameMs; const elapsedSec = Math.max(0, (now - lastFrameMs) / 1000); lastFrameMs = now; ... advanceIsland(s, now, { ... }, nowWall); ... const droneResult = tickDrones(worldState, now, prevFrameMs); ... tickRoutes(worldState, islandStates, now, elapsedSec); const orbitalDeltaMs = now - prevFrameMs; tickSatMovement(worldState, now); tickSweeperCleanup(worldState, orbitalDeltaMs); tickDebris(worldState, now); tickScannerDiscovery(worldState, orbitalDeltaMs, now); tickCommPackets(worldState); tickRepairDrones(worldState, now); tickSonarBuoys(worldState); const vehicleResult = tickVehicles(worldState, islandStates, now); });

Every per-frame call gets a performance.now() reading. Routes additionally gets elapsedSec. Orbital sweeper + scanner additionally get orbitalDeltaMs. nowWall = Date.now() is threaded to advanceIsland for the day/night cycle (which must be a wall-clock function so it survives page refresh per spec §2.7). No subsystem reads a frame counter, fixed time step, or guesses elapsed.

Findings

F-1 Debris hit probability sampled once per ticker fire YELLOW

tickDebris iterates each locked sat sitting inside a debris field and rolls one RNG draw against debrisHitProbability per tick. The RNG is seeded by ${seed}_debris_${nowMs}_${sat.id}, so different timestamps produce independent rolls — but the number of rolls per wall-clock window scales with frame rate.

src/orbital.ts:840-845const hitP = debrisHitProbability(field, sat) / protection; const rng = makeSeededRng(`${world.seed}_debris_${nowMs}_${sat.id}`); if (rng() >= hitP) { survivors.push(sat); continue; }

Behaviour under throttling. 30s alt-tab @ throttled 1Hz rAF: ~30 hit-rolls instead of ~1800. Sat lifetime in a debris cell is effectively 60× longer in a backgrounded tab.

Spec alignment. Spec §1571: "Hit probability per tick on any satellite within the cell is fragments × hit_constant × cross_section." The "per tick" is explicit — so the implementation matches spec. The question is whether spec should be revised to "per second" given the existence of background tabs.

Severity. Yellow. A player who backgrounds the tab essentially gets free debris-field traversal. Exploitable but cheesy; only relevant once T6 orbital play matters, and the player has to be actively neglecting the tab.

F-2 Scanner-sat discovery probability sampled once per ticker fire YELLOW

Same anti-pattern as F-1, in a different subsystem. tickScannerDiscovery correctly accumulates dwell using tickDeltaMs (so the saturating-exponential dwell ramp is wall-clock accurate), but the per-tick discovery roll uses the ramp output as a per-tick probability.

src/orbital.ts:974, 996-1009// Bump dwell — wall-clock correct. sat.dwellByCellKey[key] = (sat.dwellByCellKey[key] ?? 0) + tickDeltaMs; ... const rng = makeSeededRng(`${world.seed}_scan_${sat.id}_${nowMs}`); for (const isl of world.islands) { if (isl.discovered) continue; const { cellX, cellY } = tileToCell(isl.cx, isl.cy); const key = cellKey(cellX, cellY); if (!covered.has(key)) continue; const dwell = sat.dwellByCellKey[key] ?? 0; const effectiveDwell = dwell * (sat.dwellRateMul ?? 1); const p = scannerDiscoveryProbability(effectiveDwell); if (rng() < p) { ... } }

Behaviour under throttling. Dwell continues to accrue correctly (so the asymptotic probability is right), but the roll happens once per ticker fire. 30s @ throttled rAF samples the (now high) probability ~30 times instead of ~1800 — discovery still happens at higher dwell, just slower.

Spec alignment. §1604 references "p per tick"; §1934 Appendix A also "per tick". Per-spec.

Severity. Yellow. Lower impact than F-1 because the asymptote (SCANNER_ASYMPTOTE_P_PER_TICK) eventually drives discovery to certainty; throttling delays it, doesn't break it.

F-3 Comm-packet propagation: one hop per ticker fire YELLOW

tickCommPackets takes no time argument and advances every in-flight packet exactly one hop per call.

src/orbital.ts:492-515 (truncated)export function tickCommPackets(world: WorldState): CommPacket[] { ... for (const pkt of world.commPackets) { ... const next = nextHopToNearestSpaceport(graph, pkt.currentNodeId, targets); if (next !== null) { pkt.currentNodeId = next; } survivors.push(pkt); } ... }

Behaviour under throttling. A packet that would normally hop 60 times per second hops ~1× per second when the tab is hidden. End-to-end delivery latency grows linearly with the throttling factor.

Spec alignment. §1518: "moves through the comm graph one hop per tick." Explicit per-tick semantics; the implementation is faithful.

Severity. Yellow. Comm-packet delivery is a soft constraint (the spec acknowledges buffering); slow delivery during a backgrounded session resolves on resume. No state corruption.

F-4 Route dispatch bolus on long catchup YELLOW

Routes correctly receive elapsedSec as the dispatch window, and capDemand = route.capacityPerSec × ... × elapsedSec is wall-clock correct in total volume. But every batch dispatched in this one tick gets arrivalTime = nowMs + transitTimeSec × 1000 — they all arrive at the same future instant, not spread across the elapsed window.

src/routes.ts:698, 813-814const capDemand = route.capacityPerSec * capacityMul * skillCapMul * airshipMul * weatherMul * elapsedSec; ... d.route.inFlight.push({ resourceId: d.resourceId, amount, arrivalTime: nowMs + d.route.transitTimeSec * 1000, dispatchTime: nowMs, ... });

Behaviour under throttling. After a 4h alt-tab on a route with 60s transit:

  • One huge batch sized for 4h of throughput dispatches at nowMs.
  • Source inventory is correctly debited (route bookkeeping preserved).
  • Destination headroom clamps via planRouteCargo (no overflow).
  • The batch lands at nowMs + 60s — 60s after resume, all at once.

Compare to foregrounded: 4h × 60Hz = ~864000 small batches spread evenly. Same total volume, very different shape.

Severity. Yellow / cosmetic. Inventory totals are correct; only the arrival timing differs. The destination island's advanceIsland integrator handles step-function input fine (it's an event in findNextCapEvent). The visible effect: ledger shows "in-flight: huge" for one transit cycle after resume, then snaps to delivered.

Risks if a player runs without the Chrome flags

Most game state advances correctly. Routes, economy, drones, settlement, weather, day/night, persistence — all are wall-clock-driven and indifferent to rAF cadence. A player on stock Chrome can alt-tab freely and resume to a correctly-advanced world.

Frame-rate-dependent edges live in the orbital layer. Debris exposure (F-1) and comm-packet latency (F-3) lean on per-tick semantics that spec'd as per-tick. A backgrounded tab effectively reduces orbital risk and delivery speed by the throttling factor. Scanner discovery (F-2) also slows but still reaches its asymptote.

Cosmetic timing artefact in routes. The F-4 dispatch bolus is briefly visible in the ledger after a long resume but doesn't affect totals.

Page-lifecycle freeze. If Chrome puts the tab into the deep "frozen" state (which the user's flags do not entirely disable — that's a separate page-lifecycle mechanism), requestAnimationFrame stops firing. Wall-clock subsystems still resume correctly on unfreeze because they read performance.now() at resume time. Per-tick subsystems see zero rolls during the freeze interval, then resume normally.

Setinterval-driven autosave. setInterval(triggerSave, 30000) at main.ts:1557 would normally throttle to 1Hz then 1/min in a hidden tab. The visibilitychange handler at main.ts:1562 compensates by saving on every hidden transition — so the 30s autosave is just a foregrounded freshness floor, not the primary "don't lose progress" guarantee.
