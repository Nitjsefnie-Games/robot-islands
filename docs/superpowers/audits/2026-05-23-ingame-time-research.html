How in-game time is generated

Question Is in-game time derived from the current wall clock, from a per-save anchor, or from a global game-clock? Trace nowMs from the ticker, through the economy, through save/load, including the §13.3 Time Lock acceleration layer. Side-channel — source files untouched.

TL;DR
It's a dual-clock model: a per-island performance.now()-domain economy clock, with the wall-clock Date.now() threaded in as the day-night anchor.
There is no global game clock. Each island carries its own lastTick (economy.ts:272) in the per-page performance.now() domain. The ticker calls advanceIsland(state, now, ctx, nowWall) per island per frame (main.ts:1704) — now drives integration and inventory math, nowWall is forwarded ONLY to the §2.7 solar / day-night sampler so the cycle is "purely time-driven and does not depend on the player's session."
On save, both anchors are captured (savedAt = Date.now(), savedAtPerf = performance.now(), persistence.ts:217–218). On load, the deserializer computes perfShift = nowPerfMs - savedAtPerf - deltaMs (persistence.ts:312) and rebases every persisted perf-domain timestamp into the new session's domain. Offline catchup falls out of this for free — lastTick is set to nowPerfMs - deltaMs (persistence.ts:440), and the next frame's advanceIsland integrates the full gap through its existing event-driven loop. §13.3 Time Lock acceleration is just a per-island 3× multiplier on segment rates — it never decouples the per-island clock from the global frame clock.

Q1

What's the time source?

The canonical nowMs argument is produced once per frame inside the Pixi ticker callback (main.ts:1601):

Per-frame time capture (main.ts:1580 + 1601 + 1608)

Concretely: every economy and routing timestamp lives in the performance.now() domain; only the §2.7 solar / day-night sampler reads Date.now(). Production timestamps elsewhere are minted with performance.now() at the call site — drone launches (main.ts:567), vehicle launches (main.ts:581), orbital launches (main.ts:594), hover tooltip (main.ts:758), initial island states (main.ts:903, 914).

The Pixi ticker's own lastTime is not consulted — the integrator pulls performance.now() directly. Frame-delta is derived as now − lastFrameMs (main.ts:1614) and only used for some subsystem subticks.

Q2

Global clock vs. per-island lastTick

There is no global game clock. The only persisted time field is the per-island state.lastTick (economy.ts:272):

/** Wall-clock timestamp of the last advance, in milliseconds. */ lastTick: number;
— economy.ts:271–272. The comment says "wall-clock" but the value is in fact performance.now()-domain (see persistence.ts:28–41 head comment and state.lastTick = nowMs at economy.ts:1690 where nowMs is the perf-clock from main.ts:1601). The comment is stale.

The factory makeInitialIslandState(spec, nowMs) (world.ts:1033) seeds lastTick: nowMs at island creation (world.ts:1079) — at game-start the seed comes from performance.now() captured in main.ts:903 (home) and main.ts:914 (demo islands).

There is no world.tickNowMs or equivalent on WorldState. Subsystems (drones, routes, vehicles, weather, orbital) re-receive the frame's now from main.ts on every call (main.ts:1785, 1797, 1810–1815, 1830); they don't read a world-level clock field.

Q3

Two time domains, who samples each

performance.now() measures elapsed time since the page document started parsing — per-tab, monotonic, sub-millisecond, and resets to 0 on every page reload. Date.now() is Unix-epoch milliseconds — moves with the system clock, can jump on NTP sync or DST, but is stable across page reloads.

Q4

Save / load and the perfShift

Persistence captures both clocks in the snapshot (persistence.ts:163–175):

FieldDomainCaptured atSite
savedAtwall (Date.now)save timepersistence.ts:217
savedAtPerfperf (performance.now)save timepersistence.ts:218

On load (persistence.ts:292–489):

  1. deltaMs = max(0, Date.now() − snapshot.savedAt) — wall-clock outage in ms (persistence.ts:306).
  2. perfShift = performance.now() − snapshot.savedAtPerf − deltaMs (persistence.ts:312).
  3. Every persisted perf-domain timestamp is rebased by + perfShift: drone launchTime/expectedReturnTime (persistence.ts:364–365), route arrivalTime/dispatchTime (persistence.ts:371–372), vehicle launchTime/expectedArrivalTime (persistence.ts:377–378), repair-drone (persistence.ts:387–388), building placedAt/maintainedAt/toxicityExpiryMs (persistence.ts:330–339), per-island declaredAt/lastResetAt (persistence.ts:434–435).
  4. Crucially, state.lastTick is set to nowPerfMs − deltaMs (persistence.ts:440) — i.e. "as far in this session's past as the outage was long." The next frame's advanceIsland integrates the gap in one piecewise loop.
  5. operatingMs is a duration (not a timestamp) and is not shifted (persistence.ts:311).

Subtle conceptual move: perfShift is not "translate saved-perf → live-perf 1-for-1"; it's "translate so a saved timestamp at savedAtPerf lands at nowPerfMs − deltaMs" — i.e. account for the outage. A drone whose expectedReturnTime elapsed during the outage lands at or before nowPerfMs and is processed as already-arrived on the next tick. Same "1 frame or 24 h, one code path" property the lastTick remap gives the economy integrator.

Q5

Offline catchup timeline

Question 5 is essentially "what happens when the player returns after N hours?" Diagram:

Save → close tab → reload → catch-up integration

The wall-clock delta is the source-of-truth for "how long was I gone." performance.now() can't measure across reloads (it resets to ~0), so the deserializer constructs a synthetic perf-domain past via nowPerfMs − deltaMs and hands the economy a single integrate-this-gap segment. The §15.3 piecewise integrator (subdivided by day-night quadrant boundaries, ramp sub-segments, recipe cap events, etc.) handles the rest transparently.

Q6

Time Lock acceleration (SPEC §13.3)

Time Lock does not decouple the per-island clock from the global frame clock. It is implemented as a runtime rate multiplier inside the same integrator:

  • Banking: if the island has any Time Lock and bankingEnabled is set, advanceIsland short-circuits — instead of integrating production, it adds (nowMs − lastTick) / 60000 minutes into timeLockBankedMin and returns (economy.ts:1480–1487). Capped at 24 h × locks.
  • Spending: banked time is consumed as state.accelerationRemainingMin on a target island; while non-zero, every segment multiplies rates by 3× (economy.ts:1518 accelerationMul; consumed at economy.ts:1665–1675).
  • Concurrency: the integrator clamps the segment end to the moment acceleration runs out (economy.ts:1586–1588), so the piecewise-constant rate invariant still holds.

The accelerated island still pulls real-time inputs from routes/cables. Per spec §13.3, the multiplier never compounds and never exceeds 3×, "to avoid inter-island flow-conservation problems." Implementation reflects this: only accelerationMul changes, never the t-advance step.

Q8

Page-close / browser-quit

document.addEventListener('visibilitychange', …) at main.ts:1569–1571 calls triggerSave() when document.visibilityState === 'hidden' — covers tab-switch, minimize, and the in-flight portion of tab close. There is no pagehide / beforeunload handler — the spec's "primary 'don't lose 30 s of progress' guarantee" rests on this visibilitychange:hidden save plus the 30 s autosave timer (main.ts:1564 SAVE_INTERVAL_MS = 30_000).

Reconstruction on next load depends on Date.now() deltas (the deltaMs at persistence.ts:306). performance.now() deltas are not consulted across the reload boundary — they're meaningless since perf resets to 0 on every page load.

Q9

Failure modes — system-clock jumps

performance.now() is monotonic and immune to NTP/DST. Date.now() isn't. Two distinct failure modes:

  1. Mid-session NTP jump (forward or backward). The per-frame wallOffset = nowWall − now (economy.ts:1472) is recomputed every frame, so the day-night sampler shifts on the next frame. If the system clock jumps forward 2 hours, the next frame's solar multiplier is sampled 2 h later — the day-night phase visibly jumps but the economy itself only sees a single perf-clock frame delta, so inventory isn't double-credited. A backward jump (nowWall drops by Δ) is similarly absorbed — the phase rewinds.
  2. Mid-session clock backward, then save + reload. deltaMs = max(0, Date.now() − snapshot.savedAt) (persistence.ts:306) clamps a backward jump to 0 — so the catch-up synthetic-past is 0 ms and no offline progress accrues. Forward jumps before save would inflate savedAt and then yield a smaller-than-real deltaMs on the next legitimate reload, costing some offline progress. Both are bounded.

The code does NOT mix Date.now() and performance.now() for the same delta in any production-path integrator. The two domains are kept apart, joined only at wallOffset for the solar sampler. However, two UI-facing call sites violate this discipline — see Q10 below.

Q10

Bugs found — domain confusion

Three mixed-domain spots, all UI/cosmetic in impact (none corrupt the economy):

Bug A · cooldown gate in HUD reads wall-clock against perf-domain field. SEVERITY · minor

hud.ts:541 — if (canTierReset(state, Date.now()).ok) { … hint chip … }.
canTierReset at tier-reset.ts:99–101 tests nowMs − state.lastResetAt < TIER_RESET_COOLDOWN_MS. lastResetAt is in perf-domain (perfShift-ed on load, stamped at skilltree-ui.ts:589 via performance.now()). With Date.now() ≈ 1.7e12 and lastResetAt on the order of 1e4–1e7 even right after a reset, the difference is always >> 24 h — the HUD cooldown gate always evaluates as off-cooldown. Result: the HUD "tier-reset available" hint chip flashes even within the 24 h cooldown window. The actual click-handler (skilltree-ui.ts:589) correctly uses performance.now(), so the user can't actually fire a reset — but the hint is misleading.
Bug B · inspector's manual "Refresh maintenance" stamps wall-clock into perf-domain field. SEVERITY · cosmetic

inspector-ui.ts:1021 — tryRefreshMaintenance(building, def, inv, Date.now(), …). tryRefreshMaintenance (maintenance.ts:187–204) writes maintainedAt = nowMs. Every other production write of maintainedAt uses perf-domain (placement.ts:450, economy auto-maintain via maintenance.ts:172).

Reads: a global grep maintainedAt shows the only runtime read is tutorial.ts:165 (b.maintainedAt > b.placedAt) and the inspector UI display. Maintenance gating uses operatingMs (reset to 0 on refresh), not maintainedAt, so the wrong domain in maintainedAt has no functional effect — but the inspector's "Maintained X ago" readout will show a multi-decade-in-the-past value the moment the user clicks Refresh.
Bug C · weather() samples day-night phase in perf-domain. SEVERITY · cosmetic / spec-drift

Per spec §2.7, the day-night cycle is "purely time-driven and does not depend on the player's session." weather() (weather.ts:135–171) is called with perf-domain nowMs from main.ts:1944 (overlay), routes.ts:556 (route-loss), and weather-overlay.ts:142/157 (forecast). Inside weather(), dayPhaseName(t) at weather.ts:151 boosts severe-storm weights by 1.25× when the phase is night or dawn.

Because weather() samples dayPhaseName on a perf-domain t, the night/dawn boost shifts across page reloads for the same cell+seed at the same wall-clock moment. The base sequence (the seeded RNG draw chain rooted at ${seed}_weather_${cx}_${cy}) is still deterministic — but the per-step weight set isn't. A forecasted severe storm "for tomorrow at 23:00" may not appear at 23:00 after a long-outage reload, because the per-page perf clock has been rebased while the weather function still keys phase off its own perf-domain integrator. Fix would be the same pattern as advanceIsland: pass a wall-clock anchor in and key dayPhaseName off that.

None of the three corrupts inventory, drones, routes, or saves. They are all consistency holes between (a) the explicit perf-vs-wall discipline the integrator/persistence layer follows and (b) ancillary call sites that bypass it.
