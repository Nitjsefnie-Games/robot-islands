01

Problem

Phase 2's CO₂ accrual is monotonically increasing. Phase 6 will read the pool for weather thresholds; without drains, the player has no agency. The doc's §7.4 names three sinks; Phase 5 ships them.

  • wastewater_treatment at building-defs.ts:1621 exists today (T3 water-cleanup); its real-world analog (algae bioreactor / activated-sludge treatment) absorbs CO₂. The rev-16 doc says ~5 kg/cycle.
  • exhaust_scrubber at building-defs.ts:1639 exists today (post-combustion gas treatment); real flue-gas scrubbers capture 5-20% of CO₂. Per rev-16, 20 kg/cycle adjacent to a CO₂-producing source.
  • plant_a_tree doesn't exist. Per rev-16 §7.4: new T1 placeable, forest biome only, 0.1 kg/cycle. Slow but free.

2.2 exhaust_scrubber

Per rev-16 §7.4: 20 kg/cycle when adjacent to a CO₂-producing source. The adjacency gate uses the heat-resolver's 4-neighbor footprint pattern.

// building-defs.ts:1639 — extend exhaust_scrubber def.
{
  id: 'exhaust_scrubber',
  /* …existing fields… */
  co2CaptureKgPerCycle: 20,
  co2CaptureAdjacency: ['coal_gen', 'biomass_plant', 'cement_mill', 'limekiln'],
}

The adjacency field is new: co2CaptureAdjacency?: ReadonlyArray<BuildingDefId> on BuildingDef. The integrator drains only when at least one adjacent building (per 4-neighbor footprint) matches an entry. Absent → drain unconditionally (wastewater_treatment uses this).

3.1 Recipe-less sink — integrator handling

The plant_a_tree has no real recipe (no inputs, no outputs). Two implementation choices:

  • (a) Synthetic recipe. Add a no-op recipe in RECIPES.plant_a_tree with cycleSec: 60, inputs: {}, outputs: {}. The CO₂ drain hook fires on every cycle completion. The recipe-graph treats the building as a leaf node with no resource implications.
  • (b) Synthetic running gate. Treat any building with co2CaptureKgPerCycle > 0 and no recipe as "always running"; the integrator applies the drain at a fixed cadence (e.g. 60s) regardless of recipe.

Phase 5 picks (a) — synthetic recipe. Aligns with the existing infrastructure (the recipe-graph code path is well-tested). The no-op recipe is a one-line entry in RECIPES with empty inputs / outputs. The drain fires as a side-effect of recipe progress, same as the other sinks.

04

Drain integrator hook

Inside advanceIsland's segment integrator, after Phase 2's accrual hook. The drain runs per building, scaled by segment progress.

// economy.ts — append to the segment loop, after Phase 2's accrual hook.
for (const b of runningBuildings) {
  const def = BUILDING_DEFS[b.defId];
  const capture = def.co2CaptureKgPerCycle ?? 0;
  if (capture <= 0) continue;

  // Adjacency gate — exhaust_scrubber requires adjacent CO₂ source.
  if (def.co2CaptureAdjacency) {
    const hasAdjacentSource = adjacentMatchesAnyDef(
      b, state.buildings, def.co2CaptureAdjacency,
    );
    if (!hasAdjacentSource) continue;
  }

  // Per-cycle drain × cycles this segment × running factor (heat throttle, etc.).
  const recipe = resolveRecipe(b, state, nowMs);
  const cyclesThisSegment = recipe
    ? (segmentDurationSec / recipe.cycleSec) * runningFactorFor(b)
    : segmentDurationSec / 60;  // recipe-less sink fixed cadence

  const drainKg = capture * cyclesThisSegment;
  state.co2Kg = Math.max(0, state.co2Kg - drainKg);
}

The zero clamp is load-bearing. Phase 6 reads state.co2Kg for weather thresholds (rev-16 §7.3). If the integrator allowed negative values, a "preemptively scrubbed" island would have negative CO₂ contributing negative emissions when summed for world-total — Phase 6's weather logic would behave erratically. The clamp ensures CO₂ stays non-negative.

Order of operations. Per segment: Phase 2's accrual fires first (CO₂ accrues from producing buildings), then Phase 5's drain (sinks subtract). This means instantaneous net CO₂ during a segment is order-dependent in edge cases (e.g. exhaust_scrubber adjacent to coal_gen — the scrubber drains AFTER the gen accrues). Per rev-16 this is the correct order — scrubbers capture emissions in real industrial operation.

06

Test plan

6.1 Single-source accrual

coal_gen at 1 coal / 2 s producing 2.42 kg CO₂ / coal → 1.21 kg/s. After 60 s of operation, state.co2Kg = 72.6 kg.

6.2 Adjacent exhaust_scrubber drain

Same coal_gen + adjacent exhaust_scrubber at 20 kg/cycle (cycleSec inherited from recipe; assume 60 s default for the scrubber's recipe). Over 60 s: accrual +72.6 kg, drain −20 kg. Net state.co2Kg = 52.6 kg.

6.3 Standalone wastewater_treatment

No adjacency requirement. 5 kg/cycle drain over 60 s: state.co2Kg drops by 5 kg (assuming sufficient initial value). If state.co2Kg = 3 kg initial, drops to 0 (zero clamp).

6.4 plant_a_tree slow drain

No-input building. 0.1 kg/cycle at cycleSec 60 s = 0.0017 kg/s. After 10 minutes, drains 1 kg. Bounded; verifies the no-input path works.

6.5 Zero clamp

state.co2Kg = 0; wastewater_treatment runs; after 60 s state.co2Kg still 0 (not negative).

6.6 Existing suite continuity

npm test green. Phase 5 adds ~8 new tests, touches no existing.

Risks

Risk
Sev
Mitigation
A scrubber adjacent to multiple CO₂ sources double-drains.
LOW
The drain is a fixed per-cycle value; the adjacency gate is a boolean (any matching neighbor). No double-counting.
plant_a_tree's "no-op recipe" pattern is novel; existing economy code paths may not handle it cleanly.
MED
The no-op recipe pattern matches existing constructs (some maintenance recipes have empty outputs). If computeRates rejects empty outputs, fallback to the synthetic running gate (§3 option b).
A future building adds co2CaptureKgPerCycle without an adjacency check, draining everywhere.
LOW
Adjacency is optional by design; wastewater_treatment intentionally drains without one. The schema allows both shapes.
The zero clamp causes a discontinuity when the pool first hits zero — the inspector's per-tick rate display shows -X kg/s then 0 kg/s abruptly.
LOW
Cosmetic. Inspector can show "drain rate (would be X if pool larger)" if needed; Phase 5 doesn't ship that UI.
09

Non-goals

Weather coupling
Phase 6 reads state.co2Kg and applies storm-probability thresholds. Phase 5 only drains.
Biogenic vs fossil CO₂ accounting
If biomass_plant's biogenic CO₂ should be excluded from weather coupling (rev-16 §7.2 note), that logic lands in Phase 6. Phase 5's drain doesn't distinguish.
A real biological sapling resource
Rev-16 §7.4 floated "new sapling resource OR reuse wood −1/cycle". Phase 5 picks neither — plant_a_tree has no inputs. Future content can revisit.
CO₂ sequestration via deep-injection
A future high-tier sink (T4-T5) using underground reservoirs. Not in rev-16.
