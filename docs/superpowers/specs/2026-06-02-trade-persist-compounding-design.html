Design spec · robot-islands · amends Trade Offers

Trade — persisted cadence + compounding speedup

Context The Trade Offers mechanic shipped with its offer spawn cadence held in ephemeral runtime state keyed on performance.now() — which resets on every page reload. That defeats the 2-hour cadence: load, accept the immediately-spawned offer, refresh, repeat. This spec closes the infinite-trade-via-refresh exploit by persisting the cadence as an online-time cooldown (offers stay ephemeral), and adds a per-island compounding speedup — each accepted trade makes that island's next offer arrive 1% sooner.

Decision
Persist the offer cadence as a per-island online-time cooldown (offers stay ephemeral), and make each accepted trade compound the next offer 1% sooner, floored at ~1 min.
(a) Persisted online-time cooldown. Replace the ephemeral performance.now() spawn gate with IslandState.tradeCooldownMs (persisted), counting down only on online frames — visibilityState === 'visible' && document.hasFocus() — with a per-frame cap so a hidden→focused jump can't dump time. Refreshing with an offer on screen forfeits it; the already-started cooldown runs its course. Exploit closed.
(b) Per-island compounding speedup. New IslandState.tradeAcceptCount; effective cadence = max(FLOOR_MS, baseCadence × 0.99^tradeAcceptCount), FLOOR_MS = 60_000. Per-island matches the per-island Signal Exchange; the floor keeps the compounding fast-but-bounded so it can't soft-reopen the farm.
Deliberately unchanged: offers remain ephemeral and grant no XP; online-only / offline-parity (§15.5) is preserved (offline advances neither the cooldown nor the accumulator); valuation, bias, size and reach are untouched.

01

The exploit

Two independent causes, both in the spawn-gate.

The cadence gate lives only in TradeRuntime.nextSpawnAt — an in-memory Map rebuilt empty on every load — and is compared against performance.now(), which resets to ~0 on every page reload. On a fresh load, due = nextSpawnAt.get(id) ?? 0 is 0, so an offer spawns immediately for every Signal-Exchange island, regardless of how recently one was taken.

The loop: load → accept the instant offer → refresh → repeat, bypassing the 2-hour cadence entirely and reshaping inventory (and, with favorable spread nodes, gaining weight) without limit.

The original design made offers ephemeral on purpose (kept out of save / offline-replay). That decision is sound and retained. The oversight was that the cadence gate was also ephemeral. Ephemeral offers are fine; an ephemeral cadence gate is the bug.

02

Why this fix (rejected vs chosen)

Two clock models for the persisted cadence; only one keeps offline-parity.

Rejected — wall-clock next-spawn
  • Persist nextSpawnAt as real (Date.now) time. Simple and refresh-proof.
  • But offline time then counts toward the next offer — a player returning after 2h gets one immediately.
  • That softens the online-only pillar: presence should be the thing rewarded, not elapsed wall-time.
Chosen — persisted online-time cooldown
  • Per-island tradeCooldownMs counts down only on online frames (visible & focused), capped per frame.
  • Persisted, so a refresh reads the mid-countdown value — the farm is closed.
  • Offline-parity intact: an absent player neither accrues nor loses cadence progress.
  • Offers stay ephemeral — only one number per island is added to the save.
03

Cooldown & speedup lifecycle

Per Signal-Exchange island, each online frame with no active offer.

  1. Online predicate. isOnline() = visibilityState === 'visible' && document.hasFocus(). Visible covers minimize / other-tab; hasFocus() covers another window on top or focus loss. "Covered but still focused" is not JS-detectable — accepted as the limit.
  2. Decrement. tradeCooldownMs = max(0, tradeCooldownMs − onlineDtMs), where onlineDtMs = min(frameElapsedMs, ONLINE_DT_CAP_MS) and is 0 on offline / unfocused frames.
  3. Spawn. When tradeCooldownMs ≤ 0, generateOffer(...) (unchanged) and push to the ephemeral runtime list.

On resolution (not at spawn — so the accept's increment lands on the very next offer):

  • Accepted: tradeAcceptCount += 1, then tradeCooldownMs = effectiveCadence(state), then remove the offer.
  • Expired: tradeCooldownMs = effectiveCadence(state) during the prune.

effectiveCadence(state) = max(FLOOR_MS, tuningFor(mults).cadenceMs × 0.99^tradeAcceptCount). The tuningFor cadence already folds in the Logistics-Network frequency skill multiplier; the accept-speedup stacks on top. First offer stays immediate (tradeCooldownMs seeds to 0).

06

Verification

Pure-layer tests plus the migration; render layer smoke-tested manually.

Exploit closed. A fresh load reads the mid-countdown tradeCooldownMs; no offer spawns until online time elapses. Spawning then "refreshing" forfeits the offer without resetting the countdown.
Online-gating. tickTradeOffers does not decrement when passed onlineDtMs = 0; a hidden→focused jump decrements by at most ONLINE_DT_CAP_MS.
Compounding + floor. effectiveCadence halves by ~69 accepts and never drops below FLOOR_MS.
Resolution resets. Accept increments tradeAcceptCount then resets cooldown; expiry resets cooldown. Next offer reflects the new count.
Migration. A v19 fixture loads into v20 with both fields 0; v20 round-trips identity.
Green gate. Full suite + tsc -b && vite build pass before the PR.

08

Out of scope

Explicitly deferred — considered, not missed.

Offer persistence across reload
Still ephemeral by design — only the cadence cooldown persists.
Tier Reset interaction
tradeAcceptCount is treated as permanent engagement (not in §9.7's reset-cleared set). Revisit if it should reset with a tier wipe.
Global (cross-island) accumulator
Chosen per-island to match per-island cadence/state; a world-wide accumulator is a later idea.
Skill-node integration of the speedup
The 1%/accept is a flat mechanic for now, not a tunable Logistics-Network node.
