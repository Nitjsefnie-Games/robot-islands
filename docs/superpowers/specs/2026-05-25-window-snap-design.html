Window-snap — reject degenerate panel layouts

Context Two panels (settlement-panel, routes-panel) were observed in the live localStorage at w=1, h=1 — invisible 1px squares the player can't recover. Root chain: a panel mounted with collapsed CSS dimensions gets dragged before being resized; promoteToFree captures the 1×1 rect; parseLayoutBlob accepts any w > 0 on next load. Fix: floor both the load-time validation and the capture-time persistence at MIN_PANEL_W = 80, MIN_PANEL_H = 40 (well below the 200×120 user-resize minimum, but well above the degenerate 1×1 state).

Decision
Two-layer guard on persisted panel dimensions, floored at 80×40.
Layer 1 — load-time validation: parseLayoutBlob currently accepts any entry with w > 0 && h > 0. Tighten the guard to w >= MIN_PANEL_W && h >= MIN_PANEL_H. Below-floor entries are silently dropped, matching the existing pattern for malformed entries (drop, fall back to default mount position). Cleanses existing degenerate saves on the next page load.
Layer 2 — capture-time guard: promoteToFree currently persists whatever rect.width/height the panel happens to have at the moment of promotion. Add a Math.max floor against MakeDraggableOptions.minWidth/minHeight defaults (200×120) before persisting, AND apply the floored dimensions to the inline style.width/height so the rendered panel matches the persisted layout. Prevents the original root-cause path from creating new degenerate saves.
No schema bump (the on-disk JSON shape is unchanged; only the validator and capture-time clamp tighten). No new fields. Single file plus a test file.
01

Root-cause chain (verified)

Confirmed by direct inspection of the live localStorage in the user's browser via Daedalus earlier this session:

localStorage['ri-ui-layout-v1'].panels:
  "settlement-panel": { "x":79, "y":0, "w":1, "h":1, "zRank":294 }
  "routes-panel":     { "x":79, "y":0, "w":1, "h":1, "zRank":284 }

Trace:

  1. updatePanelLayout (window-manager.ts:157-172) defaults a fresh entry to {x:0, y:0, w:0, h:0, zRank:0}.
  2. The drag-end handler at window-manager.ts:287-288 only patches {x, y} — never touches w/h.
  3. If a panel is dragged BEFORE it's ever been resized AND its current getBoundingClientRect() is already 1×1 (e.g. routes-panel ships with inline style="width:1px; height:1px" as a CSS placeholder), promoteToFree at window-manager.ts:372-387 captures w=1, h=1 and persists it.
  4. parseLayoutBlob at window-manager.ts:71-77 validates w > 0 && h > 0 — 1 passes.
  5. clampToViewport at window-manager.ts:117-135 permits any w >= 1 via Math.max(1, Math.min(rect.w, viewport.w)).
  6. Result: panel renders at 1×1 forever; invisible to the player; no recovery short of manually editing localStorage or calling resetUiLayout (which wipes everything).
02

Change inventory

src/window-manager.ts (~line 18-22)
Add floor constants. Near the existing LAYOUT_STORAGE_KEY export: 
export const MIN_PANEL_W = 80;
export const MIN_PANEL_H = 40;
Floor chosen well below the 200×120 user-resize minimum (so legitimate small saves still pass) and well above the degenerate 1×1 state (so the observed bug is caught).
src/window-manager.ts:71-77 (parseLayoutBlob)
Tighten dimension validation. Replace w <= 0 and h <= 0 rejection with w < MIN_PANEL_W and h < MIN_PANEL_H. Same drop-silently behaviour for failing entries.
src/window-manager.ts:372-387 (promoteToFree)
Clamp captured rect at the floor. Replace w: rect.width, h: rect.height in the updatePanelLayout call with floored values. Same floored values fed into panel.style.width/height so the rendered panel matches what's persisted. 
const flooredW = Math.max(rect.width, opts?.minWidth ?? 200);
const flooredH = Math.max(rect.height, opts?.minHeight ?? 120);
panel.style.width = flooredW + 'px';
panel.style.height = flooredH + 'px';
updatePanelLayout(id, {
  x: rect.left, y: rect.top, w: flooredW, h: flooredH,
});
Note: promoteToFree doesn't currently receive opts. Either thread opts through from makePanelDraggable (call site at line 268-269 already has access to minWidth/minHeight resolved locally) or use the resolved minWidth/minHeight variables in scope. Implementer should pick the cleaner threading.
src/window-manager.test.ts
Add 4 unit tests.
  1. parseLayoutBlob rejects an entry with w=1, h=1 (panel dropped from result).
  2. parseLayoutBlob accepts an entry at the floor exactly (w=MIN_PANEL_W, h=MIN_PANEL_H).
  3. parseLayoutBlob rejects an entry one below the floor (w=MIN_PANEL_W - 1, h=MIN_PANEL_H OR h=MIN_PANEL_H - 1).
  4. Round-trip with mixed valid + invalid entries — only the valid ones survive.
Skipped: direct test of promoteToFree's clamp — it's DOM-bound (mutates a real HTMLElement's style and calls updatePanelLayout which writes to module-level state). Browser-smoke covers the capture path.

04

Out of scope

Snap-back animation on the cleansed panels
Visual flourish (e.g. "panel pops back into the default zone"). Not load-bearing; the cleansed panel just falls back to its zone-anchored default on next mount.
One-shot localStorage cleanup at mount time
An alternative path would scan-and-clean the existing blob at first makePanelDraggable call. Considered, rejected: parseLayoutBlob already runs at readBlob which fires on first makePanelDraggable (lazy init at line 141). So the load-time filter IS effectively the one-shot cleanup; a separate explicit cleanup pass would be redundant.
Viewport-resize edge cases beyond existing clampToViewport
window.resize already re-clamps every wired panel (window-manager.ts:416-435). No new handling needed.
Fixing the underlying CSS that lets a panel mount at 1×1
The routes-panel ships with inline style="width:1px; height:1px" per the live DOM inspection. The root CSS could be tightened so panels never mount that small — but that's a separate concern (per-panel CSS audit) and the floor here defends against any future panel with the same bug regardless. Layered defense.
