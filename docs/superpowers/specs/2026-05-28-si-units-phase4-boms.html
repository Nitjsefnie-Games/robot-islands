01

Problem

v3 audit finding: every placement cost is a coarse footprint × tier-density formula. A T1 Smelter costs 70 kg of materials yet houses a furnace whose real analog weighs 550 kg. A Nuclear Reactor's 770 kg cost is 1000× too cheap by embodied mass. The economy lies about scarcity.

Verified by direct read:

  • mine at building-defs.ts:634: { stone: 30, wood: 15 } = 45 kg vs analog ~300 kg.
  • smelter at building-defs.ts:655: { stone: 80, wood: 40, iron_ingot: 30 } = 150 kg vs ~550 kg.
  • battery_bank at building-defs.ts:867: { saltwater_cell: 4, wire: 3, steel_beam: 1 } = ~10 "units" but under §2.2 unit-mass anchors actually 340 kg.
  • nuclear_reactor placementCost (search): pre-rev-4 was ~770 kg vs ITER-class small modular reactor analog at 832 t (1000× under).

Storage gap. BASELINE_STORAGE_CAP = 2000 at constants.ts:20 applies the same 2000-unit cap to every resource regardless of category — bulk minerals, precious components, and crystals all share the same default. Under rev-16's per-category cap discipline this collapses to absurdity: 2000 ai_cores = 260 t of compute, 2000 antimatter_propellants = 2000 ng of antimatter (more than CERN produces in a century).

02

BOM methodology — rev-16 §4.1

Per-building Bill of Materials derived from a real engineering analog. Five steps per building.

  1. Identify the analog. What real-world thing does this building approximate? (e.g. small bloomery for T1 smelter, ITER for T4 fusion_core, Rocket Lab LC-1 for spaceport.)
  2. Cite the source. A named engineering reference (textbook chapter, NREL ATB, IEA fact sheet, manufacturer datasheet, peer-reviewed paper, Wikipedia + a deep-source corroboration).
  3. Derive the BOM. Mass breakdown by component: structural steel, fire brick, foundation stone, electronics, fluids. Each line is a real number of kg.
  4. Convert to game units. Each kg-mass → RESOURCE_META[r].massPerUnitKg reverse-lookup. 500 kg of steel = 500 units; 100 kg of fire-brick (≈ clay) = 100 units of clay; 5 kg of microchip = 50 units of microchip (microchip = 0.1 kg/unit).
  5. Tile-scale per area. Where the analog is rated per-area (solar arrays, concrete pads), scale by the building's footprint in tiles (1 tile = 1 m²). 4-tile spaceport's 200 t concrete pad = 50 m × 4 m × 1 m × 2400 kg/m³ ≈ 200 t.

Code-comment citation format. Per rev-16 §4.5, every building's placementCost carries a JSDoc comment naming the analog + source. Example:

// BOM source: NREL TP-6A20-72443 Residential PV BOM
// 1 m² c-Si PV: 6 silicon + 2 glass + 1 aluminum + 0.5 wire + 1 backsheet = ~10 kg
placementCost: { silicon: 6, glass: 2, aluminum: 1, wire: 1, plastic_precursor: 1 },

Round-and-ship integer policy. The derived BOM may produce fractional units (e.g. 0.5 wire = 0.25 kg under wire = 0.5 kg/unit). Round to the nearest integer unit count per resource. The mass-balance auditor (Phase 2) does not apply to placement costs (placement is a one-time event, not a recipe); rounding error here is cosmetic.

4.2 The 24h budget

~6 minutes per building. 24 h ÷ 240 buildings = 6 min. Of which 4 min for analog identification + BOM derivation, 1 min for citation lookup, 1 min for the code edit. Realistic per Hartman + IspatGuru reference-checking velocity. Buildings with already-cited rev-16 §4.3 BOMs (30) cost ~1 min each; buildings outside that set cost more — call it 8-10 min for un-templated cases. 30 × 1 + 210 × 8 = 1710 min = 28.5 h, which exceeds the 24 h budget by ~20%. Phase 4 may need 28 h in practice; the plan sets the gate at "all 240 land, even if it takes longer."

4.3 Fantasy buildings

Per rev-16 §4.3 conventions for fantasy spec-mandate materials (exotic_alloy, ai_core, reality_anchor, antimatter_propellant): the citation comment names BOTH the plausible analog (e.g. "DGX H100 130 kg per Sunbird DCIM") AND a // spec-mandate: no real analog tag for the fantasy half (ai_core inputs).

Replaces global BASELINE_STORAGE_CAP = 2000. Per-resource defaults from rev-16 §13.4, with overrides for sub-calibrated resources (helium_3, antimatter_propellant, ai_core).

Resource baseCap Rationale

ai_core (1 unit = 1 kg single chip)0 (whole-unit-only)Rev-9 implementation-time resolution: 1 unit = 1 kg single chip. Vault adds 5000 chips; reality_forge's 4 ai_core/cycle still grindy.

6.3 Algorithm change — aggregateStorageCaps

// world.ts:1018-1045 — replace the BASELINE_STORAGE_CAP loop.
// BEFORE:
for (const r of ALL_RESOURCES) caps[r] = BASELINE_STORAGE_CAP;

// AFTER:
for (const r of ALL_RESOURCES) {
  caps[r] = RESOURCE_BASE_CAP[r] ?? defaultCapForCategory(RESOURCE_STORAGE_CATEGORY[r]);
}

A new RESOURCE_BASE_CAP constant lives in storage-categories.ts (or a new file storage-baseline.ts — implementer chooses). The defaultCapForCategory helper maps the 5 categories to the §6.1 defaults. Per-resource overrides bypass the category default.

6.4 Removal of BASELINE_STORAGE_CAP

The constant at constants.ts:20 is deleted. Imports at world.ts:30 and world.ts:1003 are removed. Grep BASELINE_STORAGE_CAP across src/ to catch any remaining caller. Any test fixture that asserts "2000 default" needs updating to category-default values.

07

Reachability invariant test (binding gate)

Rev-16 §12.9.5 invariant — the player must reach 1 battery_bank within 45 minutes of game time from the rev-9 starter inventory. Phase 4's BOM rewrite shifts every building's cost; this test confirms the chain remains buildable.

describe('reachability invariant (rev-16 §12.9.5)', () => {
  it('walks rev-9 starter → 1 battery_bank in 45 min of simulated game time', () => {
    const homeSpec = makeHomeIslandSpecForReachabilityTest();
    const state = makeInitialIslandState(homeSpec, 0);
    state.inventory = {
      stone: 1200, wood: 600, iron_ore: 30, coal: 80, iron_ingot: 60, bolt: 25,
      limestone: 15, saltwater_cell: 4, foundation_kit: 1,
      // All others zeroed
    } as Inventory;

    const deadline_ms = 45 * 60 * 1000;
    const outcome = simulateOptimalPath(state, deadline_ms, 'battery_bank');

    expect(outcome.reached, 'battery_bank reachable').toBe(true);
    expect(outcome.elapsedMs, 'within deadline').toBeLessThanOrEqual(deadline_ms);
  });
});

simulateOptimalPath helper. A new test utility that walks the dependency graph from a starter inventory toward a target building, placing intermediate buildings as soon as their inputs are available, advancing the simulator state via advanceIsland through each recipe cycle. Implementation lives in src/test-helpers/reachability.ts (new file). The algorithm: BFS over recipe-graph predecessors from the target; for each predecessor, place its source building when its placement BOM is satisfied; advance simulation until the target's placement BOM is satisfied OR the deadline expires.

Binding gate. If this test fails on the post-Phase-4 BOMs, Phase 4 cannot land. Mitigations: bump starter inventory (Phase 7 territory), reduce a specific building's BOM with a citation (acceptable per rev-16 §4.5 per-building citation discipline), restructure the tutorial chain (Phase 7). NOT acceptable: apply a multiplier or early-game subsidy.

09

Test plan

9.1 Reachability invariant (binding gate)

Per §07. Must pass.

9.2 BOM citation snapshot

Every placementCost in BUILDING_DEFS has at least one preceding code-comment line containing BOM source: or analog:. Phase 4 adds the comments alongside the values; a snapshot test walks the source file looking for the comment pattern.

9.3 Mass-positive placement check

Every building's placement cost, when scored as Σ (units × RESOURCE_META[r].massPerUnitKg), is > 1 kg. Catches accidentally-zero placement costs.

9.4 Storage rescale fixture

A test that builds a silo on a fresh island and asserts every dry_goods resource's cap jumps from 100 → 200100 kg (per-resource baseCap 100 + silo capacity 200000). Same for tank, cold_storage, etc.

9.5 Sub-calibrated baseCap

helium_3 storage cap is 1 g on fresh island; a vault adds 5000 g = 5 kg = 1/3 world annual production. Asserts the per-resource override beats the category default.

9.6 Existing-suite continuity

npm test stays green. Phase 4 touches ~10-20 fixtures that hard-coded the old 2000-cap or specific placement costs.

Gameplay implication. Real embodied-mass costs make endgame buildings genuinely expensive. A Fusion Core at 418 t requires the player to have established a 100+ t/hr concrete + steel production chain — the spaceport + bulk_concrete_plant + steel_mill cascade becomes mandatory endgame infrastructure. Non-goals

Weather / CO₂ coupling
Phase 6. Phase 4 doesn't touch weather or CO₂.
Multiplicative storage caps
