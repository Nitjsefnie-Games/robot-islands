Execution contract
Sequential phase execution via subagent-driven-development on master/main. No git worktrees. Phases land in order; each phase's plan is its commit-by-commit walkthrough.
Per-phase shape. Each phase has TWO artifacts: a SPEC (the WHAT and the WHY) at docs/superpowers/specs/2026-05-28-si-units-phase<N>-*.html and a PLAN (the HOW, commit-by-commit) at docs/superpowers/plans/2026-05-28-si-units-phase<N>-*-plan.html. Both are HTML-only per the global doc-format doctrine.
Subagent strategy. Per CLAUDE.md the implementation subagent is kimi via Bash (kimi -w /root/robot-islands -p <brief> --quiet --yolo). Each phase's plan provides per-commit briefs sized for kimi to consume one commit at a time. Claude Opus reviews each subagent commit at the per-phase boundary; any phase that doesn't pass review rolls back without affecting subsequent phases.
Sequential, no breaks. Per user directive (2026-05-28): phases execute back-to-back, no human approval gate between phases (except Phase 7's playtest). Subagent-driven-development reads the next phase's plan as soon as the prior phase lands on master. The lead session monitors but does not block per-phase.
Test gate per phase. Each phase ends with npm test green + npx tsc --noEmit clean. A red test surfaces immediately; the implementer fixes before the next phase begins.
Phase 7 human playtest. The only manual gate. Per rev-16 §11.4 tutorial UX cannot be subagent-verified. ~8 h of in-browser playthrough at https://islands.nitjsefni.eu/ after Phase 7 subagent work lands. Expect 2-4 fix commits during the playtest.

02

Dependency DAG

Sequential execution per user directive. The natural dependency order:

  • Phase 1 → all others. Schema fields (RESOURCE_META, RECIPE_SPECULATIVE, heat KW fields, co2Kg, lat/lon) are foundations everything else reads.
  • Phase 2 → Phase 3, 4, 5, 6. Recipe rewrites + new resources land before placement-cost BOMs (Phase 4), CO₂ sinks (Phase 5), and weather coupling (Phase 6) can use the new chains.
  • Phase 3 → Phase 6. Phase 3's heat throttle + drag retable doesn't depend on Phase 4-5, but Phase 6's full-suite retune needs the engines stable.
  • Phase 4 → Phase 7. The reachability invariant uses Phase 4's BOMs; Phase 7's tutorial walks through them.
  • Phase 5 → Phase 6. CO₂ drains land before weather coupling so the climate-crisis chain has an exit path.
  • Phase 6 → Phase 7. The map picker fires before the tutorial; Phase 7's step 1 is the picker.
  • Phase 7 — final. No phase depends on Phase 7's tutorial; it's the player-facing wrapper.

All phases land in numeric order; no parallel execution. Sequential execution simplifies the dependency contract and avoids merge conflicts on the long-lived files (recipes.ts, building-defs.ts, economy.ts) that multiple phases touch.

03

Schema decisions baked in

Locked at brainstorm time; not subject to per-phase reconsideration.

  • Single RESOURCE_META exhaustive Record with required massPerUnitKg + terminal, optional volumePerUnitL + energyPerUnitKWh. Compile-time enforcement.
  • Sparse RECIPE_SPECULATIVE — only fantasy recipes appear. Real-chemistry recipes carry no entry; the auditor's contract is "absent = must mass-balance."
  • BuildingDef.heatSource.thermalKW? + BuildingDef.heatDemandKW? — both optional. Boolean N:1 fallback when either is absent.
  • IslandState.co2Kg, SerializedWorld.totalCo2Kg, SerializedWorld.playerLat: number | null, SerializedWorld.playerLon: number | null — all introduced by Phase 1's v14 → v15 migration.
  • Per-resource terminal tag (not per-recipe-output) per rev-16 §10.9 resolution.
  • ai_core sub-calibration: 1 unit = 1 kg single chip per rev-9 implementation-time resolution (a).
04

Conventions enforced across all phases

  • Every commit message ends with Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com> when claude authors; Co-Authored-By: Kimi K2.6 <noreply@kimi.com> when kimi authors the implementation commit.
  • Numbers come from rev-16 verbatim. The implementer does NOT re-derive recipe ratios, BOMs, or calibration constants. Rev-16 §X.Y is the canonical source.
  • No multipliers on placement costs. Per rev-16 §4.2 — every BOM is its derived embodied mass; no early-game-cheap reductions, no narrative-cost inflations.
  • HTML-only docs. Per global CLAUDE.md doctrine — specs, plans, audits are HTML; persisted to docs/superpowers/{specs,plans}/; published via docs_hub.py.
  • Mass-balance auditor as vitest assertion (Phase 2). Walks every recipe; fails if Δ ≠ 0 absent exogenousFlow or RECIPE_SPECULATIVE exemption.
  • Reachability invariant as vitest binding gate (Phase 4 introduces, Phase 7 extends). 45-min target; parametric env var TUTORIAL_DEADLINE_MIN for tuning.
  • File:line citations in spec + plan use <a class="cite">file.ts:line</a>.
  • Verification badges per spec template: vbadge v verified, vbadge u unverified projection, vbadge x corrected from prior draft.

06

Execution kickoff

When all 7 specs + 7 plans are reviewed and approved by the user, the lead session invokes /superpowers:subagent-driven-development with the Phase 1 plan as the starting point. The flow:

  1. Read plans/2026-05-28-si-units-phase1-schema-plan.html.
  2. For each commit in the plan: dispatch a kimi subagent with the per-commit brief.
  3. Review the commit on master; if green, proceed to next commit.
  4. At Phase 1 completion, proceed to Phase 2's plan.
  5. Continue through Phase 7.
  6. Phase 7 subagent commits land; pause for the 8 h human playtest.
  7. Apply playtest fix commits; declare full rework complete.

Rollback discipline. Any phase can roll back by reverting its commits — schema, recipes, BOMs, engines all stripe additive changes (Phase 1's schema is the only schema bump; subsequent phases just populate fields and add behaviour). A red Phase N rolls back only Phase N; Phases 1..N-1 stay in place.
