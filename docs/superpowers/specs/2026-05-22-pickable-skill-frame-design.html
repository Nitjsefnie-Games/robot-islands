Pickable-skill yellow frame on the HUD island selectbar

Context The top-HUD island selectbar (mountIslandBar in src/hud.ts:235) currently surfaces per-island power tone but offers no across-the-world cue when a player has skill points ready to spend somewhere. With multi-island save lengths now reaching T3+ and the per-island skill tree (§9.3) holding 225 nodes, unspent points routinely accumulate on the island the player isn't looking at. This spec defines a single global "something to claim" glow that wraps the island-bar panel when any populated island has at least one skill-tree node it could buy right now — reusing the existing canSpend predicate, the existing --ri-warn theme token, and the existing per-frame refresh path.

Decision
Wrap the #island-bar panel in a yellow outer glow (CSS box-shadow) whenever NODE_CATALOG.some(n => canSpend(state, n.id).ok) is true for any populated island.
Reuse the existing canSpend predicate at src/skilltree.ts:641–685 — it already encodes the full pickable rule (cost, tier, depth prereq, branch lock). Walking the catalog with an early-exit makes the check trivial; a memoized cache key (level, points, unlocked-size, sub-path-progress hash) is reserved as a fallback if measurement ever shows it.
Differentiate from the existing per-trigger data-tone="warn" power-low yellow by using an outer glow (soft box-shadow) on the .ri-panel.topbar wrapper, not a flat border — same colour, different visual register, on a different DOM element (parent of the trigger). The two signals can co-occur without merging.

Per-island dot inside the dropdown

When the dropdown opens, each option row carries a second small dot (right side of the row) lit yellow if THAT island has a pickable skill. Lets the player tell which island the panel-level alert is about without leaving the bar.

03

State matrix — when the frame appears

Four cross-cutting per-island states; the frame is driven solely by whether hasPickableSkill is true for at least one populated island. Declaration state is incidental.

State unspentSkillPoints specializationRole / declaredAt Ready skill-tree node exists? Frame Per-island dot
A · idle 0 declared or null — irrelevant n/a (no points) off off
B · declaration pending only 0 specializationRole === null at T3+ n/a (no points) off off
C · pickable skill > 0 any yes — some canSpend(state, n.id).ok ON ON (that island)
D · both > 0 specializationRole === null at T3+ yes ON ON (that island)
E · points but nothing buyable > 0 any no — tier-locked / depth-prereq / branch-locked / fully owned off off

Property the design gets for free. canSpend(state, nodeId) (src/skilltree.ts:641–685) reads only unlockedNodes, unspentSkillPoints, level, and subPathProgress. It does NOT read specializationRole or declaredAt. Row B (declaration pending alone) therefore cannot flip the predicate — no extra exclusion logic is needed. Row D (both signals) lights the frame for the skill point only; the awaiting- declaration cue already exists inside the skill-tree modal at src/skilltree-ui.ts:786 ('◇ AWAITING DECLARATION') and is unchanged.

06

The two helpers, in code

Both helpers are pure, no PixiJS, no DOM — same architectural layer as the rest of skilltree.ts. The walker lives in hud.ts only because it depends on WorldState; if reused later it can move down into a shared module.

// src/skilltree.ts — append near the existing canSpend / nodeById exports.

/**
 * True if `state` has at least one skill-tree node it can buy right now.
 * Composition: NODE_CATALOG.some(canSpend.ok) with early exit. Used by
 * the HUD island-bar to surface a global "claim available" cue without
 * the player opening the skill-tree modal.
 *
 * The rule is canSpend's: enough points, tier met, depth prereq satisfied,
 * sub-path not branch-locked, node not already owned. canSpend reads
 * unspentSkillPoints / unlockedNodes / level / subPathProgress only —
 * so an island with role declaration pending but zero spendable points
 * (or no ready nodes) is correctly false.
 */
export function hasPickableSkill(state: IslandState): boolean {
  for (const node of NODE_CATALOG) {
    if (canSpend(state, node.id).ok) return true;
  }
  return false;
}

// src/hud.ts — inside mountIslandBar's update(), after the per-option
// loop that already runs for each populated island.

let anyPickable = false;
for (const spec of populated) {
  const state = world.islandStates?.get(spec.id);
  if (!state) continue;
  const pickable = hasPickableSkill(state);
  const opt = optMap.get(spec.id);
  if (opt) opt.dataset.pickable = pickable ? 'true' : 'false';
  if (pickable) anyPickable = true;
}
bar.classList.toggle('has-pickable', anyPickable);

09

Verification

Vitest covers the predicate; the visual contract is verified against the live build via the Daedalus screenshot path.

Unit — true case. Construct a fresh L1 IslandState, manually set unspentSkillPoints = 1. hasPickableSkill(state) returns true (mining.1 is buyable).
Unit — empty case. L1, 0 points. hasPickableSkill returns false.
Unit — branch-locked case. Commit one sub-path partially (3 points spent, sub-path not complete), then award one spare point. Two sub-cases: (a) the committed sub-path's next-depth node is buyable at the current tier → hasPickableSkill is true (player can still progress the committed sub-path); (b) the committed sub-path's next-depth node is tier-locked AND every sibling in the same branch is therefore branch-locked AND no other branch has a buyable node → hasPickableSkill is false. Asserts the walker composes with canSpend's branch-lock semantics, not a separately-coded version of them.
Unit — declaration-only case. L15 (T3), specializationRole = null, unspentSkillPoints = 0, every node already owned. hasPickableSkill returns false. Confirms declaration alone never flips the predicate.
Unit — world walker. 3-island world; only the middle island has pickable. The walker returns true. Set that middle island's points to 0 → walker returns false.
Build. npm run build passes strict + noUnusedLocals.
Visual — frame on. Boot, level home to L5, take a level-up. Browser tab reload after npm run build → daedalus screenshot of the top-HUD shows the yellow outer glow on #island-bar.
Visual — frame off after spend. Open the skill tree, spend every point until none remain spendable. Reload tab → screenshot shows the glow gone, trigger pill still in its prior power-tone state.
Visual — coexistence. Force a brownout on the active island (e.g. disable power buildings) AND retain a pickable skill. Screenshot shows trigger pill with yellow border (power-warn) AND panel with yellow glow (pickable) — visually distinguishable.
Visual — per-island dot. 3-island save, two with pickables. Open the dropdown via the bar's trigger; daedalus screenshot shows yellow right-side dot only on the two pickable rows.

Risks

Risk
Sev
Mitigation
Yellow-on-yellow collision with the trigger pill's existing data-tone="warn" power-low border (ui.css:280). If a player has a power brownout AND a pickable skill, two yellow signals appear at once.
MED
Different DOM elements (panel vs. inner trigger pill), different visual register (outer soft glow vs. inner flat 1-px rim), different scale (panel-width vs. pill-width). Acceptable: the two signals semantically reinforce ("this island needs your attention for two reasons"). Mitigation if playtest reads them as merged: bump the glow to 0 0 0 2px var(--ri-warn) + larger blur, or swap the inner trigger border to a different colour token. Do NOT introduce a new colour without a follow-up theme review.
Per-frame predicate cost grows with island count × 225 nodes × canSpend (which itself loops sub-path nodes for the depth-prereq check). Worst case at 20 islands × no pickable: ~20 × 225 × ~10 = ~45k checks/frame.
LOW
Early-exit on the first ok-true makes the common (something pickable) case cheap. The worst case (nothing pickable, max islands) is still <1ms on modern hardware against simple Set/Map lookups. If measurement ever shows it, memoize on the tuple (level, unspentSkillPoints, unlockedNodes.size, subPathProgressSerialized) per island — invalidates only on a level-up, spend, or reset.
A player who has fully maxed every island sees the frame disappear permanently. Could read as "the feature is broken."
LOW
Expected behaviour — there's nothing to claim. The skill-tree modal already shows "complete" sub-paths via the existing ◉ complete caption (skilltree-ui.ts:934). No mitigation needed.
The right-side per-island dot competes for space inside .ri-island-opt, which uses flex: 1 on the name and a level tag (hud.ts:308–314). Long island names may push the dot off-screen on narrow viewports.
LOW
Dot is the last child after the level tag and inherits flex's natural ordering; the level tag already uses ri-mono ri-muted in a fixed slot. CSS: .ri-dot--pickable { flex-shrink: 0; margin-left: 4px; }. If extreme names overflow, the existing dropdown is already min-width: 100% on a flex column (ui.css:291) so the row grows; no truncation regression.
A second, future "needs-attention" signal (route deficit, building stalled) wants the same outer-glow real estate.
LOW
The class is named has-pickable, not has-attention. Future signals add their own class (e.g. .has-stall) and pick a different colour token (--ri-danger for stalls). Composition is per-CSS-rule; no design lock-in.
11

Out of scope

Explicitly deferred. Named so the reader knows it was considered.

Animation / pulse
A pulsing animation on the glow would make it more attention-grabbing, but pulses are already the danger-tone signal (.ri-dot[data-tone="danger"] pulses at ui.css:256). Keep pickable a steady soft glow; reserve pulse for danger semantics.
Click-the-frame-to-jump shortcut
Making the frame itself clickable to auto-jump to the offending island and open the skill tree would be a nice affordance, but blurs the line between status indicator and control. The per-island dot already points to the source; the user can click that row. Defer until playtest shows the two-click flow is friction.
Hover tooltip on the frame
A tooltip listing which islands and which nodes are pickable. Useful but expensive to render every hover-frame at scale. The dropdown already discloses which islands; opening the skill tree per-island discloses which nodes. Defer.
Aggregating "awaiting declaration" into the same frame
The task explicitly excludes declaration. The existing in-modal '◇ AWAITING DECLARATION' cue at skilltree-ui.ts:786 is sufficient. A separate top-HUD declaration cue could be added later but is a different design problem (declaration is one-way per §9.4; pickable skills are reversible via Tier Reset). Don't conflate.
Persisting "I've seen this" dismissal
Some players will leave points unspent on purpose (saving for a future tier). A "dismiss this alert" affordance would silence the frame until next level-up. Not worth the persistence churn until playtest shows the always-on glow is nagging rather than informing.
Memoization of the predicate
The early-exit walker is cheap enough in the common case that memoization would be optimising before measuring. Code is in the spec's risks section for when/if needed; not in scope for the initial implementation.
