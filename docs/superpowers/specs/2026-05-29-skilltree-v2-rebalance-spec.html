Context Builds on the 2026-05-29-skilltree-v2-audit foundation and v1 (2026-05-25-skilltree-rebalance, schema v14, shipped). v1 killed the multiplier runaway by capping every pool at ×10 — but did it by adding nodes, which is why per-node gains are now imperceptible (553 nodes total). v2 keeps the caps, cuts the node counts, consolidates cross-branch duplication, and adds material-input "magic" to refinement.

01

Goal & non-goals

Goal. (a) Cut every subpath to ≤2 filler chains so per-node gains are perceptible; (b) stop branches re-implementing levers that already have a home (the cross-branch dilution the effect-kind audit hid); (c) add a "magical" material-input-efficiency lever to refinement.

Non-goals. No change to v1's caps (×10 / √10 / ÷10 / ×3-xp) or the C^(w/N)−1 formula — keep deriveMagnitudes() + its CI test. No graph/UI/cost-model/aura/crystal/socket redesign. De-noding is a filler-chain operation; notables/keystones change only where a rehomed lever forces it.

03

The de-noding mechanism (Q1)

Filler nodes per subpath are generated from <SUBPATH>_FILLER_ARCHETYPES: FillerArchetype[] in src/skilltree-archetypes.ts, each fed through generateFillerNodes(). One archetype = one effect kind ramped over count depths = one filler chain.

Current bloat: most subpaths carry 3–4 archetypes (Mining = extraction-rate + dry_goods-cap + mineYield + mineRareTrickle). v2 rule: each array drops to ≤2 archetypes, chosen per the ownership map. Subpaths whose levers rehome elsewhere may fall to 1 or 0 chains — sparse stays sparse.

Magnitudes follow automatically: deriveMagnitudes() sizes each node so the chain's product hits its filler share of the (unchanged) cap. Fewer nodes → punchier nodes, no formula change.

"≤2" counts distinct lever-families, not raw archetype rows. A single lever sliced into specific variants (storage's per-category caps, storageCategoryCapMul:<cat> × 4) is one family and doesn't trip the budget. The rule targets a subpath hoarding unrelated generic levers — the dilution v2 removes.

06

Magical input-efficiency lever (Q4)

New SkillEffect kind { kind:'recipeInputMul'; reduce:true } — multiplies recipe input quantities down (outputs unchanged) at runtime in the economy fold. Home: refinement, as the 2nd filler chain in smelting/chemistry/electronics; three chains feed one shared pool capped ÷1.5. Gated T3+, with elevated baseCost/costGrowth so maxing it costs ~1.5–2× a normal pool.

Correction to the audit (verified). "Exempt like RECIPE_SPECULATIVE" is the wrong mechanism. src/mass-balance.test.ts audits the static RECIPES table and skips entries keyed by RecipeId. recipeInputMul is a runtime multiplier — it never edits the static table, so the auditor can't see it and no exemption is needed. The "magic" is fiction-level. If we want an explicit invariant ("only the magical lever may push effective inputs < outputs"), that's a new, separate test — decided: not added (your vote).
07

Magnitudes (Q2)

No formula change. After the chain cuts, re-run deriveMagnitudes() against the smaller per-pool N; the CI guard skilltree-magnitudes.test.ts re-asserts each pool's product equals its cap. The magic pool registers as a reduce-pool with cap 1/1.5. Per-node magnitudes rise purely as a consequence of smaller N.

08

Persistence migration

Removing filler node ids invalidates saved unlockedNodes → schema bump + ladder reset (preserve buildings/inventory), like v1's v13→v14. Current committed schema = SCHEMA_VERSION = 16 (persistence.ts:79); since v2 runs after throughput-floors and more bumps may land first, the target is "current committed + 1 at implementation time", not a hard-coded number — re-read persistence.ts when implementing.

Follow AGENTS.md discipline: SerializedSnapshotV<N> alias, migrateV<N>toV<N+1> (refund SP, clear unlockedNodes, keep buildings/inventory/xp), wire into loadWorld, add to SUPPORTED_LOAD_VERSIONS, tests.
