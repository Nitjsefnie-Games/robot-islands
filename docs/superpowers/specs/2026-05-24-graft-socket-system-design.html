Graft socket system — crafted Skill Crystals on a hand-curated socket grid

Activate the 44 GRAFT_SOCKET_CATALOG entries that currently render as inert dashed circles in the graphview. Players craft Skill Crystals at a new T4 Skill Forge building; each crystal is attached to a single eligible socket, where it adds a small permanent mini-tree (3-5 nodes) of pure-multiplier or unique-effect nodes that integrate with the existing buyNode / cheapest-path solver. Free swap; persists per island. PoE cluster-jewel idiom applied to a sparse, pre-wired socket grid.

Scope is large — 3-5 days estimated end-to-end. Touches: new building def, new recipes, new SkillEffect surface, IslandState shape, persistence schema bump v8 → v9, graphview interactive socket UI, new modal for crystal selection, balance tuning. This spec breaks the work into 6 sub-tasks that can land independently across multiple sessions; ship Task 1-2 first to gate the rest.
Why crystals + sockets, not just more permanent nodes? The hand-curated catalog already has 80 notables + 30 keystones. Crystals add player-driven late-game choice without expanding the fixed catalog further — they are inventory-craftable + freely swappable, which is the only "build-changing" mechanism that doesn't require respec (and v1 has no respec). Players reaching the level where sockets are reachable (sub-path depth 7-11) get a meaningful "I can specialize THIS island differently" lever.

01

System overview

inventory: [ Mining Crystal × 2 ] [ Smelting Crystal × 1 ] [ … ] │ ▼ attach via socket modal graphview: ┌─────────────┐ │ socket ◯ │ ← unbound (current behaviour) └─────────────┘ ── attach Mining Crystal ──▶ ┌─────────────┐ │ socket ⬢ │ ← bound; mini-tree of 3-5 │ ╱│╲ │ crystal-defined nodes │ ◉─◉ ◉─◉ │ becomes reachable via │ ◉ │ cheapest-path solver. └─────────────┘

Crystal kinds are catalog-defined (NOT random per-save); the mini-tree's shape + node effects are constants. What's player-driven is which crystal goes in which socket — a build-shaping choice.

Eligibility: each socket has a subPathId in GRAFT_SOCKET_CATALOG. A Mining Crystal can only attach to sockets with subPathId === 'mining'. This keeps the player from mismatching crystals to thematically wrong sub-paths.

Reachability: sockets in the layout sit past the deepest catalog node of their sub-path (see skilltree-layout.ts). To buy any crystal mini-tree node, the player must first own the socket — which requires owning the deepest pre-socket node of that sub-path. So crystals reward sub-path investment.

02

Data model

Extend skilltree-graph.ts with a CrystalCatalog type. Each catalog entry is a small graph fragment that gets instantiated against a specific socket at attachment time.

// src/skilltree-graph.ts — additions
export type CrystalId = string & { readonly __crystalBrand: unique symbol };

export interface CrystalDef {
  readonly id: CrystalId;
  readonly displayName: string;
  readonly tier: 1 | 2 | 3;        // crafting cost + power scales with tier
  readonly eligibleSubPaths: ReadonlyArray<SubPathId>;
  // Mini-tree fragment — relative to the socket. Node ids and edge ids
  // are SUFFIXES; the engine prefixes them with the socket id at bind.
  readonly nodes: ReadonlyArray<{
    readonly idSuffix: string;          // e.g. 'core', 'a', 'notable'
    readonly cost: number;
    readonly magnitude: number;
    readonly effect: SkillEffect;
    readonly description: string;
    readonly position: { dx: number; dy: number }; // relative to socket centre
  }>;
  readonly edges: ReadonlyArray<{
    readonly fromSuffix: string;  // 'socket' refers to the socket itself
    readonly toSuffix: string;
    readonly cost: number;
  }>;
}

Per-island binding state on IslandState:

interface IslandState {
  // ... existing fields ...
  readonly socketBindings: Map<string, CrystalId>;  // socket.id → crystal.id
  // Crystals are consumed from inventory at bind time. To support free
  // swap, the engine RESTORES the previous crystal to inventory when a
  // socket is rebound. See §06.
}

05

Graphview UI for sockets

The graphview today draws sockets as dashed circles (8-segment dashes) with no interactivity. Add:

  1. Click an empty socket → opens a DOM crystal-picker modal centred on the socket. Lists every crystal in the active island's inventory that's eligible (i.e. its eligibleSubPaths includes the socket's subPathId). Player picks one; engine binds it.
  2. Click a bound socket → opens the same modal pre-populated with the current crystal selected, plus an "Unbind (returns crystal to inventory)" button. Confirm before unbind if any of the bound crystal's mini-tree nodes are owned — unbinding refunds the SP spent on those nodes and clears them from state.unlockedNodes.
  3. Render bound sockets with the crystal's mini-tree nodes (via the existing drawNodes path) + edges (via drawEdges). The mini-tree's nodes appear in DEFAULT_GRAPH.nodes at runtime — see §06.
  4. Hover an empty socket → tooltip "Empty graft socket — craft a Skill Crystal at a Skill Forge and click to attach."
  5. Hover a bound socket → tooltip with crystal name + sub-tree node count + total SP cost.

Modal styling reuses the existing .ri-skillgraph-overlay conventions; the picker is a child overlay that doesn't dismiss the graphview.

06

Engine integration

The graph the cheapest-path solver sees is the union of DEFAULT_GRAPH + every island-bound crystal's mini-tree. Two implementation options:

  1. Lazy graph construction. A new effectiveGraph(state): Graph returns DEFAULT_GRAPH with each bound crystal's nodes + edges appended (with their IDs prefixed by the socket id). Solver calls effectiveGraph(state) at the start of costToUnlock / buyNode. Cost: ~O(n) per call where n = total mini-tree size; negligible.
  2. Persistent graph mutation. Mutate DEFAULT_GRAPH when binding / unbinding. Risky — defeats the readonly nature, and breaks parallel-island reasoning (each island has its own bindings).

Pick option 1. Pure, per-island-safe, and the per-tick cost is dominated by the existing Dijkstra anyway.

Binding write path:

bindCrystal(state, socketId, crystalId):
  if state.inventory[crystalId] === 0 throw
  state.inventory[crystalId] -= 1
  const prev = state.socketBindings.get(socketId)
  if prev:
    state.inventory[prev] = (state.inventory[prev] ?? 0) + 1
    // Refund SP for any owned mini-tree nodes from the previous crystal
    refundSPForMiniTreeNodes(state, socketId, prev)
  state.socketBindings.set(socketId, crystalId)

Effect fold: the existing effectiveSkillMultipliers walks state.unlockedNodes — if mini-tree nodes are in that set, their effects fold naturally with no special-casing. The crystal IS the catalog entry for those nodes.

Persistence + migration

  • migrateV8toV9(s): adds socketBindings: [] to every island state. Lossless; existing saves get empty bindings (no crystals attached anywhere). Inventory items already serialize naturally; no new field there.

08

Balance — discussion

  • Why one crystal per sub-path? Mirrors the sub-path / branch axis the player already understands. 15 crystals × 3 tiers = 45 distinct items, manageable but not overwhelming.
  • Why T4 forge? T4 is when players have stable mid-game logistics + can afford rare resources. Crystals unlock at T4 → reach socket depth (7-11) → bind. ~10-15 levels of progression before crystals matter.
  • Magnitude: tier-3 crystals top out at ×1.5 for a single notable + a few small nodes. Compares to: a hand-curated notable in the main catalog is typically ×1.2-1.3. Crystals are stronger than catalog notables (justifies the crafting cost) but not game-breaking.
  • Per-island scope: bindings are PER ISLAND. Each island can specialise differently; the player isn't locked into one global build.
  • Refund on unbind: player can experiment without permanent loss. Lowers the cost-of-mistake floor; raises engagement.

10

Out of scope

  • Random / generated crystal stats. Every crystal is catalog-defined; nothing rolls per save. PoE-style "stat-rolled" jewels would compound balance complexity and need an itemiser system.
  • Cross-island crystal sharing. Crystals are inventory items on the crafting island; transferring them between islands uses the existing route system if applicable. No special "crystal share" mechanic.
  • Crystal upgrade / fusion. No "combine 3 T1 crystals into a T2 crystal" recipe. Each tier is crafted independently from rarer inputs.
  • Socket count expansion. The 44 hand-curated sockets are fixed. Adding more requires updating GRAFT_SOCKET_CATALOG + the layout module; not a player-facing growth lever.
  • Quest / drop crystal sources. All crystals are crafted at the Skill Forge. A future spec could add quest drops; not v1.
  • Build planner integration. Saved-builds (separate audit item, deferred) would also need to capture socket bindings. Out of scope until build-planner ships.
