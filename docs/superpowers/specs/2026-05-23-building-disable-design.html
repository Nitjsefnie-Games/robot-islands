Per-building enable/disable toggle

Context Today, the only ways a player can stop a particular building from working are demolition (irreversible, partial scrap refund) or indirect starvation (let it run out of inputs). There is no “mothball this thing for a while” switch. This spec defines a per-building disabled?: boolean flag toggled from the inspector. A disabled building is functionally non-existent: no power flow, no recipe in/out, no gate provisioning, no maintenance accrual. Toggling either direction is free, instantaneous, and lossless. The user has confirmed the behaviour; this spec surfaces only the residual design forks for a final pick-and-paste cycle before code lands.

Decision
Add disabled?: boolean to PlacedBuilding; gate the economy tick on it; surface a single Enable/Disable button in the inspector.
The user spec locks the runtime contract. What this design call covers: where the UI lives, what a disabled building looks like, how it composes with the five orthogonal “not running right now” states already in the codebase (construction, ocean-pause, maintenance-degrade, route-draining, in-flight vehicles), and what happens to owned dispatches (routes, drones, helicopters, satellites) when their source is disabled mid-flight.
Insertion chokepoint: state.buildings.filter((b) => !b.invalid && isOperational(b)) at src/economy.ts:617 is the single line every economy pass (rates, gates, power, heat) feeds off. Appending && !b.disabled there knocks out power, recipe, and gate-provider semantics in one stroke. Maintenance accrual at src/economy.ts:1627 iterates raw state.buildings and needs a second skip — that's the only other in-engine site that has to change. UI sites that surface “is there a Spaceport / Reality Forge / Lattice Node on this island” (orbital.ts, lattice.ts, inspector-ui.ts, drones.ts, antenna.ts, lighthouse.ts — the “provider scan” pattern) each grow a !b.disabled filter.

Research — where the flag lands

File-line citations and quoted helpers for every site the change touches. Implementer reconciles with the picks below before writing.

1.1 — PlacedBuilding field, alongside existing mutables

src/buildings.ts:26 declares the interface. The mutable per-instance fields already include cargoLabel (§4.6 relabel), paused (§4 ocean-pause, set by the tick), constructionRemainingMs (§9.3, ticked every segment), operatingMs (§4.7 maintenance accrual), maintainedAt (§4.7 auto-maint timestamp), eternalServitor (§13.3 one-way flip), tier (§14.2 spaceport upgrade), toxicityExpiryMs (§4.5), and invalid (footprint terrain mismatch). The new field slots in identically: 

/** Player toggle (§NEW). When `true`, the building is functionally
 *  non-existent for this tick: no power flow, no recipe in/out, no
 *  gate provisioning, no `operatingMs` accrual. Toggling either way
 *  is free and instantaneous (no resources gained / lost). Optional
 *  so legacy saves load as `false` (enabled) by default. */
disabled?: boolean;

1.2 — Power, recipe execution, and gate provisioning — one chokepoint

src/economy.ts:617 filters state.buildings once per tick into validBuildings: 

const validBuildings = state.buildings.filter((b) => !b.invalid && isOperational(b));

That single array drives all downstream passes inside computeRates:

  • Heat assignment — resolveHeatAssignments(validBuildings, ...) at economy.ts:665.
  • Gate provisioning — checkGates(b, validBuildings, defs, ...) at economy.ts:847, 973, 1017. The neighbor pool is validBuildings, so a building filtered here is invisible to matchesGate (src/adjacency.ts:279).
  • Recipe pass-1 / pass-2 / pass-4 — for (const b of validBuildings) { ... } at economy.ts:702, 952, 1014.
  • Power balance — pass-3 sums powerProduced / powerConsumed only over validBuildings (economy.ts:950–1011).
  • Buff-adjacency — computeBuffStack(b, validBuildings, defs) at economy.ts:799.

Appending && !b.disabled to the filter at economy.ts:617 covers three of the five runtime bullets (no power, no recipe in/out, no gate provisioning) plus heat-source disabling, buff-source disabling, and adjacency-neighbor invisibility. One line.

1.3 — Maintenance accrual — the second site

src/economy.ts:1627 iterates state.buildings directly (NOT validBuildings) to advance operatingMs: 

for (const b of state.buildings) {
  const wasUnderConstruction = (b.constructionRemainingMs ?? 0) > 0;
  if (wasUnderConstruction) { tickConstruction(b, dtMs); continue; }
  // ... recipe-output filter ...
  accrueOperatingTime(b, dtMs);  // ← bumps operatingMs
}

This is intentionally raw because §4.7 says “Idle buildings, stalled buildings, and inactive buildings ... accrue maintenance time the same as actively-producing ones.” The user spec contradicts §4.7 for the disabled case: a disabled building does NOT accrue. Adding if (b.disabled) continue; after the construction check is the second engine edit.

1.4 — Provider-scan sites — the anti-pattern

Anti-pattern
The codebase has ~12 provider-scan sites that walk state.buildings / spec.buildings outside computeRates to ask “does this island have a spaceport?” / “a reality_forge?” / “a lattice_node?” / etc. Each is a separate touch the disabled-filter has to be appended to. There's no shared helper. Adding a hasOperationalBuilding(state, defId) helper that encapsulates !b.invalid && isOperational(b) && !b.disabled would be a small refactor worth bundling.

1.5 — Existing “not-running” states — orthogonality

Five states already mean “this building isn't producing right now.” The new disabled is the sixth, and is player-toggled (the others are engine-driven). Compose semantics:

StateSourceCompose with disabled
constructionconstructionRemainingMs > 0 (§9.3)Orthogonal — see picker p_constructed_disable.
ocean-pausepaused === 'anchor-depopulated' / 'terrain-lost' (§4)Engine-set; disabled OR paused both halt the building. No conflict.
maintenance-degradedmaintenanceFactor(b, def) < 1 (§4.7)Disabled freezes operatingMs — degradation pauses but does NOT reset on enable.

1.6 — Inspector pattern (where the new button slots)

src/inspector-ui.ts already exposes four per-building controls in this house style: the cargo-label dropdown (cargoLabelControls at :654), the Convert to Eternal Servitor button (convertBtn at :857), the maintenance refresh button (refreshBtn at :985), and the demolish button (demolishBtn at :1028). All four follow the same idiom: a document.createElement('button') with a uniform styled(...) stamp, an addEventListener('click', ...) that mutates state and calls paint(). The new disableBtn mirrors this verbatim — same styling, same flow.

1.7 — Persistence round-trip

src/persistence.ts:329 serialises buildings via a shallow spread ({ ...b, ...(timestamp shifts) }). A new optional disabled?: boolean field rides through automatically — no schema bump, no migration. Legacy saves load with the field absent (treated as false). Forward-compat is intrinsic.

1.8 — Render layer

src/buildings.ts:219 (renderBuildings) draws each placed building as: drop shadow + per-tile desaturated fill + glyph. A disabled building gets a visual cue applied here — the exact form is p_visual_cue below. The construction overlay (building-alerts-overlay.ts:82) already shows the precedent for an “overlay glyph + tint” pattern, so the disabled cue can plausibly live in either layer (renderer-level for an alpha/dashed stroke, alerts-overlay for a corner badge).

Picker 03
p_routes_pending_inbound
In-flight route cargo when the source is disabled
A route's sourceBuildingId (routes.ts:107) is the building that “owns” the route — demolishing that building drains the route (drainRoutesForBuilding at routes.ts:899). What about cargo that's already mid-flight at the moment of disable?
current: deliver_anyway
Picker 04
p_routes_disabled_source
What happens to a route whose sourceBuildingId is disabled
Routes have an explicit draining flag set by demolition. Disabling is reversible, demolition is not — so the policy should differ.
current: route_pauses
Picker 05
p_owned_subordinates
Drones / helicopters / satellites in transit when their owner-building is disabled
A Drone Pad has launched drones currently in-flight. A Spaceport has satellites tracked in orbital state. A Launch Tower has a queued launch about to fire. The owner-building is disabled mid-operation. What happens to those subordinates?
current: complete_then_block

p_constructed_disable
Can a building be disabled while under construction?
Construction is the only existing state with a “progress counter”. Disable should be either orthogonal (freeze the counter) or disallowed.

07

Out of scope

Considered, deferred
  • Per-island “disable all of category X” bulk action.
  • Schedule-based disable (cron / day-night-driven).
  • Auto-disable on starvation (the recipe already stalls; auto-disable would be a separate UX pattern).
  • Alert-overlay corner badge for disabled state (orthogonal to p_visual_cue) — consider if user reports the sprite-level cue is too quiet.
Not in this spec
  • Demolition flow changes — demolish remains separate from disable.
  • Construction-cost refund tweaks — disable doesn't refund anything.
  • Multi-select disable (no multi-select UI exists yet).
