Per-pad selection for drone launches

Context The Drone Ops panel currently uses the first operational Drone Pad on the active island for every launch — reticle origin, range ring, preview line, dispatch spawn position. The per-island concurrency cap rejects a second launch while any drone is in flight, so a 2-pad island is functionally a 1-pad island. This spec adds an explicit pad picker and switches the cap rule to per-pad, making multi-pad placements strategic.

01

Goal · non-goals

What this change adds and what it deliberately leaves alone.

In scope
  • Dropdown in Drone Ops panel listing operational pads on the active island.
  • Canvas highlight of the selected pad (matching the cyan reticle palette).
  • Range ring · preview line · DIST/PATH · reticle reachability colour anchor on the selected pad centre.
  • Concurrency cap re-keyed to per-pad: arm-button shows PAD BUSY when the selected pad has an in-flight drone, ALL PADS BUSY when every pad is busy.
  • Default selection: first operational pad in placement order; resets on active-island switch.
  • Fallback: when the selected pad is destroyed or becomes non-operational, switch to the next available pad on the same island.
Non-goals
  • No per-pad fuel queue. Fuel still pulls from the island inventory at launch time.
  • No automatic pad rotation ("round-robin" auto-launch).
  • No cross-island pad selection. The picker only enumerates pads on the currently active island.
  • No Drone-record schema change. Drone.originX/originY already pin the spawn to the launching pad's centre.
  • No persistence migration. selectedPadId is UI state, not saved.
  • No change to T4 Launch Tower pulse or T5 Path Drone Foundry — those are separate dispatch buildings (§11.7 dispatch-capacity table).

User-facing UX

A compact <select> in the Drone Ops dock, sitting just above the existing TIER row so the player reads which pad before which drone tier. Visual highlight on the canvas confirms the choice in world space.

Dock placement & label format

The pad picker is a sibling row in statBlock at drones-ui.ts:286-389, inserted above the TIER row so it reads top-to-bottom as pad → tier → fuel → outbound → flight → dist. The picker mirrors the existing tierSelect pattern (compact <select> styled with the same border/elev/accent rule) so a player who already learned the tier picker meets a recognisable affordance. verified

Label format chosen: Pad #1 / #2 / … in placement order (index into state.buildings filtered to defId === 'dronepad' + isOperationalBuilding). Rationale: tile coordinates would read as Pad @ (5, 12) — accurate but noisy at 248px panel width, and the player rarely reasons about pad positions in island-local tiles. Ordinal labelling matches how the player thinks ("the first pad I built, the second pad I built"). The canvas highlight is the primary spatial cue; the dropdown label is the textual cue. verified panel width 248px at drones-ui.ts:192.

Keyboard accessibility. Native <select> handles Tab focus, Up/Down arrow navigation between options, Enter to commit. No custom roving-focus logic needed — the existing tierSelect at drones-ui.ts:336 is the same pattern and gets the same a11y for free. verified

Canvas highlight. A new selectedPadHighlightLayer (PixiJS Container, world-space, added to the world container so it pans/zooms) draws a 2px cyan rounded-rect outline on the selected pad's footprint when Drone Ops is visible. Colour: VISION_BLUE (matching the reticle and range ring at drones-ui.ts:653, 614). Unselected pads stay neutral — the existing building render at buildings.ts:294-367 already draws them and we leave that untouched, so "unselected" reads as "the normal pad appearance". verified

Data model

Pure UI-side state. No save schema field, no Drone-record field.

Lifecycle. On the first refresh() after mount, on every active-island switch, and whenever the selected pad becomes non-operational (destroyed, set invalid, set disabled), selectedPadId is reassigned to the first id in state.buildings.filter(b => b.defId === 'dronepad' && isOperationalBuilding(b)).map(b => b.id). If that list is empty, selectedPadId = null and the arm-button gate falls through to NO DRONE PAD. verified isOperationalBuilding signature at buildings.ts:150-157.

Interaction with selectedTier. Independent. The tier picker lives at drones-ui.ts:336-369 and is unchanged. Tier and pad are orthogonal choices: switching tier does not reset pad, switching pad does not reset tier. verified

Drone-schema confirmation. Drone.originX/originY at drones.ts:43-44 already capture the launch spawn permanently. They round-trip through persistence at persistence.ts:499-517 (drone serialisation) and persistence.ts:666 (deserialise) with no per-pad field. No schema change is required. verified

New helper: selectedPadCentre

Replaces the existing dronePadCentre at drones-ui.ts:66-77. Same return shape ({ x, y } | null), but takes a pad id and falls back to the first operational pad when the id is null/unknown:

export function selectedPadCentre(
  spec: IslandSpec,
  state: IslandState,
  padId: string | null,
): { x: number; y: number } | null {
  const ops = state.buildings.filter(
    (b) => b.defId === 'dronepad' && isOperationalBuilding(b),
  );
  if (ops.length === 0) return null;
  const pad = ops.find((b) => b.id === padId) ?? ops[0]!;
  const def = BUILDING_DEFS[pad.defId as BuildingDefId];
  return {
    x: spec.cx + pad.x + shapeWidth(def.footprint) / 2,
    y: spec.cy + pad.y + shapeHeight(def.footprint) / 2,
  };
}

The legacy dronePadCentre(spec, state) entry point is kept as a thin wrapper — selectedPadCentre(spec, state, null) — so the unit tests at drones.test.ts:472-488 that directly import dronePadCentre stay green without rewriting. verified

04

Engine contract

The single load-bearing engine change: dispatchDrone trusts the caller's originX/originY, and its concurrency cap becomes pad-aware via coordinate comparison.

Caller is the source of truth for origin

The current dispatchDrone body at drones.ts:441-451 contains a defensive-override block:

// Current (drones.ts:441-451) — silently overrides caller's pad with first pad.
let spawnX = originX;
let spawnY = originY;
const originSpec = world.islands.find((i) => i.id === origin.id);
if (originSpec) {
  const dronepad = findOperationalBuilding(origin.buildings, 'dronepad');
  if (dronepad) {
    const dpDef = BUILDING_DEFS[dronepad.defId as BuildingDefId];
    spawnX = originSpec.cx + dronepad.x + shapeWidth(dpDef.footprint) / 2;
    spawnY = originSpec.cy + dronepad.y + shapeHeight(dpDef.footprint) / 2;
  }
}

This block is deleted. Under per-pad selection it would silently force every dispatch back to the first pad regardless of which one the player picked — making the entire feature a no-op. Reverting to the caller-supplied originX/originY aligns with the function's docstring at drones.ts:326-329 ("The originX/originY are read from the home spec by the caller … We store them on the drone so the tick loop doesn't have to re-look-up by id."). verified

After deletion, the function body falls through to the spawnX = originX; spawnY = originY defaults at drones.ts:441-442 and proceeds directly to the Drone record construction at drones.ts:453-473. verified

Pad-aware concurrency cap

The current cap at drones.ts:354-358 rejects any second launch from an island while another drone is active from that island:

// Current cap — per-island.
for (const d of world.drones) {
  if (d.fromIslandId === origin.id && (d.status === 'active' || d.status === undefined)) {
    return { ok: false, reason: 'already-in-flight' };
  }
}

Becomes pad-aware by comparing d.originX/d.originY against the caller-supplied originX/originY with a tolerance:

// New cap — per-pad. PAD_MATCH_EPS = 0.5 (pads are SHAPES.single at integer
// tiles, so centres differ by ≥ 1.0 between distinct pads).
const PAD_MATCH_EPS = 0.5;
for (const d of world.drones) {
  if (d.fromIslandId !== origin.id) continue;
  if (d.status !== 'active' && d.status !== undefined) continue;
  if (Math.abs(d.originX - originX) < PAD_MATCH_EPS &&
      Math.abs(d.originY - originY) < PAD_MATCH_EPS) {
    return { ok: false, reason: 'already-in-flight' };
  }
}

Epsilon rationale: Drone Pad is SHAPES.single (1×1) at building-defs.ts:736 — pad centres are spec.cx + pad.x + 0.5 and spec.cy + pad.y + 0.5 with pad.x, pad.y integer tile coords. Two distinct pads on the same island differ by ≥ 1.0 in at least one axis, and the spawn coords on a Drone record are direct float-sum copies of the launch-time centre, so a sub-tile epsilon is robust. 0.5 leaves comfortable headroom; the only finer-than-1.0 jitter would come from a future non-integer-tile placement, which is out of scope. verified

Reason code stays 'already-in-flight' — the existing DispatchResult union at drones.ts:248-250 is unchanged. The UI gate distinguishes PAD BUSY vs ALL PADS BUSY by scanning world.drones for matches against every operational pad centre, before calling dispatchDrone; the engine never has to carry that distinction. verified

UI gate pseudocode

// drones-ui.ts refresh() — replaces the per-island inFlight check at lines 1052-1054.
function padBusyState(): 'free' | 'selected-busy' | 'all-busy' {
  const ops = originSpec.buildings.filter(
    (b) => b.defId === 'dronepad' && isOperationalBuilding(b),
  );
  if (ops.length === 0) return 'free';  // arm gate falls through to NO DRONE PAD
  const active = deps.world.drones.filter(
    (d) => d.fromIslandId === origin.id &&
           (d.status === 'active' || d.status === undefined),
  );
  const busyPadIds = new Set<string>();
  for (const p of ops) {
    const px = originSpec.cx + p.x + 0.5;
    const py = originSpec.cy + p.y + 0.5;
    if (active.some((d) => Math.abs(d.originX - px) < 0.5
                            && Math.abs(d.originY - py) < 0.5)) {
      busyPadIds.add(p.id);
    }
  }
  if (busyPadIds.size === ops.length) return 'all-busy';
  if (selectedPadId && busyPadIds.has(selectedPadId)) return 'selected-busy';
  return 'free';
}

08

Migration

None. Explicit confirmation.

No persistence schema bump. Drone.originX/originY already exist at drones.ts:43-44 and round-trip through the v13 snapshot at persistence.ts:499-517 and back at persistence.ts:666-674. The pad-aware cap rule is a pure runtime computation against fields already on the wire. selectedPadId is in-memory UI state in mountDronesUi's closure — not saved, intentionally rebuilt from "first operational pad" on every fresh session and on every active-island switch. SUPPORTED_LOAD_VERSIONS is unchanged. verified

Backward-compat saves. A v13 save with a drone whose originX/originY were minted under the old single-pad UI (i.e. equal to pad-A's centre on a multi-pad island) will load and behave correctly under the new cap: the drone's coords match pad-A's centre to within PAD_MATCH_EPS, so attempting another launch from pad-A is rejected and launches from pad-B are allowed. verified

10

Out of scope

Considered, named, deferred — so the reader knows what was thought about and put down.

Per-pad fuel queues / per-pad inventory
Fuel continues to come from the launching island's inventory pool, regardless of which pad fires the drone. A "pad has its own fuel tank" model would require a schema bump on PlacedBuilding and a new draw rule in the economy tick — material, separate spec.
Auto-rotate / round-robin launching
A "launch from the next free pad" auto-pick mode is a UX nicety; the explicit picker is the load-bearing part of this change. If playtesters want auto-pick, add a sentinel option (e.g. Auto · next free) at the top of the dropdown in a follow-up.
Cross-island pad selection
Launching from another island's pad while the active island is something else is a different feature (remote-dispatch UX). The dropdown enumerates only deps.getOriginSpec()'s pads.
Persisted selectedPadId
Sticky last-selected-pad across reload is deferred; the scope locks "no persistence change." A localStorage read at refresh() time would be the minimal addition if it becomes desired.
Per-pad labelling by world-tile coords
"Pad @ (x, y)" labelling considered and rejected at §02 — too noisy for a 248px panel, less mnemonic for the player than ordinal placement order.
Launch Tower / Path Drone Foundry per-pad selection
Out of scope by the §11.7 dispatch-capacity table — Launch Tower (T4 pulse) and Path Drone Foundry (T5) are independent dispatch buildings with their own single-slot caps. Each currently occurs at most once per island in production play; if multiples become common a similar picker can be added behind their own scope.
