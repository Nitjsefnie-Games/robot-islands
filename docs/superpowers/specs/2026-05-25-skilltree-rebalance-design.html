Skill-tree rebalance — cap-per-pool via calibrated magnitudes

Context The current catalog produces multiplier runaway: a player at L128 ends up with ×10⁹ on extraction-category recipe rates after Dijkstra threads them through 47 nodes that all carry recipeRateMul:extraction, plus stacking sub-path bonuses, plus xpGainMul compounding into the level-up loop (L30 → L129 observed in minutes). This spec rebalances every multiplier-style skill effect so each output quantity caps at ×10 (or ÷10 for divisors, +N for additive) via calibrated per-node magnitudes — no runtime clamp. The cap is achieved by sizing each node's +X bonus such that the multiplicative product of all bought-in-pool nodes hits the cap exactly. Shared effects (e.g. extraction across Mine + Logger + Drill) split their pool budget 50/50 between the shared and per-building effect kinds.

Decision
Cap every multiplier-style pool at ×10 by re-magnitudeing every catalog node. No runtime clamp; magnitudes do all the work.
Per-effect-kind magnitude formula: for an effect with N nodes contributing to a pool capped at C, with the effect's share of the pool budget = w (in [0,1] summing to 1 across the pool's contributing kinds), per-node magnitude is m = C^(w/N) - 1. The natural multiplicative product across all N bought nodes is then (1+m)^N = C^w, and the joint product across all contributing kinds in the pool is ∏ C^w_k = C^∑w_k = C^1 = C exactly. No clamp needed.
Shared pools (Mine, Logger, Drill, Aqua, Patronage, T5 extractors) each include recipeRateMul:extraction plus a per-building yield bonus, split 50/50. Drill, Aqua, Patronage, T5 extractors get NEW yield-bonus effect kinds (drillYieldBonusMul, aquacultureYieldBonusMul, patronageYieldBonusMul, t5ExtractorYieldBonusMul) so they have symmetric structure to Mine + Logger.
Shrink-each pools (S3 storage, S4 power) each side caps at √10 ≈ 3.162 so the product stays ≤ ×10 without joint-cap logic. xpGain capped tighter at ×3 (meta-amplifier; tighter prevents the level-up runaway from re-emerging). Divisors reach ÷10 max (matching the multiplier cap on the inverse axis). Additive caps: parallelBuildCapAdd tops at +4 (5 total slots), launchSuccessAdditive tops at +90 percentage points.
Cost curve + SP grant: unchanged. The cap-per-pool design subsumes the runaway — infinite SP just hits cap faster. No costForDepth or skillPointsForLevelUp rework. Less change, lower risk.
Migration v13 → v14: reset every island's level=1, xp=0, and skill-tree fields (unlockedNodes=[], unspentSP=0, totalSpent=0). Preserve buildings, inventory, maintenance state, drones, routes, satellites, all other state. Player keeps their built world but starts the skill ladder fresh against the new magnitudes.
01

Background — the three root causes

Verified via direct catalog inspection earlier this session (see 2026-05-25-skilltree-rebalance-pools docs-hub artifact for the pool-picker that ingested these findings).

(a) Over-concentration of recipeRateMul:extraction. 47 catalog nodes carry that exact effect, spread across mining / forestry / drilling / aquaculture / patronage sub-paths. Cheapest-path traversal sweeps in many of them, stacking multipliers like ×1.5 × ×1.3 × ×1.25... = ×10⁹ at endgame.

(b) Flat per-buy cost. costForDepth(d) = round(1.5^(d-1)) at skilltree.ts:345-347 — cost is per-depth only, no buy-count escalation. Whole-sub-path total ~874 SP regardless of how many already bought.

(c) Geometric SP grant past L~50. skillPointsForLevelUp(L) = floor(1.1^L) at skilltree.ts:356-358. At L70 a single level grants ~790 SP; at L100, ~13,780. Since max node cost = 1.5^11 ≈ 86, every level past mid-game grants more SP than the most expensive node — every cheapest-path is affordable instantly.

This spec addresses (a) directly by re-magnitudeing every node so the worst-case sum across any pool tops out at ×10. (b) and (c) become moot — even with infinite SP, the cap is the cap. Cost curve and SP grant untouched.

06

Cost curve + SP grant — leave untouched

Original root causes (b) and (c) become moot under the cap-pool design.

The original diagnosis flagged (b) flat per-buy cost and (c) geometric SP grant past L~50 as separate issues. With per-pool caps via calibrated magnitudes, the player has a hard ceiling regardless of how much SP they accumulate. Past the cap, additional purchases give 0 marginal benefit. The "infinite SP" problem self-resolves.

No changes to: costForDepth (skilltree.ts:345), skillPointsForLevelUp (skilltree.ts:356), cumulativeSkillPointsForLevel (skilltree.ts:366). Whole-tree cost remains ~874 SP per sub-path; SP grant remains 1.1^L. Mid-late game still hits the cap fast, but that's the design — caps ARE the gameplay ceiling.

If post-ship playtesting reveals "hit the cap too fast" feels bad, a follow-up spec can tame the SP grant. Out of scope here.

07

Persistence — migration v13 → v14

Per the AGENTS.md "bump = migrate" policy. Schema bump because every island's level + skill-tree state needs to be reset (old SP allocations no longer match the rebalanced magnitudes).

Bump: SCHEMA_VERSION = 14, STORAGE_KEY = 'robot-islands:save:v14', extend SUPPORTED_LOAD_VERSIONS to include 13 + 14.

migrateV13toV14(s: SerializedSnapshotV13): SaveSnapshot:

function migrateV13toV14(s: SerializedSnapshotV13): SaveSnapshot {
  return {
    ...s,
    v: 14,
    islandStates: s.islandStates.map(state => ({
      ...state,
      level: 1,
      xp: 0,
      skillTree: {
        ...state.skillTree,
        unlockedNodes: [],
        unspentSP: 0,
        totalSpent: 0,
      },
    })),
  } as unknown as SaveSnapshot;
}

Preserved fields (NOT reset): buildings, inventory, maintenance state, drones (in-flight + scanBuffer), routes, vehicles, satellites, revealedCells, generatedCells, all timestamp / phase / tutorial / settlement state. Player keeps the world they built; only the skill-progression ladder resets.

Loader chain at src/persistence.ts:

if (snapshot.v === 13) {
  snapshot = migrateV13toV14(snapshot as unknown as SerializedSnapshotV13);
}

Tests (in src/persistence.test.ts): v13 fixture with non-trivial level/xp/skillTree loads as v14 with level=1, xp=0, empty skillTree, but buildings/inventory preserved.

10

Out of scope

Graft-socket system
Brainstorm paused mid-session; resumes separately. Magnitude design here doesn't preclude graft content (graft fills add to the same effect kinds; magnitudes already calibrated for the post-graft pool size).
Cost-curve / SP-grant rework
Per §06. Cap-pool design subsumes the runaway. Tunable in a follow-up if mid-late game feels too "race-to-cap".
Renaming existing yield-bonus effects for symmetry
Current naming: mineYieldBonusMul, loggerYieldBonusMul. New: drillYieldBonusMul, aquacultureYieldBonusMul, etc. Pattern matches; no rename needed.
Per-T5-extractor specialisation
All 6 T5 extractor variants share one t5ExtractorYieldBonusMul effect kind. A future spec could split into per-variant bonuses for build identity (e.g. "Aetheric Conduit specialist") at the cost of 5 more effect kinds + archetypes.
Conditional-bonus + adjacency-bonus folding
conditionalBonus (during-storm / during-night / networked-to-N) and exoticAdjacency (pairBoost) effects exist but aren't pool-capped here. They modify pool products at evaluation; if they push past ×10 in edge cases, fold into the cap calculus in a follow-up. Out of scope until playtesting surfaces a problem.
