Design spec · robot-islands

Route cargo modes — composable selection + source-floor gates

Context Today an inter-island route carries cargo one of two ways: a fixed filter (one resource) or filter = any with an ordered priorityList that ships the first available resource — one resource per tick. This spec replaces that two-concept model with a single unified cargo list plus a mode: four selection modes (priority, waterfall, split, balanced) and an optional per-resource source-floor gate. It also retires the misleading "priority" framing — priority becomes one mode of four, not the whole concept.

Every existing route migrates to mode: 'priority' with its current resource(s) as the cargo list — strict priority is today's behaviour, so migrated routes run identically.
01

Background

What the routing tick does today, and the two limitations this spec addresses.

A route's per-tick cargo is chosen by selectResource (routes.ts:451): it walks the candidate list — [filter] or priorityList — and returns the first resource with stock at the source and headroom at the destination. The dispatch runs in three phases: Phase 1 builds one (route, resource, desired) demand per route; Phase 2 groups demands by (source, resource) and scales for route-vs-route contention proportionally to capacity; Phase 3 executes.

Two limitations motivate this work. (1) A route can only ever move one resource per tick — there is no way to say "carry wood and stone 2:1", or "fill wood, then top up with stone". (2) SPEC §2.4 already describes a within-tick waterfall ("move as much of priority-1 as capacity allows, then continue to priority-2 with remaining capacity") that the code never implemented — so the spec and the build disagree. The waterfall mode below closes that gap.

02

Unified data model

One cargo list and one mode replace the filter / priorityList split.

Today — two concepts
  • filter: ResourceId | null
  • priorityList: ResourceId[] (used only when filter === null)
  • Behaviour is implicit: filtered = fixed; any = strict single-pick
  • No weights, no gates, no waterfall
New — one concept
  • mode: CargoMode — explicit behaviour
  • cargo: CargoEntry[] — one list, all routes
  • A single-resource route is just a one-entry list
  • Per-entry weight and sourceFloorPct
type CargoMode = 'priority' | 'waterfall' | 'split' | 'balanced';

interface CargoEntry {
  resourceId: ResourceId;
  weight?: number;          // split only; positive; default 1
  sourceFloorPct?: number;  // optional gate; 0–100; absent = no gate
}

// Route loses `filter` and `priorityList`, gains:
mode: CargoMode;
cargo: CargoEntry[];

Migration. Both persistence.ts (save load) and any in-memory route construction convert old-shape routes: a route with filter !== null becomes { mode: 'priority', cargo: [{ resourceId: filter }] }; a route with filter === null becomes { mode: 'priority', cargo: priorityList.map(r => ({ resourceId: r })) }. Because priority mode reproduces selectResource's first-viable-wins logic exactly, every migrated route behaves identically to before. New routes default to mode: 'priority', cargo: [].

03

The four selection modes

Every mode first computes the tick's viable entries: a cargo entry is viable when the source has stock (inv > 0), the destination has headroom (destinationHeadroom > 0), and its source-floor gate passes (§04). Modes differ only in how they divide the per-tick capacity budget across viable entries.

priority — strict, single-pick (today's behaviour)

The first viable entry in list order takes the whole tick's capacity. Everything below it is ignored this tick. Identical to today's selectResource.
cargo [wood, stone, coal] · wood viable → ship wood, full capacity. stone/coal untouched.

waterfall — fill, then spill

Walk viable entries in list order with a capacity budget. Each entry takes min(remaining budget, its source stock, its destination headroom); the budget decrements and the remainder spills to the next entry. Stops when the budget is exhausted or the list ends. This is the behaviour SPEC §2.4 already specifies.
budget 10 · wood: only 3 in stock → ship 3, budget 7 · stone: ship 7 → budget 0. coal untouched.

split — weighted ratio

Each viable entry has a weight (default 1). The tick's capacity is divided across viable entries proportionally to normalised weight. A non-viable entry simply drops out of the set, and its share rescales across the rest — so a missing resource never wastes capacity.
cargo wood×2, stone×1 · both viable → wood 2/3 capacity, stone 1/3. stone missing → wood 3/3.

balanced — ship what the destination needs most

Single-pick, like priority, but the pick is the viable entry whose destination has the lowest fill ratio (destInv / destCap) — the resource the destination is most short of. Ties break by list order. No fixed list order or weights needed; the route self-tunes to keep a colony stocked.
dest has wood 90/100, stone 10/100 → ship stone (0.10 < 0.90), full capacity.
One tick of capacity — how each mode fills it

First viable entry takes the entire tick. Lower entries wait.
waterfall · fill, then spill
Entry 1 (wood) is stock-limited; leftover capacity spills to entry 2.

Single-pick the entry the destination is most short of (lowest inv ÷ cap).
All four operate on the same per-tick capacity budget — clay and olive are just two example resources. The modes differ only in how that budget is divided across the route's viable cargo entries.
04

Source-floor gate

An optional per-entry modifier, independent of mode.

A CargoEntry may set sourceFloorPct (0–100). When set, the entry is viable only while the source island holds at least that percentage of its storage cap of that resource — inv / cap ≥ sourceFloorPct / 100. Below the floor, the entry is treated exactly like a missing resource: skipped by priority/waterfall/balanced, dropped from the split set. This lets a route export only a colony's surplus without draining the stock its own production chain depends on. Absent sourceFloorPct = no gate (always passes the stock check). The gate composes with every mode — it only changes the viable-entry set, never the allocation maths.

Source-floor gate — a binary on / off per entry
When the source holds ≥ the floor % of its cap, the entry is viable and ships normally. Below the floor it is treated exactly like a missing resource — skipped by priority / waterfall / balanced, dropped from the split set. The gate only changes the viable set; it never skims a partial amount.
05

Tick integration

Only Phase 1 of the dispatch changes.

Today Phase 1 emits exactly one demand per route. New Phase 1 calls a pure mode-dispatch function — planRouteCargo(route, world, states, elapsedSec) — that returns an array of (resourceId, desired) demands: one element for priority and balanced, one-or-more for waterfall and split. Phase 2 (capacity-proportional contention) and Phase 3 (execute) are unchanged — they already group demands by (source, resource), and a single route never emits two demands for the same resource, so the grouping stays correct.

Known imprecision (accepted). Waterfall computes each entry's take from source/destination state at tick start, then spills the remainder. If route-vs-route contention in Phase 2 later scales a higher entry's demand down, the capacity that scaling freed is not retroactively handed to lower entries that tick. The route under-ships slightly and self-corrects on the next tick. This matches the approximation already present in today's Phase-1 design (which also reads pre-contention state) and is not worth a multi-pass solver.

Dispatch pipeline — only Phase 1 changes
Phase 1 now emits one or more (resource, amount) demands per route — exactly one for priority & balanced, several for waterfall & split. Phase 2 (capacity-proportional contention) and Phase 3 (execute) are untouched; they already group demands by (source, resource).
06

UI

The route ledger editor in routes-ui.ts gains a mode selector and an adaptive cargo editor.

Each route row gets a mode dropdown (priority / waterfall / split / balanced). The cargo editor below it adapts to the mode: priority and waterfall show a drag-orderable list (order is meaningful); split shows the same list with a numeric weight field per row; balanced shows an unordered set (order irrelevant). Every row, in every mode, has an optional source-floor % field. Adding / removing resources works as today. Edits mutate the route in place and take effect next tick. This replaces the current renderPriorityEditor (routes-ui.ts:812), which only handled the any-route ordered list.

07

Naming

The field priorityList is removed, not renamed — it is superseded by cargo. "Priority" survives only as the name of one of the four modes, which is honest: it is genuinely a priority-ordered behaviour. This resolves the earlier observation that "priority" was a misleading name for the whole concept — the whole concept is now "cargo mode", and priority is one mode within it. Helper names follow: reorderPriorityList → reorderCargo, and similar.

08

Verification

The mode-dispatch logic is pure and unit-testable; routes.test.ts already exercises the dispatch phases.

Per-mode allocation tests. One suite per mode: priority single-pick, waterfall fill-and-spill (incl. a higher entry short on stock), split weighted division + rescale-on-shortage, balanced lowest-fill-ratio pick + tie-break.
Source-floor gate tests. Entry above floor is viable; below floor is treated as missing; absent sourceFloorPct always passes — verified against each mode.
Migration test. An old-shape filtered route and an old-shape any route both convert to priority mode and dispatch byte-identically to the pre-change result.
Contention still works. Two routes (any mix of modes) demanding the same (source, resource) still split proportionally to capacity in Phase 2.
npm run build + npm test green. Strict TypeScript flags every stale filter / priorityList reference.

10

Out of scope

Destination-ceiling gate
A "stop shipping X once the destination is N% full" gate was considered; the chosen gate axis is source-floor only. A destination ceiling can be added later as a second gate type — the CargoEntry shape leaves room.
Even-split as a distinct mode
"Split capacity equally" is just split with all weights equal — no separate mode needed.
Balancing-split for demand mode
balanced is single-pick most-needed, not a capacity split weighted by shortfall. The split variant overlaps with split and was dropped.
