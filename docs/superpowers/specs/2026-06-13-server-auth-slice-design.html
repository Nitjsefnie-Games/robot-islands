Design spec · robot-islands · server migration slice 1 of 5

Auth service + user store

Context The TODO.md server-authoritative migration moves the simulation off the browser; the server will own all state and validate client intents (anti-cheat + future multiplayer). That work decomposes into ~5 slices, each with its own spec → plan → build. The owner chose to build auth first (accounts are the identity everything else keys on). This spec decides: a standalone Node + Fastify server with email+password accounts and revocable server-side sessions, backed by the box's existing PostgreSQL — and nothing of the game yet.

Decision
A standalone Node + Fastify auth server, scrypt-hashed passwords, revocable server-side sessions, on the box's existing PostgreSQL.
scrypt (Node built-in) needs zero native deps; a server-side sessions table gives instant revoke/ban, which the anti-cheat rationale driving the whole migration wants. Fastify is TS-first and grows cleanly into the WebSocket intent loop of later slices.
This slice deliberately does not host game state, open a socket, touch the Vite client, or wire SMTP (email verification / password reset are deferred).

01

Background — what exists on this box

Verified during brainstorming, so the design targets the real setup rather than a guess.

  • Postgres 17.9 is up (cluster 17 on 5432; cluster 13 on 5433 is down). pg_hba: local all all peer, host all root 127.0.0.1 trust, host all all 127.0.0.1 md5.
  • A robot_islands role exists (login, has a password, non-super) and owns an already-provisioned robot_islands_dev database.
  • Box convention for DB access: services run as root via socket peer (postgresql:///<db>) — zero password to manage. We follow it.
Existing box auth — wrong shape to reuse
  • Human creds live in goonbot.users.config, keyed by Discord integer user_id.
  • One user (the owner); no email, no self-signup.
  • PBKDF2-HMAC-SHA256 / 200k; stateless HMAC session cookie with a shared secret.
  • docs-hub piggybacks on it for the owner's single login.
This slice — own, stronger store
  • Robot-islands' own email-keyed users table (multiplayer-ready).
  • scrypt (memory-hard) instead of PBKDF2.
  • Server-side sessions table → revocable / bannable.
  • Independent of the goonbot identity; not reused.

Repo layout (npm workspaces)

Convert the root to npm workspaces so the server is its own package and can import the pure src/ layer in later slices.

robot-islands/
  package.json            # + "workspaces": ["server"]  (client config unchanged)
  src/                    # unchanged
  server/
    package.json          # deps: fastify, @fastify/cookie, @fastify/rate-limit, pg
                          # devDeps: typescript, vitest, @types/node, @types/pg, tsx
    tsconfig.json         # strict, noUncheckedIndexedAccess, noUnusedLocals/Parameters
    .env.example          # DATABASE_URL, PORT, COOKIE_SECURE
    migrations/0001_init.sql
    src/
      index.ts            # boot Fastify, register plugins + routes, run migrations, listen
      config.ts           # parse + validate env
      db.ts               # pg Pool from DATABASE_URL
      migrate.ts          # numbered-SQL migration runner (tracks schema_migrations)
      crypto/password.ts  # scrypt hash/verify (pure, dep-free)  + .test.ts
      crypto/token.ts     # opaque token gen + sha256 (pure)     + .test.ts
      auth/users.ts       # user repo: create / findByEmail / findById
      auth/sessions.ts    # session repo: create / findValid / revoke / purgeExpired
      auth/cookie.ts      # set/clear session cookie helpers
      auth/guard.ts       # preHandler: cookie -> valid session -> req.user
      auth/routes.ts      # POST signup|login|logout, GET me
      auth/auth.test.ts   # integration tests against robot_islands_test

Separation of concerns mirrors the src/ discipline: pure crypto (no DB, no Fastify) → DB repos (no Fastify) → HTTP. Each unit is independently testable.

04

Data model (Postgres)

Databases: robot_islands (prod, to create), robot_islands_dev (exists), robot_islands_test (to create). Migration 0001 enables pgcrypto + citext.

CREATE TABLE schema_migrations (
  version    integer PRIMARY KEY,
  applied_at timestamptz NOT NULL DEFAULT now()
);

CREATE TABLE users (
  id         uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  email      citext UNIQUE NOT NULL,
  password   text NOT NULL,           -- PHC: scrypt$N=...,r=...,p=...$salt_b64$hash_b64
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now()
);

CREATE TABLE sessions (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id      uuid NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  token_hash   bytea NOT NULL UNIQUE,  -- sha256(opaque token); raw token NEVER stored
  created_at   timestamptz NOT NULL DEFAULT now(),
  expires_at   timestamptz NOT NULL,
  last_seen_at timestamptz NOT NULL DEFAULT now(),
  revoked_at   timestamptz            -- NULL = active; set to revoke/ban
);
CREATE INDEX sessions_user_id_idx ON sessions(user_id);
  • citext → case-insensitive unique email.
  • password is a self-describing PHC-style string so scrypt params upgrade without a schema change.
  • Storing only sha256(token) means a DB read can't resume sessions.
05

Crypto

Two pure, dep-free modules — the only security-critical math, unit-tested without a DB.

  • Password (crypto/password.ts): scryptSync(pw, salt, 32, {N:2**15, r:8, p:1, maxmem:64MB}), random 16-byte salt, encoded scrypt$N=32768,r=8,p=1$<salt>$<hash>. verify parses params, recomputes, compares with timingSafeEqual. Length bounds (12–200) enforced at the route layer.
  • Session token (crypto/token.ts): randomBytes(32) → base64url (cookie value); hashToken = sha256 (bytea). Lookup hashes the incoming cookie and matches token_hash.
06

HTTP surface (/api/auth)

MethodPathBodySuccessFailure
POST/api/auth/signup{email, password}201 + Set-Cookie + {id,email}409 taken; 400 invalid
POST/api/auth/login{email, password}200 + Set-Cookie + {id,email}401 generic
POST/api/auth/logout— (cookie)204 + cleared cookie (revoked)204 even if none
GET/api/auth/me— (cookie)200 {id,email}401
  • Cookie: ri_session = raw opaque token, HttpOnly, Secure (when COOKIE_SECURE=1), SameSite=Lax, Path=/, Max-Age 30d.
  • Validation: Fastify JSON schemas — email format, password length 12–200.
  • Rate limit: @fastify/rate-limit on /login + /signup, per-IP (e.g. 10/min).
  • Login failures generic; an unknown email still runs a dummy scrypt verify to defeat timing oracles.
  • guard.ts: cookie → hashToken → findValidByTokenHash (active, unexpired) → req.user; else 401; touches last_seen_at.
07

Configuration & deployment

  • Env (server/.env, gitignored; .env.example committed): DATABASE_URL (default postgresql:///robot_islands), PORT (5180), COOKIE_SECURE (1).
  • Migrations run on boot, applied in version order inside a transaction, recorded in schema_migrations.
  • systemd: robot-islands-server.service runs node server/dist/index.js on 127.0.0.1:5180, Restart=on-failure. Does not touch robot-islands-dev.service. Unit committed under server/deploy/.
  • nginx: deferred — add location /api/ proxy on the islands.nitjsefni.eu vhost when the client consumes the API.
08

Testing (vitest)

  • Pure unit (no DB): password hash≠plaintext, verify roundtrip, wrong-pw fails, PHC parse; token uniqueness/length, hash determinism, raw≠hash.
  • Integration (against robot_islands_test): signup→/me; duplicate email→409; login wrong→401; login right→cookie+/me; logout revokes→/me 401; expired session rejected; revoked session rejected; raw token absent from sessions.
  • Test-DB guard: tests refuse any DATABASE_URL whose db name doesn't end in _test; TRUNCATE … RESTART IDENTITY CASCADE between tests.
  • Root npm test stays green; server tests run via npm test -w server.
