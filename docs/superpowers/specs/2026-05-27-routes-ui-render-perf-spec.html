Decision
Three independently revertable phases. Skip-gate first, world-space migration second, texture-tiled dash stroke third. All other render hotspots stay out of scope.
Phase 1 — Skip-gate. Cache a per-frame routesCacheKey(routes) string. When the key matches the last invocation, paintLayer bails before touching any Pixi Graphics. Still screen-space; still per-frame redraw on changes. Expected to idle ~30-40 % of frames (any frame where camera, routes, and in-flight counts haven't moved).
Phase 2 — World-space migration + static/animated split. Two new sibling Containers parented under the existing world container: routesStaticLayer (built once per route from tileToWorldPx(building.x, building.y)) and routesAnimatedLayer (dashed overlay). Pixi's stage transform on world handles pan/zoom, so the cache survives camera moves. resolveScreenPos is removed from the renderer's geometry path.
Phase 3 — Texture-tiled stroke + UV animation. Dashed lines use a small PIXI.Texture per route type as the stroke fill. Each frame, paintLayer advances tilePosition.x on the animated Graphics (one number-write per route) — zero geometry rebuild during animation. This is the phase that lands the < 1 % self binding gate.
Why texture-tiled stroke, not a GLSL shader. A GLSL custom shader can move dash phase to a uniform write — even cheaper than a texture UV update — but pulls in a per-route Mesh / Filter, custom shader source, and a feature-flag for WebGL fallback. At the live save's ~10 routes the texture-stroke path's per-route UV write is negligible (Risk 4 below). Shader-uniform fallback is named in §10 as a future option if a re-profile at 100+ routes demands it.

01

Architecture

Three PixiJS scene-graph layers, two of them new. Routes leave screen-space and live under the existing world container so the Pixi stage transform handles pan/zoom for free.

world (existing)
├── … existing children (islands, buildings, ocean, drones)
├── routesStaticLayer    ← new, built once per route, cached
└── routesAnimatedLayer  ← new, texture-stroked dashed lines, UV animates

(label / chevron / pulse overlay that depends on screen-space
 positioning stays in the existing screen-space UI container —
 unchanged. The chevron *count* is a route-cache-key field; the
 chevron *placement* remains screen-space.)

Per route, one entry in Map<RouteId, RouteRenderState> living on the RouteRenderer instance:

  • the static Graphics object (world-coord line geometry, built once)
  • the animated Graphics object (texture-tiled stroke, tilePosition.x animates)
  • a reference to the shared dashed-stroke Texture for this route type
  • the last-applied cacheKey for invalidation
  • (optional) a reference to a screen-space chevron container — built / torn down with the route, but positioned in the UI container at paint time

What is NOT in the new architecture. The route line geometry is the only path being moved; everything else in routes-ui.ts retains its current call shape. Other per-frame routes-panel work is out of scope (see §11).

02

Components

One new module owns the new layers and the cache. Two existing files lose code; one existing file gains a JSDoc'd helper export.

RouteRenderer — src/routes-renderer.ts (new, ~150 LOC)

Owns the two new PIXI.Container instances (routesStaticLayer, routesAnimatedLayer) and the per-route cache Map<RouteId, RouteRenderState>. Public surface:

export class RouteRenderer {
  readonly staticLayer: PIXI.Container;    // addChild'd into world
  readonly animatedLayer: PIXI.Container;  // addChild'd into world

  constructor(world: PIXI.Container, world: WorldState);
  update(routes: readonly Route[], nowMs: number): void;
  dispose(): void;
}

RouteRenderState — same file, plain interface

interface RouteRenderState {
  staticGraphics: PIXI.Graphics;
  animatedGraphics: PIXI.Graphics;
  cacheKey: string;
  chevronContainer?: PIXI.Container;  // optional UI-container overlay
}

getDashedStrokeTexture(routeType) — src/routes-dash-texture.ts (new, ~25 LOC)

Module-scope cache of one PIXI.Texture per route type. Built lazily on first request (drawn into a small PIXI.RenderTexture — dash length + gap pattern in the route-type colour). Held at module scope for the session lifetime; never destroyed. With the current route-type catalogue, the pool is 3-5 textures (see §10 Risk 3).

export function getDashedStrokeTexture(
  renderer: PIXI.Renderer,
  routeType: RouteType,
): PIXI.Texture;

routesCacheKey(routes) — src/routes.ts (new export, ~10 LOC)

Pure function next to the Route type declaration. Walks routes once, joins `${routeType}|${fromBuildingId}|${toBuildingId}|${inFlight.length}` for each route with a "" separator (any non-overlapping codepoint works; the separator is invisible and chosen so it can never appear in an id). Returns the joined string.

/**
 * Render cache-key for the routes-renderer. Encodes every Route field
 * that affects rendered geometry / chevron count.
 *
 * Fields included:
 *   - route.type           (selects dashed-stroke texture + colour)
 *   - route.from           (building id → world-coord endpoint)
 *   - route.to             (building id → world-coord endpoint)
 *   - route.inFlight.length  (chevron count; items themselves NOT keyed)
 *
 * If you add a visual field to Route, you MUST update this function
 * AND the not-visual whitelist in routes-renderer.test.ts. The
 * whitelist test fails the build for unclassified fields.
 */
export function routesCacheKey(routes: readonly Route[]): string;

paintLayer body in routes-ui.ts

Collapses to the delegating shape below — ~5 lines. resolveScreenPos import removed.

function paintLayer(nowMs: number): void {
  routeRenderer.update(world.routes, nowMs);
}

Why one module, not three. RouteRenderer, RouteRenderState, and the per-route paint logic share a private cache and ticker contract. Splitting across files forces an export surface that doesn't pay for itself at this size. The dashed-texture factory is split out because it has a different lifecycle (module-scope singleton pool, never disposed) than the per-route Graphics (created/destroyed with routes).

03

Data flow

Single entry point. Fast path = one cache-key compare + one number write per route. Slow path = diff against the cache and rebuild only changed entries.

nowMs ──► RouteRenderer.update(routes, nowMs)
            │
            ├─ key = routesCacheKey(routes)
            ├─ if key === this.lastKey:
            │     for each anim entry: anim.tilePosition.x = nowMs * SCROLL_SPEED_PX_PER_MS
            │     return                              ← FAST PATH (90 %+ of frames)
            │
            ├─ key !== this.lastKey: diff
            │   ├─ for route in routes not in cache:        BUILD static + animated
            │   ├─ for route in cache not in routes:        DESTROY entry
            │   ├─ for route in both with changed key:      REBUILD static, adjust animated
            │   └─ for route in both unchanged:             NO-OP
            │
            ├─ update anim UV offset (same as fast path)
            └─ this.lastKey = key

What invalidates the cache

  • route.type changes (different dashed texture)
  • route.from or route.to changes (different building endpoint → different world-coord line)
  • a route is added or removed from world.routes
  • route.inFlight.length changes (chevron count — items themselves are NOT in the key because only the count is rendered)

What does NOT invalidate the cache

  • camera pan / zoom (handled by Pixi stage transform on world)
  • nowMs advancing (animation is UV-offset, not geometry)
  • routes panel visibility toggle (the layers exist regardless of panel state; panel toggle does not enter RouteRenderer.update)
  • DOM ledger refresh inside routes-ui.ts (out of scope, separate signature-cache)

First-frame behaviour. this.lastKey initialises to null. The first update() call short-circuits the equality check, falls through to the diff path, builds every route from scratch, and sets this.lastKey. No special case in the code.

04

Cache key & correctness

The cache is correct iff routesCacheKey captures every Route field that affects the rendered output. The contract is enforced two ways — by JSDoc + a grep-able marker, and by an automated whitelist test.

Cache-key completeness

JSDoc on routesCacheKey (§02) explicitly lists the four cache-key fields. A grep-able comment on the Route type declaration in src/routes.ts reminds maintainers that adding a visual field requires touching the key:

// VISUAL-FIELD-MARKER: any new field on this interface that
// affects rendered route geometry, dash style, or chevron count
// MUST be added to routesCacheKey() AND classified in the
// not-visual whitelist in routes-renderer.test.ts.
export interface Route {
  id: RouteId;
  type: RouteType;
  from: BuildingId;
  to: BuildingId;
  inFlight: ReadonlyArray<Batch>;
  /* …other fields… */
}

Whitelist enforcement test

A test (§07 row "Whitelist") iterates Object.keys(routes[0]) for a representative live route and asserts every key is either in the cache-key set or on a not-visual whitelist (e.g. id — used for cache lookup but not rendered; throughput stats; any future non-visual metadata). A new field on Route that's neither in the cache key nor on the whitelist fails the test.

No in-place mutation surface

Routes are wholesale added / removed / replaced; they are not mutated in place. The only sub-array that is mutated is inFlight, and its length is in the cache key. Items pushed or popped from inFlight change the key; mutations to an item's internal fields do not — that is correct, because item internals do not affect rendering.

Optional debug flag — ?routes-debug=1

When the URL carries ?routes-debug=1, the fast path additionally recomputes routesCacheKey against a structured-clone snapshot of the previous routes input and console.warns on divergence. This catches the case where a real mutation slips past the cache key. Off by default — the snapshot has non-trivial cost. u

Fast-path UV-write ceiling

At the live save's ~10 routes the per-route tilePosition.x = write is negligible (one numeric assignment per route per frame). At 100+ routes a single shader uniform replaces the per-route writes — out of scope for this spec, called out as a future fallback in §10 Risk 4.

05

Phase deliverables

Each phase is a single shippable commit with its own measurable gate. Phase 3 is the only one with a binding gate (paintLayer self < 1 %); the earlier phases are observability + foundation. Sequencing details (commit order, exact edit locations) are the writing-plans skill's job — the spec defines the artifacts each phase must deliver.

Phase 1
Skip-gate
routesCacheKey(routes) exists and is called once per frame. paintLayer caches the last key and bails before any Pixi Graphics.clear / redraw when the key matches. Drawing is still screen-space; resolveScreenPos still in the code path. Cache survives only frames where nothing relevant changed.
Gate: Graphics.clear called at most once across two update()s with identical inputs. Visual smoke-test shows zero behavioural change.
Phase 2
World-space + static/animated split
RouteRenderer exists. routesStaticLayer and routesAnimatedLayer are addChild'd into world (not the screen-space UI container). Line endpoints built from tileToWorldPx(building.x, building.y); no resolveScreenPos in the geometry path. Per-route static Graphics built once and reused across camera moves.
Gate: panning the camera does NOT call Graphics.clear on any route's static layer. Z-order verified by Daedalus screenshot — routes above buildings, below drones.
Phase 3 — binding
Texture-tiled stroke + UV animation
Dashed lines are drawn with a PIXI.Texture-filled stroke (one shared texture per route type, fetched from getDashedStrokeTexture). Per frame, animatedGraphics.tilePosition.x is the only mutated field on the route's Pixi state. Zero Graphics.clear during steady-state animation.
Gate: paintLayer self < 1 % on the live save re-profile. Secondary observation: refresh inclusive < 10 %. Dashed-scroll visually identical to baseline.

Phase ordering rationale. Phase 1 is the cheapest reversible win and unblocks measurement of "how often does the renderer actually need to run." Phase 2 detaches the cache from camera state — without it, every pan re-invalidates and the cache is useless. Phase 3 is where the Pixi geometry cost disappears. Reverting Phase 3 keeps Phases 1 + 2 productive; reverting Phase 2 collapses to Phase 1; reverting Phase 1 returns the file to HEAD.

06

API change surface

Mostly internal. Two new files house the renderer; one existing file gains a public-export helper; one existing file loses a body; one bootstrap file gains an init line.

src/routes-renderer.ts (new)
Adds RouteRenderer class + RouteRenderState interface. Owns the two new world-parented Containers + per-route cache. Public surface: constructor, update(routes, nowMs), dispose(). ~150 LOC.
src/routes-dash-texture.ts (new)
Adds getDashedStrokeTexture(renderer, routeType). Module-scope pool, one Texture per route type, never destroyed. ~25 LOC.
src/routes.ts
Adds routesCacheKey(routes): string export + JSDoc. Adds the VISUAL-FIELD-MARKER comment on the Route type declaration. ~10 LOC of new code; no behavioural change.
src/routes-ui.ts
paintLayer body deleted, replaced by routeRenderer.update(world.routes, nowMs). resolveScreenPos import removed. The two screen-space Graphics previously cleared at the top of paintLayer are removed; the chevron / pulse overlay (if it keeps a screen-space surface) is migrated to RouteRenderer's optional chevronContainer.
src/main.ts
Bootstrap wiring. Construct RouteRenderer(world, worldState) after world is ready; addChild its two layers in the documented z-order (§08). Pass the renderer instance to routes-ui.ts (replaces the existing UI-container handle).
src/routes-renderer.test.ts (new)
Correctness tests. One test per phase contract + whitelist enforcement. See §07.
src/routes-renderer.perf.test.ts (new)
Perf-gate test. 50 routes, steady-state frames, per-update wall-clock < 0.5 ms. Un-skipped (runs in normal npm test). See §07.

No persistence change. The renderer is runtime-only. No fields land on WorldState / IslandState; no schema bump; persistence.ts untouched.

Perf-gate test (un-skipped)

it('paintLayer with 50 routes stays under 0.5 ms wall-clock on steady-state frames', () => {
  const routes = makeNRoutes(50);
  const renderer = new RouteRenderer(...);
  renderer.update(routes, 0);              // warm
  const t0 = performance.now();
  for (let i = 0; i < 100; i++) renderer.update(routes, i * 16);
  const dt = (performance.now() - t0) / 100;
  expect(dt).toBeLessThan(0.5);
});

08

Z-order & lifecycle

Concrete addChild order in main.ts and the texture / Graphics lifecycle rules.

Z-order inside the world container

Pixi child order is paint order. Bottom → top:

  1. ocean / fog (existing)
  2. islands + buildings (existing)
  3. routesStaticLayer (new — Phase 2)
  4. routesAnimatedLayer (new — Phase 2)
  5. drones (existing — kept on top so chevrons and dashes never occlude active swarm visuals)

The plan must specify the exact addChild call sequence in main.ts. Visually verifiable per the §07 visual smoke-test.

Texture lifecycle

  • 3-5 PIXI.Texture instances held at module scope inside routes-dash-texture.ts (one per route type).
  • Built lazily on first call to getDashedStrokeTexture for a given route type.
  • Never destroyed during the session — they are session-singletons. RouteRenderer.dispose() does NOT touch them.
  • Memory ceiling is bounded by route-type count. With the current catalogue this is < 1 MB of GPU memory total (small dash patterns at low resolution).

Per-route Graphics lifecycle

  • Built on first appearance of a route in world.routes inside the slow-path diff (§03).
  • Destroyed (graphics.destroy()) when a route disappears from world.routes in the slow-path diff.
  • Rebuilt (destroy + build) on cache-key change for an existing route id (e.g. its from or to changed).
  • RouteRenderer.dispose() iterates the cache, destroys every per-route Graphics, then removes the two layers from the world container. Used at app teardown only.

Risks

Risk
Sev
Mitigation
Stale geometry from a missed cache-key field. A future change adds a visual Route field without updating routesCacheKey, so the renderer silently serves stale geometry until the next non-visual change happens to bump the key.
MED
JSDoc on routesCacheKey lists the four key fields. VISUAL-FIELD-MARKER comment on the Route type. Whitelist enforcement test (§07) fails the build for unclassified fields. Optional ?routes-debug=1 flag (§04) re-runs the key against a deep snapshot and warns on divergence.
Z-order regression. Phase 2 moves routes from a screen-space UI container into world; getting the addChild order wrong causes routes to occlude drones or sit underneath buildings.
MED
§08 specifies the exact addChild order. Daedalus visual smoke-test per phase. The Phase 2 commit body must include a screenshot diff against the baseline.
Texture upload memory creep. PIXI.Texture instances accumulate at module scope and are never destroyed.
LOW
Texture pool is bounded by route-type count (3-5 today). Small dash patterns at low resolution — total GPU footprint < 1 MB. Module-scope retention is intentional and documented in routes-dash-texture.ts's file header. RouteRenderer.dispose() does not touch them; they survive session lifetime.
Fast-path UV-write cost at high route counts. At 100+ routes the per-route tilePosition.x write becomes measurable.
LOW
Live save is ~10 routes today. Fallback documented: replace the per-route writes with a single shader uniform on a custom PIXI.Filter or PIXI.Mesh. Implementation deferred until a re-profile shows the regression; out of scope for this spec.
Animation phase desync — routes that share a texture but should animate at different speeds or colours look identical.
LOW
One PIXI.Texture per routeType (not one shared across all types). Per-type SCROLL_SPEED constant in routes-renderer.ts drives the UV-offset multiplication; different types get different speeds without rebuilding geometry.
Phase 3 lands but the gate is missed — paintLayer self stays > 1 %.
LOW
Phases are independently revertable. If the binding gate fails, the diagnostic surface is small (one new module + one bootstrap change); the next step is the shader-uniform fallback named in Risk 4. The phase ordering ensures Phases 1 + 2 remain productive even if Phase 3 is reverted.
11

Non-goals

Explicitly out of scope so the reader can confirm these were considered and deliberately deferred.

refreshStats, refreshFormReadout, repaintLedger
Per-frame DOM-side caching for the routes-panel ledger and stats readouts is a separate thread. Those functions are called from inside refresh alongside paintLayer; this spec only touches the Pixi geometry path. The remaining ~10 % inclusive on refresh after this work is exactly where that follow-up will land.
WebGL instancing rewrite of the route renderer
A custom PIXI.Mesh + instanced attributes would push all routes into a single draw call. Rejected as overengineering for the current scope (~10 routes steady-state). If a future re-profile at 100+ routes shows the texture-stroke path saturating, this is the next radical step.
GLSL custom-shader dash animation
Moving dash phase to a shader uniform write is cheaper than a texture tilePosition update, but adds a custom PIXI.Filter / Mesh + a WebGL feature-flag fallback. Texture-tiled stroke meets the < 1 % gate at current route counts. Shader path noted in §10 Risk 4 as a future fallback.
Other CPU hotspots in the cpuprofile
inFlightSumFor @ routes.ts:410 at 1.31 % self, effectiveTierShift @ skilltree.ts:1253 at 1.14 % self, and other < 2 %-self callsites are tracked separately. They sit outside the routes-renderer path; fixing them is a different commit chain.
New buildings or game-design changes
This spec is render-perf only. Visual output of the routes is preserved verbatim — same colours, same dash patterns, same scroll direction and speed. Players see no change.
