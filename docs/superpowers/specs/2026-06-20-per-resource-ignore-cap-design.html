Design spec · robot-islands · §4.6 / §15.3

Per-resource Ignore Cap — replaces Force Run

Context Two overlapping mechanisms control "a full output bin must not stall the producer": the per-building forceRun boolean (§4.6, blunt — voids the valuable primary too) and the global automatic OUTPUT_CAP_EXEMPT byproduct set (§15.3 A, invisible to the player). This spec collapses both into a single, user-configurable per-(building, output-resource) Ignore Cap flag, and adds slag to the default-on set.

Decision
One per-(building, output-resource) Ignore Cap flag, stored as sparse per-building overrides over a global default-on set.
The global OUTPUT_CAP_EXEMPT set stays as the default-on state (now 9 resources, adding slag). Each building stores only deviations in ignoreCapOverrides?: Partial<Record<ResourceId, boolean>>. The single effective predicate isOutputCapExempt(b, r) = b.ignoreCapOverrides?.[r] ?? OUTPUT_CAP_EXEMPT.has(r) subsumes both old code paths. An untouched building carries no field and behaves exactly as today.
Fully configurable: the player can turn a default-on byproduct OFF or a default-off primary ON. The P4 byproduct no-stall invariant becomes the player's choice, not a hard floor — deliberately, per the brainstorming decision.
01

What changes, conceptually

Two mechanisms become one, with a strictly larger configuration space.

Before
  • forceRun: one toggle per building, exempts ALL outputs (voids the primary too).
  • OUTPUT_CAP_EXEMPT: global, automatic, 8 byproducts, no player control.
  • No way to say "overflow steel but throttle on full slag".
After
  • One Ignore Cap toggle per output resource per building.
  • The 9 default-on resources (8 + slag) just seed the initial checkbox state.
  • Force Run ≡ "every output ON"; recoverable, plus everything finer.

Verification

How the implementer confirms correctness — runnable, per the test surface.

flow-solver.test.ts. A building with one exempt + one non-exempt output gates only on the non-exempt bin; all-exempt matches old force-run.

persistence.test.ts. v29 (forceRun true / false / absent) → v30 migration; forced building gets all-outputs overrides; v30 round-trips identity.
inspector-ui.test.ts. One checkbox per output; toggling dispatches the right resource+value; no subsection for output-less buildings.
intents.test.ts (server). set-ignore-cap accepted for a real output, rejected for a non-output resource / unknown building; writes the override.
discovery-signature.test.ts. Overrides change the fingerprint deterministically; equal overrides ⇒ equal signature.
Full suite + typecheck. npm test green; cd server && npm run typecheck clean.
06

Risks

Risk
Sev
Mitigation
Player turns a byproduct exemption OFF and re-introduces the P4 stall (e.g. full slag throttles steel).
LOW
Deliberate per the "fully configurable" decision — it is the player's choice and is locally reversible. Default-on keeps the safe behavior unless explicitly changed.
Migration mis-enumerates a def's outputs (rotateOutputs), so a force-run building loses an exemption.
MED
Migration imports the catalog and includes rotateOutputs entries; persistence test asserts a forced multi-output building gets every output.
Renaming the intent/field breaks an in-flight client or stored intent.
LOW
Intents are transient (not persisted as the field); the persisted field is migrated. Server rejects the old intent name cleanly.
07

Out of scope

Considered, deliberately deferred.

"Ignore cap on ALL outputs" master toggle
The per-resource checkboxes already cover it; a bulk toggle is sugar, not capability.
Per-floor / per-recipe-variant granularity
Ignore Cap is per output resource of the building, not per active floor or rotating-recipe phase.
NON_STORED_OUTPUTS (co2) changes
Atmosphere scalar handling is untouched; co2 stays non-stored and is never an Ignore Cap target.
