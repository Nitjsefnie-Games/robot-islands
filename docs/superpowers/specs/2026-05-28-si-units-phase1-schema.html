01

Problem

The v3 physics-balance audit found 13 findings (F16–F28) rooted in one root cause: the game's "unit" has no SI calibration. Recipes can't mass-balance, the heat engine can't meter joules, the drag curve can't respect P ∝ v³, and the placement-cost auditor can't verify that a Steel Mill consumes more mass than a Steel Mill weighs. The rev-16 foundation doc closes all 13 findings via a phased rework starting with this schema upgrade.

What's broken today

  • No mass calibration. recipes.ts has 252 resources in ALL_RESOURCES at recipes.ts:417; none carries a kg-per-unit value. v
  • No orphan-output discipline. The v3 F12 dead-resource test flags any resource produced without a recipe consumer. Today's catalog has several legitimate orphans (mercury, natural_gas, scanner_sat) that fail the test; the workaround has been to delete or rename rather than tag intent.
  • No heat metering. heat.ts at heat.ts:124-205 implements an N:1 boolean adjacency gate; a 200 W Plasma Heater serves N×60 W Coke Ovens at zero incremental cost (v3 F16). v
  • No CO₂ pool. IslandState at economy.ts:216 has no field tracking emissions. Phase 2's recipe restoichiometry produces real CO₂; without an accrual field, the chemistry is unanchored.
  • No location. SerializedWorld at persistence.ts:131 has no playerLat / playerLon. Phase 6 real-astronomy solar needs both.

What Phase 1 does NOT change

  • No behaviour change. The new fields are declared but unread (with one exception — terminal-tag enforcement via the F12 rewrite). No recipe rates shift, no placement costs move, no save semantics differ.
  • No value backfill. massPerUnitKg is populated for every resource (compile-time required) but the values are Phase 1 best-effort from rev-16 §2.2 — the §2.2 table's per-resource kg numbers are encoded here, but the recipe restoichiometry that uses these numbers lands in Phase 2.
  • No new resources. The 7 new resources rev-16 introduces (co2, refinery_gas, wood_tar, water_vapor, aviation_kerosene_crude, mill_scale, calcium_sulfonate, air, cryo_coolant_vented) ship in Phase 2 alongside the recipes that produce them. Phase 1 prepares ALL_RESOURCES + RESOURCE_META so adding them in Phase 2 is a one-line entry per resource.
  • No persistence break. A v14 snapshot loads cleanly under v15 via the additive migration. The wire format gains fields but loses none.

3.1 RESOURCE_META — single exhaustive record

New module-level constant in recipes.ts, placed beside the existing XP_WEIGHT table at recipes.ts:738 (same locality, same idiom).

// recipes.ts — new module-level constant, near XP_WEIGHT.
export interface ResourceMeta {
  /** Mass of one unit in kilograms. Required for every resource —
   *  even fantasy resources carry an anchor (e.g. 1 unit ai_core = 1 kg
   *  single chip per rev-16 §10 implementation-time resolution (a)). */
  readonly massPerUnitKg: number;
  /** Volume of one unit in litres. Optional — only fluids and gases
   *  declare this. Cross-check: if volumePerUnitL set, massPerUnitKg must
   *  match the substance density within ±3 % per rev-16 §2.2 saltwater note. */
  readonly volumePerUnitL?: number;
  /** Specific energy of one unit in kWh. Optional — only energy carriers
   *  (batteries, capacitors, casimir_energy, plasma_charge). */
  readonly energyPerUnitKWh?: number;
  /** Terminal classification — three values per rev-16 §3.5.1. */
  readonly terminal:
    | 'consumed'
    | `expansion-hook:${string}`
    | 'gameplay-sink';
}

  coal:        { massPerUnitKg: 1, terminal: 'consumed' },
  // …all 252 entries…
  // Volume-billed fluids — both mass + volume declared.
  saltwater:    { massPerUnitKg: 1.03, volumePerUnitL: 1, terminal: 'consumed' },
  fresh_water:  { massPerUnitKg: 1,    volumePerUnitL: 1, terminal: 'consumed' },
  crude_oil:    { massPerUnitKg: 0.85, volumePerUnitL: 1, terminal: 'consumed' },

Exhaustiveness enforcement. Readonly<Record<ResourceId, ResourceMeta>> is strict by construction — TypeScript refuses any literal that omits a known ResourceId key. A new resource added to ResourceId at recipes.ts:46-415 without a sibling entry in RESOURCE_META fails npm run build. No runtime fallback is needed.

F12 dead-resource invariant rewrite

The v3 audit's F12 test currently fails any resource produced without a recipe consumer. Under rev-16 §3.5.4 the test changes shape: tagged-'expansion-hook' + tagged-'gameplay-sink' always pass; tagged-'consumed' still requires a consumer.

Locate the existing test

F12 lives in recipes.test.ts (or economy.test.ts; the canonical location is identified by a grep for 'dead resource' / 'F12'). The rewrite preserves the failing-on-untagged contract; the only relaxation is on intentional-orphans.

describe('F12 — produced resources have a terminal classification', () => {
  it('every ResourceId has a RESOURCE_META entry with a terminal tag', () => {
    for (const r of ALL_RESOURCES) {
      const meta = RESOURCE_META[r];
      expect(meta, r + ' missing RESOURCE_META').toBeDefined();
      expect(meta.terminal, r + ' missing terminal').toBeDefined();
    }
  });

  it("every consumed resource has at least one recipe consumer", () => {
    const producedBy = new Map<ResourceId, string[]>();
    const consumedBy = new Map<ResourceId, string[]>();
    for (const [recipeId, recipe] of Object.entries(RECIPES)) {
      if (!recipe) continue;
      for (const r of Object.keys(recipe.outputs)) {
        producedBy.set(r as ResourceId, [...(producedBy.get(r as ResourceId) ?? []), recipeId]);
      }
      for (const r of Object.keys(recipe.inputs)) {
        consumedBy.set(r as ResourceId, [...(consumedBy.get(r as ResourceId) ?? []), recipeId]);
      }
    }
    for (const r of ALL_RESOURCES) {
      const meta = RESOURCE_META[r];
      if (meta.terminal === 'consumed' && producedBy.has(r)) {
        // Tagged consumed AND actually produced — must have a consumer.
        expect(consumedBy.has(r), r + ' tagged consumed but unconsumed').toBe(true);
      }
      // expansion-hook and gameplay-sink resources always pass.
    }
  });
});

Why a tagged-'consumed' resource can be unproduced

A resource can be tagged 'consumed' without any current producer (extractor / recipe) — this is the "starter-inventory-only" pattern (e.g. a tutorial seed resource). Such resources fail no invariant — the gate is on produced AND consumed === 'consumed', not on production unconditional. The pre-rev-16 F12 test flagged this as dead-input; the post-rev-16 test does not.

v14 → v15 migration

Additive only. Old snapshots load by gaining new fields with safe defaults. No field renames, no removals, no per-resource backfill. The migration is mechanical.

Schema-version bump

// persistence.ts:79
export const SCHEMA_VERSION = 15 as const;

// persistence.ts:87 — add 14 to supported.
export const SUPPORTED_LOAD_VERSIONS: ReadonlySet<number> =
  new Set([7, 8, 9, 10, 11, 12, 13, 14, 15]);

// persistence.ts — companion to migrateV13toV14 at line ~420.
export interface SerializedSnapshotV14 {
  readonly v: 14;
  /* …full v14 shape — captured precisely so the migration input is
   *  typed, not 'any'. Mirrors the SerializedSnapshotV13 idiom. */
}

export function migrateV14toV15(s: SerializedSnapshotV14): SaveSnapshot {
  return {
    ...s,
    v: 15,
    world: {
      ...s.world,
      totalCo2Kg: 0,
      playerLat: null,
      playerLon: null,
    },
    islandStates: s.islandStates.map((entry) => ({
      ...entry,
      state: { ...entry.state, co2Kg: 0 },
    })),
  };
}

// persistence.ts — wire into loadWorld's migration chain at line ~603.
if (snapshot.v === 14) {
  snapshot = migrateV14toV15(snapshot as unknown as SerializedSnapshotV14);
}

Defaults rationale

  • co2Kg = 0 per island. Pre-rev-16 saves had no emissions; the pool starts at zero. Phase 2's accrual integrator adds from t=0 forward.
  • totalCo2Kg = 0 globally. Same rationale.
  • playerLat = null, playerLon = null. Sentinel value meaning "the player hasn't picked a location yet." Phase 3's sin-scale solar ignores these; Phase 6 prompts the map picker on the first session after migration when both are null. Picker writes once; persisted thereafter. A null pair after Phase 6 ships is the new-fresh-game state.

Live-save migration check

/tmp/robot-islands-save.json (the user's 27.5 MB live save dump, preserved across sessions) must round-trip through v14 → v15 cleanly. Phase 1's test suite adds a persistence-load.test.ts case that loads the live dump and asserts every island has co2Kg === 0 and the world has totalCo2Kg === 0 + null lat/lon post-migration.

Storage key

The STORAGE_KEY constant at persistence.ts:71 is 'robot-islands:save:v14'. Two options for the bump: (a) update to '…:v15' and accept old browsers picking up the v14 key via the legacy-key probe (loadWorld already walks supported versions); (b) keep the key as :v14 and trust the in-snapshot version field. Phase 1 uses option (b) — the storage key is a probe hint, the in-snapshot v field is authoritative. The string-display value at persistence.ts:75 (STORAGE_KEY_DISPLAY) stays as 'robot-islands:save' (already version-stripped).

07

lat/lon scaffold (Phase 6 prep)

Phase 6 ships real-astronomy solar driven by the player's chosen geographic location and the wall-clock time. Phase 1 declares + persists the fields so Phase 6 doesn't need a second schema bump.

Why declare in Phase 1 vs Phase 6. The migration cost is fixed-per-bump (define SerializedSnapshotV14, write migrateV14toV15, extend SUPPORTED_LOAD_VERSIONS, add a test fixture). Bundling lat/lon into the Phase 1 bump is ~10 LOC; deferring to Phase 6 would require a second migration v15 → v16 with the same setup cost. Per the "bump = migrate" policy (AGENTS.md persistence section) the chain stays shorter when fields are bundled.

Phase 1 doesn't add a picker. The map-picker UI ships in Phase 6. Phase 1 only declares the fields, seeds them to null on fresh state, and adds a JSDoc comment naming the future consumer.

Lat/lon validation. Phase 1 doesn't validate range — the map picker (Phase 6) will only emit values in [-90, +90] / [-180, +180]; Phase 1's type is number | null without bounds, matching the existing-codebase pattern (the codebase doesn't enforce numeric bounds anywhere in the persistence layer).

Non-goals

Deferred to later phases. Naming these explicitly so the reviewer confirms they were considered.

Mass-balance auditor
Phase 2 ships the vitest auditor that walks every recipe, sums input × massPerUnitKg against output × massPerUnitKg, and asserts equality unless RECIPE_SPECULATIVE or exogenousFlow tag exempts the recipe. Phase 1 declares the fields the auditor will read; Phase 1 does not write the auditor.
Thermal-kW throttle
Phase 3 ships the proportional-throttle resolver in heat.ts that reads thermalKW / heatDemandKW and replaces the boolean N:1 gate. Phase 1's heat fields are declared but unread.
CO₂ accrual integrator
Phase 2 wires the per-recipe CO₂ accrual into advanceIsland's integrator. Phase 1's co2Kg field is declared and seeded to 0; nothing increments it yet.
Map picker UI
Phase 6 ships the static-SVG world-map picker that prompts on null lat/lon. Phase 1's fields are declared and persisted; no UI surfaces them.
Per-resource massPerUnitKg usage
Phase 1 populates the values from rev-16 §2.2 because the exhaustive Record requires it. No production code reads the values in Phase 1; Phase 2's auditor is the first consumer. The values are sanity-checked against rev-16 §2.2 in a snapshot test but not used elsewhere.
New resources (mill_scale, co2, refinery_gas, etc.)
The 7+ new resources rev-16 introduces ship in Phase 2 alongside the recipes that produce them. Phase 1's RESOURCE_META covers the current 252; Phase 2 extends both ResourceId and RESOURCE_META in one commit per added resource.
Cohort recipe rewrites
The §3.9 cohort (steel_beam, heavy_cable, sheet_metal, pipe, wire, spring, bearing) ships in Phase 2. Phase 1 doesn't touch RECIPES at all — only adds metadata that the Phase 2 auditor will read against the new recipe shapes.
Per-recipe-output terminal tagging
Rev-16 §10.9 resolved this to per-resource tagging (one tag per resource, not per output of each recipe). Phase 1 honors that resolution; the per-recipe-output variant is explicitly out of scope unless a future audit surfaces an H₂-from-chlor-alkali-style conflict.
