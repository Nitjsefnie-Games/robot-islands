Mass Building Actions — multi-select & batch operations

Context Today every building action (destroy, upgrade, move, enable/disable, ignore-cap) runs one building at a time through the single-building inspector. This spec adds a multi-select model (shift-click + shift-drag box, single-island scope) and a mass-action mode in the inspector that batches the existing per-building gateway operations — no new authoritative ops, no schema change.

Decision
Selection state lives in main.ts; the inspector hosts a delegated mass-panel; mass actions loop existing gateway calls.
main.ts already owns the canvas click/drag handlers, the selection-outline layer, and all gateway wiring, so the selection set belongs there. The inspector visually hosts the mass-action UI (per the chosen approach) but delegates its body to a focused new inspector-multi.ts, keeping the already-2445-line inspector-ui.ts from absorbing batch logic.
Deliberately not done: cross-island selection, group rotation, and any new MutationGateway method or snapshot-schema bump. Mass actions are sequenced existing intents, so they add zero new trust surface.
01

Selection model

main.ts holds the selection set, scoped to one island.

selection: Set<buildingId>       // building ids on selectionSpec
selectionSpec: IslandSpec | null  // the single island the selection lives on
GestureEffect
Plain click on a buildingSingle-select (clears set); opens inspector single-mode (existing path).
Shift-click on a building on selectionSpecToggle that id in/out of selection.
Shift-click on a different islandIgnored (single-island scope).
Shift-drag from empty startRubber-band box mode; on release adds every building on selectionSpec whose footprint intersects the box.
Plain click on empty oceanClear selection + close inspector (existing).
EscapeClear selection (existing dismiss path).

The selection-outline layer iterates selection and draws one outline per building (today it draws exactly one). The rubber-band box is a new lightweight overlay rect, following the existing ghost-drag / bend-drag overlay pattern. selection.size ≤ 1 → single inspector; ≥ 2 → multi-mode.

02

Approach chosen

Where multi-mode lives, and why the inspector delegates rather than absorbs.

Rejected — all inline in inspector-ui.ts
  • Grows the largest file (already 2445 lines).
  • Tangles single-building and batch responsibilities.
  • Move-group has to loop the single beginRelocate.
Chosen — delegate to inspector-multi.ts
  • Selection owned by main.ts (already owns handlers + gateway).
  • Inspector visually hosts the panel; body delegated to a focused new file.
  • Honors "one responsibility per file" (AGENTS.md).
  • Group move gets a dedicated placement-ui mode.
03

Mass-action semantics

Each action operates only on selected buildings for which the single-building op is currently legal (operational, not queued); ineligible ones are silently skipped and the panel reports the effective count.

ActionBehavior
DestroyOne window.confirm showing aggregate scrap + 50% refund (existing preview helpers, summed). Then demolishBuilding per building, drainRoutesForBuilding per building, one rebuildWorldLayers(); clear selection + close.
Upgrade (n fit)Pure planMassUpgrade picks the lowest-floor, affordable buildings that fit free build + queue slots (see below). main.ts then calls applyUpgrade per returned id. Button shows "Upgrade (n fit)"; disabled at n=0.
EnableSeparate button. setBuildingActiveFloors(id, 0) for every selected building.
DisableSeparate button. setBuildingActiveFloors(id, allFloors) for every selected building with any active floor; drains routes like the single op, one rebuild.
Ignore-capOne checkbox per resource in the union of selected buildings' outputs. Tick → setIgnoreCap(id, resource, true) for every selected building that outputs it; checkbox shows checked when all such buildings already have it set.
MoveGroup-drag rigid cluster; all-or-nothing drop (see §05).

Mass-upgrade algorithm — planMassUpgrade(state, selectedIds) → buildingId[], pure:

  1. freeSlots = (parallelBuildSlots − inProgressBuildCount) + (queuedBuildSlots − queuedBuildCount). If ≤ 0, return [].
  2. Eligible selected buildings (operational, below max floor), sorted ascending by current floorLevel (laggards first); deterministic tie-break.
  3. Walk the list with a running copy of state.inventory: if a candidate's upgradeCost fits the running inventory, select it and deduct; else skip and continue. Stop at freeSlots selected or list end.
  4. Return the selected ids (≤ freeSlots).

Skipping (not stopping) on an unaffordable laggard keeps cheaper upgrades flowing into the remaining slots — mirroring what sequential single-clicks would do.

04

Move — group drag

A new group-relocate mode in placement-ui.ts, sibling to the single beginRelocate.

  • Selection becomes a rigid cluster following the cursor, preserving each member's relative offset.
  • Each frame, validate the whole cluster as a unit via pure validateGroupRelocate(spec, state, members, dx, dy): every member's new footprint must be in-island, pass biome/tier gates, and not overlap any non-selected building or another member. Ghost renders green (all valid) / red (any invalid).
  • All-or-nothing drop: on a fully-valid drop, relocateBuilding per member, each paying its own relocateFee (50% invested cost, matching single-move). Invalid drop → no-op.
  • Group rotation is out of scope for v1.

06

Verification

Pure-layer unit tests (render code stays read-only, untested per convention).

planMassUpgrade. Lowest-floor-first; skip-unaffordable with running-inventory depletion; respects freeSlots = running + queue; [] when no slots / nothing affordable; excludes max-floor and under-construction.
Box hit-set. Footprint-intersecting buildings included; partial overlaps count; non-selectionSpec islands excluded.
validateGroupRelocate. Rejects member leaving island / hitting biome gate / overlapping a non-selected building or sibling; accepts a clean translation; fee = sum of per-member relocateFee.
Ignore-cap union. Checked-state derivation: all output-bearing buildings set → checked.
07

Out of scope (v1)

Cross-island selection
Slots, inventory, and upgrade costs are per-island; single-island scope keeps the algorithm simple. Revisit only if demanded.
Group rotation during move
v1 preserves orientation; rotating the cluster is deferred.
New gateway / schema / migration
Mass actions are sequenced existing intents; no authoritative-surface or persistence change.
Inherently single-building inspector features
Cargo relabel, land reclamation, servitor conversion, time-lock, etc. stay single-only.
