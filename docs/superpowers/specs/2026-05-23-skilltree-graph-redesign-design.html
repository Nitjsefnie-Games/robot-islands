Design spec · skilltree · graph redesign

Skill graph — full redesign as a directed PoE-density graph

Throw out the current 225-node linear catalog and any code handling commitment, branch-lock, and tier-gates inside the skill tree. Replace with a directed graph in which standard boost nodes form depth-ramping templated chains, hand-curated notable nodes carry mid-power named effects, and rare keystone nodes gate rule-breaking unlocks behind AND-prereqs. The graph spans 5 branches × 4 sub-paths × ~30-40 nodes each for ~600-800 total nodes, with cross-sub-path and cross-branch threshold-bridges activating soft progression gates without per-bridge prereq nodes.

Authority Replaces and supersedes SPEC.md §9.3 (Skill Tree Structure) for the current implementation. This design retains the spec's headline goals (per-island progression, branch-thematic structure, every node theoretically reachable, depth 1-2 cheap and depth deepest game-warping) while overhauling the underlying mechanism from linear-chain-with-branch-lock to directed-graph-with-cost-paths. Audit verdict 2026-05-23-skilltree-depth-analysis v1 (Findings 1-5, 7, 8) drives the redesign.

Structural skeleton — 5 × 4

Five thematic branches, each containing four sub-paths. A 6th branch (Space) is left implicit — when SPACE-expansion content lands, it attaches via graft sockets on Orbital's outer rim and/or as a new branch slotted in. The data model is branch-count-agnostic.

Entry model — branch-roots, all owned at start

An island owns the 5 branch-root nodes from level 1, free. The 4 sub-paths within a branch all radiate outward from their shared branch-root — i.e. the first depth-1 node of Mining, Forestry, Drilling, and Robotics each connect by an edge to the single Extraction root. This gives the player a clear "you are here" anchor per branch and makes the rare cross-branch bridges (which connect outer-rim nodes in different branches) feel like genuine departures from the home territory.

03

Edges & topology — mixed AND/OR semantics

Three edge regimes, designer-controlled per node.

Edge regimeWhere usedPlayer meaning
Single in-edgeFiller and notable nodes. The vast majority."Buy the predecessor, then this is the next standard step." A directed-chain progression — but the chain branches, so each node may have several outgoing edges.
OR-prereq (single edge from any of N sources)Bridge endpoints (cross-sub-path within a branch, and cross-branch). The destination node accepts ANY one of N incoming edges."Reach this node by any of N routes. Take the cheapest from your current frontier." Alternate-entry semantics; the silent-cheapest-path solver naturally chooses.
AND-prereq (all of N edges)Keystone nodes. The destination requires ALL N incoming source nodes to be owned."This unlock demands genuine investment in 2-3 different upstream points." Synergy: the keystone's effect leverages the convergence of multiple prerequisites.
Threshold-bridges (Borderlands-inspired) are a parallel edge primitive, not a node type. A bridge edge has a normal SP cost AND a threshold attribute: "spend ≥ N points in the source branch OR ≥ M points in the destination branch to activate this edge." Until the threshold is met, the bridge is effectively unowned-and-unreachable; once met, it becomes part of the solver's graph and gets considered for cheapest-path. This lets entire branch-to-branch corridors open up softly as the player invests, without scattering "unlock-this-bridge" prereq nodes around.

Internal sub-path topology

Each sub-path is a directed sub-graph rooted at the branch-root. Standard nodes form roughly tree-like chains 4-7 levels deep with sibling branches at each level. Notables sit at junction points (often where two filler chains converge). Keystones sit at the deepest convergence points, requiring AND-prereqs across multiple notable predecessors. ~3-5 OR-bridges (within-branch, between sibling sub-paths) sprinkle the middle layers; cross-branch bridges hit deepest-tier nodes only.

04

Cost model — node-click + silent cheapest path

Every edge carries an SP cost. Nodes themselves are free; ownership is implied by paying for any incoming edge. The player initiates a purchase by clicking an unowned node; the system finds the cheapest path of unowned edges from the player's current owned frontier to that node, charges the path's total edge-sum, and marks every intermediate node and edge along that path as owned.

// pseudocode
function costToUnlock(target: NodeId, owned: Set<NodeId>, graph): { path: Edge[]; cost: number } {
  // Multi-source Dijkstra: roots are all currently-owned nodes;
  // edge weight is each edge's SP cost; unowned-frontier-to-target.
  // Threshold-bridges contribute their edge weight ONLY IF activated.
  return shortestPath(sources=owned, target, graph, ignoreEdgesViolatingThresholds(...));
}

function buyNode(target: NodeId, state: IslandState): void {
  const { path, cost } = costToUnlock(target, state.unlockedNodes, graph);
  if (state.unspentSkillPoints < cost) throw 'insufficient-points';
  state.unspentSkillPoints -= cost;
  for (const edge of path) state.unlockedNodes.add(edge.target);
}

The UI shows each unowned node's cost-to-unlock from the current state, updated live as the player accumulates nodes. Bridge edges show their threshold status (locked / unlocked-and-available).

Intermediate-effect activation. When a path purchase auto-owns intermediate nodes, those nodes' effects activate immediately. Owned = active; there is no "owned-but-dormant" state. This keeps the semantics simple: cost = full commitment.

AND-prereq purchases

Keystone nodes can only be purchased when ALL of their incoming source nodes are already owned. The UI greys out keystones whose AND-prereqs aren't satisfied. Once satisfied, the keystone's own edge cost (paid at the convergence point — typically a fixed flat number, see PLACEHOLDER §09) is the unlock price; no path-finding needed since the player must already own the prereqs.

Threshold-bridge activation

A bridge edge from sub-path A to sub-path B activates when spentIn(branchOf(A)) ≥ threshold_A OR spentIn(branchOf(B)) ≥ threshold_B. Computed at purchase-evaluation time (no eager re-scoring of all bridges per tick). The cost-to-unlock UI re-renders affected unowned nodes' costs when a threshold is crossed.

07

PoE / D4 / LE patterns adopted

Three structural patterns borrowed from established skill-tree designs, each addressing a specific axis of "complex but navigable."

7.1 Threshold-bridges (Borderlands 3)

Bridges between regions (sub-paths within a branch, branches across the rim) are edges in the graph, not separate prereq nodes. Each bridge edge carries:

  • cost: number — SP cost like any edge
  • threshold: { branch: BranchId; minSpent: number } | { branch: BranchId; minSpent: number }[] — activation condition (OR across listed thresholds)

The cheapest-path solver considers a bridge edge ONLY when at least one of its thresholds is met. Below threshold, the edge is effectively absent from the graph. Removes per-bridge prereq-node clutter (PoE has explicit unlock nodes for cluster jewel sockets; we don't need to).

7.2 Graft sockets (PoE Cluster Jewels)

Reserve ~6-12 graft sockets on each branch's outer rim (the deepest layer of each sub-path). A graft socket is a position in the graph with a defined attachment point and no node attached yet. Late-game content adds nodes by attaching a 5-10 node sub-cluster to a specific socket.

Use cases:

  • SPACE expansion content — flagged by user as imminent; attaches modular content to Orbital's outer rim without rebalancing the core.
  • Network Consciousness milestones (§9.6) — could unlock graft-cluster content tied to the milestone (e.g. 5-island milestone grafts a "Robotics breadth" cluster onto Robotics' rim).
  • Crafted T5 jewels — late-game craft recipe outputs a "jewel" item that the player sockets into one of N graft positions, attaching a small chosen sub-cluster.

Engine: a graft socket has { id, branchId, subPathId, attachmentDepth }. When content attaches, a new node-and-edges sub-graph is appended; ownership and cost behave identically to hand-designed nodes.

7.3 Glyph-locality (Diablo 4 Paragon)

Selected notable nodes carry an aura effect: their declared multiplier applies not just to their own boost, but also to all adjacent owned nodes' effects. Spatial layout becomes a real decision — clustered builds get an extra ~1.1×-1.2× multiplier on the cluster as a whole.

Mechanic:

  • An aura-bearing notable declares aura: { radius: 1 | 2; bonus: number; appliesTo: EffectKindFilter }.
  • Radius = graph-edge distance (1 = direct neighbours; 2 = nodes within 2 edges).
  • At effectiveSkillMultipliers aggregation time, owned nodes' contribution is multiplied by Σ aura-bonuses from any aura-bearing notables within whose radius they sit.
  • Pairs naturally with the cheapest-path solver: building toward an aura notable inherently clusters your owned nodes near it, so the aura's payoff scales with depth.

Use sparingly — maybe 10-20% of notables are aura-bearing. The rest are flat named effects.

08

Curation strategy — hybrid

Hand-design the ~105+ keystones + notables (the soul). Archetype-template the ~500+ filler via cost+magnitude formulas.

8.1 Filler templates

Each sub-path declares ~5-8 filler archetypes, where each archetype = { effectKind, baseMagnitude, magnitudeGrowth, baseCost, costGrowth, count }. Examples:

// Mining sub-path filler archetypes
[
  { effectKind: 'recipeRateMul', category: 'extraction', baseMag: 0.04, growth: 1.45, baseCost: 1, costGrowth: 1.4, count: 6 },
  { effectKind: 'storageCategoryCapMul', category: 'raw-ore', baseMag: 0.05, growth: 1.4, baseCost: 1, costGrowth: 1.5, count: 5 },
  { effectKind: 'mineYieldBonusMul', baseMag: 0.05, growth: 1.5, baseCost: 2, costGrowth: 1.6, count: 4 },
  ...
]

Generator function: generateFillerNodes(archetypes, depths) → SkillNode[]. Deterministic; the same archetypes always produce the same nodes. The output array is committed to the catalog (not regenerated at runtime) so designers can hand-edit specific nodes after generation if needed.

8.2 Notable curation

Each notable is hand-written with a unique name, themed description, and specific effect. Notables can use ANY effect kind including the new ones. ~5 notables per sub-path × 20 sub-paths = ~100 hand-written notables. Each notable's data is ~10 lines of TypeScript — total ~1000 LoC of curated content.

8.3 Keystone curation

Each keystone is hand-written with extensive thought: payload effect (typically one of the 4 newly-wired + 3 new kinds), AND-prereq specification (which 2-3 notable predecessors must be owned), thematic name, and player-visible description. ~1.5 keystones per sub-path × 20 = ~30 hand-written keystones. Each is ~15-20 lines of TS — total ~500 LoC.

8.4 Bridge & socket curation

Bridges are hand-placed at design time, not generated. ~3 bridges per sub-path × 20 = ~60 within-branch bridges, plus ~10-15 cross-branch bridges. ~75 bridges total. Each is a ~5-line declaration.

Graft sockets are similarly hand-placed: ~10 per branch × 5 = ~50 reserved sockets. Each is a ~3-line declaration with no payload attached at v1.

SP-economy sanity check (PLACEHOLDER): at ~1 SP/level baseline + xp-softcap past lvl 50, a level-100 island ends with ~100-200 SP. At average ~3-4 SP/edge in the graph, that's ~30-60 nodes owned out of ~600+. Mid-game islands own <10% of the tree; very-late-game islands own ~20-30%; only insanely-long-grinded islands approach full mastery. Compares to PoE: a fresh lvl 90 character owns ~85-95 of ~1300 nodes (~7%). Our ratio is similar at lvl 100, more permissive at lvl 200+.

11

Open questions for the plan

Questions to resolve when the implementation plan is written. None block writing the plan; the plan will surface them as decision points.

  1. How does the UI render ~600-800 nodes? Graph view is the locked direction (audit Picker 8); concrete UI design is its own spec (probably 2026-05-23-skilltree-graphview-ui-design.html as a follow-up).
  2. Hex grid or freeform layout? PoE1 is freeform; PoE2 shifted toward themed weapon-shaped constellations. Affects render code, not data model.
  3. How does Specialization role (§9.4) reshape the graph? Out of scope here. Could unlock a 5th sub-path per branch ("ascendancy" style) — designed in a separate spec.
  4. Per-node respec? Deferred; if added, becomes a paid action at T2/T3 resource cost (Last-Epoch-style), implemented as a new refundNode function that reverses cheapest-path purchases.
  5. Aura stacking semantics with overlapping auras? Placeholder: additive within a node, capped at +50%. Final value to playtest.
  6. Graft socket attachment UI? When a cluster jewel (or equivalent) is acquired, how does the player place it? Drag-to-socket vs auto-attach vs choice-prompt. Probably a UI spec follow-up.
  7. Network Consciousness graft unlocks? §9.6 milestones could unlock graft-cluster content. Specifies which milestones unlock which sockets. Design choice deferred.
  8. Exact threshold values per bridge? Tuning concern; placeholder ranges in §09. Playtest-driven.
  9. How many cross-branch bridges to ship at v1? ~10-15 hand-placed feels right; final count emerges from designing the keystones (cross-branch keystones tend to need bridges to reach across).
Out of scope for THIS spec: The full hand-written content of every keystone, notable, and bridge. The spec describes the data model and curation strategy; the catalog content itself gets authored in a separate "skilltree-content" task or set of tasks during implementation.
