Net-flow ⇄ power-brownout joint fixpoint

Context The §15.3 net-flow solver was built to kill the +0/+0.0x rate flicker by gating producers/consumers so a pinned bin runs at net exactly 0. But brownout (powerFactor) is computed after the solve and applied only to power-consumers, so it perturbs the solver's balance asymmetrically and the flicker returns through the power side. This spec replaces that documented one-segment lag with a joint g ⇄ pf per-pool fixpoint.

Decision
Solve the net-flow gate g and the brownout scalar pf as one damped per-pool fixpoint, so pinned bins net to 0 under the realized brownout-scaled flows.
A "power pool" shares one brownout scalar (a unified §5.3 cable component, or a single island). The fix is one shape regardless of topology: pre-scale each power-consumer's flow coefficients by pf, solve g, recompute pf from the g-scaled draws, iterate. Because pass 4 realizes base × g × pf, the solver then balances exactly what pass 4 produces — pinned bins net to 0, no drift, no flicker.
A no-brownout early-out keeps the common case (ΣP ≥ ΣC ⇒ pf = 1) byte-identical to today at one solveFlow call. Cross-island convergence is per-tick, not per-segment — the shared scalar means no lockstep-per-segment machinery is needed.
01

Problem

Why the solver's pinned-bin guarantee breaks under brownout, and why this is a spec contradiction rather than only a code bug.

The solver computes g on nominal (gate-1) coefficients with no knowledge of power. Brownout is computed after the solve (pass 3) and applied in pass 4 (economy.ts:1747) only to power-consuming buildings (consumesPower ? powerFactor : 1), faithful to §5.1.

When a pinned resource's producers and consumers differ in power-dependence, powerFactor scales the two sides of that bin asymmetrically, so realized net at the pinned bin is no longer 0. The bin drifts off its boundary, the next segment's constraint set flips, and g/pf swing — the +0/+0.0x flicker the solver was built to remove, reborn through the power side. The introducing design doc named this an accepted approximation:

Documented approximation: powerFactor computed after the solve; brownout-reduced draw re-throttles suppliers next segment (clamps pin the bin meanwhile). Same lag as today. — 2026-06-10-net-flow-economy-design.md:109-111

A one-segment-delayed negative-feedback loop is the textbook recipe for oscillation. "Same lag as today" was the error: the solver removed the storage-side flicker but left the power-side lag, assuming it benign.

Spec as written — contradictory
  • §15.3 (SPEC.md:2050): a pinned bin runs at net exactly 0, emits no cap event.
  • §5.1 (SPEC.md:607): power_factor multiplies the rate of every consumer — asymmetric across a bin's participants.
  • Both cannot hold under active brownout.
After this spec
  • Pinned bin nets to 0 under the realized (brownout-scaled) flows.
  • g and pf co-solved per pool — consumer-only scaling no longer breaks the §15.3 invariant.
  • No flicker; O(1) segments at a brownout-pinned steady state.
02

Core insight — a power pool shares one brownout scalar

The cross-island story collapses: within a unified component the brownout factor is a single shared scalar, so there is no per-segment lockstep machinery to build.

Within a unified §5.3 cable component the brownout factor is a single shared scalar — min(1, producedTotal / consumedTotal) over the aggregate, applied identically to every member's power-consumers (economy.ts:1721-1725). Define a power pool = the set of buildings sharing one brownout scalar this tick: a unified cable component, or a single island (no cables, or its component's gate fails and it falls back to local, economy.ts:1726).

Regardless of topology the fix is one shape — a joint fixpoint over (g vector, pf scalar) per pool:

// per power pool, iterate to mutual consistency:
pf := min(1, ΣproducesW / Σ(consumesW_i × throughputFrac_i(g)))   // one scalar over the pool
g  := solveFlow(...)  with each power-consumer's flow coefficients pre-scaled by pf
      // pass 4 realizes base × g × pf, so the solver balances exactly the realized flows
repeat until ‖Δg‖∞ < ε and |Δpf| < ε   // damped; MAX_ITERS cap → fail open

throughputFrac_i(g) is the existing §5.1 throughput-scaled draw (effectiveMul × g × heat), so pf stays consistent with the realized g. Resource pooling is orthogonal: §13.3 lattice and §15.1 shared-network shape the solveFlow input (the flowSiblings union / pooled inventory) but not the power story — the same fixpoint wraps whatever flow problem they hand it. "Everything in one pass" is one mechanism with a wider aggregation set, not three reworks.

03

Architecture

A new pure leaf module owns the fixpoint; existing callers swap their solveFlow call for it.

New src/flow-power-fixpoint.ts — sits one level above flow-solver.ts; no PixiJS, no DOM, no economy.ts import (same leaf discipline). Given a pool's flow problem(s) + per-building power spec (producesW, consumesW, the throughputFrac rule) + cap/zero constraints, returns { gates, powerFactor, converged }. It is a thin damped outer loop over a low-dimensional scalar; the inner exact solve is flow-solver.ts, reused unchanged.

  • No-brownout early-out: if the first solve (pf = 1) leaves the pool at ΣP ≥ ΣC, that is already the fixpoint — return after one iteration. Common case costs the same as today and yields byte-identical gates.
  • Convergence: iterate until ‖Δg‖∞ and |Δpf| are below ε, or MAX_ITERS → converged = false, return the last clamped iterate (fail open, mirroring flow-solver.ts's FLOW_MAX_SWEEPS + post-100 damping). The g→pf map is a contraction-flavoured negative feedback, so it settles in a handful of iterations.
  • Cable component: N independent per-island flow blocks (resources don't cross cables) sharing one pf aggregated over the union of member buildings; each outer iteration solves the blocks against the current shared pf, then re-aggregates. Solo / lattice-union / shared-net-union are a single block.

05

Verification

Concrete, runnable checks — the failing tests come first (TDD).

No-brownout regression. ΣP ≥ ΣC ⇒ one iteration, pf = 1, gates identical to solveFlow.
Asymmetric-power pinned bin. Power-consuming producer + non-power consumer at a zero-pinned input; power-consuming consumer + non-power producer at a cap-pinned output — realized net = 0 under returned (g, pf).
Convergence / fail-open. converged = true within MAX_ITERS for worked cases; pathological pool returns clamped [0,1] iterate, no NaN, no throw.
Flicker regression (economy.test.ts). Steady-state rate/utilization constant across many segments under brownout; 1 h advance at a brownout-pinned bin = O(1) segments.
Mass balance. No conjuring at a zero-pinned input, no silent discard at a cap-pinned output, under brownout.
Cross-island. Two-island cable component in brownout converges to one shared pf; member gates balance local pinned bins under it.
Full suite green. npm test and npm run build clean under strict TS.

Out of scope

Explicitly deferred — considered, not missed.

Per-segment component-pf convergence
Re-converging the component scalar at every integration-segment boundary would require lockstep grouped advancement of cable components. Ruled out: the shared scalar + per-tick convergence already removes the 5 Hz-visible flicker; per-segment is strictly-more-correct but invented complexity here.
