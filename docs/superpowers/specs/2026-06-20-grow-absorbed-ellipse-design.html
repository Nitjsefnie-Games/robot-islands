Design spec · robot-islands

Grow absorbed ellipses via Land Reclamation

Context Today, Land Reclamation Hub grows only a merged island's primary ellipse (ellipses[0]); absorbed lobes (extraEllipses) are frozen geometry forever (SPEC §3.4 / §15.1). This spec lets the player grow any constituent of a merged island, choosing it through a per-lobe picker in the Hub inspector, with each lobe capped by its own origin biome.

01

Background & current behavior

What the geometry does today and why merged islands feel stuck.

An IslandSpec carries a primary ellipse (majorRadius / minorRadius at offset 0,0) plus zero or more extraEllipses appended by §3.6 merges. A tile belongs to the island iff inscribed in any constituent (islandConstituents walks primary-then-extras). Geometry consumers — islandsOverlap, islandTileCount, terrainAt — already honor every constituent.

But Land Reclamation only grows the primary: expandIsland mutates spec.majorRadius / minorRadius and nothing else (SPEC §15.1: "Land Reclamation grows ellipses[0]'s radii"). Absorbed lobes are frozen at their merge-time size forever. The landReclamationCost union path already charges only tiles not already covered by another constituent, so the cost math is constituent-aware; only the choice of which ellipse grows is hardcoded to the primary.

03

Data model (§15.1)

One new field, derived caps, no frozen numbers.

extraEllipses[i] gains biome: BiomeType — the origin biome of that absorbed constituent. The cap is always derived (BIOME_MAX_RADII[biome]), never stored as a number, so a future change to the cap table propagates automatically — symmetric with the primary, which already derives its cap from spec.biome.

The ConstituentEllipse view returned by islandConstituents also carries biome: the primary entry's is spec.biome; each extra's is entry.biome ?? spec.biome (the ?? spec.biome fallback makes legacy in-memory specs safe before migration runs).

// world.ts — IslandSpec.extraEllipses entry
extraEllipses?: Array<{
  readonly biome: BiomeType;   // NEW — origin biome, cap-only
  readonly major: number;
  readonly minor: number;
  readonly rotation: number;
  readonly offsetX: number;
  readonly offsetY: number;
}>;
04

Engine API — constituent-indexed

The three reclamation functions take a constituent index; index 0 reduces to today's behavior exactly.

  • landReclamationCost(spec, index, axis) → union delta of growing constituent index by +1: scan that constituent's new bounding box in island-local coords (offset by its offsetX/Y), charge only tiles inscribed in the post-grow union but in no pre-grow constituent. index 0 is exactly the current primary path.
  • canExpandConstituent(spec, state, index, axis) → same rejection precedence (no-hub → axis-at-max → insufficient-resources). Cap is per-constituent: primary uses spec.biome; extra i uses extraEllipses[i].biome. Current radius read from the chosen constituent.
  • expandConstituent(spec, state, index, axis) → index 0 mutates spec.major/minorRadius as now; extra i rebuilds extraEllipses[i] with major/minor +1 (entries are readonly — same rebuild pattern performMerge already uses for buildings). Defensive no-op on a failed re-check, as today.

Thin expandIsland / canExpandIsland wrappers (= constituent 0) may be kept to limit caller churn, or callers migrate to the indexed form — implementer's call during planning.

05

Merge & persistence migration

Record origin biome on append; default legacy saves to the absorber's biome.

Merge (§3.6, island-merge.ts). performMerge step 1 stamps biome on every appended constituent: the absorbed primary→extra gets absorbed.biome; each propagated extra (recursive merge history) keeps e.biome ?? absorbed.biome. Biomes carry forward correctly through chained merges.

Persistence (v27 → v28). extraEllipses round-trips through the generic ...s island spread (only one comment-reference in persistence.ts; no explicit field enumeration), so the migration is a focused walk:

// migrateV27toV28: stamp biome on legacy extra ellipses
for (const isl of snapshot.world.islands)
  for (const e of isl.extraEllipses ?? [])
    if (e.biome === undefined) e.biome = isl.biome;  // = absorber's

Defaulting legacy extras to the island's own (absorber's) biome reproduces today's behavior exactly — before this change every lobe was implicitly capped by the absorber's biome. Add 28 to SUPPORTED_LOAD_VERSIONS, add the SerializedSnapshotV27 alias, wire migrateV27toV28 into the chain (per the AGENTS.md bump=migrate checklist).

06

Mutation gateway / trust surface

One new intent field; server re-validates with the same pure functions.

The expand-island intent gains an optional constituentIndex (default 0). LOCAL calls expandConstituent directly; REMOTE sends the intent and the server re-runs the same canExpandConstituent / expandConstituent to validate, so LOCAL and REMOTE stay in lockstep (SPEC Appendix C). An out-of-range or stale index fails the server's re-check and is rejected — no trust placed in the client's index. Legacy intents without the field validate as index 0.

07

UI — lobe picker & badges

A scrollable per-constituent list in the Hub inspector, plus map badges that identify each row.

The Hub inspector's single primary +MAJ/+MIN pair becomes a scrollable list — one row per constituent in islandConstituents order. Each row shows its badge number, biome label, both radii against their derived caps, and per-axis buttons gated/labeled via canExpandConstituent + landReclamationCost (now correctly passing the full constituent set, building on the cost-preview fix already on master):

Land Reclamation — 3 lobes              ⤓ scrolls if tall
┌──────────────────────────────┐
│ #1 · Plains    r20/28 · r18/28  [+MAJ][+MIN] │
│ #2 · Volcanic  r8/14  · r8/14   [+MAJ][ CAP] │  ← minor at cap
│ #3 · Volcanic  r10/14 · r9/14   [+MAJ][+MIN] │
└──────────────────────────────┘

A new render-layer overlay (lobe-badge-overlay.ts, per the *-overlay.ts convention) draws a numbered badge at each constituent's center (world→screen) while the Hub panel is open. Badge #N ↔ row #N; display is 1-based, internal index 0-based (#1 = primary). Render code stays read-only against state per the architecture split.

09

Verification

Pure-layer tests only; render layer (picker, badges) stays untested per the repo convention.

Cost union-delta for an extra. Growing extraEllipses[i] charges only tiles not already in any constituent (incl. overlap with the primary / siblings).
Per-biome cap gating. A Volcanic lobe rejects with axis-at-max at 14 while the Plains primary still expands toward 28.
Targeted mutation. expandConstituent(i) grows exactly entry i and leaves the primary + sibling lobes byte-identical; index 0 still grows the primary.
Merge stamps biome. Appended and recursively-propagated extras carry the correct origin biome after a chained merge.
Migration. A v27 fixture with biome-less extras loads into v28 defaulting each to the island biome; v28 round-trips identity.
Gateway parity. LOCAL and REMOTE produce identical state for the same constituentIndex; an out-of-range index is rejected server-side.
Full suite green. npm test (with Postgres up) and npx tsc -b --force clean.

11

Out of scope & spec edits

Deliberately deferred, named so they read as considered-not-missed.

Per-lobe terrain / biome generation
Terrain stays the absorber's biome per the §3.6 note. The new field is cap-only; per-lobe terrain is a separate, larger change.
Ellipse rotation
rotation remains metadata-only across all geometry; absorbed lobes still enter at rotation 0. Untouched here.
Map-click lobe selection / color-coding
Rejected in favor of the picker + numbered badges. Could be revisited if badges prove insufficient in playtest.

Spec edits shipped with the code: §3.4 (reclamation grows any constituent; per-lobe origin-biome cap; union-delta cost of the chosen constituent), §3.6 (appended constituents record origin biome; reaffirm terrain stays absorber-biome), §15.1 (extraEllipses.biome field, cap-only).
