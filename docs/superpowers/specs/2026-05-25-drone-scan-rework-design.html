Drone scan rework — tier-keyed corridor + survive-and-report

Context Three drone-scan bugs surfaced while preparing the skill-tree rebalance: (1) every drone tier T1–T4/T6 uses one scan-radius constant DRONE_SCAN_RADIUS_TILES = 8 regardless of tier, contradicting SPEC.md §2.2 (T1 narrow / T2 modest / T3 wider); (2) for T1–T4/T6 the per-tick reveal loop adds corridor cells to world.revealedCells only when the cell centre is in antenna range right then, so out-of-range corridor cells are silently dropped with no buffer or return-trip flush; (3) a drone launched from an island with no antennas anywhere reachable scans nothing recoverable even if it returns safely. All three must be fixed before the rebalance because they distort the cap-magnitude conversation for droneScanRadiusMul and commRangeMul.

Decision
Per-tier scan-radius table + unified buffer-and-flush across all drone tiers.
Replace the single DRONE_SCAN_RADIUS_TILES = 8 with a per-tier lookup DRONE_TIER_SCAN_RADIUS: Record<DroneTier, number>. Tier 1 / 2 / 3 take values 2 / 4 / 8 (user-locked). Tier 4 is the omnidirectional pulse (separate disk mechanic, unaffected). Tier 5 keeps its existing DRONE_T5_SCAN_RADIUS_TILES = 12. Tier 6 needs a pick (proposed 16; flagged for approval below).
Replace the per-cell antenna-range check with a per-drone scanBuffer: Set<string> of cell keys (and the existing darkModeDiscoveries for islands continues to do the same job). On every tick, every corridor cell goes into the buffer regardless of antenna range. A separate flush step adds cells from the buffer to world.revealedCells when (a) the drone re-enters antenna range, or (b) the drone reaches terminal status 'returned'. Drone destroyed in dark (status 'lost') → buffer discarded. This unifies T1–T6 (the existing T5 dark-mode-islands buffer becomes the general pattern; cells join it).
The no-antenna case (origin island has no antennas, no other populated island's antennas reach the corridor) falls out as a degenerate of (b) — the drone is always out of range, always buffering, flushes on return. No special-case code needed.
01

Spec citations

What SPEC.md says about the scan mechanic, with line numbers.

§2.2 Discovery via Drones (SPEC.md:93-111). Drones scan a corridor along their path; corridor width depends on tier and scan radius. The tier list explicitly differentiates: T1 narrow, T2 modest, T3 wider, T4 omnidirectional pulse (disk).

§11.5 Drone tiers (SPEC.md:1216-1222). Repeats the per-tier scan-corridor enumeration. T2 = "scan corridor radius W", T3 = "scan corridor radius 2W" — explicit 1:2 ratio between consecutive tiers. The 2/4/8 ladder doubles each step, matching the 1:2 doubling for the only adjacent pair the spec quantifies.

§11.6 T5 Path-Drawn Drone, "Telemetry and dark mode" (SPEC.md:1232-1235). "Within visibility range of any populated island: the drone transmits live. Discovered islands appear on the map immediately as the drone passes them. Beyond visibility range: the drone enters dark mode. It continues recording everything it scans, but transmits nothing. All findings are reported only on return." This is the canonical telemetry rule and applies to all drone tiers per user clarification — currently only the island-discovery half of it is implemented for T5; nothing is implemented for T1–T4/T6.

02

Spec vs. code, before / after

Two-column compare for each of the three fixes.

Fix 1 — Tier-keyed scan radius

Current code
  • DRONE_SCAN_RADIUS_TILES = 8 (single constant) at drones.ts:86.
  • Used uniformly for T1, T2, T3, T4, T6 at drones.ts:376 via the ternary isPathDrawn ? DRONE_T5_SCAN_RADIUS_TILES : DRONE_SCAN_RADIUS_TILES.
  • Tier-pick is decoupled from scan width — only range (DRONE_TIER_EFFICIENCY) and weather vulnerability (DRONE_TIER_MULTIPLIERS) scale with tier.
After
  • New DRONE_TIER_SCAN_RADIUS: Record<DroneTier, number> next to the existing tier tables.
  • T1 = 2, T2 = 4, T3 = 8, T4 = 0 (pulse; unused), T5 = 12 (existing), T6 = 16 (proposed; needs approval).
  • Dispatch reads the per-tier value; multiplies by originSkill.droneScanRadius as today.

Fix 2 — Corridor-cell buffer + flush

Current code
  • T1–T4/T6 branch at drones.ts:674-684: each corridor cell is added to world.revealedCells only if pointInSignalRange(ranges, center.x, center.y) at that moment.
  • Out-of-range cells: silently dropped. No buffer. No return-trip flush.
  • T5 branch at drones.ts:663-671: same per-cell antenna check applies even though the island-discovery side has dark-mode buffering.
After
  • New scanBuffer: Set<string> field on the Drone interface (cell keys).
  • Per-tick: every corridor cell joins scanBuffer unconditionally.
  • Flush rule A: when the drone is in any antenna's range at tick end, drain scanBuffer + darkModeDiscoveries into world.revealedCells / island.discovered; clear buffers.
  • Flush rule B: when the drone status transitions to 'returned', drain unconditionally (origin antenna may or may not exist — both cases handled).
  • Status 'lost' (destroyed by weather): buffer discarded with the drone. Matches spec: "all that data is lost" (SPEC.md:1237).
  • The T5 island branch's existing in-range / out-of-range split is replaced by this unified logic; the island-discovery and cell-reveal sides become two views of the same buffer-and-flush pipeline.

Fix 3 — No-antenna case

Current code
  • If no populated island has antennas in range of the corridor, pointInSignalRange returns false for every cell, every tick.
  • Result: drone flies, returns, and reveals nothing — even though it scanned a corridor.
  • No code path acknowledges this — the case is just the worst slice of bug 2.
After
  • Falls out of Fix 2 with zero additional code. Drone always buffers; flushes on return; if no antenna ever helps, the return-flush still drains.
  • Tested explicitly: launch from an island with no antennas, no neighbours with antennas, fly out, return → buffered cells appear in world.revealedCells.

Mitigation

LOW
Set operations are O(1) per insert. Worst case is a T5 path drone covering thousands of cells — Set + flush is still < 1ms per drone per tick. No new allocations per cell.
