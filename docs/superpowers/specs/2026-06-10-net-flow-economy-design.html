Design spec · robot-islands · 2026-06-10

Net-flow economy: demand-coupled throttling replaces the binary output stall

Context A producer whose output bin is at cap stalls entirely today (§15.3 "binary outputAvail"), even when a consumer is drawing from the same bin in real time. The result: rates flicker between 0 and a sliver, the upstream stall oscillates downstream consumers, and every cap-hit emits an integration event forever. This spec replaces the binary stall with an exact demand-coupled throttle: producers at a full bin rescale to precisely what consumers draw, and one utilization factor per building drives effective rate, XP, power draw, and (new) maintenance wear.

Decision
Solve the producer↔consumer rate fixed-point exactly, in a new pure module, and make one utilization factor per building drive rate, XP, power, and wear.
An exact solver (SCC condensation + topological propagation, ε-converged iteration only inside true feedback loops) was chosen over bounded relaxation ("4 passes and hope") at the owner's explicit request: overkill over underkill. Island building counts are tens, so exactness costs nothing measurable — and the steady state at a pinned bin emits zero integration events, where today it emits one per segment forever.
Deliberately NOT done: power brownout is not a solver variable. The powerFactor × flow interaction keeps today's one-segment lag (see §04).
01

Problem

Three symptoms, one cause: outputAvail takes storage at face value.

outputAvail (economy.ts:532) is binary per §15.3: if any output bin is at cap, the whole building stalls — no inputs, no outputs, no XP, no power draw. With a live consumer on the same bin the engine oscillates: stall → consumer drains a hair below cap → producer wakes at full rate → bin refills in one segment → stall. Visible flicker (+0 / +0.01), even through the 60 s rolling average when the duty cycle is long.

  • Input-side collateral: inputAvail is already continuous (supply/demand flow-through, economy.ts:575), but the binary output stall upstream makes its supply term oscillate, so consumers flicker too.
  • Event storm: every cap-hit lands a boundary event (findNextCapEvent); a factory parked at caps re-segments forever. Fixing this is a real performance win, especially for offline catchup.
  • Wear mismatch: operatingMs accrues at full wall-clock speed for every constructed productive building (economy.ts:2172) — a stalled or 1%-utilized building wears like a fully-loaded one.
02

Before / after

Locked behavioral rules, fixed during brainstorm with the owner.

Today (binary)
  • Any output bin at cap → building rate 0, flickering against its consumers.
  • Full bin + 3/s consumer + 5/s producer → average ~3/s realized via stall/wake oscillation, displayed as flicker.
  • Stalled building wears at full wall-clock rate ("can't escape maintenance pressure by capping output").
  • Cap-pinned bins emit one integration event per segment, forever.
After (net-flow)
  • Producers at a full bin throttle to exactly the consumers' draw: 5/s producer, 3/s consumer → producer runs at u = 0.6, bin pinned at cap, net exactly 0.
  • Min rule for multi-output recipes: the most-constrained output sets the building's rate. Strictly no waste — mass balance holds, no XP-at-cap farm.
  • Input side unchanged in spirit (already continuous): bin at 0, 3/s inflow, 5/s demand → consumer at 60%. Generalized into the same solver.
  • One utilization factor per building drives effective rate, XP, power draw, and (new) maintenance wear.
  • Pinned bins emit no events; steady-state factories integrate in long segments.
03

Flow-solver contract (new pure module flow-solver.ts)

One responsibility: given building flow specs and per-resource regimes, return the exact gate factor per building.

ElementDefinition
Input: building specPer building i: nominal rate (pass-1 tentative, static multipliers folded in), per-resource production coefficients p[i][r] and consumption coefficients c[i][r] in units/sec at gate 1. Exogenous-atmosphere inputs excluded, matching inputAvail.
Input: resource regimePer resource r: AT_CAP (inv ≥ cap), AT_ZERO (inv ≤ 0), or BUFFERED (in between — the buffer absorbs, no constraint). Uses the same inv()/cap() helpers, so the §13.3 Lattice inventory/caps override works unchanged.
OutputGate g[i] ∈ [0,1] per building — replaces the ia × oa product.
Cap constraintFor r AT_CAP: realized production of r may not exceed realized consumption of r. If violated at current gates, all producers of r share one throttle θ[r] = consumption / production (proportional sharing).
Zero constraintFor r AT_ZERO: realized consumption may not exceed realized production: φ[r] = production / consumption. This is today's inputAvail flow-through, relocated into the solver.
Min ruleg[i] = min(1, min θ[r] over i's AT_CAP outputs, min φ[r] over i's AT_ZERO inputs) — the most-constrained stream governs the whole building (owner's choice over surplus-vanishes).
ComplementarityA constraint binds only while it would actually be violated. If producer X is throttled below demand by a different output, resource r's cap constraint may deactivate (bin starts draining) and other producers of r relax back up. The solver resolves the active set consistently — not approximately.
MaximalityAmong gate vectors satisfying all constraints with shared per-resource factors and the min rule, the solver returns the greatest one (no unnecessary throttling). Start at g = 1, recompute θ/φ from realized flows, take mins; the operator is monotone non-increasing, so the iteration converges to the greatest fixed point.
AlgorithmBuild the building↔resource graph over constrained resources only; condense to SCCs; process components in topological order — exact closed-form propagation through the DAG (virtually every real factory). Inside an SCC (true feedback loops, e.g. A↔B): damped fixed-point iteration to ε = 1e-9 with a hard iteration guard. Deterministic, no RNG.
Worked example — cap throttle
Iron bin AT_CAP. Mine produces 5/s, Workshop consumes 3/s → θ[iron] = 3/5, Mine g = 0.6, bin stays pinned, net = 0, no event. Mine earns 0.6× XP, draws 0.6× power, wears 0.6×. With no consumer at all: θ = 0, Mine idles at g = 0 (today's stall, minus the flicker, minus the wear).
Worked example — zero flow-through (owner's case)
Iron bin AT_ZERO. Mine produces 3/s, Workshop demands 5/s → φ[iron] = 3/5, Workshop ticks at 60%. Identical to today's inputAvail; now it no longer oscillates because the Mine's supply no longer flickers.
Worked example — min rule
A building outputs alloy (AT_CAP, small consumer draw) and slag (BUFFERED). g = θ[alloy]: slag production slows with the alloy throttle. Nothing overflows, nothing vanishes — a full byproduct bin still chokes the building, but continuously instead of binary.
Worked example — deactivating constraint
Building B outputs r1 (AT_CAP, demand 2/s of B's 10/s) and r2 (AT_CAP, zero demand). Min rule gives g = θ[r2] = 0 … but then r1's supply is 0 < 2/s demand, r1's bin drains — its cap constraint deactivates. The active-set solve settles on g = 0 (r2 still binds). Conversely if r2 had a 5/s consumer, g = min(2/10, 5/10) = 0.2, r1 pinned, r2 drains below cap and deactivates: g relaxes to… still 0.2 (r1 binds). The solver, not segment-by-segment luck, picks the consistent answer.

Rare-find trickles stay outside the solver's scope — clamp-pinned by applyRates as today (cap overflow of a bonus trickle vanishes; unchanged behavior).

Furnace coal burn is solver-visible on the cap side (owner decision 2026-06-10): the §5.2 burn (coalPerCycle × servedCount / 30s, folded into net.coal post-recipe) appends one synthetic consumer entry { consumes: { coal: burnPerSec } } per billing furnace, so a coal producer at a pinned coal bin throttles to recipe-draw + burn — without it the cap flicker loop returns for coal specifically. The synthetic entries are skipped when coal is zero-constrained: the existing binary fuel-starvation recompute (Fix 4.1) owns that regime per §5.2's all-or-none heat gate; proportional fuel/heat is explicitly deferred (see §10).

Integration into computeRates

The four-pass shape survives; pass 2 becomes the solver call.

PassTodayAfter
1 — tentativeBase rates, gates, per-building mulsUnchanged; also assembles solver coefficients.
2 — availabilityBinary outputAvail + one-shot inputAvailReplaced by solveFlow(...) → g[i]. outputAvail/inputAvail retire from the rate path (kept only if a helper still needs the regime test).

4 — effectivebaseRate × ia × pf × accel × variance × perBuildingMulbaseRate × g[i] × pf × accel × variance × perBuildingMul; BuildingRate gains a utilization field (see §05) so HUD/inspector/advanceIsland read the engine's own number.

Documented approximation (unchanged from today): powerFactor is computed after the flow solve, so a brownout that reduces a consumer's draw does not re-throttle its suppliers within the same segment. The applyRates clamps pin the bin and the next segment's solve picks it up — the same one-segment lag the engine already accepts for power. Making W a solver commodity is out of scope (§10).

05

Maintenance wear, XP, events

  • Wear ∝ utilization. accrueOperatingTime(b, dtMs × u). Definition (by exclusion, so it stays precise wherever individual factors are folded): u is the product of every dynamic gate composed into the building's effectiveRate — solver gate g[i], powerFactor for consumers, heat throttle, adjacency-gate multipliers — excluding the maintenance factor itself (a degraded building must not wear slower — that feedback would flatten the §4.7 ramp into an asymptote) and excludes time-acceleration and variance (both unchanged from today's accrual). The pass-3 comment "a stalled building still wears down at full rate; the player can't escape maintenance pressure by capping output" (economy.ts:1270) is explicitly superseded by owner decision — idle buildings now wear 0.
  • XP needs no new code. XP accrues from pass-4 production (accrueXp), which now carries the throttled rate: a 0.6-utilized Mine earns 0.6× XP. No XP-at-cap exploit exists because nothing is overproduced.
  • Events. Rates remain piecewise-constant between events, so findNextCapEvent keeps its contract. Pinned bins have net exactly 0 — no event source. One adjustment: the §4.7 maintenance-boundary candidate becomes tMs + (boundary − operating) / u per building (skip when u = 0), since wear now ticks at u-speed. u is constant within a segment, so the mapping is exact.
  • Persistence: no schema bump. operatingMs keeps its meaning (accumulated wear-ms); it just accrues slower. Old saves load unchanged — v22 stays.

08

Verification

Solver unit tests — §03's four worked examples, 3-stage chain with middle bin capped, A↔B cycle, deactivating constraint, randomized constraint-satisfaction property test.
Steady state at cap — mine 5/s + workshop 3/s, bin at cap: across many segments, mine utilization stays 0.6 (no oscillation), bin pinned exactly at cap, XP accrues at 0.6×, power draw at 0.6×.
Event-count regression — same scenario: a 1-hour advance produces O(1) segments, not O(cap-hits). Assert segment count via the existing safety counter or a test hook.
Offline ≡ online invariant — 24 h catchup in one advanceIsland call ends bit-close to the same state as 24 h of small ticks (existing invariant, re-asserted under net-flow).
Wear proportionality — building at u=0.5 for 2 h accrues 1 h operatingMs; u=0 accrues none; maintenance boundary event lands at the stretched time; degraded building (maintenanceFactor 0.75) still wears at duty-cycle speed, not 0.75×.
Mass balance — mass-balance.test.ts passes unmodified.
Full suite + build — npm test green, npm run build clean under strict tsconfig.
Visual smoke-test — build, reload tab, screenshot via Daedalus: inspector rates on a capped chain hold steady (no +0/+0.01 flicker).

10

Out of scope

Power as a solver commodity
Coupling the W balance into the flow solve would close the brownout lag but turns the solver into a genuine LP. Deferred; revisit only if the one-segment lag is ever player-visible.
UI exposure of utilization
BuildingRate.utilization makes a "running at 60%" badge trivial for inspector/HUD, but UI work is a separate change after the engine lands.
Proportional fuel/heat (continuous furnace starvation)
A partially-fueled furnace serving partial heat would remove the binary starve/unstarve oscillation at zero coal stock, but rewrites §5.2's locked all-or-none heat semantics and pulls heat.ts into scope. Cap-side burn demand ships now; this follow-up is deliberate.
Maintenance threshold retuning
Wear slows globally; whether 12 h/16 h/… thresholds should tighten in compensation is a balance question for a later pass with play data.
