Design spec · robot-islands-cluster · §4.5

Cluster bonus — rework of the category-adjacency buff

Context Today the §4.5 buff multiplies a building's recipe rate by 1 + n × 0.10, where n is that building's count of distinct same-category 4-neighbours — a positional reward that small islands struggle to earn. This spec replaces it with a cluster bonus: every building in a connected same-category cluster gets the same 1 + (k − 1) × 0.10, scaled by the whole cluster's size k. Branch feat/cluster-bonus, approved in brainstorming, pending implementation plan.

Bonus follows the whole connected same-category cluster, uniformly, holes ignored.
A cluster is a maximal set of same-category buildings joined by 4-adjacency (the existing touchesBorder test). Every member gets the identical multiplier 1 + (k − 1) × 0.10. This intentionally and substantially shifts layout strategy — the point is to let an undeveloped island consolidate a few same-category buildings into one block and reach the full multiplier instead of being capped by short rows and edge positions.
"No gaps" = connectivity only (R1). An enclosed empty tile does not break a cluster; a ring of 8 around a hole is one valid 8-cluster. The function is renamed categoryAdjacencyMul → clusterBonusMul; the CATEGORY_ADJACENCY_RATE constant keeps its name and 0.10 value.
01

Problem

Why the positional formula penalises the islands that most need help.

The count in 1 + n × 0.10 is local to each building's own footprint border, so the bonus is positional: an interior building beats an edge building, which beats a corner. With only a handful of same-category buildings, you can rarely give any one of them more than one or two neighbours, so the multiplier stays near ×1.1 even when every building you own is part of one tight block. The reward for "I packed my whole economy into one cluster" is diluted by the geometry of which building happens to sit where.

02

Before vs. after

The mechanic stops being positional and becomes a uniform per-cluster property.

Before — per-neighbour count
  • 1 + n × 0.10, n = focal building's own same-category 4-neighbours.
  • Positional: centre of a block beats its edges and corners.
  • Small islands cap out near ×1.1 — short rows can't earn neighbours.
  • Resolver: categoryAdjacencyMul.
After — per-cluster size
  • 1 + (k − 1) × 0.10, k = size of the whole same-category cluster.
  • Uniform: every member of a cluster gets the identical multiplier.
  • One consolidated block earns the full multiplier regardless of shape.
  • Resolver renamed: clusterBonusMul (+ batch clusterBonusMuls).
03

Cluster definition & formula

A cluster is a maximal same-category, 4-connected component. Connectivity is the only requirement.

mul = 1 + (k − 1) × CATEGORY_ADJACENCY_RATE[category]      // rate = 0.10
k    = size of the focal building's same-category 4-connected cluster
k = 1 (isolated) → ×1.0     ·     rate-0 category → ×1.0 (short-circuit kept)

Two buildings link iff one's footprint border tile overlaps the other's footprint (the §4.4 4-neighbor test). Diagonal-only contact does not link. A multi-tile building is one node, counted once. Connectivity is within a single category — a different-category building between two same-category buildings does not bridge them. Enclosed empty tiles are irrelevant (R1).

08

Out of scope

Considered and deliberately deferred.

Solid-fill / hole-penalty rule (R2)
Rejected in brainstorming in favour of R1 — connectivity only. Enclosed holes never affect the bonus.
Gating adjacency & exotic-pair skill bonus
checkGates (heat, cooling) and computeBuffStack's exotic-pair term keep direct-neighbour semantics — untouched.
Cross-island lattice clusters
Clusters remain per-island physical components, exactly as the category term is today.
Rate tuning
Rate stays 0.10; CATEGORY_ADJACENCY_RATE's shape and values are unchanged. Re-balancing is a separate concern.
