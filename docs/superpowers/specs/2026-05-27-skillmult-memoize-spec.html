Memoize effectiveSkillMultipliers per advanceIsland tick

Context A live save replayed through scripts/profile-economy.ts shows effectiveSkillMultipliers consuming 72.8% of CPU inclusive inside advanceIsland. It is recomputed from scratch on every call to cap(), which fires once per resource per pass per segment. The set of unlocked skill nodes is constant within one advanceIsland call, so the multiplier object is constant in that scope. Memoize it once, thread it through the inner loop, and the home-island per-frame budget drops from 6.6 ms mean / 21 ms p99 back under 60 FPS without touching any gameplay surface.

Decision
Cache the base SkillMultipliers once per advanceIsland; thread an optional mult arg through cap().
Option A — RatesContext threading. Compute baseMult = effectiveSkillMultipliers(state) at the top of advanceIsland. Add baseMult?: SkillMultipliers to RatesContext. Extend cap(state, r, override, opts, mult?) with an optional 5th arg. Hot-path callers inside economy.ts opt in by passing ctx?.baseMult; UI / orbital / drones / routes callers keep the legacy 4-arg call and pay the per-call cost (acceptable — they're outside the hot path).
Why not Option B (version cache). layerConditionalBonuses at economy.ts:588-613 mutates the returned SkillMultipliers in place. A version-keyed cache that returns the same object reference would be poisoned by the first computeRates call (which layers conditional bonuses), and the next cap() would read a polluted base. Fixing that requires either defensive deep-copy on every cache hit (defeating the speedup) or a wider rewrite of the conditional-bonus layering contract. Option A sidesteps the entire landmine by giving the hot path a separate, never-mutated baseMult handle.
Reservation. The picker in §10 is the final call. Option B's broader call-site coverage is a real upside; if the user is willing to accept the conditional-bonus copy-on-write refactor, B becomes viable.

Root cause

Walk the call chain to see why a single advanceIsland call computes the same multiplier object dozens of times.

StepCall siteCalls per advanceIslandVerified
1. advanceIsland loop economy.ts:1607 S segments (1 in a typical frame; up to ~hundreds in a 24h catchup) v
2. computeRates (per segment) economy.ts:1619 1 per segment — itself calls effectiveSkillMultipliers once at line 733 v
3. outputAvail (per building) economy.ts:446 ~B buildings × ~K outputs/recipe per segment → each output calls cap() v
4. findNextCapEvent (per resource) economy.ts:1323 ~R resources with non-zero net rate per segment → each calls cap(); also separately effectiveSkillMultipliers at line 1351 for the maintenance-threshold ladder v
5. applyRates (per resource) economy.ts:1387 ~R per segment → each calls cap() v
6. cap() body economy.ts:381 1 effectiveSkillMultipliers per cap() invocation. v

For the dumped save's home island (B = 59, R non-zero rates ~50-80 mid-tick), each segment runs cap() on the order of 200–400 times. effectiveSkillMultipliers rebuilds the adjacency graph (buildAdjacency on graph.edges — the default catalog has 100+ edges) and re-folds every unlocked node every time. The work is invariant in the scope of a single segment, let alone a single tick: state.unlockedNodes and state.unlockedEdges are not mutated by computeRates, applyRates, accrueXp, or findNextCapEvent. v

Invariant
Within one advanceIsland call, the base SkillMultipliers object is constant. Level-ups happen via levelUpIfReady → they increment state.level and state.unspentSkillPoints, but they do not mutate unlockedNodes / unlockedEdges. The player can only spend points via buyNode / keystone activations, which happen on UI actions outside the tick loop. v skilltree.ts:760,1363,1375,1399 — the only mutation sites are user-driven.
03

Proposed solutions

Two viable shapes. They differ in where the cache lives and which callers benefit.

Option A — RatesContext threading RECOMMENDED

Shape. Compute the base multiplier once at the top of advanceIsland, stash on RatesContext, pass through to cap().

  • Data-model: RatesContext.baseMult?: SkillMultipliers (read-only, never mutated).
  • Call-site changes: 5 inner-loop sites in economy.ts (lines 400, 446, 1105, 1323, 1387) opt in.
  • Mutation surface to track: none. Cache lives in a stack-allocated ctx object.
  • UI / orbital / routes / drones callers untouched — they pay the per-call cost (~24 non-test sites total) but are well outside the hot path.
  • Conditional-bonus mutation is naturally handled: computeRates at line 733 still does effectiveSkillMultipliers(state) itself, then layers; the per-segment local skillMul stays distinct from the per-tick baseMult threaded into cap().
  • Risk of stale cache: zero. The cache cannot outlive one tick.
  • Test impact: 1 new memoization-correctness test; existing 2197 tests pass unchanged.
Option B — Version-keyed cache on state

Shape. Add state._skillMulVersion: number and state._skillMulCache: SkillMultipliers | null. Bump the version on every unlockedNodes / unlockedEdges / socketBindings mutation. effectiveSkillMultipliers returns the cache when its cached version matches.

  • Data-model: 2 new private fields on IslandState; persistence migration to v14 to ignore them (they're cache, not state).
  • Call-site changes: zero. Every existing call benefits, including UI / orbital / routes / drones.
  • Mutation surface to track: every unlockedNodes.add, unlockedNodes.delete, unlockedEdges.add, unlockedEdges.delete, socketBindings.set, socketBindings.delete, and state.unlockedNodes.clear() in tier-reset.ts. Enumerated in §4.
  • Conditional-bonus mutation landmine. layerConditionalBonuses at economy.ts:588 mutates the returned SkillMultipliers in place — the comment at line 586 documents this explicitly. A cache that returns the same object reference would be poisoned on the first call; the next cap() would read polluted storageCap / storageCategoryCap values. Fix paths: (a) defensive deep-copy on every cache hit (defeats the speedup), or (b) rewrite layerConditionalBonuses to copy-then-mutate (wider refactor; touches 2 call sites at economy.ts:733, 1644).
  • Risk of stale cache: high. One missed mutation site corrupts the tick.
  • Test impact: 1 invalidation-correctness test + 1 cache-coherence test per mutation site (~7 tests). Existing tests need audit for hidden mutations.
Critical — option B has a concrete correctness bug, not just a hypothetical one
layerConditionalBonuses mutates its mul argument in place (economy.ts:601-610). Today, cap()'s call chain returns a fresh object every time, so it always sees the unlayered base. If Option B returns the cached object, the very first computeRates call (which layers conditional bonuses) leaves the cache with storageCap permanently multiplied by the conditional-bonus factor. Subsequent cap() calls inside findNextCapEvent and applyRates would compute caps using the inflated value. This is not a version-tracking issue; bumping the version doesn't help because nothing mutates unlockedNodes between the layering and the next read.

Recommendation: Option A

Option A is recommended on two grounds, in order of weight:

  1. Correctness. Option A's baseMult on RatesContext is read-only by construction: the only consumer is cap(), which only reads storageCap and storageCategoryCap. The per-segment computeRates at economy.ts:733 still builds its own local skillMul for layering, so the per-tick cache and the per-segment layered copy never touch.
  2. Simpler audit. Option A adds one field to RatesContext; the diff fits on one screen. Option B adds two state fields, bumps the persistence schema, demands a complete enumeration of mutation sites (§4 below), and forces a separate refactor of layerConditionalBonuses.

Option B's upside — UI / orbital / routes / drones callers also benefit — is real but immaterial. Those sites fire on player input or per-frame UI render, not per cap() invocation; the profile shows they contribute negligibly to wall-clock time.

05

API change (for Option A)

The cap() signature gains a fifth positional argument. Behaviour is back-compat by construction: when mult is undefined, the function falls back to recomputing.

// Before (economy.ts:373)
export function cap(
  state: IslandState,
  r: ResourceId,
  override?: Record<ResourceId, number>,
  opts?: { ignoreGrace?: boolean },
): number;

// After
export function cap(
  state: IslandState,
  r: ResourceId,
  override?: Record<ResourceId, number>,
  opts?: { ignoreGrace?: boolean },
  mult?: SkillMultipliers,   // NEW — pre-computed base mult
): number;

Body change. The single line const mult = effectiveSkillMultipliers(state); at economy.ts:381 becomes const mult = passedMult ?? effectiveSkillMultipliers(state);. Nothing else inside cap() changes.

RatesContext extension

// economy.ts:108
export interface RatesContext {
  // ... existing fields ...

  /** Pre-computed base SkillMultipliers for this advanceIsland call. Read-only.
   *  When present, cap() reads from this instead of recomputing on every
   *  invocation. Populated by advanceIsland; consumers must NOT layer
   *  conditional bonuses onto this object — that's a separate per-segment
   *  local in computeRates (line 733). */
  readonly baseMult?: SkillMultipliers;
}

06

Cross-island Lattice non-interference

§9.6 Network Consciousness and §13.3 Omniscient Lattice introduce cross-island effects. Per-island memoization must not conflate these.

effectiveSkillMultipliers(state, graph) is keyed strictly on this island's unlockedNodes and the (global) skill graph. The §9.6 NC global production buff is applied separately at economy.ts:686 via ctx.ncBuff, and the §13.3 Lattice inventory / cap / cross-island-neighbor overrides come in via ctx.inventory, ctx.caps, and ctx.crossIsland. None of these feed effectiveSkillMultipliers. v

network-consciousness.ts reads only world.islands + state.level to derive the milestone; it never touches unlockedNodes. v network-consciousness.ts:1-40

Conclusion: per-island memoization is safe regardless of NC or Lattice activation. Each island's advanceIsland call computes its own baseMult at the top; islands do not share baseMult objects.

Test plan

Existing test surface + targeted additions for the invalidation guarantee.

Existing tests that must continue passing

  • src/economy.test.ts — 700+ tests. Particularly the cap/storage path: v sites at lines 3003, 3029 use batteryCapacityWs(state, effectiveSkillMultipliers(state)) directly; these don't pass through RatesContext and are unaffected.
  • src/skilltree.test.ts — the multiplier-folding identity tests must round-trip unchanged.
  • src/skilltree-integration.test.ts — end-to-end skill purchase + economy advancement.
  • src/battery-ladder.test.ts — line 84 reads effectiveSkillMultipliers(state) directly; unchanged.
  • src/persistence.test.ts — no schema change, no migration impact.

New tests

  1. Memoization correctness. Construct an IslandState with a non-trivial unlocked set. Call advanceIsland once; assert the storage cap reflects the skill multiplier. Mutate state.unlockedNodes.add(storage-mul-node). Call advanceIsland again; assert the cap now reflects the new multiplier (no stale cache). This is the load-bearing invariant test.
  2. RatesContext threading. Construct an IslandState. Patch effectiveSkillMultipliers with a call-count spy. Run one advanceIsland tick across N segments. Assert call count equals 2 + maintenance-threshold reads — the per-tick base + per-segment local-with-bonuses + the line 1351 maintenance-ladder read; no per-cap() calls. (Spy needs vi.spyOn on the module.)
  3. Conditional-bonus non-pollution. Construct a state with a conditional-bonus node that multiplies storageCap. Tick once. Assert cap(state, r) from outside the tick (UI-style call) returns the unlayered base value, not the conditional-bonus-multiplied value. Guards against the Option-B failure mode if the implementation ever drifts back toward shared mutable mult.

Perf-regression gate

In scope (Phase 5, optional). Add a perf smoke test that loads a synthetic fixture (the same shape as the live dump — 50+ buildings, 200+ inventory keys, L25) and asserts advanceIsland for one frame completes in <5 ms wall-clock on the CI box. The threshold is generous (~3× the projected post-fix mean) so it trips on regressions but not on transient noise.

Sensitivity. The win is proportional to cap() call count per segment. For pure storage-cap-bound islands (lots of resources at cap, lots of outputAvail / findNextCapEvent early-exits per segment), the win is even larger. For thin islands (few buildings, few resources), the win is smaller in absolute ms but the GC churn reduction is the same %.

What the projection assumes. Each effectiveSkillMultipliers call costs ~1 ms on the home island (the cpuprofile suggests this from the 72.8% inclusive / ~50–100 calls/segment math); replacing dozens of these with a single call saves the bulk. The residual ~10% covers the one call we still make at the top of advanceIsland plus the per-segment layered copy in computeRates.

09

Non-goals

Optimising buildAdjacency itself
35% of self-time today. Likely follow-up — cache the adjacency map on the static graph once at module-init, not per call. Out of scope here because once memoization lands, buildAdjacency only fires once per advanceIsland instead of dozens of times, demoting it from "top hotspot" to "occasional cost". Re-profile before deciding it still matters.
Reducing allocation churn beyond the memoization
blankMultipliers allocates a fresh object on every call (plus 100+ keys via Object.fromEntries). With memoization it allocates once per tick; the surplus reduction is what brings GC from 10.7% to ~3%. Going below 3% would require an object-pool refactor — not worth the complexity.
Optimising rendering / PixiJS
The profile is worker-thread economy only. Rendering hotspots are a separate investigation.
Option-B-style cache for UI / orbital / routes / drones call sites
~16 cold-path sites. They fire at human cadence; per-call cost of ~1 ms is invisible. Could be addressed in a follow-up if a different profile (e.g. opening the inspector at 60Hz) ever surfaces them — but only after the layerConditionalBonuses mutation contract is rewritten.
