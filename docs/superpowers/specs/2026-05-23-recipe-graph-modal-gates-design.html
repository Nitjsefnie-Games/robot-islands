Recipe graph: convert from full-screen modal to a draggable floating panel, plus a new GATES column

Context v1 of this spec read the brief inverted — ignore it. The current recipe-graph window is a full-screen modal with a dark scrim that blanks the rest of the HUD; that is the problem, not the goal. The user wants it to behave like the Drone Ops panel: a draggable, resizable floating panel that sits alongside the rest of the HUD, no scrim, no centred-card sizing. This benefits directly from the draggable-windows feature shipped earlier in this session (commits 28fe30d through 0bb1a4a — src/window-manager.ts + mountPanel now wires makePanelDraggable for non-chrome panels, with localStorage persistence at ri-ui-layout-v1). Task (2) — the new GATES column — is unaffected by the inversion and its pickers carry over wholesale.

As a panel rather than a centred 960px modal, the recipe window needs a default footprint. mountPanel accepts minWidth / maxWidth; makePanelDraggable defaults minWidth: 200 and minHeight: 120 but is happy with anything passed in. After the first drag/resize the saved layout overrides (ri-ui-layout-v1). Heads up: if you also pick GATES placement = "2nd column" in Picker 05 the row becomes 5 columns wide; the narrow default below gets tight.

Picker 03
Close-button semantics (header X)
The current modal X dismisses fully via handle.hide() → scrim.style.display = 'none'. For a panel via mountPanel the equivalent is panelHandle.setVisible(false), which hides the element and removes it from the zone stack (siblings re-flow up). Pickers differ in what "close" means for a free (dragged) panel.
Current: setvisible_false

Today's window — what the inversion replaces

All references are to commit 267fa62.

The window is mounted at app boot from src/main.ts:1000-1003:

1000const graphUi = mountGraphUi(document.body);
1001defineAction(reg, 'toggle-graph', () => {
1002  graphUi.toggle();
1003});

mountGraphUi currently calls mountModal at graph-ui.ts:163, which produces a centred card over a full-screen scrim:

163const handle: ModalHandle = mountModal(parentEl, {
164  title: 'RECIPE GRAPH',
165  subtitle: '/ §6 + §7',
166  onClose: () => handle.hide(),
167  buildBody(body) {
168    body.style.display = 'flex';
169    body.style.flexDirection = 'column';
170    body.appendChild(container);
171  },
172});

Body content: a sticky <input type="search"> plus per-category section dividers and rows rendered as a 4-column CSS grid (graph-ui.ts:81):

81rowEl.style.gridTemplateColumns = '260px 1fr 1fr 60px';

The four columns today are:

  • BUILDING (260px) — displayName + Tn chip + optional (recipeKey) note.
  • INPUTS (1fr) — "N resource, …" or —.
  • OUTPUTS (1fr) — same shape.
  • CYCLE (60px) — 17s, right-aligned.

Dismiss vectors today: scrim click (ui-modal.ts:97-99), Escape via the shared dismiss-modal action (main.ts:1370-1390), header X (ui-modal.ts:64-69), Y hotkey toggle (input.ts:89). The first two go away post-migration (no scrim; Esc cascade no longer touches this panel per Picker 04 default).

02

Before ↔ after

Two-column compare of the modal shell (today) against the floating-panel shell (post-migration), under the recommended defaults.

Today — modal (the problem)
  • Mounted via mountModal at graph-ui.ts:163.
  • Full-screen scrim with backdrop-filter: blur(2px) (ui.css:534) blanks the rest of the HUD.
  • Centred card min(960px, 92vw), max-height: 86vh (ui.css:544-555) — can't be moved out of the viewport centre.
  • Not draggable, not resizable. Player must dismiss the window to see the world underneath.
  • Joins the dismiss-modal Esc cascade at main.ts:1379.
Post-migration — floating panel
  • Mounted via mountPanel({ id: 'recipe-graph-panel', zone: Zone.R, order: 3 }).
  • No scrim. HUD remains fully interactive while the panel is open.
  • Initial 720×520 footprint (per Picker 02 default); player can drag the header to reposition and the BR grip to resize.
  • Layout persists per-panel via ri-ui-layout-v1 in localStorage (window-manager.ts:18); Reset Window Layout in Settings reverts (settings-ui.ts).
  • Drops out of the Esc cascade. Y still toggles via the input registry.
  • Same family as Drone Ops / Routes / Settlement / Inspector.
03

Change inventory (under the recommended defaults)

Three files touched. Assumes every picker stays at its default.

src/graph-ui.ts
Replace the mountModal body with a panel skeleton + mountPanel. Build a .ri-panel root with a .ri-panel__head (title + X close button) and a .ri-panel__body that wraps the existing container (search + rows). Replace mountModal(parentEl, …) with mountPanel(panel, { id: 'recipe-graph-panel', zone: Zone.R, order: 3, minWidth: 480 }). The close-button click calls panelHandle.setVisible(false). Remove the maxHeight: '70vh' on container — the panel itself is now resizable and the body should fill it. Drop the onClose wiring that pointed at the modal handle.
src/graph-ui.ts (row grid)
Insert GATES as the 2nd column. Change gridTemplateColumns from '260px 1fr 1fr 60px' to '180px 150px 1fr 1fr 60px' (5-column grid). New gatesCell div per row renders row.gates as inline monospace text with status-colour classes (olive=met, clay=pending, gray=N/A). Extend the search haystack at graph-ui.ts:135-144 with gate kind + parameter strings so text search hits gates too.
src/recipe-graph.ts
Add a pure gates field to RecipeTableRow. New type GateEntry = { kind: 'tier'|'t5'|'t6'|'biome'|'tile'|'coastal'|'heat'|'adjacency'; param: string }. buildRecipeTableRows populates gates from def.tier, def.requiredBiomes, def.requiredTile, def.coastal, def.requiresHeat, def.gates. Pure; no DOM; no IslandState dependency (status colour is decided at render time in graph-ui.ts using the active-island getter).
src/main.ts
Pass { getState: activeState } at mount; drop graph from the Esc cascade. Line ~1000: mountGraphUi(document.body, { getState: activeState }). Line ~1379: remove graphUi.hide() from the dismiss-modal action body (the recipe panel is no longer a modal). Keep the toggle-graph action and the Y binding unchanged — graphUi.toggle() now flips setVisible on the panel handle instead of mounting/unmounting a modal.

No CSS changes. The existing .ri-modal-scrim / .ri-modal rules stay in ui.css for the seven sibling modals that still use them — we're just no longer one of them. .ri-panel, .ri-panel__head, .ri-resize, .ri-free are all defined. Persistence and resize wiring come along for free via mountPanel → makePanelDraggable.

Implementation notes the picker structure doesn't surface:

  • The current sticky search input lives at the top of the modal body. Post-migration it can live in the panel header (alongside the X) or stay at the top of the panel body. I'd default to keeping it at the top of the body so the header chrome stays consistent with sibling panels.
  • GraphUi.toggle() already tracks visibility internally; that pattern carries over — PanelHandle doesn't expose isVisible(), so the wrapper stays the same.
  • makePanelDraggable resolves the drag handle via .ri-panel__head (see window-manager.ts:347-352). The new panel skeleton must use exactly that class on the header div.
  • First-mount placement: when ri-ui-layout-v1 has no entry for recipe-graph-panel the panel stacks at Zone.R, order: 3 beneath Settlement. Once dragged it becomes "free" (setPanelFree(true)) and the rect persists.

07

Out of scope / deferred

Things this spec deliberately does not decide.

Mermaid-style visual graph view
The window briefly was a Mermaid-rendered DAG (commit 649a7f3) before being refactored to a searchable table (fde23f4). Reviving the graph view as a tab is out of scope.
Runtime status of tile / coastal / heat / adjacency gates without a placement
Four of the eight gate dimensions are placement-time predicates; they have no live "met / not met" answer for a catalog row. The GATES column renders the requirement only.
Click-to-build / click-to-locate from a row
A natural follow-up: clicking a row could route to the build catalog or to a placed instance. Pickers here are about static display.
Runtime sortable column headers
Picker 10 chooses one default. A full sort-by-any-column UX is a separate interaction surface.
Persisting search / filter text across sessions
Panel position/size persist via ri-ui-layout-v1; search input does not. Adding that is independent of (1) and (2).
Showing skill-tree multiplier effects per row ("MODS" sister column)
Multipliers are per-recipe but not gates — they change rate, not availability. Sister column for a follow-up.
Reset Window Layout interaction
resetUiLayout() (Settings → Reset Window Layout) restores the recipe panel to its default zone slot alongside Drone Ops / Routes / Settlement. Covered by the existing window-manager wiring; no new work here.
