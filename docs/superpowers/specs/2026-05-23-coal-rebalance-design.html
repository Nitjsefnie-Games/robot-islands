Coal loop rebalance — make mine_on_coal + coal_gen self-sustain at 1:1

Context The audit at docs/superpowers/audits/2026-05-23-coal-loop-balance.html showed the coal pipe is broken at every ratio: one mine outputs 1 coal / 17 s = 0.0588 coal/s, one generator burns 1 coal / 2 s = 0.5 coal/s, so coal-balance demands ≥ 8.5 mines per gen — but 8.5 mines at 40 W each cost 340 W to power the 100 W gen, a guaranteed −240 W deficit per generator. This spec mirrors the Heavy Logger / Biomass Plant rebalance (commit 9cda2aa) to land the coal pair on net-positive fuel AND net-positive power at 1:1, then exposes the lever choices as interactive pickers so the user can pick the exact knob set.

Methodology choices to preserve:

  1. Tune the extractor, not the generator. Generator side stays at original spec; only the extractor recipe + power draw move.
  2. Match the generator's burn cadence with extractor cycle math. biomass burns 1 / 2 s = 0.5 / s; logger now produces 9 / 10 s = 0.9 / s. Modest fuel headroom (+0.4) lets the loop power downstream consumers too.

03

Target profile + the algebra behind the default

Match the biomass loop's net shape at 1:1 with modest excess in both pipes, so coal stays an early-tier loop instead of a free-lunch.

Target: at 1 mine + 1 gen, both balances clear zero with modest positive excess. Brief asked for ~+0.3 coal/s and +30 W as a target shape. We adopt slightly more headroom on power side (+60 W) because coal_gen produces 100 W natively vs biomass's 80 W; 9 coal / 10s on the mine matches biomass's fuel shape (+0.4) exactly.

The two-variable system. Let N = mines per generator (1:1 ratio means N = 1). For a single gen + N mines:

coal_net = N · (output_per_cycle / cycleSec_mine) − (1 / cycleSec_gen)
power_net = (1 · produces_gen) − N · consumes_mine

Plugging the default combo (9 / 10s mine, 1/2s gen, 40 W draw, 100 W produces) at N=1:

coal_net = 1 · (9 / 10) − (1 / 2) = 0.9 − 0.5 = +0.4 coal/s
power_net = 100 − 40 = +60 W

Both clear zero with comfortable headroom. The +0.4 coal/s flows downstream into smelter (1 coal / 27 s ≈ 0.037/s per smelter) and workshop (1 coal / 33 s ≈ 0.030/s), so this single pair supports ~6 smelters or ~13 workshops on coal alone — intentional, so the coal loop genuinely powers the iron-bolt chain.

05

Anomalies & caveats

Things that perturb the headline math; flagged so the implementer doesn't trip on them.

Anomaly 1 — skill tree mineYieldBonusMul stacks on coal src/skilltree.ts:521 defines a depth-2 mining node with effect mineYieldBonusMul ("Mine vein depth +10%"). The runtime multiplier (state.skill.mineYieldBonus) applies to every mine recipe rate, not just mine_on_ore — because there's no recipe-id discrimination in the lookup path. A maxed mining sub-path (typically up to ~1.3×) bumps coal/s proportionally. With the default combo, this shifts the 1:1 fuel net from +0.4 to +0.67 coal/s; the picker rationale numbers are base values, pre-skill-tree.
Anomaly 2 — shared mine building def BUILDING_DEFS.mine backs both mine_on_ore and mine_on_coal via runtime recipe resolve (building-defs.ts:600–622). Editing BUILDING_DEFS.mine.power.consumes — i.e. picker 03 — also changes the iron-mine power cost. The default (keep_40W) avoids the spillover. Anything else needs a separate sanity-check pass on iron / steel / bolt chain power budgets.
Anomaly 3 — coal_gen's power model is steady-state, not energy-per-cycle coal_gen.power.produces = 100 means "100 W constant while running", not "100 J per coal". If the recipe is fuel-starved, the actual gen output is the steady-state burn rate × 100. So a naive cycleSec: 2 → 17 rebalance (the seemingly natural way to match mine throughput) drops effective gen output to ~11.76 W (audit §3). The slow_generator picker's rationale assumes both cycleSec AND produces are scaled together to preserve "Joules per coal".
Anomaly 4 — downstream consumers expect +0.4 to flow through Smelter (1 coal / 27 s ≈ 0.037/s) and workshop (1 coal / 33 s ≈ 0.030/s) both burn coal as smelting/manufacturing input (not generator input). At default, one coal pair supports up to ~6 smelters or ~13 workshops plus the generator. This is intentional shape — coal is meant to be both fuel and process-input — but the implementer should know the loop isn't sized for gen-only consumption.
