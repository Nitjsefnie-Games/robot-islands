Design spec · robot-islands · §2.7

Real-sun day-night cycle

Context At 21:43 local time in Brno (the in-game location), the HUD showed Dawn 78% · solar 0.0× — a fake "Dawn" label next to the real sun's zero output. The cause is a second, synthetic clock that drives the labels, tint, weather, and a gameplay gate independently of the player's location, while solar power already follows real astronomy. This spec migrates the behavioral consumers onto the real sun and keeps the synthetic clock only as the no-location fallback.

Decision
Add real-sun functions alongside the synthetic ones; switch every behavioral consumer; keep the synthetic clock as the null-location fallback.
solarMultiplier already computes the true sun at the player's coordinates via SunCalc — power is correct. The fix is to stop the synthetic UTC-quadrant clock from driving the phase label, the screen tint, weather modulation, the during-night gate, and the integrator's segment boundaries.
We do not delete the synthetic helpers: they keep their passing test suite and become the deterministic fallback when no location is set (fixtures / pre-picker frames). The label readout becomes phase + countdown ("Night · sunrise in 7h12m"), not a percentage, because real phases are not equal-length.
01

Root cause

Numeric confirmation that the reported "Dawn 78%" is the synthetic clock, not the real sun.

dayPhase(now) maps Unix-epoch wall-clock time through a fixed EPOCH_PHASE_OFFSET = 0.375 into four equal 6-hour quadrants — with no dependence on the player's location. At 19:43 UTC (= 21:43 CEST in Brno):

phase = (19.717/24 + 0.375) mod 1 = 0.1965   → quadrant 'dawn'
frac  = (0.1965 × 4) mod 1        = 0.786    → "Dawn 78%"  (exactly as reported)
real sun altitude at Brno         = below horizon → solar 0.0×

The label and the solar multiplier disagree because they read different clocks. SPEC §2.7 documents the synthetic model and is now out of date relative to the intended behavior (track the real sun).

02

Two clocks, before & after

What drives each consumer today, and what will drive it after the change.

Before — synthetic UTC quadrants
  • HUD phase label & % from dayPhase/dayPhaseName.
  • Screen tint cross-fade from synthetic phase distance.
  • Weather severe-storm boost keyed to synthetic Night/Dawn.
  • during-night economy gate from synthetic dayPhaseName.
  • Integrator segment boundary from synthetic nextPhaseBoundaryMs.
  • Only solarMultiplier (power) was real.
After — real sun at player coordinates
  • Label name from realPhaseName; countdown from nextSunEvent.
  • Tint base + cross-fade from real altitude near h=0 / h=−6°.
  • Weather boost keyed to real Night/Dawn at the player location.
  • during-night gate from realPhaseName.
  • Integrator boundary from nextRealPhaseBoundaryMs.
  • Synthetic clock retained as null-location fallback only.
03

New pure functions in daynight.ts

All take (nowMs, lat: number | null, lon: number | null), no PixiJS/DOM. Null lat/lon delegates to the synthetic clock.

realPhaseName — altitude-based, never NaN (getPosition always returns a number):

// null lat/lon → dayPhaseName(nowMs)  (synthetic fallback)
h      = SunCalc.getPosition(nowMs, lat, lon).altitude        // radians
rising = SunCalc.getPosition(nowMs + 60_000, lat, lon).altitude > h
CIVIL  = -6° = -0.10472 rad
  h >= 0                 → 'day'
  h >= CIVIL &&  rising  → 'dawn'
  h >= CIVIL && !rising  → 'dusk'
  h <  CIVIL             → 'night'

nextSunEvent → { kind: 'sunrise' | 'sunset'; atMs } | null. Earliest getTimes sunrise/sunset strictly after now over today+tomorrow; its kind is automatically the meaningful one (sun up → next is sunset; down → sunrise). null on null lat/lon or polar (Invalid Date / NaN guarded).

nextRealPhaseBoundaryMs → earliest of the four real transitions (sunrise, sunset, civil dawn, civil dusk) strictly after now. Fallbacks (never NaN): null lat/lon → nextPhaseBoundaryMs(now); polar (no candidate) → now + QUADRANT_MS.

04

Phase thresholds

The altitude bands that define each phase. Geometric horizon for sunrise/sunset; civil twilight (−6°) for the dawn/dusk bands (user-approved).

05

Consumer migration — the 5 sites

Each behavioral consumer switches to a real-sun function with the player's coordinates. nextSolarBoundaryMs (32 fixed segments) is untouched — solar integration is already faithful.

hud.ts:384–387
Label → phase + countdown. realPhaseName for the name, nextSunEvent for "sunrise/sunset in Xh Ym". Null event → omit countdown. Drop the % readout.
daynight-tint.ts:61 (currentTint)
Tint → real phase + altitude fade. Base tint by realPhaseName; cross-fade window expressed in altitude near h=0 / h=−6° instead of synthetic phase distance. lat/lon plumbed from the tint mount in main.ts.
economy.ts:639 (during-night gate)
Real night check. realPhaseName(nowMs, world.playerLat, world.playerLon) === 'night'. world already in scope.
economy.ts:1786 (advanceIsland)
Real integrator boundary. nextRealPhaseBoundaryMs(t + wallOffset, lat, lon) − wallOffset. Keeps the §15.3 piecewise-constant invariant — the now-real night boolean stays constant within each segment. lat/lon reachable (same source computeRates' line-784 solarMultiplier uses).
weather.ts:179 (weather loop)
Real phase modulation. realPhaseName(t, lat, lon) — requires the signature change in §06.
06

weather() signature change

weather() has 38 references across 9 files. Optional trailing params keep every existing caller and test untouched.

New signature: weather(seed, cx, cy, nowMs, biome?, totalCo2Kg = 0, lat = null, lon = null). Inside the loop, realPhaseName(t, lat, lon) with null/null delegates to dayPhaseName(t) — so every caller and test that omits lat/lon keeps its current deterministic behavior (weather.test.ts stays green).

Only the callers that must reflect the real sun and have the player location are updated, so simulation and display agree: economy.ts:634 (during-storm gate), routes.ts:577, hover-tooltip.ts:287/293, weather-overlay.ts:144/159. Re-run findReferences before editing to confirm no positional-arg caller breaks.

Determinism (accepted): re-anchoring to the real sun changes weather history for the real-location path — existing seeds replay differently. Performance: the loop runs up to 1,000,000 iterations calling the phase fn each time; the implementer must measure a representative iteration count first and memoize phase per integration step if 2× SunCalc/iteration is material — decide on the measurement, not assumption.

07

Verification

New tests target the real-sun functions; the synthetic and solarMultiplier suites must stay green unmodified.

realPhaseName, Brno late-May post-dusk instant. Returns 'night' (the bug regression — must NOT be 'dawn').
realPhaseName rising vs falling near horizon. Distinguishes 'dawn' from 'dusk'; equator equinox noon → 'day'.
Null lat/lon identity. realPhaseName/nextRealPhaseBoundaryMs with null equal dayPhaseName/nextPhaseBoundaryMs.
Polar (84°N, Dec 21). realPhaseName never 'day'; nextSunEvent null; nextRealPhaseBoundaryMs finite (≤ now + QUADRANT_MS); no NaN/throw.
weather() real path. Brno night instant boosts severe-storm weight vs day; omitting lat/lon reproduces the synthetic result.
npm test green. Full suite passes; synthetic-clock + solarMultiplier suites unchanged.
Browser smoke. Build + reload islands.nitjsefni.eu; the HUD reads a sane phase + countdown for the current Brno time.

09

Out of scope

Considered and deliberately deferred.

Deleting the synthetic helpers
dayPhase/dayPhaseName/nextPhaseBoundaryMs/EPOCH_PHASE_OFFSET are retained as the null-location fallback (user chose "keep", not "retire").
Per-cell (per-longitude) sun variation
The player has a single global location; weather time-of-day uses that one location, not per-island longitude.
Changing solarMultiplier / nextSolarBoundaryMs
Already correct and location-aware; untouched.
