Battery ladder — T2 / T3 / T5 power-storage progression

Generalise the existing T5 Singularity Battery mechanic into a 3-tier ladder so solar-dependent islands can buffer day surplus into night before reaching T5. SPEC.md §285 already promises this (“plan for night-time stockpile via Battery (T2) or Singularity Battery (T5)”) but only the T5 ships in code. Adds Battery Bank (T2) and Capacitor Bank (T3) as new building-defs.ts rows + a new batteryCapacityMul skill effect.

Gap today: on a solar-only T1/T2 island, any daytime power surplus is wasted — there's nowhere to store it. Night falls and production stops. The player either over-builds solar to push past night-time consumption (wasteful) or stalls. SPEC.md §285 + §433 names a T2 Battery as the intended fix; this spec defines it.
Single mechanic, three buildings. All three battery types contribute additively to one per-island energy buffer (state.batteryStoredWs) with one summed capacity (state.batteryCapacityWs). The existing charge / discharge integration in economy.ts:1610-1693 stays the same — only the capacity computation widens to include the new building types.

01

Generalised buffer mechanic

Today's state shape: state.singularityStoredWs: number, capacity computed from the count of placed Singularity Batteries × SINGULARITY_BATTERY_CAPACITY_WS.

New shape: a single per-island buffer with capacity summed across all three battery building types.

// economy.ts — replace SINGULARITY_BATTERY_CAPACITY_WS with a per-def table
export const BATTERY_CAPACITY_WS: Readonly<Record<BuildingDefId, number>> = {
  battery_bank:         2_000 * 3600,        // 2 kWh — T2 (partial-night buffer)
  capacitor_bank:       60_000 * 3600,       // 60 kWh — T3 (short-night coverage)
  singularity_battery:  50_000_000 * 3600,   // 50 MWh — T5 (unchanged)
};

export function batteryCapacityWs(state: IslandState, mul: SkillMultipliers): number {
  let cap = 0;
  for (const b of state.buildings) {
    if (b.constructionTime !== undefined) continue; // not yet built
    const per = BATTERY_CAPACITY_WS[b.defId] ?? 0;
    if (per > 0) cap += per;
  }
  return cap * mul.batteryCapacity;
}

Rename state.singularityStoredWs → state.batteryStoredWs at the v8 → v9 schema bump (see §05). The charge / discharge integrator at economy.ts:1610-1693 already operates on a single capacity value; replace the in-line capacity calc with a call to batteryCapacityWs(state, mul) and the rest works unchanged.

03

Recipes + crafting

Both new buildings have placement-time costs (paid out of inventory; no crafting chain like the Singularity Battery Factory's singularity_battery_unit intermediate). Rationale: T2/T3 batteries should be quick to deploy. The T5 Singularity Battery's heavy crafted-unit gate stays as-is to preserve its endgame feel.

The Battery component recipe (SPEC §725) is already in recipes.ts — Lithium + Electrolyte + Plastic casing → Battery — and the Battery Factory (T3) already exists in building-defs.ts:1232. Both new battery buildings consume the existing Battery component, so no new recipes are needed; the recipe tree is closed.

04

Skill-tree integration

Add one new effect kind:

// skilltree.ts — extend SkillEffect
| { readonly kind: 'batteryCapacityMul' };

The fold contributes to SkillMultipliers.batteryCapacity (new, default 1). Eligible nodes: at minimum a notable in Power Systems (refinement branch), and arguably a keystone “Off-grid Capable” that gates on owning at least 3 batteries of any tier. Concrete catalog additions:

  • powerSystems.notable.electrochemistry — “Electrochemistry Lab — battery capacity ×1.30”. Stat: batteryCapacityMul magnitude 0.30.
  • powerSystems.notable.fastDischarge — “Fast Discharge — battery discharge rate uncapped during brownout”. Removes the integrator's discharge cap when the buffer is > 50% full. (Engine change in economy.ts:1088-1096.)
  • powerSystems.keystone.offGrid — “Off-grid Capable — battery capacity ×2.00; standby draw ×0.50”. AND-prereq: own at least 3 of battery_bank | capacitor_bank | singularity_battery on this island.

The keystone's AND-prereq introduces a new gate shape (count-of-building rather than count-of-node). Extend KeystonePrereq with an optional requiresBuildingCount field, or model as a synthetic “virtual node” owned-when-3-batteries-placed. Recommended: virtual node, since it keeps canBuyKeystone uniform.

  • SerializedSnapshotV8 alias capturing today's shape.
  • migrateV8toV9(s) → SerializedSnapshotV9: rename singularityStoredWs → batteryStoredWs in each island state (lossless; data identity preserved). No new field added at migration time.

  • Test: v8 fixture round-trips through the chain and produces a v9 with the renamed field.

No effect on existing islands' gameplay — they keep their Singularity Battery buffer; the only change is the field's name + that T2/T3 battery buildings can now contribute.

06

HUD / inspector surfaces

  • Power HUD strip (hud.ts islandBar power section): show stored / capacity as a percentage chip — e.g. BATT 47% in muted green when above 20%, amber 20-10%, red below 10%.
  • Inspector panel on any battery building: detailed view with current Ws, capacity Ws, average charge rate over last 60s, projected hours-to-empty at current draw.
  • Brownout indicator: when discharge is active, flash the BATT chip and add a tooltip “discharging at N W to cover deficit”.
07

Balance + numerics — discussion

Partial-night coverage is the design intent for T2. 2 kWh holds ~1-2 hours of a small base's draw (~1 kW). The player won't bridge the full night with one Battery Bank — that's deliberate. They'll either chain 3-5 banks for fuller coverage, OR accept tactical brownouts during the deepest part of night and rely on the buffer to keep critical paths (Workshop, Logger) ticking. Either play pattern is more interesting than "over-build solar until night doesn't matter".

Why ×30 jump T2 → T3? 60 kWh covers a multi-building T3 base across the bulk of the night even at heavy draw. Players reaching T3 are running smelters / chemistry plants and an arsenal of peripherals; the T3 step needs to feel like a real upgrade, not just "place 30 more battery banks".

Why ×830 jump T3 → T5? T5 Singularity Battery is apex — carries entire industrial loops across multi-hour offline catch-ups. Keep its scale unchanged; the design isn't to nerf endgame storage but to make pre-endgame storage exist.

Crafting cost sketch is placeholder — pin in SPEC.md Appendix A when the §14 tuning pass lands (same convention as the existing T5 Singularity Battery cost).

08

Out of scope

  • Energy trade routes between islands. Batteries are strictly local — no inter-island power transfer. (Spec §13.3 covers shared-inventory across networked islands but excludes power buffering, intentionally.)
  • Custom charge / discharge curves. All battery types charge and discharge at the same per-Watt rate; differentiating efficiency by type would be a separate balance pass.
  • Aging / wear mechanics. Batteries don't degrade with cycle count. Add only if maintenance-system playtests show low-engagement maintenance loops on power buildings (separate spec).
  • Auto-build prioritisation. Players manually place batteries; no “auto-add battery when projected outage detected” helper.
  • Mirror-Sat interaction. The §14.3 Mirror Sat bonus already mitigates the night problem at T6+. Battery ladder is the pre-T6 fix; no Mirror-Sat integration changes here.
