Design spec · robot-islands · skilltree computed per-tier magnitudes · 2026-05-26

Skill-tree per-tier magnitudes via runtime derivation

Context The 2026-05-25 skill-tree rebalance shipped with uniform per-effect-kind magnitudes: every keystone, notable, and filler of e.g. routeCapacityMul carries the same magnitude: 0.1103. The caps land correctly (×10 / √10 / ×3 hold across all 37 effect kinds) but the tier hierarchy collapsed — a keystone feels indistinguishable from a depth-1 filler, and every depth in a filler chain has identical magnitude. Plus the catalog now carries ~280 hardcoded magnitude: literals + 36 archetype baseMag values that must be re-derived by hand whenever a node is added or removed. This spec replaces hardcoded magnitudes with runtime-derived per-tier magnitudes using a 50/30/20 keystone/notable/filler weight split + 1.10 per-depth growth within filler chains. Magnitudes become a pure function of (tier, effect-kind catalog counts, pool cap, filler depth) — no authored numbers per node.

Decision
Compute per-node magnitudes at module load from (tier, effect-kind counts, pool cap, filler depth). Drop the §03 hardcoded magnitude table entirely.
Tier inference: NodeId containing .keystone. → keystone. NodeId matching an archetype's idPrefix.<depth> → filler. Everything else hand-curated → notable. Optional explicit tier?: NodeTier override on a RawSkillNode for ambiguous cases.
Weight split: 50% of the log-budget goes to keystones (uniform within the tier), 30% to notables, 20% to fillers. If a tier has zero nodes for a given effect kind, its weight is redistributed proportionally to the nonzero tiers (e.g. recipeRateMul:manufacturing with 0K/1N/8F: the 50% keystone weight splits 60/40 between notable + filler → notable gets 30+30=60%, filler gets 20+20=40%, lone notable lands at ×3.98).
Filler chain: within an archetype with count nodes and growth g = 1.10, bisection-solve baseMag such that ∏ (1 + baseMag · g^i) for i = 0 .. count-1 equals the filler tier's allocated log-budget. Result: depth-1 starts small, deeper depths progressively stronger by 10% per step.
Cap preservation: the product across all nodes of an effect kind equals the pool's cap target exactly (×10 singleton, √10 shared-pool side, ×3 xpGain). Same cap math as the 2026-05-25 rebalance — only the per-node distribution changes. Lone-keystone or lone-notable effect kinds get pivotal big nodes (e.g. debrisProtectionMul keystone ×4.22 — accepted as desirable concentration of identity).
Authored numbers per node: zero. Catalog rows drop the magnitude field. Archetypes drop baseMag (keep growth, count, baseCost, costGrowth). Adding a new node auto-rebalances every other node of the same effect kind. Drift is structurally impossible.
01

Background — why uniform magnitudes feel wrong

Verified by inspection of the live catalog post-rebalance (commit chain through 5c738df).

Symptom: a player who buys a routeCapacityMul keystone at SP cost 5 sees ×1.110 to route capacity, identical to buying a depth-1 filler at SP cost 1. The keystone label and the high cost feel undercut by the unchanged magnitude. Same shape across 37 effect kinds.

Within a filler chain: the rebalance set growth: 1.0 (flat) so depth-1, depth-4, and depth-8 of e.g. mining.recipeRate all grant the same ×1.0248. A player has no progression-feel within a chain — every node is interchangeable.

Source-of-truth fragility: the 2026-05-25 plan hand-shipped 280+ magnitude: literals into skilltree-catalog.ts plus 36 archetype baseMag values. Any new node breaks the cap unless the engineer re-runs scripts/skilltree-magnitudes.ts and redistributes magnitudes by hand. The vitest invariant catches drift after the fact but can't prevent it.

02

Tier model

A three-tier classification that the runtime derives from each node's ID.

export type NodeTier = 'keystone' | 'notable' | 'filler';

Inference rules (applied in order, first match wins):

RuleTestTier
Explicit overridenode has a tier field setthat value
KeystoneNodeId contains the substring .keystone.'keystone'
FillerNodeId matches <archetype.idPrefix>.<number> for some archetype'filler'
Notable(fallthrough)'notable'

Why this works: existing hand-curated keystones already live in ROLE_ABSORPTION_KEYSTONES and RULE_BREAKER_KEYSTONES arrays with ids like 'smelting.keystone.foundryMastery'. Existing filler archetypes generate ids like 'mining.recipeRate.3'. The remaining hand-curated rows (notables) have descriptive non-keystone ids like 'deepVeinSurveying'. No id renaming required.

03

The formula

Per-effect-kind magnitude derivation. Same pool cap target as the 2026-05-25 rebalance.

Inputs:

  • C = pool cap target (×10 singleton, √10 shared-pool side, ×3 xpGain).
  • (K, N, F) = count of keystone / notable / filler nodes carrying this effect kind.
  • (w_K, w_N, w_F) = (0.50, 0.30, 0.20) — base tier weights, must sum to 1.
  • g = 1.10 — per-depth growth within a filler archetype chain.

Redistribution for empty tiers: for any tier with zero nodes, drop its weight to 0 and renormalize the remaining weights to sum to 1. Example: (K=0, N=2, F=14) → effective (w_K', w_N', w_F') = (0, 0.60, 0.40).

Per-node magnitudes:

m_keystone = C^(w_K' / K) − 1     // uniform within keystone tier
m_notable  = C^(w_N' / N) − 1     // uniform within notable tier
m_filler(depth) = baseMag · g^(depth − 1)
  where baseMag is found by bisection such that
  ∏ (1 + baseMag · g^i) for i in [0, F-1] = C^w_F'

Invariant: the product across all bought nodes equals C exactly. By construction:

(1+m_K)^K · (1+m_N)^N · ∏(1+m_filler(d)) for d in [1, F]
  = C^w_K' · C^w_N' · C^w_F'
  = C^(w_K' + w_N' + w_F')
  = C^1
  = C

04

Engine integration

A single new module exporting one pure function. No engine surface changes elsewhere.

New module: src/skilltree-derive-magnitudes.ts. Exports:

export type NodeTier = 'keystone' | 'notable' | 'filler';

export interface RawSkillNode extends Omit<SkillNode, 'magnitude'> {
  readonly tier?: NodeTier;
}

export const POOL_TARGETS: Readonly<Record<string, number>> = { /* moved from scripts/skilltree-magnitudes.ts */ };

export const TIER_WEIGHTS = { keystone: 0.50, notable: 0.30, filler: 0.20 } as const;
export const FILLER_GROWTH = 1.10 as const;

export function deriveMagnitudes(
  rawNodes: ReadonlyArray<RawSkillNode>,
  archetypeIdPrefixes: ReadonlyArray<string>,
): ReadonlyArray<SkillNode> { /* ... */ }

Algorithm (one pass):

  1. Bucket each rawNode into its NodeTier via the inference rules in §02. Group by effectKey(node.effect) where effectKey applies the same composition used by scripts/skilltree-magnitudes.ts (recipeRateMul:<category>, etc.).
  2. For each effect-kind bucket: count K, N, F. Look up C = POOL_TARGETS[key]. For effects with no pool target (e.g. placeholder, structural, conditionalBonus), magnitudes stay 0 and are not derived.
  3. Redistribute weights for empty tiers per §03.
  4. Compute m_K and m_N from closed form. Bisection-solve baseMag for the filler chain (target C^w_F', 80 iterations is sub-1ms and yields ~1e-24 precision).
  5. For filler nodes: parse the node's depth from its ID suffix (last numeric component) and apply baseMag · g^(depth − 1).
  6. Emit each rawNode with its computed magnitude attached.

Wiring: in src/skilltree-catalog.ts, replace the current FULL_CATALOG aggregation with

export const FULL_CATALOG = deriveMagnitudes(
  RAW_FULL_CATALOG,
  ALL_ARCHETYPE_PREFIXES,
);

where RAW_FULL_CATALOG assembles the same nodes the catalog currently aggregates, but built from sources that emit RawSkillNode (no magnitude). generateFillerNodes returns RawSkillNode[].

Performance: ~550 nodes, 37 effect-key buckets, ~36 bisections (80 iters each). Total work ≈ a few thousand multiplications. Sub-10ms at module-load. Production traffic is unaffected — derived FULL_CATALOG is a frozen const.

05

Catalog + archetype source shape changes

Hardcoded numbers come out; structural metadata stays.

Hand-curated catalog rows (src/skilltree-catalog.ts): drop the magnitude field from every SkillNode literal. Type becomes RawSkillNode. Where the tier inference would mis-bucket a row, add an explicit tier: 'notable' or tier: 'keystone' — but in practice no overrides are needed because the existing id conventions already disambiguate.

Filler archetypes (src/skilltree-archetypes.ts): drop the baseMag field from FillerArchetype. Keep idPrefix, effectKind, effectExtra, subPath, growth, count, baseCost, costGrowth. The interface comment is updated to reflect that baseMag is no longer authored — it's solved at deriveMagnitudes time. Default growth: 1.10 across all archetypes (the 2026-05-25 rebalance had set this to 1.0; reverting to 1.10 per the design).

Description strings: existing rows say e.g. 'Deep Vein Surveying — ×1.0625 mine yield per Mine' with the literal multiplier embedded. Under the computed model the multiplier may shift any time a node is added elsewhere in the same effect kind. Decision: strip the literal multiplier from descriptions; leave pure flavor text (e.g. 'Deep Vein Surveying — high-yield seam analysis'). The UI tooltip layer derives the current ×factor at render time and appends it. Out of scope for v1: implementing the UI render-time formatting — if the existing tooltip surface already shows magnitude, that work continues to function; otherwise the description-without-number ships and a follow-up wires the runtime ×factor into the tooltip.

What stays authored per node: id, subPath, depth, cost, effect (the kind + extra), description, optional tier. Magnitude is the only field that becomes derived.

06

Tests

New unit tests for deriveMagnitudes: given a synthetic raw catalog with known (K, N, F) counts per effect kind, assert per-tier per-node magnitudes match the closed-form (for keystone + notable) and the bisection result (for filler depth 1 and last). Tolerance ±1e-6.
Invariant test stays: src/skilltree-magnitudes.test.ts already asserts per-effect-kind product ≈ cap target. Under the computed model this is structurally guaranteed but the test remains as a regression sentinel — it catches a refactor that breaks the invariant by accident.
Existing tests that hardcode specific magnitudes (e.g. skilltree.test.ts fold tests, economy.test.ts numeric expectations like 'Coal Gen produces 107.01W') are recomputed under the new tier-split numbers. The hand-curated 0.0701 for powerProductionMul becomes either m_K, m_N, or a filler-depth magnitude depending on which node the test uses.
Tier-inference test: verify the inference rules (.keystone. in id → keystone, archetype-prefix-suffix-digit → filler, else notable) by feeding sample IDs and asserting the bucket.
Sanity check at module-load (optional, dev-only): assert FULL_CATALOG.every(n => typeof n.magnitude === 'number'). Fails fast if deriveMagnitudes misses any node.
07

Persistence

No schema bump.

Saves serialize unlockedNodes as NodeId[] — magnitudes are never persisted. effectiveSkillMultipliers reads node.magnitude from the live FULL_CATALOG at fold time, so existing saves automatically pick up the new derived magnitudes on load.

Players who completed nodes pre-rebalance had their unlockedNodes wiped by the v13→v14 migration on the 2026-05-25 ship — those islands restart at level 1 with empty unlockedNodes. Players starting fresh under v14 (the current shape) keep their progression; their unlocked nodes simply gain the new derived magnitudes seamlessly. No additional migration is needed for this spec.

Risks

Risk
Sev
Mitigation
Tier-inference mis-bucket: a hand-curated notable accidentally gets a .keystone. substring in its ID (e.g. a flavor-name collision) and inflates the keystone tier. Or a row's id happens to match an archetype prefix + digit and gets mis-classified as filler.
MED
Add a dev-mode sanity check that every node's inferred tier matches a sentinel list of known-good buckets per effect kind (or simply log a warning when a notable has the same prefix as a filler archetype). Tier-inference unit tests cover the standard cases. Explicit tier override is available for forced disambiguation.
Module-load cost regression: derive runs once but if anything else imports a re-export of FULL_CATALOG before it stabilises, the load order matters.
LOW
Keep deriveMagnitudes a pure function with no side effects. Module dependency graph already cleanly loads catalog before economy. Total work is sub-10ms.
Description strings show stale literal multipliers if not stripped. Players see "×1.0625" on a node whose actual derived factor is ×1.075 (because catalog grew).
MED
Strip the literal multiplier from descriptions in this spec's task list. Either keep descriptions flavor-only OR derive the ×factor at render time. The latter is a follow-up if the current tooltip surface doesn't already format magnitude.
Per-effect-kind catalog growth gives players a small "shift" in magnitudes mid-save: adding a 56th extraction node tomorrow nudges every other extraction node's magnitude downward slightly.
LOW
Pre-release game; per-save drift is acceptable. The drift is downward (every node gets smaller as more share the budget) so a player never gains over the cap from a content patch — they only see the per-node value redistribute. If post-ship playtesting flags this as confusing, lock magnitudes per save version (defer to a future spec).
Tooling drift: scripts/skilltree-magnitudes.ts currently duplicates the cap-check logic. Under the computed model it could either be repurposed (sanity-check the engine's output) or removed (no longer needed since deviation is impossible by construction).
LOW
Keep the script — repoint it to call deriveMagnitudes and pretty-print the resulting per-effect-kind product table. Useful for debugging or for the design conversation when adding new pools.
09

Out of scope

Configurable per-effect-kind tier weights
All 37 effect kinds use the same (0.50, 0.30, 0.20). A future spec could let an effect kind opt into a punchier (0.70, 0.25, 0.05) or gentler (0.40, 0.30, 0.30) split for design variety. Defer until playtesting surfaces a need.
Per-archetype custom growth
All archetypes use growth: 1.10. A future spec could let mining filler grow gentler than smelting filler, etc. Same defer rationale.
UI tooltip ×factor derivation
If the current tooltip surface already shows magnitude from node.magnitude, it continues to work (the derived magnitude lives on the same field). If the tooltip parsed the description's ×1.XXXX literal, that path breaks once descriptions are flavor-only — a follow-up wires the runtime ×factor into the tooltip.
Save-locked magnitudes
Currently magnitudes derive at every module load, so a save loaded under newer code sees newer magnitudes. If this drift becomes uncomfortable, a future spec could freeze magnitudes per save-version.
