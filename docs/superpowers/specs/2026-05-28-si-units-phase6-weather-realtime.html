01

Problem

Three coupled changes share this phase because they share the environmental-loop subsystem.

1.1 Solar curve doesn't reflect the player's reality

Current solarMultiplier(nowMs) at daynight.ts:83 is a trapezoidal curve: 6h ramp up, 6h flat at 1.0, 6h ramp down, 6h flat at 0. Integrates to a 50% capacity factor. Doesn't depend on where the player is geographically; a player in Reykjavik gets the same solar pattern as a player in Singapore. Doesn't depend on the calendar date; December and June produce identical curves.

Rev-16 §7.1 proposed a sinusoidal 22% CF curve as an improvement. The user's mid-stream directive supersedes that: real wall-clock time = in-game time, with the player choosing a geographic location on a map, and full real sunrise/sunset behaviour. Phase 6 ships this.

1.2 No weather feedback from CO₂

Phases 2 + 5 wired accrual + drains; state.co2Kg moves but nothing reads it. Rev-16 §7.3 specs the consumer: storm probability + heatwave events keyed on threshold bands. Phase 6 wires this into the existing weather logic in weather.ts.

1.3 Coal_gen is still wrong

Per rev-16 §10.3 + §2.1 corrected: at realistic ~40% coal-plant thermal efficiency, 12.5 MW thermal → 5 MW electrical = 50 unit. Current building-defs.ts's coal_gen produces 100 unit (10 MW electrical) — implies impossible-for-coal ~80% efficiency. Phase 6 retunes to 50 unit per the rev-9 resolution.

02

SunCalc + altitude formula

SunCalc.js gives sun position (altitude + azimuth) for any (date, lat, lon) triple. Solar intensity scales as sin(altitude) for altitude ≥ 0; below horizon means night.

2.1 Dependency add

// package.json
"dependencies": {
  /* …existing… */
  "suncalc": "^1.9.0",
}

SunCalc surface (verified at https://github.com/mourner/suncalc):

  • SunCalc.getPosition(date: Date, lat: number, lon: number): { altitude: number; azimuth: number } — altitude in radians, azimuth in radians. Pure function; no I/O.
  • SunCalc.getTimes(date: Date, lat: number, lon: number): { sunrise: Date; sunset: Date; solarNoon: Date; ... } — useful for the inspector UI ("sunrise in 2h 14m").

2.2 New solarMultiplier

// daynight.ts — replace solarMultiplier wholesale.
import SunCalc from 'suncalc';

/** Real-astronomy solar insolation factor at the player's geographic
 *  location and the current wall-clock time. Returns 0 below horizon,
 *  sin(altitude) above. Drives computeRates's solar power
 *  multiplier per §15.3 the same way the trapezoidal curve did.
 *
 *  Pre-Phase-6 callers passed solarMultiplier(nowMs); the new
 *  signature requires lat + lon. Callers must thread
 *  these through; the world snapshot carries them as playerLat /
 *  playerLon. When either is null (no picker run yet), this
 *  returns 0 — solar buildings produce nothing until the player picks
 *  a location. The map picker is a session-start blocker in main.ts. */
export function solarMultiplier(
  nowMs: number,
  lat: number | null,
  lon: number | null,
): number {
  if (lat == null || lon == null) return 0;
  const pos = SunCalc.getPosition(new Date(nowMs), lat, lon);
  if (pos.altitude <= 0) return 0;
  return Math.sin(pos.altitude);
}

Range check. pos.altitude ∈ [-π/2, +π/2]; sin(π/2) = 1 at sun directly overhead (equator at solar noon on equinox). sin(altitude) ∈ [0, 1] for altitude ≥ 0.

Removed: SOLAR_AMPLITUDE, sinusoidal-22%-CF, rev-16 §7.1's amplitude scaling. The amplitude factor 0.69 from rev-16 §7.1 doesn't apply here — the real-astronomy curve produces the realistic capacity factor naturally (e.g. 20-30% annual CF at mid-latitudes because nights and shallow-angle hours are real and accounted for). The 22% target was an aggregate of cloud + soiling losses; Phase 6 doesn't simulate those — clear-sky is the realistic ceiling.

2.3 dayPhase / dayPhaseName retain

Existing dayPhase and dayPhaseName at daynight.ts:45 return abstract phase values used by other systems (e.g. the weather logic, the maintenance scheduler). Phase 6 retains these — they're driven by the in-game-time-of-day modulo, not by astronomy. The two systems are independent: phase-of-day for gameplay rhythm; astronomy for solar power.

03

Map picker UI

One-time location selection. Static SVG world map, equirectangular projection. Click coordinate → (lat, lon). Persists to world.playerLat / playerLon.

3.1 New module — src/map-picker.ts

Vanilla DOM (no PixiJS — matches the existing hud.ts + ui.ts patterns). The picker is a modal layer over the canvas; closes on click.

// map-picker.ts — schematic
export function showMapPicker(
  onPick: (lat: number, lon: number) => void,
): void {
  const modal = document.createElement('div');
  modal.className = 'map-picker-modal';
  modal.innerHTML = `
    <h2>Where do you live?</h2>
    <p>Click your location on the map. Real sunrise / sunset will follow real time at that location.</p>
    <svg viewBox="0 0 360 180" class="map-picker-svg">
      <!-- Inline world equirectangular SVG. Public-domain Natural Earth land outline. -->
      <path d="..." fill="#3A3833" />
    </svg>
    <p class="map-picker-readout"></p>
  `;
  const svg = modal.querySelector('svg')!;
  svg.addEventListener('click', (e) => {
    const rect = svg.getBoundingClientRect();
    const xPx = e.clientX - rect.left;
    const yPx = e.clientY - rect.top;
    // Equirectangular: viewBox 0..360 = lon -180..180, 0..180 = lat 90..-90.
    const lon = (xPx / rect.width) * 360 - 180;
    const lat = 90 - (yPx / rect.height) * 180;
    document.body.removeChild(modal);
    onPick(lat, lon);
  });
  document.body.appendChild(modal);
}

3.2 SVG world-map asset

A static public-domain SVG world map. Source: Natural Earth Public Domain Map Dataset (https://www.naturalearthdata.com/), 1:110m cultural-vector boundaries. Either bundle the SVG inline (~30 KB minified) or load from public/world-map.svg. Inline is simpler for a one-time picker. The SVG fits in map-picker.ts as a const string.

3.3 Styling

Reuse existing CSS variables from ui.css — modal background var(--ivory), border var(--clay). The picker matches the existing UI aesthetic.

04

First-session prompt flow

After Phase 1's migration, every fresh world + every migrated v14 save has playerLat = playerLon = null. Phase 6's main.ts bootstrap checks the pair and prompts the picker if either is null.

// main.ts bootstrap — after world load, before the canvas renders.
if (world.playerLat == null || world.playerLon == null) {
  showMapPicker((lat, lon) => {
    world.playerLat = lat;
    world.playerLon = lon;
    saveWorld(world);
    startGame(world);
  });
} else {
  startGame(world);
}

4.1 Settings panel "Change location" affordance

Per the global CLAUDE.md "Keep the human in the loop" doctrine — the player may want to relocate. The settings panel (existing settings-ui.ts) gains a Change location button that re-invokes showMapPicker and overwrites the saved values.

Weather coupling — rev-16 §7.3

CO₂ thresholds drive storm probability + new heatwave events. Bands per rev-16 §7.3.

5.1 Thresholds

Global CO₂ band (kg) Effect
< 100No penalty.
100 – 10000+10% storm probability per biome.
> 10000+30% storm probability + 5% chance per day of "heatwave" event (solar bonus, but per-tile crop yield −20%).
> 100000"Climate-crisis" event chain — weather penalties stack; only out is wastewater_treatment + plant_a_tree.

5.2 Integration with existing weather.ts

The existing storm-spawn logic lives in weather.ts (~587 lines). The integration point is the storm-roll function (whatever its name; grep stormProbability). Phase 6 multiplies the existing baseline by the CO₂-band multiplier from the table above.

function co2WeatherMultiplier(totalCo2Kg: number): number {
  if (totalCo2Kg < 100) return 1.0;
  if (totalCo2Kg < 10000) return 1.1;
  if (totalCo2Kg < 100000) return 1.3;
  return 1.6;  // climate-crisis tier
}

// In the existing storm roll:
const baseProb = /* existing biome × tier × season formula */;
const totalCo2 = sumIslandCo2(world);
const co2Mul = co2WeatherMultiplier(totalCo2);
const adjustedProb = baseProb * co2Mul;

5.3 Heatwave event (new)

A new weather event type. Fires once per game-day on a 5% roll when totalCo2Kg > 10000. Effects:

  • Solar output ×1.2 for the day (heat = more direct insolation, slightly more efficient PV — rough proxy).
  • Per-tile crop yield (forest tile wood extraction, plant_a_tree CO₂ capture) ×0.8 for the day.
  • UI surface: HUD banner "Heatwave today — solar +20%, plants stressed −20%".

5.4 Climate-crisis event chain (new)

At totalCo2Kg > 100000, a recurring event chain fires. The chain is a sequence of escalating storms + heatwaves + drought events until the player drives CO₂ below 100000 via sinks. Per rev-16 §7.3 narrative: "the only out is wastewater_treatment + plant_a_tree."

5.5 sumIslandCo2 helper

export function sumIslandCo2(world: World): number {
  let sum = 0;
  for (const state of world.islandStates.values()) {
    sum += state.co2Kg ?? 0;
  }
  return sum;
}

Computed on demand (not stored). World snapshot's totalCo2Kg field is updated by this helper at save-write time so persisted snapshots have a consistent value; runtime reads call the live aggregator.

Coal_gen retune — rev-16 §10.3

Single-value edit. coal_gen's power.produces drops 100 → 50 unit. Rationale: at ~40% real coal-plant thermal efficiency, 12.5 MW thermal × 0.40 = 5 MW electrical = 50 unit under the 100 kW/unit anchor.

// building-defs.ts — coal_gen entry, find via grep coal_gen.
{
  id: 'coal_gen',
  /* …existing fields… */
  power: { produces: 50 },  // was 100; rev-16 §10.3 resolution
  /* …rest unchanged… */
}

Gameplay implication. Players whose grid relied on coal_gen's 100-unit output now see half that. Multiple coal_gens or a switch to alternative sources (biomass_plant, wind_turbine, solar, geothermal) needed. The balance retune in §06 of this phase verifies the existing progression remains playable.

Alternative (rejected): double burn rate. Per rev-16 §10.3, option B keeps output at 100 unit by doubling coal burn to 1 coal/s. Phase 6 picks option A (50 unit, 1 coal/2s) because the 2-s cycle is load-bearing for smelter timing.

07

Boundary integrator — sub-segmentation

The new astronomy curve is continuously varying; the piecewise-constant integrator must sub-segment finely enough to integrate accurately.

Current SOLAR_RAMP_SEGMENTS = 8 at daynight.ts:73. Trapezoidal had 8 sub-segments per 6h quadrant for the 0→1 ramp. The new astronomy curve is non-linear (sin of altitude) and continuously varying over the entire day. Phase 6 keeps the 8-per-quadrant resolution as a first pass — the post-implementation profile compares it against higher resolutions (16, 32) to confirm 8 is sufficient. If not, bump.

7.1 nextSolarBoundaryMs adjustment

nextSolarBoundaryMs(nowMs) at daynight.ts:131 currently returns the next sub-segment boundary based on the trapezoidal-quadrant model. Phase 6 replaces with a uniform sub-segment grid: divide the 24h day into SOLAR_RAMP_SEGMENTS × 4 = 32 intervals (or whatever the new resolution is); return the next interval boundary.

export function nextSolarBoundaryMs(nowMs: number): number | null {
  const totalSegmentsPerDay = SOLAR_RAMP_SEGMENTS * 4;
  const segmentMs = DAY_DURATION_MS / totalSegmentsPerDay;
  const nowSegment = Math.floor(nowMs / segmentMs);
  return (nowSegment + 1) * segmentMs;
}

Why not sample at sunrise / sunset exactly: The integrator's segment math expects regular boundaries; introducing astronomy-derived boundaries would complicate the math and tie the integrator to SunCalc's precision. The uniform-grid approach is simpler and the worst-case integration error stays bounded by the segment width.

12

Non-goals

Cloud cover simulation
The astronomy curve is clear-sky. Per rev-16 §7.1's 22% CF discussion, real CF includes ~30-40% cloud cover loss. Phase 6 ships clear-sky; future content could layer cloud probability per latitude.
Daylight saving time
SunCalc uses UTC + lat/lon; DST doesn't apply. Player's local clock might be DST-adjusted but solar position is timezone-independent.
Multiple per-island locations
Phase 6 picks ONE location for the world (shared across all islands). Future content could allow per-island latitude offsets for thematic biomes.
Tilted PV arrays
Real PV arrays tilt to maximise insolation. Phase 6 treats solar buildings as horizontal panels (sin(altitude) intensity). A tilt-optimization is future content.
Atmospheric refraction
SunCalc handles standard refraction; Phase 6 doesn't customize. Edge cases (sun visible below astronomical horizon at sunrise) are SunCalc-handled.
Sunpath visualization in UI
The HUD could show "sunrise in 2h 14m" via SunCalc.getTimes. Phase 6 doesn't ship this UI; the data is there if a future phase wants it.
