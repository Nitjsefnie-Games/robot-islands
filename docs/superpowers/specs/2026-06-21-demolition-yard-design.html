Demolition Yard — a continuous Scrap faucet

Context Scrap currently comes from one place only: manually demolishing a finished building (~30% of cost as Scrap + 50% material refund, §6.7) — a one-shot, finite faucet. The bootstrap planner (scripts/bootstrap_planner_v3.py) models a synthetic {building}_scrapper that automates the place-then-demolish loop, but no such building exists in-game. This spec adds it: a placeable Demolition Yard that runs the net economics of one place+demolish cycle of a selectable target building, continuously.

Decision
A building with a per-instance selectable target, whose recipe is derived from the target's placement cost via the existing §6.7 formulas.
The derived recipe equals one place+demolish cycle value-for-value, matching the planner formula exactly. It reuses the canonical scrap/refund math in placement.ts so the Yard and manual demolish cannot drift. The per-instance scrapTarget field mirrors the established cargoLabel pattern.
It deliberately does not instantiate real building instances, tiles, build slots, or touch the construction queue — that fully-literal variant is out of scope.
01

Economics

The Yard's recipe is derived from the target def's base (floor-0) placementCost.

Output: scrap = floor(0.3 · Σ placementCost) — identical to demolishBuilding's scrapReturned. Inputs: for each cost resource with count n, consume n − floor(n/2) (full place cost minus the 50% demolish refund); only positive nets are listed. Cycle: the target tier's base construction time, BASE_CONSTRUCTION_MS_BY_TIER[targetDef.tier] / 1000 s.

# scripts/bootstrap_planner_v3.py, lines 491-504
_scrap_out = floor(0.3 * sum(_cost.values()))
_consumed  = {r: n - (n // 2) for r, n in _cost.items() if n - (n // 2) > 0}
_cycle     = BASE_CONSTRUCTION_S_BY_TIER[_tier] / PARALLEL_BUILD_SLOTS  # slots = 1

Worked example. Target = T1 Mine (30 stone + 15 wood, 30 s construction): inputs 15 stone + 8 wood, output 13 scrap, cycle 30 s → 0.43 scrap/s. High-tier targets have long construction times → deliberately slow; construction time is the honest rate limiter. With no scrapTarget set, the Yard is idle — like an unlabeled Crate.

02

Why net-recipe over fully-literal

Fully-literal instances
  • Must allocate tiles + build slots for a transient building.
  • Routes through the construction queue, refund clamps, demolish primitive.
  • Large new mechanic touching placement.ts, the queue, the renderer.
Net-recipe (chosen)
  • One derived Recipe object — no tiles, slots, or queue.
  • Reuses the existing economy / flow-solver path with zero new integration.
  • Value-for-value identical to the place+demolish loop.

Verification

scrapRecipeForTarget unit tests. Matches planner formula for Mine / Smelter / a T2 building; undefined for zero-scrap basket.
resolveRecipe tests. undefined when unset; derived recipe when set.
Economy integration. Scrap accrues at expected rate; inputs consumed; throttles to demand.
Persistence. v31 fixture → v32 clean; v32 round-trips; scrapTarget survives.
Full build + typecheck. npm run build and server npm run typecheck clean.

Out of scope
