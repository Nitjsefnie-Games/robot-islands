Transport + intent protocol

Context Slices 1–2 gave us accounts and a server that owns + persists game state with offline catch-up. Slice 3 adds the two-way channel: the client sends intents (place, spend, dispatch, trade, …); the server validates each against the same pure rules, applies it authoritatively, persists, and pushes back the projection. This is the anti-cheat core — the client can ask, but only the server's pure-function outcome is real. The browser client is not cut over yet (slice 4).

Decision
An authenticated WebSocket carrying an intent envelope; a dispatch table validates each intent by RE-RUNNING the existing pure function on authoritative server state, then persists + broadcasts the projection.
Validation = re-running the pure rule on server state — the client's numbers are never trusted; success/failure comes from the pure function's outcome on authoritative inventory/state. Rejected intents persist nothing.
Scope: the transport + envelope + validate/persist/broadcast mechanism, wired for a 9-intent core subset, with all 30 cataloged so the rest are a mechanical follow-up. Client cutover is slice 4; hardening under-validating pure functions + the as unknown as casts is slice 5.

Background — verified intent surface

A src/ audit enumerated 30 player-initiated state-mutating actions (the wire intents) and the automatic/tick functions that must NOT be intents (advanceIsland, tickDrones, tickRoutes, tickVehicles, tick* orbital, tickTradeOffers, performMerge, …) — those are driven by the server tick loop, never the client. Every mutating action already has a pure entry function the server reuses unchanged (slice-2's cross-workspace import).

04

Intent envelope & flow

client → server:  { "type":"place-building", "payload":{ islandId, defId, x, y, rotation }, "seq":42 }
server:           game = loadAndCatchUp(pool, userId, now)     // authoritative
                  result = INTENTS["place-building"].apply(game, payload, now)  // re-runs placeBuilding
                  if result.ok:  saveSnapshot(...); broadcast projection
                  else:          nothing persisted
server → client:  { "seq":42, "ok":true,  "projection":{...} }
            or:   { "seq":42, "ok":false, "error":"insufficient resources" }
  • Authoritative legality: the handler runs the pure fn on server state; over-spend/illegal → ok:false, nothing persisted. Where a pure fn trusts its caller, the handler adds the minimal affordability pre-check (deeper hardening = slice 5).
  • One in-flight intent per account so two intents can't race the saves row.
  • Determinism-as-bandwidth: weather/day-night are NOT sent — the client renders them from (seed, t). Projection carries only non-deterministic authoritative state.

06

Trust surface (feeds slice 5)

  • Several pure fns deduct cost only if the caller checked affordability (the client UI did). Server-side the handler pre-checks affordability from authoritative inventory before applying; slice 5 hardens the functions + the as unknown as readonly-mutation casts.
  • accept-trade and the XP path are TODO-flagged sensitive: the handler validates the offer exists in authoritative state and the give-side inventory suffices before applying.
07

Testing & verification

  • Per wired intent: legal applies + mutates expected state; illegal/unaffordable rejected + state & saves unchanged.
  • WS: unauthenticated upgrade rejected; envelope round-trip with matching seq + projection; rejected intent leaves the stored snapshot byte-unchanged; queued intents apply in order (no saves race).
  • Live: a scripted WS client authenticates, sends place-building → ok:true + projection; sends an unaffordable intent → ok:false; psql shows the save advanced only for the accepted intent.
