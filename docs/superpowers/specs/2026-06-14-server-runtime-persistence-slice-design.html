Server runtime + persistence

Context Slice 1 gave us accounts + sessions. Slice 2 makes the server own game state: it hosts the existing pure simulation layer (advanceIsland, §15.3) and moves persistence from the browser's IndexedDB to Postgres, keyed by account. This is the in-place port strategy of TODO.md ("the pure layer ports as-is"), not a rewrite. The client is still untouched (cutover is slice 4).

Decision
A per-account server runtime that deserializes, offline-catches-up via advanceIsland, and persists to a Postgres saves table — reusing the pure src/ functions unchanged.
The integrator is dt-agnostic (a 24h gap is one integrate call) and pixi-free; profile-economy.ts proves the whole persistence+economy chain imports and runs headless. So the server reuses serializeWorld / deserializeWorld / the migrateV* chain / advanceIsland with zero divergence.
Deliberately NOT here: the intent/WebSocket protocol (slice 3), the client cutover (slice 4), hardening (slice 5), and the world.ts/island.ts pure-render split (TODO #6 — the transitive PixiJS import is import-only and harmless).

Data model — migration 0002_saves.sql

CREATE TABLE saves (
  user_id        uuid PRIMARY KEY REFERENCES users(id) ON DELETE CASCADE,
  snapshot       jsonb NOT NULL,            -- a full SaveSnapshot (client shape)
  schema_version integer NOT NULL,          -- snapshot.v at write time
  created_at     timestamptz NOT NULL DEFAULT now(),
  updated_at     timestamptz NOT NULL DEFAULT now()
);

One save per account (PK = user_id). snapshot is the exact object serializeWorld produces — client and server share one format (key for slice 4). schema_version is denormalized from snapshot.v for cheap version queries.

05

Migration-chain reuse (single-sourced)

Slice 2 does not modify the migration logic's behavior. game/persistence.ts reads the saves.snapshot jsonb (possibly old version), runs the same version-dispatch + migrateV* chain the client uses, then calls deserializeWorld. If a pure "migrate snapshot object to current" helper isn't already separable from the IDB-bound loadWorld, the plan adds one small pure export to src/persistence.ts (migrateSnapshotToCurrent(raw)) and refactors loadWorld to call it — keeping client + server on one code path (with SPEC + tests updated).

06

Time & offline catch-up

  • On save: stamp savedAt = savedAtPerf = Date.now().
  • On load: deserializeWorld(snapshot, Date.now(), Date.now()) — perfShift becomes the real elapsed time; each IslandState.lastTick is shifted so the next advanceIsland(state, Date.now()) integrates exactly the offline gap.
  • loadAndCatchUp advances every populated island then re-serializes + persists, so catch-up is durable. Same path for a 1-second and a 30-day gap.

08

Testing (vitest, against robot_islands_test)

  • Save roundtrip: save then load returns an identical SaveSnapshot; schema_version matches snapshot.v.
  • Migration on load: an older-version fixture is migrated to SCHEMA_VERSION and deserializes cleanly.
  • Offline catch-up: a snapshot saved N hours ago advances on load; split-invariance sanity.
  • New game: POST /new creates a save; second call → 409.
  • State: GET /state returns a projection; 404 before new; 401 unauthenticated.
  • Pure-layer reuse smoke: importing serializeWorld/advanceIsland from src/ compiles + runs (no pixi instantiation).
