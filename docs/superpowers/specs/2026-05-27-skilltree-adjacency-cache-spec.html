Cache buildAdjacency and per-state computeAuraAmplifiers

Context After the first-tier per-tick memoization of effectiveSkillMultipliers (commit a4ff6a4 and follow-ups), a re-profile of the same live save still shows skill-tree-graph adjacency as the dominant CPU cost. buildAdjacency is a pure function of static graph.edges rebuilt on every call; computeAuraAmplifiers is a pure function of state.unlockedNodes (plus the static graph) but also rebuilds from scratch. This spec adds two layers of caching that collapse roughly half of advanceIsland's CPU into a WeakMap hit.

Decision
Two-layer cache. Layer 1 is a module-level WeakMap<Graph, …> for the static-input pieces; Layer 2 is a version-counter-keyed per-state cache for the unlocked-node walk.
Layer 1 — Graph-keyed module cache. buildAdjacency(graph) and a sibling graphById(graph) (the Map<string, SkillNode> currently rebuilt at skilltree.ts:1136-1137) become memoized on a module-level WeakMap keyed by the Graph object. The hot path always passes DEFAULT_GRAPH — one cache entry, populated lazily on first call, lives for the process lifetime. The graph-view UI passes a fresh Graph from effectiveGraph(state) when crystals are bound; those entries are reclaimed by GC once the UI render finishes (the WeakMap does not pin them).
Layer 2 — Per-state aura-amp cache (option 2A). Add auraAmpVersion: number to IslandState and bump it at every unlockedNodes / unlockedEdges mutation site (9 sites enumerated in §04). computeAuraAmplifiers stashes its Map<NodeId, number> result on the state along with the version it was computed at; a subsequent call with the same version returns the cached map directly.
Why 2A over 2B (Set-identity). Option 2B would key a WeakMap on the unlockedNodes Set object, but it requires either (a) replacing every .add()/.delete()/.clear() with a fresh new Set(...) assignment, which is a much wider edit, or (b) keying on Set identity while accepting cache poisoning when the Set mutates in place. 2A is explicit, fits the existing Set-mutation idiom, and the version bump is a single statement at each enumerable site.

02

Root cause

Walk the call chain. Every term is verified against the cited file:line.

  1. advanceIsland(state, nowMs) at economy.ts:1600 calls effectiveSkillMultipliers(state) exactly once to compute baseMult. v
  2. effectiveSkillMultipliers at skilltree.ts:903 calls computeAuraAmplifiers(state, graph) exactly once at line 950, then walks state.unlockedNodes applying each node's effect modulated by the per-node aura amplifier. v
  3. computeAuraAmplifiers at skilltree.ts:1132:
    • Line 1134 — calls buildAdjacency(graph).
    • Lines 1136-1137 — builds byId: Map<string, SkillNode> by iterating graph.nodes.
    • Lines 1139-1150 — walks state.unlockedNodes and runs nodesWithinRadius (BFS) for each aura-bearing node.
  4. buildAdjacency at skilltree.ts:1154-1170 iterates every edge in graph.edges (excluding AND-prereq edges) and builds a fresh undirected-neighbour map. v

The waste: graph is DEFAULT_GRAPH on the hot path — a module-level const defined at skilltree.ts:529-534 and never mutated. graph.edges never changes after the module loads. Yet the adjacency map is rebuilt on every advanceIsland, and the byId map is rebuilt the same number of times.

Layer 2's state.unlockedNodes changes only at the 9 enumerated mutation sites in §05 — none of which fire during a tick (economy.ts:1595-1601 documents this explicitly: "unlockedNodes / unlockedEdges do not mutate during this advanceIsland call"). So the aura-amp result is also stable across long stretches of tick activity.

03

Layer 1 — Graph-keyed module cache

Two pure functions of the static graph become WeakMap-memoized at module scope. No invalidation logic because the cache key is the input object itself.

Why WeakMap<Graph, …> and not let _adj = null

Two distinct callers reach buildAdjacency:

  • Hot path: effectiveSkillMultipliers with the default graph = DEFAULT_GRAPH at skilltree.ts:905. One cache entry forever.
  • UI path: skilltree-graphview.ts calls effectiveGraph(state) (e.g. skilltree-graphview.ts:363, 391, 420, 502, 672, 744) which, when crystal sockets are bound, allocates a fresh Graph object per call at skilltree.ts:598-603. v

A let _defaultAdj: Map | null = null single-slot cache would silently miss the UI path; a regular Map<Graph, …> would pin each UI-render graph in memory forever. WeakMap covers both: hot-path entries are kept alive by DEFAULT_GRAPH's module-level reference; UI-path entries are reclaimed by GC once the UI render ends and the transient graph goes out of scope.

Shape of the change

// At skilltree.ts module scope, near the existing DEFAULT_CATALOG.
const _adjCache = new WeakMap<Graph, Map<NodeId, NodeId[]>>();
const _byIdCache = new WeakMap<Graph, Map<string, SkillNode>>();

function buildAdjacency(graph: Graph): Map<NodeId, NodeId[]> {
  const cached = _adjCache.get(graph);
  if (cached) return cached;
  const adj = new Map<NodeId, NodeId[]>();
  for (const e of graph.edges) { /* …existing body…  */ }
  _adjCache.set(graph, adj);
  return adj;
}

function graphById(graph: Graph): Map<string, SkillNode> {
  const cached = _byIdCache.get(graph);
  if (cached) return cached;
  const byId = new Map<string, SkillNode>();
  for (const n of graph.nodes) byId.set(n.id as string, n);
  _byIdCache.set(graph, byId);
  return byId;
}

Both helpers are module-private; no signature change. computeAuraAmplifiers already calls buildAdjacency(graph) at line 1134 — the only modification there is to replace the inline byId construction at lines 1136-1137 with a call to graphById(graph).

Correctness invariant

The cache is correct iff the cached function is pure in graph. Both helpers are: buildAdjacency reads only graph.edges; graphById reads only graph.nodes. Neither the Graph instance nor the edges / nodes arrays are mutated anywhere in the codebase (DEFAULT_GRAPH is const; effectiveGraph at skilltree.ts:540-606 always constructs a new object on the crystal-binding branch). v

04

Layer 2 — Per-state aura-amp cache

The result of computeAuraAmplifiers depends on state.unlockedNodes in addition to the static graph. Layer 1 makes the inputs cheap; Layer 2 makes the output cheap too.

New fields on IslandState

export interface IslandState {
  /* …existing fields… */
  unlockedNodes: Set<NodeId>;
  unlockedEdges: Set<EdgeId>;
    /** Bumped on every mutation of unlockedNodes / unlockedEdges.
   *  Cache-key for Layer 2 (aura-amp) and any future
   *  pure-skill-state derivation. Starts at 0; resets via the
   *  persistence migration in §07. */
  auraAmpVersion: number;
  /** Memoized result of computeAuraAmplifiers(this, DEFAULT_GRAPH).
   *  Null on cold start / post-migration. Valid iff
   *  auraAmpCacheVersion === auraAmpVersion. Stripped at serialize time;
   *  not part of the persisted snapshot. */
  auraAmpCache: Map<NodeId, number> | null;
  auraAmpCacheVersion: number;
}

Read/store at computeAuraAmplifiers

function computeAuraAmplifiers(state: IslandState, graph: Graph): Map<NodeId, number> {
  // Cache hit: same state.unlockedNodes / unlockedEdges as last call,
  // AND the graph we'd resolve to is the default (the only path that
  // can rely on state-versioning — UI paths with crystals pass a
  // transient `graph` and skip the cache).
  if (graph === DEFAULT_GRAPH
      && state.auraAmpCache !== null
      && state.auraAmpCacheVersion === state.auraAmpVersion) {
    return state.auraAmpCache;
  }
  const amp = new Map<NodeId, number>();
  const neighbours = buildAdjacency(graph);  // Layer 1 hit
  const byId = graphById(graph);            // Layer 1 hit
  /* …existing aura-walk body, unchanged… */
  if (graph === DEFAULT_GRAPH) {
    state.auraAmpCache = amp;
    state.auraAmpCacheVersion = state.auraAmpVersion;
  }
  return amp;
}

Why gate on graph === DEFAULT_GRAPH. The version counter tracks state mutation, not graph identity. A UI render that constructs a transient effectiveGraph with crystal-bound mini-tree nodes would produce a different aura-amp map for the same auraAmpVersion. The simplest correct rule is: only store and read the cache when the graph is the canonical DEFAULT_GRAPH. UI paths pay the unmemoized cost (Layer 1 still helps them) — they're not the hot path.

07

Persistence interaction

The new fields live on the runtime IslandState only — they must NOT round-trip through the save file. The existing serialize/deserialize idiom already strips the Set fields; this spec extends it to strip the cache fields too.

Why the cache must not persist

  • Map<NodeId, number> stringifies to {} in JSON. Round-trip would silently corrupt the cache.
  • The cache is rebuildable in O(unlockedNodes × radius) on first call after load. There's nothing to gain by persisting it.
  • auraAmpVersion could persist, but resetting it to 0 on load is equally correct (the cache is rebuilt on first call regardless), and skipping it means no new schema migration is needed.

Serialize side — persistence.ts:482

// Before:
const { unlockedNodes, unlockedEdges, socketBindings, ...rest } = state;

// After:
const {
  unlockedNodes, unlockedEdges, socketBindings,
  auraAmpVersion: _v,
  auraAmpCache: _c,
  auraAmpCacheVersion: _cv,
  ...rest
} = state;

The _-prefixed bindings exist solely to prevent the spread from carrying the cache fields into ...rest. They are intentionally unused and pass noUnusedLocals via the underscore prefix.

Deserialize side — persistence.ts:731-732

// Inside the load path that reconstitutes IslandState from a snapshot:

  ...s,

No schema bump

The wire format does not change. v14 snapshots written before this work remain readable; new snapshots written after this work are byte-compatible with v14 readers (the cache fields are runtime-only and stripped on serialize). No new migrateV14toV15 function is required. v

08

Test plan

Three correctness tests are sufficient. The perf-regression test is documented but lives in the plan, not the spec (Phase 6 below records the actual measured win).

Test 1 — Mutation invalidates the cache (Layer 2)

it('auraAmp reflects new aura node after unlock', () => {
  const s = makeInitialIslandState(spec, 0);
  // Compute once to warm the cache.
  effectiveSkillMultipliers(s);
  const beforeMul = effectiveSkillMultipliers(s).recipeRate['mining'];
  // Unlock an aura-bearing notable that affects mining.
  s.unlockedNodes.add('mining.aura.notable' as NodeId);
  // Spec compliance: the mutation site bumps the version. In a real
  // purchase this is automatic via spendPoint / buyNode; the test must
  // bump explicitly because it's poking the Set directly.
  s.auraAmpVersion++;
  const afterMul = effectiveSkillMultipliers(s).recipeRate['mining'];
  expect(afterMul).toBeGreaterThan(beforeMul);
});

The test asserts the spec contract: after any cache-invalidating mutation, the next call returns a freshly computed result reflecting the new state. The explicit s.auraAmpVersion++ mirrors what buyNode does for real mutation sites.

Test 2 — Cache hit on repeated unmodified call

it('computeAuraAmplifiers reuses cached result on identical state', () => {
  const s = makeInitialIslandState(spec, 0);
  s.unlockedNodes.add('mining.aura.notable' as NodeId);
  s.auraAmpVersion++;
  effectiveSkillMultipliers(s);
  const cachedMap = s.auraAmpCache;
  expect(cachedMap).not.toBeNull();
  // Second call — must NOT replace the cache instance.
  effectiveSkillMultipliers(s);
  expect(s.auraAmpCache).toBe(cachedMap);  // reference equality
});

Reference-equality on the Map instance is the observable signal of a cache hit without exposing a counter on the production API.

Test 3 — Layer 1 returns the same instance per Graph

it('buildAdjacency returns the same Map instance for DEFAULT_GRAPH across calls', () => {
  // buildAdjacency is module-private; test via the observable
  // side-effect: computeAuraAmplifiers' result for an unrelated state
  // shape is unchanged across calls, AND the byId-walk in
  // effectiveSkillMultipliers produces identical results.
  const s = makeInitialIslandState(spec, 0);
  const m1 = effectiveSkillMultipliers(s);
  const m2 = effectiveSkillMultipliers(s);
  expect(m1.recipeRate).toEqual(m2.recipeRate);
});

A direct instance-identity assertion on buildAdjacency's return value would require exporting the function for the test or using vi.spyOn on the module internals. The behavioural assertion (results match) is equivalent for the spec's purpose: incorrect caching would either return stale data on a follow-up call (caught by Test 1) or surface as a result divergence (caught here).

Perf check (lives in plan Phase 6, not as a CI gate)

After implementation, re-run scripts/profile-economy.ts /tmp/robot-islands-save.json. Binding gate documented in the plan: home-island mean tick must drop to <1.8 ms (a 1.5× win from the current 2.67 ms baseline) or the implementation is broken. The measured numbers go in the final commit body.

Risks

Risk
Sev
Mitigation
A new mutation site for unlockedNodes / unlockedEdges is added in a future change without bumping auraAmpVersion, silently serving stale aura-amp data.
MED
JSDoc on the two Set fields names the bump contract. A grep-based lint (in plan Phase 5 as a test fixture) walks the codebase asserting every unlockedNodes.add|delete|clear / unlockedEdges.add|delete|clear line has a sibling auraAmpVersion++ in the same function.
The cached Map<NodeId, number> on state.auraAmpCache is returned to callers who then mutate it.
LOW
Only one caller — effectiveSkillMultipliers at skilltree.ts:950 — and it only .get()s from the map. No call site writes to the returned map. Spec calls this out; an explicit Object.freeze(amp) or returning a ReadonlyMap type is unnecessary overhead.
A test that mutates unlockedNodes directly without calling the purchase API forgets to bump the version, and the test then reads stale results.
LOW
Test 1 above demonstrates the pattern explicitly. Tests touching the Set directly remain in the minority — most test code uses spendPoint / buyNode which bump automatically.
Layer 1 WeakMap leaks because effectiveGraph's transient Graph objects accidentally get pinned by a UI ref.
LOW
WeakMap by definition does not prevent GC. A UI path that holds its Graph reference for a render and drops it sees the cache entry collected naturally. The cache is bounded by however many transient graphs are simultaneously referenced, which is O(1) in practice (one panel open at a time).
The auraAmpVersion counter overflows.
LOW
JS number is double-precision float. Counter would need 2^53 mutations to lose precision. A session that purchases one skill node per second for ~285 million years is the failure mode. Out of scope.

Non-goals

Explicitly out of scope so the reader can confirm these were considered and deliberately deferred.

The findNextCapEvent stale-ctx bug
economy.ts:1690 calls findNextCapEvent(state, net, t, nowMs, ctx) with bare ctx rather than the effectiveCtx that carries baseMult. That bug is being addressed separately. It reduces the call frequency of effectiveSkillMultipliers but does not eliminate the per-call buildAdjacency cost — this spec is load-bearing regardless.
Caching anywhere else in effectiveSkillMultipliers
The remaining ~3.7% self time after this work is the unlocked-nodes effect-application walk at skilltree.ts:952-1107. It depends on the same state.unlockedNodes as Layer 2 — a Layer 3 would memoize the final SkillMultipliers bundle on the same version counter. That's a future optimization to revisit after measuring the post-Layer-2 profile.
UI / rendering path optimization
Pixi rendering is read-only against state and does not call effectiveSkillMultipliers on the hot path. No render-side change in scope.
Persisting auraAmpVersion
Reset to 0 on load. Cold cache on session resume costs one BFS walk on the first advanceIsland call. Not worth a schema bump.
Wider WeakMap coverage for other pure graph derivations
Functions like spentInBranch at skilltree.ts:1406 and isBridgeActive at skilltree.ts:1402 are not hot enough to justify the same treatment based on the current profile. Revisit if a future profile shows them.
