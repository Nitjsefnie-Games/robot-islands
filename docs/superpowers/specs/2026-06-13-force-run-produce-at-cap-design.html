Force-Run toggle — produce-at-cap for XP

Context Under the net-flow economy, a building whose output bin is full has its flow-solver gate driven to zero — it stops, earns no XP, and accrues no wear. There is no way to keep a maxed-out production line running just to keep levelling. This spec adds a per-building Force Run toggle that keeps it running for XP, voiding the overflow at the cap.

Decision
Exclude force-run buildings from the flow-solver's cap:r constraint; the existing integrator does the rest.
The event-driven integrator already clamps inventory to [0, cap] (overflow auto-voided) and already refuses to emit a recurring event for an at-cap bin with positive net flow (no infinite loop). XP and wear both follow automatically once the gate stays open. So the only behavioral change is one carve-out in the solver.
Force Run is a real tradeoff: the building still consumes its inputs, draws power, and wears down. The overflow output is always voided — caps stay hard. It is not a free XP faucet.
01

Goal

What the toggle does, and the cost model.

Add a per-building Force Run toggle in the inspector. When on, the building keeps running even with a full output bin. It genuinely operates — consumes inputs, draws power, accrues maintenance wear — and the overflow output is voided at the cap. The XP is the payoff; the inputs, power, and wear are the cost.

02

Default vs. Force Run

At a full output bin, with no consumer drawing the capped resource.

Default (today)
  • Flow-solver gate g → 0.
  • Production stops; inputs not consumed.
  • No XP earned.
  • No power draw, no maintenance wear.
Force Run on
  • Exempt from the cap:r constraint; gate stays open.
  • Runs at full rate; inputs consumed, power drawn.
  • XP earned from production.
  • Wears normally; overflow output voided at the cap.
03

Why the change is small

The integrator already supports producing-into-a-full-bin.

Overflow is auto-voided. applyRates clamps every inventory to [0, cap] (economy.ts:2147). A force-run producer's net-positive flow into a full bin is clamped at the cap each segment — the excess is discarded.

No infinite loop. findNextCapEvent skips a boundary event for a bin at/over cap with positive net flow (economy.ts:2068, headroom <= 0 ⇒ continue).

XP and wear follow automatically. XP accrues from production; wear accrues as dt × utilization. Both are nonzero exactly when effectiveRate > 0. The single behavioral change: exclude a force-run building from the cap:r constraint so its gate isn't dragged to 0.

04

Change inventory

Every file the implementation touches.

buildings.ts
Add forceRun?: boolean to PlacedBuilding. Absent / false = today's behavior.
flow-solver.ts
Add ignoreOutputCap? to FlowBuildingSpec. When set: skip the building's cap:r key assignment, and skip it from the producer entries / consumer target inside update('cap:r'). zeroConstrained + power factor unchanged.
economy.ts
Pass the flag through. In buildFlowBuildings, set ignoreOutputCap: b.forceRun. Nothing else changes.
inspector-ui.ts
Force Run toggle button. In the maintenance section beside the floor steppers; makeExpandButton style; pending + dispatchAction + deps.onSetForceRun + paint(). Shown only for resource-producing buildings; hidden under construction.
main.ts
Wire onSetForceRun. Set building.forceRun and trigger the save path onSetActiveFloors uses.
persistence.ts
Additive field — round-trips for free. Like disabledFloors / cargoLabel. No schema bump expected; confirm no strict-shape rejection.
SPEC.md
§4.6 + §4.7. Document the cap-throttle override, and note force-run does not escape maintenance (nonzero duty ⇒ wears normally).
05

The flag

/** Force-run: keep producing for XP even when an output bin is at cap.
 *  Absent / false = default (throttle to consumer draw at a full bin).
 *  The building still consumes inputs + power and accrues wear; overflow
 *  output is voided at the cap. */
forceRun?: boolean;
06

Verification

Concrete, runnable.

flow-solver test. Force-run producer keeps gate > 0 at a capped output while a normal producer of the same resource gates to 0; non-force producers still solve to consumer draw.
economy test. Force Run on at a full bin: XP accrues, inventory stays exactly at cap (overflow voided), wear accrues, inputs consumed, building stops when inputs run dry.
persistence test. forceRun: true round-trips identity; legacy save (absent) loads with Force Run off.
npm run build & npm test. Clean under strict tsconfig; all tests green.
07

Risks

Risk
Sev
Mitigation
Excluding a building from cap:r mishandles a resource with multiple producers (some force, some not).
MED
Skip force-run buildings from both the key and the constraint's entries/target, so θ governs only normal producers; covered by the flow-solver test.
Recurring zero-length event at a pinned full bin (integrator freeze).
LOW
Already handled — findNextCapEvent skips at-cap bins with positive flow; the clamp pins inventory at cap.
Persistence rejects the new optional field.
LOW
Field is additive and absent = off; verify no strict-shape validation, add a no-op migration only if required.
08

Out of scope

Global / multi-building Force Run
Per-building only — matches the "a button" request and the floor-disable precedent.
Letting output exceed cap
Overflow is always voided; caps stay hard.
A separate XP-only code path
XP comes from genuine (then-voided) production, not a bespoke faucet.
