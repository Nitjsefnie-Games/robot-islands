Skill graph — graph-view UI design

Replace the existing 4-column text-list rendering in src/skilltree-ui.ts with a PixiJS-rendered, full-page, algorithmically-laid-out node graph. Renders the ~494 nodes (~384 filler + 80 notables + 30 keystones) + ~700 edges + 27 threshold-bridges + 44 graft sockets as a pannable/zoomable canvas. Click-to-buy via the existing buyNode / buyKeystone engine surface.

Locked decisions (chat-brainstorm): algorithmic layout · PixiJS rendering · full-page overlay (not a draggable panel).
01

Layout — algorithmic, deterministic seed

Layout computed once at app boot, cached in memory. Recomputed only if catalog changes. NOT recomputed per-frame.

Algorithm: two-stage. Stage A places branch-roots in a regular pentagon (5 branches → 5 positions evenly distributed on a circle of radius R = 600 world-px). Stage B runs a Fruchterman-Reingold-style force-directed simulation on each sub-path's internal sub-graph, anchored to its branch-root. ~200 iterations is enough for ~500 nodes to settle. Connected sub-paths within a branch are positioned along a radial arc emanating from the branch-root.

Determinism: seed the simulation with a fixed RNG (e.g. mulberry32 with seed = hash of catalog node-count). Layout is reproducible across reloads — players see the same graph every time.

Manual overrides: the catalog can optionally specify {x: number, y: number} per node. Stage B's simulation uses those as fixed-position anchors. v1 doesn't ship any overrides; designer can pin specific keystones later if force-directed mis-places them.

Cross-sub-path and cross-branch bridges render as long edges connecting nodes that aren't in the same sub-graph cluster. Force-directed pulls them slightly toward each other but the branch-root anchoring keeps clusters separated.

Implementation: place algorithm in a new module src/skilltree-layout.ts. Pure (no PixiJS). Returns Map<NodeId, {x, y}>. Tested with a small fixture graph + snapshot assertion on node positions.

02

Rendering — PixiJS canvas, full-page overlay

Full-screen overlay covering the entire viewport (above the HUD z-layer). Dismissible via Esc or a corner X button. Replaces the existing skill-tree modal entirely.

Mount point: new src/skilltree-graphview.ts mounts a PixiJS Application at full window size. Inherits the existing pure-render separation (this file is render-layer; consumes layout from skilltree-layout.ts and state from economy.ts / skilltree.ts).

Layering (PixiJS Containers): background → edges → bridges (inactive) → bridges (active) → nodes → labels → tooltip. Each layer is one Container so z-order is deterministic.

Performance budget: ~500 nodes × ~700 edges × ~30 bridges = ~1230 Graphics primitives. PixiJS 8 batches these into <10 draw calls. Target 60 fps idle; <16ms per frame on hover-redraw.

03

Interaction

Pan: click + drag empty area. Moves the world container.

Zoom: mouse wheel. Logarithmic (1.1× per notch). Min 0.3×, max 3×. Centers on cursor.

Click node: if affordable + unowned + (keystone ⇒ prereqs satisfied), purchase via buyNode / buyKeystone. Confirm dialog if cost > 20 SP (configurable threshold). On purchase, refresh the canvas (re-color affected nodes/edges); the layout doesn't recompute.

Hover node: tooltip with name, description, cost-to-unlock (live computed from costToUnlock for non-keystones; flat cost for keystones), aura ring (if applicable), highlight all incoming/outgoing edges in --rust. For aura-bearing notables, also highlight all nodes within the aura radius.

Hover edge: tooltip showing cost (for paid edges) or threshold status (for bridges). No interaction beyond that.

Dismissible: Esc key, X button in top-right, or click outside the canvas (the latter may interfere with pan — gate behind a modifier? Esc is the safe primary).

HUD shows: top-left: Unspent SP: N + Owned: N/494. Top-right: filter chips (Show: filler / notables / keystones / bridges / sockets — all checked by default). Bottom-center: legend strip.

05

Out of scope (deferred)

  • Manual override coords in the catalog — v1 ships pure force-directed.
  • Build planner (Picker 8 option C: side-panel "save build, compare two builds") — v2 spec.
  • Graft socket attachment UI — sockets render as empty placeholders; attaching cluster jewels to them is a separate feature.
  • Mobile/touch interaction — desktop only at v1.
  • Search / filter beyond chip toggles — no text search box at v1.
06

Open questions for the plan

  • Force-directed iteration count + spring/repulsion constants — needs playtesting. Start with k_repulsion=8000, k_spring=0.1, 200 iters.
  • How to handle node-label overlap at zoomed-out levels? Show labels only above 1× zoom; below that show only branch-root labels.
  • Should the "Open Skill Graph" button also act as a back-compat fallback to the old list view? Probably no — clean replacement.
  • Test strategy: layout has snapshot tests (deterministic seed); rendering layer doesn't unit-test the PixiJS scene; manual browser smoke in T20-equivalent.
