Closing the resource graph: CO₂ atmosphere, byproduct-gas consumers, and 43 orphan sinks

Context A recipe + building-cost inspection found 43 resources that are produced by a real chain but consumed by nothing, plus a class of byproduct gases that silently throttle their own producers (a capped output bin stalls the furnace that fills it), and a CO₂ double-booking bug (the co2 resource and the co2Kg climate scalar diverge — capture drains only the scalar). This spec closes the entire graph with real production chains, unifies CO₂ into a single global atmosphere, and turns the orphaned self_replication_module into a relief valve for the §4.9 exponential upgrade-cost curve. Delivered in three independently-mergeable phases; correctness lands first.

Problem & locked decisions

Three distinct problems, one graph. The catalog grew faster than its consumption side: content phases (ocean §3, Phase 10/11/12, the §7.2 alloy cluster) shipped producers without wiring the consumers.

Of 262 catalog resources, 117 never appear as a recipe input. Subtracting the ones consumed by non-recipe systems (60 skill crystals via skilltree-crystals.ts; satellites / repair via orbital.ts; biofuel/aviation_kerosene via the fuelForTier vehicle ladder) leaves 43 genuinely orphaned resources plus a class of byproduct gases that accrue with no sink. genesis_cell is the sole legitimately-terminal resource (spec: no win state).

Phase 1

Correctness — CO₂ + byproduct gases

Decision (locked)
CO₂ becomes a single global atmospheric quantity — an emission-side twin of the existing air flow — never an inventory resource. Capture and feedstock recipes debit it as atmospheric flows.
Today co2 is double-booked: recipe output accrues to both inventory['co2'] (capped liquid_gas) and the per-island state.co2Kg scalar, while capture drains only the scalar. The inventory copy has no sink, so it fills its bin and the flow solver stalls every CO₂-emitting furnace. Unifying to one global counter fixes the double-booking and the throttle in a single move.
The only climate reader, weather.ts:152 sumIslandCo2(), already returns a sum of per-island values — so the global counter is the natural shape it was already reaching for.
01

Root cause — CO₂ double-booking & the throttle

Two parallel ledgers for the same quantity, plus a capped bin nothing drains.

1.1 The double-booking

In advanceIsland, a recipe's co2 output takes two paths:

// economy.ts:2148 — Path A: net rates → inventory (co2 is a capped liquid_gas)
state.inventory[r] = clamped;                 // r === 'co2' lands here

// economy.ts:2352 — Path B: same production also feeds the climate scalar
state.co2Kg += (production.co2 ?? 0) * dtSec;

Capture only touches Path B (state.co2Kg = Math.max(0, state.co2Kg − drainKg), economy.ts:2380). The inventory['co2'] copy is never drained. Because Path A and Path B use the same production.co2 × dtSec, they accrue identically — the bug is purely that one of the two is never consumed.

1.2 The throttle (generalises beyond CO₂)

Every byproduct gas is a capped liquid_gas with no sink (storage-categories.ts): co, co2, refinery_gas, wood_tar, water_vapor, cryo_coolant_vented — and the solid byproduct mill_scale. Per the §15.3 flow solver, a producer at a full output bin runs at net 0. So on a long run / offline catch-up, every emitter eventually self-throttles on its own exhaust:

  • co2/co: blast_furnace, coke_oven, steel_mill, smelter, limekiln, oxygen_converter, charcoal_kiln
  • refinery_gas: coke_oven, naphtha_cracker
  • wood_tar: coke_oven, charcoal_kiln
  • water_vapor: brick_kiln, charcoal_kiln
  • cryo_coolant_vented: cryogenic_generator
  • mill_scale: sheet_mill, pipe_mill, beam_mill, cable_mill, bearing_assembler, spring_press, metal_rolling_mill
This is a correctness bug, not cosmetic. It is arguably higher priority than the orphan content — a long-idle steel chain can grind to a halt on a full CO₂/CO bin. Phase 1 fixes it for the whole class.
02

CO₂ as a single global atmosphere

One world-level counter; co2 leaves the inventory system and joins the air/atmosphere exogenous-flow family.

2.1 Data model

  • Add WorldState.atmosphericCo2Kg: number — the single global counter.
  • Remove IslandState.co2Kg (economy.ts:346).
  • Remove co2 from ALL_RESOURCES inventory semantics: drop its RESOURCE_STORAGE_CATEGORY entry and zero its inventory role. It stays a ResourceId only as the label for an atmospheric flow.

2.2 Atmospheric-flow semantics (mirrors air)

The integrator already special-cases air as drawn from the atmosphere, not inventory (economy.ts:647, :1545, :1888: exogenousFlow === 'atmosphere' && id === 'air'). co2 becomes the emission-side twin: 

air   : atmospheric SOURCE — consumed from the atmosphere, never inventory
co2   : atmospheric SINK/SOURCE — credited/debited on the GLOBAL counter, never inventory

emission (recipe output co2)        →  atmosphericCo2Kg += rate·dt
combustion (exogenousFlow CO₂)       →  atmosphericCo2Kg += exogenousFlowKg·rate·dt   (unchanged path)
capture (co2CaptureKgPerCycle)       →  atmosphericCo2Kg −= drainKg                    (was state.co2Kg)
feedstock (recipe input co2)         →  atmosphericCo2Kg −= rate·dt   (the H₂/inputs side stays inventory)

A recipe declaring co2 as an input consumes it from the global counter via this hook — not from inventory (which no longer holds it). So feedstock recipes (§3.2) are "mixed-source": their non-CO₂ inputs come from inventory normally; their CO₂ input debits the atmosphere. This keeps one source of truth.

2.3 Why this kills the throttle

co2 no longer occupies a per-island capped bin, so no emitter can stall on it. The global counter is unbounded — it is the climate-pressure signal. It only ever decreases via capture or feedstock consumption.

weather.ts retarget: sumIslandCo2(world) → return world.atmosphericCo2Kg ?? 0 (or keep the name as a thin alias). No coefficient rescale expected: Path-A and Path-B accrued identically today, so the global counter tracks the same magnitude the climate model already saw — minus the inventory-cap clamping that used to (wrongly) bound it.

03

CO₂ sinks — global capture + feedstock chains

Capture is the bulk sink; feedstock chains give CO₂ a productive reason to exist and reward the climate loop.

3.1 Global capture

Existing co2CaptureKgPerCycle buildings (e.g. exhaust_scrubber, wastewater_treatment) now debit atmosphericCo2Kg instead of the per-island scalar. Capture anywhere lowers the one global number — the well-mixed-atmosphere model (decision #3). No allocation rule needed: there is only one counter to decrement. The co2CaptureAdjacency gate (requires a neighbouring emitter) is preserved.

3.2 Feedstock consumers (new, mixed-source recipes)

Phase 2

Closures — ocean / alloy / chemistry / materials

Decision (locked)
Route each orphan into demand that already exists — one refine step, one recipe-input add, or one placement-cost edit. Mint a new building only where no existing one fits (decision #6).
The engine is 1:1 recipe-per-building, so a new recipe generally needs a building or a resolveRecipe variant. 13

self_replication_module — free-floor tokens vs §4.9

The one orphan whose natural sink isn't another recipe. Decision #9: it becomes a relief valve for the §4.9 exponential upgrade-cost curve.

13.1 The curve it relieves

SPEC §4.9 floor upgrade cost
  floors 2–10 : cost = ceil(0.8 · placementCost)
  floors >10  : cost = ceil(0.8 · 1.15^(L−10) · placementCost)     ← exponential, unbounded
  benefit     : ×(1 + L)                                          ← linear

Past floor 10 you pay 1.15ⁿ more materials for each +1 of linear throughput. That gap is the soft cap that stops infinite floor-stacking.

13.2 Free-floor tokens

Consuming one self_replication_module waives the entire material cost of one floor upgrade. The module is a deep T4 chain (self_replication_lab: ai_core 1 + microchip 8 + electric_motor 4 + computing_module 2), so it's scarce by construction.

Why this preserves the soft cap (resolving the concern on this option): the token waives materials only. Construction time still applies — base_ms × (level + 1), serialized through the island's limited running-build slots — so a free floor-50 upgrade still takes ~51× base, one at a time. And module supply is ai_core-gated (the cryogenic_compute_center chain), bounding lifetime skips. Two of the three soft-cap dimensions — build-time and module scarcity — survive untouched; only the raw-material dimension is relieved. Relief valve, not bypass.

13.3 Mechanics

  • applyUpgrade (placement.ts): if the player opts to spend a token and holds ≥1 self_replication_module, debit one module and set the upgrade's material cost to 0; the build still enqueues / runs through the normal slot + timer path (§4.8 / the stacked-upgrade queue).
  • Eligibility: any floor (players self-select the expensive ones; no L>10 restriction needed since modules are scarce). Flagged for sign-off — see §15.
  • Refund: cancelling a token-funded upgrade refunds the module (consistent with §4.8 100% cancel refund), not materials.
  • SPEC §4.9 gains a "free-floor token" clause; a new UI affordance on the inspector upgrade button toggles "spend module" when one is available.

14

Mass-balance / auditor impact

Decision #8 holds the strict line: every new recipe mass-balances; exogenousFlow tags are reserved for genuine physics (combustion CO₂, atmosphere, evaporation). The recipe-shape changes in P2/P3 (new refine recipes, new inputs on existing recipes) will re-trip the mass-balance auditor — that's expected, and each new recipe must pass it. Phase-1 byproduct loops are the first test of this: a condense (water_vapor → fresh_water) and a reliquefy (cryo_coolant_vented → cryo_coolant) must balance against the original vent ratios.

Pre-work check: before P1, confirm whether the auditor + economy tests would even catch a capped-exhaust stall today (they may not — the throttle is a long-run/offline emergent, not a single-tick failure). If not, add a regression test that runs an emitter to a full former-CO₂ bin. This is the gate that proves the bug existed and is fixed.
