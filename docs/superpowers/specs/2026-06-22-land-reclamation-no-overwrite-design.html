Design spec · robot-islands · §3.4 / §3.6

Land Reclamation must not overwrite already-placed land

Context On a merged island, growing a constituent ellipse via the Land Reclamation Hub overwrites a sibling constituent's already-placed land: the grown ring steals the sibling's tiles and their terrain. The cause is that overlap precedence is resolved by constituent index (“earliest wins”) rather than by placement time. This spec adds a placement-order ownership ledger so the grower yields every tile another constituent already holds.

Decision
Yield, don't overwrite — resolve contested tiles by an append-only placement-order ledger.
Growth still happens, but a contested tile keeps the constituent that inscribed it first in time; the grower gains only genuinely-new ocean tiles — exactly what landReclamationCost already charges for. A per-spec ownershipLedger records each original-and-grown claim in placement order, so the resolution is fully correct including the grown-vs-grown case.
It deliberately does not recover pre-fix overlaps in existing saves (no history exists) — those load unchanged — and it does not bump the persistence schema (the field is additive, optional, and forward/backward compatible).
01

Root cause

What the reader needs to know to evaluate the fix.

Overlap precedence is resolved by constituent index order in three sites that all walk islandConstituents(spec) and take the first inscriber: the attachTerrainAt terrain closure (world.ts:294), constituentBiomeAt (world.ts:234), and the computeIslandTiles dedup (island.ts:233).

performMerge makes the larger island the absorber, so it becomes the primary (index 0) and the smaller island a later extraEllipses entry. Growing the primary — the common case — inscribes tiles the absorbed lobe already owns, and because the primary is scanned first it claims those tiles and overwrites the lobe's land and terrain. Generally: any earlier-index constituent grown into a later-index one steals its tiles.

It is also internally inconsistent: landReclamationCost (land-reclamation.ts:124, SPEC §3.4) already treats “tiles already covered by another constituent” as not charged — so the cost model says the grower did not gain the tile while the renderer says it now owns it. That disagreement is the overwrite.

02

What bounds the fix

Three verified facts that keep the change small and safe.

The fear
  • Precedence affects the tile set, so the footprint/overlap/vision all shift.
  • Needs a per-tile ownership map (heavy persistence).
  • Client and server need separate persistence work.
The reality (verified)
  • Only terrain/biome ownership of contested tiles is wrong — the tile set is always the union, independent of precedence.
  • Original footprints are pairwise disjoint at merge (+1 growth hits adjacency before overlap); overlap only ever arises from post-merge growth.
  • serializeWorld spreads every spec field, and the server shares the same serializeWorld/deserializeWorld — one optional field round-trips everywhere, no schema bump.
03

Data model — the ledger

An append-only list of ownership claims on IslandSpec, in placement order.

// constituent indexes islandConstituents() (0 = primary, N = extraEllipses[N-1])
interface OwnershipClaim {
  readonly constituent: number;
  readonly major: number;
  readonly minor: number;
}
// on IslandSpec / SerializedIslandSpec:
readonly ownershipLedger?: ReadonlyArray<OwnershipClaim>;

Absent ⇒ implicit baseline = constituents in index order at current radii (≡ today's behavior). So single-ellipse and merged-but-never-grown islands store nothing (lean saves, mirroring the route-waypoints “persist only when present” convention), and existing saves render identically.

A constituent may appear multiple times (its baseline claim plus one per later growth). Only consecutive same-constituent claims coalesce — non-consecutive ones stay separate, and that separation is what encodes “this growth came after that other constituent's growth,” making the grown-vs-grown case correct. Invariant: the last claim for each constituent equals its current radii (asserted by a test; the resolver self-heals if violated).

05

Verification & tests

Pure-layer tests in src/; no renderer.

Reported-bug regression. Grow primary toward an adjacent lobe; contested tiles still report the lobe's biome and terrain, and cost charged 0 for them.
Grown-vs-grown ordering. Lobe grows into the gap first, then the primary grows over it; the lobe keeps its reclaimed tiles.
Ledger maintenance. Consecutive same-constituent growths coalesce; interleaved growths stay separate in order; last-claim-equals-current-radii holds.
Merge maintenance. Merging ledgered islands appends and index-remaps (incl. recursive merge); merging never-grown islands leaves the ledger absent.
Union invariant. islandTileCount / islandLocalTiles identical with and without a ledger for the same geometry.
Persistence round-trip. A ledgered spec round-trips byte-identically; a ledger-absent snapshot resolves via the implicit baseline. Then npm test (client + server, needs Postgres) green.

Out of scope

Explicitly deferred — considered, not missed.

Recovering pre-fix overlaps in existing saves
No placement history exists; affected tiles load unchanged. The fix applies from the first post-load growth onward.
Universe Editor constituent reorder/remove
Would invalidate ledger constituent indices; the editor must rebuild or clear the ledger. Tracked separately.
