Skill-tree audit — over-noding, per-node dilution, and a missing efficiency lever

Context Builds on 2026-05-25-skilltree-rebalance-design (v1, schema v14, implemented). Data: live catalog survey via vite-node against src/skilltree.ts (current schema v15). This audit is the foundation for the skill-tree rebalance v2 spec.
v1 fixed the multiplier runaway (×10⁹ → cap every pool at ×10 via calibrated magnitudes) — but by adding nodes (its rule: "N≥5 per effect kind to avoid cliff magnitudes"). That deliberate "many tiny nodes" choice is exactly why per-node gains are now imperceptible (54 power nodes → ×10 = +4.4 %/node). The caps are right; the node counts & per-node feel are wrong, and one core lever — material-input efficiency — doesn't exist. v2 reverses the tradeoff: same ×10 caps, far fewer/punchier nodes, redundant paths merged, plus a new "magical" input-efficiency subpath.
01

How to measure over-noding

Raw effect-kind totals mislead without these three rules — the analysis v1 lacked.

  1. Generic + specific STACK. A generic effect applies on top of every specific variant, so effective nodes for one outcome = generic + that variant (storage generic storageCapMul 16 adds to each storageCategoryCapMul:<cat>).
  2. Group by ECONOMIC OUTCOME, not effect-kind name. Reducing consumer draw and raising producer output are the same lever ("power headroom"); recipe-rate and per-extractor yield both raise extraction throughput.
  3. Cap ≠ count. v1 capped each pool's total at ×10 but left counts high, so per-node = cap^(1/N). At N=54 that's +4.4 %/node — a rounding error, not an upgrade.

Structural root cause: redundant paths boosting one outcome — extraction (category-rate and per-extractor yield), storage (generic and per-category), power (consume and produce). The fix is pick one path per lever, then cap the survivor.
05

Missing lever — material-input efficiency ("magical")

  • Gap: skills scale time (recipeRateMul), yield, power, storage — none scale the input→output ratio (powerConsumptionMul is power, not materials).
  • Proposal: new effect kind recipeInputMul {reduce:true} — multiplies recipe input quantities (outputs unchanged) — as a subpath summing to ÷1.5 input at full (≈33 % less material), across a small number of meaningful nodes.
  • Why "magical": input below output violates the SI kg mass-balance the auditor enforces → must be exempted (like RECIPE_SPECULATIVE). That intentional break is the in-fiction "magic" and the reason to segregate it.
06

v2 thesis & recommendations (deltas from v1)

  1. Keep v1's cap-per-pool framework, the C^(w/N)−1 formula, pool definitions, magnitude CI test, and the ×10 / √10 / ÷10 / ×3-xp caps.
  2. Reverse "N≥5 / many tiny nodes." Contract each pool to ~15–20 nodes; accept punchier per-node magnitudes (the cliffs v1 avoided are now wanted). Re-derive magnitudes for the smaller N.
  3. Merge redundant paths per outcome — keep ONE path per lever and cap it (extraction: per-extractor yield vs generic rate; storage: per-category vs generic; power: already net-capped, just cut the count).
  4. Add the magical input-efficiency subpath (recipeInputMul, ÷1.5, auditor-exempt).
  5. Migrate — removing node ids invalidates unlockedNodes → schema bump + reset the ladder like v1's v13→v14 (preserve buildings/inventory).
  6. Interaction: input-efficiency (÷1.5) + rate (×10) stack with the floor mechanic (×(1+L)) from the throughput pass — verify the combined ceilings are intended.
07

Open questions for the v2 spec

  • Node-count target — flat ~15–20, or core 15–20 / niche 3–4?
  • Per-node band — ~1.2–1.6×/node, or a few hand-curated "notable" jumps + minimal filler?
  • Which path to keep for extraction (category-rate vs per-extractor yield) and storage (generic vs per-category)?
  • Magical subpath size/curve to reach ÷1.5; gated behind a tier?
  • Sequencing — after throughput/floors + tutorial (default), or sooner?
