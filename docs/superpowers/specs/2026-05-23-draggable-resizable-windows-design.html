Design spec · Robot Islands · HUD chrome · v1

Draggable, resizable, persisted HUD windows

Context Today every floating HUD panel auto-docks into a zone via mountPanel(el, { zone, order }) in src/ui-zones.ts. Zones are non-overlapping screen regions (TL/TR/BL/BR/TC/L/R) and within a zone panels stack along the primary axis with a fixed gutter. The player can't move them, can't resize them, and any preference vanishes on reload. This spec defines a small window manager that adds drag + resize + per-browser persistence to every floating panel (modals stay scrim-centered, chrome stays anchored) without breaking the existing zone-stack fallback when no saved layout exists.

Decision
Add a new module src/window-manager.ts that augments every floating panel with header-drag + bottom-right-corner-resize affordances, persists the per-panel { x, y, w, h, zRank } in a versioned localStorage key (ri-ui-layout-v1), and integrates with ui-zones.ts so that “free” (saved-layout) panels are skipped when the zone manager computes its stack cursor.
Mounting stays one-line: mountPanel reads the layout store on insertion; if an entry exists, it applies the saved rectangle and marks the record free (zone stack skips it). If no entry exists, the existing zone-stack layout runs unchanged — everyone with no saved layout sees the current UI.
Modals (mountModal from src/ui-modal.ts) are out of scope: they are scrim-centered overlays with no anchor to persist. Chrome panels — #action-strip (TR) and #island-bar (TC) — are also out of scope by default, because they're orientation anchors the player needs to find at fixed coords.

01

Background — panel inventory + zone system

What exists today, with file:line citations.

Zones are an enum in src/ui-zones.ts:41-56: TL, TR, BL, BR, TC, L, R. Each has a primary axis (ZONE_AXIS, ui-zones.ts:59-62): TL/TR/TC/L/R stack down, BL/BR stack up. A zone's first panel sits flush with the viewport edge plus a VIEWPORT_GUTTER = 12 px gap; each subsequent panel offsets by prev.height + PANEL_GAP (8) (ui-zones.ts:64-65, 121-171).

Placement is auto. The function layoutZone(zone, panels) in ui-zones.ts:121-171 assigns position: fixed + edge anchors based on the zone id. TC uses left: 50%; transform: translateX(-50%) (ui-zones.ts:148-150) — relevant for the chrome-exemption discussion. ResizeObserver (ui-zones.ts:193-194) re-stacks whenever a panel's intrinsic size changes; window.resize re-stacks the whole world (ui-zones.ts:174).

All panels are DOM, not PixiJS. The zone manager appends to document.body if a panel has no parent (ui-zones.ts:181). mountPanel records each panel under its id, stamps data-ri-panel + data-ri-zone attrs, and returns a PanelHandle { el, requestLayout, setVisible, destroy } (ui-zones.ts:178-221).

Existing chrome: .ri-panel in src/ui.css:86-93 declares a translucent dark background (var(--ri-panel) = rgba(14, 18, 26, 0.88)), a 1px var(--ri-border-strong) border, 6px radius, an 8-blur backdrop filter, and var(--ri-shadow-panel) (ui-tokens.ts:78). .ri-panel__head (ui.css:95-108) is the standard 22-px header used by hud-economy; the side-docks (inspector-ui.ts:299-340, drones-ui.ts:179-220, settlement-ui.ts, routes-ui.ts) build identical-looking stamped headers inline. No existing drag or resize hooks; the only drag-drop in the codebase is the cargo-row reordering inside the routes panel (routes-ui.ts:848-919) which is HTML5 draggable, intra-panel, and unrelated.

Modals use a separate primitive (mountModal in src/ui-modal.ts) with a scrim-centered layout, Z.modal = 60 (ui-tokens.ts:92), and an explicit show/hide/toggle handle. Modals are out of scope.

Persistence lives in src/persistence.ts. SaveSnapshot (persistence.ts:239-251) carries { v, savedAt, savedAtPerf, world, islandStates }; no uiLayout field today. Storage key robot-islands:save:v4, schema v5 (persistence.ts:105, 113-159), via idb-keyval. Layout could live there (per-world) or in a separate localStorage key (per-browser) — see Picker 03.

The draggable + resizable mechanic

The exact pointer choreography and the zone-manager handshake that prevents the two systems from fighting for the same panel each frame.

Pointer choreography

Drag uses pointerdown on the header element → setPointerCapture → pointermove updates style.left/top → pointerup writes {x, y, zRank} to the layout store and releases capture. While dragging, document.body.style.userSelect = 'none' and cursor: grabbing. A first-time pointerdown flips the record's free flag in ui-zones.ts's records map; the zone's next layoutZone pass skips the panel from the cursor accumulator, and sibling panels in the zone re-flow upward to fill the gap.

Resize uses the same pointer-capture idiom on a dedicated .ri-resize handle child appended once by window-manager.ts on first mount. The handle is 14×14 absolute-positioned at the panel's bottom-right (right: 0; bottom: 0), with three diagonal cyan stripes drawn via CSS gradients. cursor: nwse-resize. pointermove clamps w ≥ minWidth ?? 200 and h ≥ minHeight ?? 120. pointerup writes the new {w, h} to the store.

Click-to-front fires on any pointerdown inside the panel (capturing, {passive: true}): bump globalZCounter, write the panel's zRank, recompute every panel's style.zIndex = Z.panel + (rank % 20). Modal scrim at Z.modal = 60 (ui-tokens.ts:92) always wins.

Zone-manager handshake: the PanelRecord interface (ui-zones.ts:87-93) gains a free: boolean field. layoutZone(zone, arr) (ui-zones.ts:121-171) filters by !rec.free when computing the cursor stack; free panels are still position: fixed but their top/left/right/bottom are written by the window-manager, not by layoutZone. The existing ResizeObserver firing on a free panel must NOT trigger layoutZone for that panel — the manager gates RO callbacks during an active drag/resize via a per-panel activeMutation flag.

03

Persistence shape + reset

The serialized layout, the storage key, the load/save lifecycle, and the reset surfaces.

Storage: localStorage.getItem('ri-ui-layout-v1') — one key, JSON-encoded value, synchronous read on first mountPanel. Per-browser, world-agnostic. Bump key suffix on schema change.

// Versioned envelope; bump `v` and key suffix in lockstep.
interface UiLayoutBlob {
  readonly v: 1;
  readonly panels: Readonly<Record<string, PanelLayout>>;
  readonly globalZCounter: number;
}

interface PanelLayout {
  /** Pixel offset from viewport top-left, clamped on load. */
  readonly x: number;
  readonly y: number;
  /** Saved pixel size. Honors the panel's per-mount minWidth/minHeight. */
  readonly w: number;
  readonly h: number;
  /** Last-interacted timestamp for stacking; modulo 20 → z-index offset. */
  readonly zRank: number;
}

Load lifecycle: on mountPanel(el, opts), window-manager.ts reads the blob once (cached in module scope) and looks up blob.panels[opts.id]. If found and opts.id is not in CHROME_PANEL_IDS: clamp (x, y) against window.innerWidth/Height, apply style.left/top/width/height, mark the PanelRecord.free = true so the zone manager skips it, and write the saved zRank into the panel's data-zrank. If not found: the panel falls through to the existing zone-stack layout unchanged.

Save lifecycle: pointerup on header drag or .ri-resize handle commits to the in-memory blob and writes to localStorage. Writes are debounced ~250 ms via setTimeout so a fast drag emits one write, not dozens.

Reset: the reset-ui-layout action (registered in input.ts, default Shift+Alt+R) and the Settings panel button both call resetUiLayout() in window-manager.ts. It removes the localStorage key, clears blob.panels in module memory, flips every record's free back to false, clears the inline style.top/left/width/height/zIndex on each affected panel, and calls requestZoneLayout() (already exported at ui-zones.ts:226-228) to re-stack from scratch.

04

Z-order + focus

How overlapping windows resolve.

Base: Z.panel = 20 for every floating panel (ui-tokens.ts:88). Focused window: z-index = Z.panel + (rank mod 20) so focused panels live in [20, 39] — comfortably below Z.toast = 40, Z.modal = 60, Z.tooltip = 80. globalZCounter is the monotonic source: every focus event increments it; each panel's zRank is the counter snapshot at its last focus. Z-index is recomputed across all floating panels on each focus event to keep rank % 20 meaningful when the counter rolls past 20.

Focus event: any pointerdown in the panel's bounding box, captured in capturing phase so it fires even when an inner button stopPropagation()s. Drag and resize both trigger it as a side effect of their own pointerdown.

Chrome interaction: action-strip (TR) and island-bar (TC) keep Z.panel base, never participate in click-to-front (they're not in the focus pool). The HUD economy panel and the four side-docks are the only focus-pool members.

Change inventory

Every file the implementation touches.

src/window-manager.ts (new)
New helper module. Exports makePanelDraggable, resetUiLayout, CHROME_PANEL_IDS, plus the PanelLayout / UiLayoutBlob types. ~250 LOC. Pure DOM + localStorage; no PixiJS, no IDB.
src/ui-zones.ts:87-93, 109-119
Augment PanelRecord with free; filter from cursor stack. Add free: boolean field (default false). In layout(), skip records where rec.free === true when building the per-zone bucket. Export a setFree(id, free) helper so window-manager.ts can flip the bit on first drag.
src/ui-zones.ts:178-221 (mountPanel)
Wire window-manager on mount. After the record is registered and the initial scheduleLayout() runs, call makePanelDraggable(el, opts.id, { minWidth: opts.minWidth, maxWidth: opts.maxWidth }). The helper internally decides whether the panel is in CHROME_PANEL_IDS (no-op) or a window (drag + resize + persistence). mountPanel's API is unchanged.
src/ui.css:86-93, +new rules
Add .ri-resize grip + .ri-panel[data-free="true"] cursor states. The grip uses CSS gradients for the three diagonal stripes; positioned absolute, bottom-right. Header gets cursor: grab via .ri-panel__head, [data-ri-drag-handle]:hover { cursor: grab; } (the new data-ri-drag-handle attr is stamped on the picked-up header element so stamped headers in inspector-ui / drones-ui / etc. participate without each gaining the class).
src/input.ts (default bindings)
Register reset-ui-layout action. Wire to resetUiLayout() from window-manager.ts. Default key: Shift+Alt+KeyR. Conforms to AGENTS.md "every key goes through the registry"; UI button (settings-ui) calls the same dispatcher.
src/settings-ui.ts

06

Helper module signature (src/window-manager.ts)

The complete public surface, typed.

// src/window-manager.ts
//
// Augments floating panels mounted via `ui-zones.mountPanel` with:
//   - header-drag to free-position
//   - bottom-right grip to resize
//   - per-browser persistence to localStorage (key: ri-ui-layout-v1)
//   - click-to-front z-order via a global rank counter
//
// Chrome panels (action-strip, island-bar) are explicitly excluded; modals
// don't go through mountPanel at all so they're naturally out.

/** localStorage key. Bump the trailing `:v1` when changing PanelLayout shape. */
export const LAYOUT_STORAGE_KEY = 'ri-ui-layout-v1';

/** Panels in this set are skipped by makePanelDraggable — they remain
 *  zone-anchored chrome. Drives Picker 07's default. */
export const CHROME_PANEL_IDS: ReadonlySet<string> = new Set([
  'action-strip',
  'island-bar',
]);

export interface PanelLayout {
  readonly x: number;
  readonly y: number;
  readonly w: number;
  readonly h: number;
  readonly zRank: number;
}

export interface UiLayoutBlob {
  readonly v: 1;
  readonly panels: Readonly<Record<string, PanelLayout>>;
  readonly globalZCounter: number;
}

export interface MakeDraggableOptions {
  /** Minimum width in px the user can resize to. Default 200. */
  readonly minWidth?: number;
  /** Minimum height in px the user can resize to. Default 120. */
  readonly minHeight?: number;
  /** Pick a non-default drag-handle child. Default: the element matching
   *  `.ri-panel__head, [data-ri-drag-handle]` inside the panel. */
  readonly handle?: HTMLElement;
}

/** Augment `panel` with drag + resize + persistence. Idempotent: a second
 *  call on the same panel id is a no-op. Called from `mountPanel` after
 *  the panel is in the DOM and registered with the zone manager. */
export function makePanelDraggable(
  panel: HTMLElement,
  id: string,
  opts?: MakeDraggableOptions,
): void;

/** Clear every persisted layout entry; restore zone-stack defaults for
 *  every floating panel. Triggered by the `reset-ui-layout` input action
 *  and the Settings UI button. */
export function resetUiLayout(): void;

/** Pure: clamp a rectangle to the viewport so the header is reachable.
 *  Exported for unit tests; called internally on load and on resize. */
export function clampToViewport(
  rect: { x: number; y: number; w: number; h: number },
  viewport: { w: number; h: number },
  minVisible?: number, // default 80
): { x: number; y: number; w: number; h: number };

/** Pure: validate a JSON-parsed blob; returns null on shape mismatch so the
 *  caller falls back to an empty layout (same pattern as persistence.ts'
 *  `isValidSaveSnapshot`). Exported for tests. */
export function parseLayoutBlob(raw: unknown): UiLayoutBlob | null;

The pure helpers (clampToViewport, parseLayoutBlob) are testable without a DOM. The impure makePanelDraggable / resetUiLayout side-effect on the live localStorage and on the DOM, and are covered by Daedalus visual-smoke per AGENTS.md's preview-server doctrine (no HMR; build, reload, screenshot).

08

Risks

Risk
Sev
Mitigation
ResizeObserver loop on a free panel. The zone manager registers a ResizeObserver per panel that calls scheduleLayout() on size change (ui-zones.ts:193-194). If a user resize fires the RO and the RO re-triggers layoutZone, the panel's style.left/top could be reset under the player.
MED
Make layout() early-out per record when rec.free === true for that zone, AND have window-manager.ts set el.dataset.riActiveMutation = '1' on pointerdown / clear on pointerup. The RO callback bails if the dataset flag is set.
Header drag conflicts with the in-header close button. Every stamped header has a child × button (inspector-ui.ts:342, drones-ui.ts:222, settlement-ui.ts:202, routes-ui.ts:154) and routes-ui has list-row HTML5 drag.
MED
pointerdown handler returns early if event.target.closest('button, input, select, textarea, [draggable="true"]') !== null. Tested via the verification "inner interactions" case.
localStorage quota / disabled. Some browsers throttle, some users disable storage. localStorage.setItem can throw QuotaExceededError.
LOW
Wrap every setItem in try/catch; on failure, console.warn and keep the in-memory blob. Layout still works for the session; only persistence is lost. Same pattern as persistence.ts's save fallthrough (persistence.ts:938).
Touch / tablet handling. pointerdown covers touch and mouse uniformly, but pinch-zoom can fire pointermove events with two pointers.
LOW
pointerType === 'mouse' || event.isPrimary === true gate. Multi-touch on tablets is deferred to a follow-up (§09).
TC chrome panel uses transform: translateX(-50%). If someone adds island-bar to the in-scope set later without clearing the transform, dragging it will be off by half its width.
LOW
Solved by the chrome blocklist default. Documented in the CHROME_PANEL_IDS comment in window-manager.ts.
z-index overflow. A long session of constant focus-switching grows globalZCounter without bound.
LOW
Modulo by 20 keeps the actual z-index bounded. The rank counter itself grows but is a 53-bit integer — overflow is >10^15 clicks away.
09

Out of scope

Explicitly deferred. Named so the reader knows it was considered, not missed.

Modals (skilltree, inventory, settings, buildings, construction, orbital, graph)
Mounted via mountModal in src/ui-modal.ts; scrim-centered with a fixed .ri-modal max-width. The drag/resize/persistence story for modals is a different shape (centered-by-default, occasional resize) and a separate spec.
Chrome panels (action-strip, island-bar)
Orientation anchors. Picker 07's recommended default. If a future user-research note shows people want to move them, the unlock is to remove the id from CHROME_PANEL_IDS — and for island-bar, also clear the transform: translateX(-50%) on first drag.
Touch / mobile multi-pointer
The Robot Islands UI is implicitly desktop-only today (keyboard registry, hover-tooltip, side docks). Multi-touch drag-resize is its own follow-up. Single-pointer touch works via pointerType === 'touch' and isPrimary; pinch-zoom + drag-at-the-same-time is out.
Snap-to-zone (Picker 04 alternative)
"If dropped within N px of a zone anchor, re-dock." Magnetic feel, but not necessary for the minimum useful behaviour. Available as a follow-up if playtest shows the clamp-only default is too sticky.
Layout import / export
Power-users moving between machines could export the JSON blob and paste it on the other browser. Trivial to add later (one button next to the Settings Reset). Not in v1.
Edge resize handles (left/right/top/bottom only)
Picker 02's "corners+edges" option is deferred. Bottom-right grip covers the actual ergonomic need (panels never need to grow upward in this game).
