01

Problem

Two engines violate energy/momentum conservation. v3 audit findings F16 (heat) and F18 (drag).

Heat — F16

Current heat.ts at heat.ts:124-205 implements an N:1 adjacency gate: a single Plasma Heater serves any number of adjacent Coke Ovens at zero incremental cost. A 200 W Plasma Heater (well, 200 kW under the rev-4 anchor) producing 184 kW thermal serves N×60 kW Coke Ovens at full throughput, regardless of N. Energy violation by design. Real industrial thermal coupling has finite supply; deficit propagates as proportional throttling.

Drag — F18

Current SHIP_STATS at settlement.ts:119:

export const SHIP_STATS: Record<VehicleTier, VehicleStats> = {
  1: { speed: 0.25, tilesPerFuel: 0.5,  maxKits: 1, failureRate: 0.02,  weatherMultiplier: 1.0 },
  2: { speed: 0.30, tilesPerFuel: 2,    maxKits: 2, failureRate: 0.015, weatherMultiplier: 0.9 },
  3: { speed: 0.40, tilesPerFuel: 3.5,  maxKits: 2, failureRate: 0.01,  weatherMultiplier: 0.8 },
  4: { speed: 0.50, tilesPerFuel: 5,    maxKits: 2, failureRate: 0.005, weatherMultiplier: 0.7 },
};

Both speed AND tilesPerFuel grow across tiers — fuel/sec at T4 vs T1 is (0.50/5) / (0.25/0.5) = 0.20 — fuel-per-second drops by 5× while speed doubles. This violates cruise drag (P ∝ v³ for ships and helicopters per rev-16 §6 + Geography of Transport Systems). Real cruise drag means T4 ship should burn ~64× more fuel/s than T1 at the new T4 speed.

02

Heat throttle algorithm

Rev-16 §5.1 BEFORE/AFTER pseudocode applied verbatim. The new algorithm has three phases: discover adjacency edges (unchanged), compute per-source budgets (new), apply throttle in computeRates (new).

2.1 New HeatAssignments shape

// heat.ts:52 — extend HeatAssignments with the throttle map.
export interface HeatAssignments {
  readonly hasHeat: Map<string, boolean>;
  readonly coalConsumersByFurnace: Map<string, number>;
  readonly assignedSource: Map<string, string>;
  /** Per-consumer throttle factor ∈ [0, 1] from the rev-16 §5.1
   *  proportional-budget resolver. Missing key reads as 1.0 (the
   *  backward-compat fallback for consumers without heatDemandKW). */
  readonly heatThrottleFactor: Map<string, number>;
}

2.2 Resolver — proportional throttle

Append a fourth phase to resolveHeatAssignments (after the existing priority A/B/C assignment): walk per-source consumer lists, sum demand, compute throttle.

// heat.ts — appended to resolveHeatAssignments after the existing loop.
const heatThrottleFactor = new Map<string, number>();

// Group consumers by their assigned source.
const consumersBySource = new Map<string, PlacedBuilding[]>();
for (const consumer of sortedConsumers) {
  const srcId = assignedSource.get(consumer.id);
  if (!srcId) continue;  // no source → hasHeat already false; throttle stays absent
  const list = consumersBySource.get(srcId) ?? [];
  list.push(consumer);
  consumersBySource.set(srcId, list);
}

// For each source: sum demand, compute ratio, propagate per-consumer.
for (const [srcId, consumers] of consumersBySource) {
  const src = buildings.find((b) => b.id === srcId)!;
  const srcDef = BUILDING_DEFS[src.defId];
  const supply = srcDef.heatSource?.thermalKW;

  if (supply == null) {
    // Backward-compat: source has no thermalKW → boolean N:1 behaviour preserved.
    // Every consumer this source serves throttles at 1.0.
    for (const c of consumers) heatThrottleFactor.set(c.id, 1);
    continue;
  }

  const demand = consumers.reduce((acc, c) => {
    const d = BUILDING_DEFS[c.defId].heatDemandKW ?? 0;
    return acc + d;
  }, 0);

  if (demand <= 0) {
    // No measured demand from any consumer → boolean-equivalent: all run.
    for (const c of consumers) heatThrottleFactor.set(c.id, 1);
    continue;
  }

  const ratio = Math.min(1, supply / demand);
  for (const c of consumers) {
    // Multiple sources potentially feeding the same consumer: take the max ratio.
    const prev = heatThrottleFactor.get(c.id) ?? 0;
    heatThrottleFactor.set(c.id, Math.max(prev, ratio));
  }
}

return { hasHeat, coalConsumersByFurnace, assignedSource, heatThrottleFactor };

2.3 Boolean fallback contract

Source without thermalKW: every consumer gets throttle 1.0 (existing behaviour preserved). Consumer without heatDemandKW: consumer's contribution to demand is 0; doesn't affect other consumers' ratio computation; the consumer itself runs at 1.0. MIN_HEAT_FACTOR threshold: per rev-16 §5.1 pseudocode, consumers below a minimum ratio (e.g. 0.1) are gated off entirely. Phase 3 ships MIN_HEAT_FACTOR = 0.1 as a const in heat.ts; the threshold is exposed for tests but not configurable at runtime.

04

computeRates integration

The boolean hasHeat gate in computeRates becomes a multiplicative heatThrottleFactor applied to recipe rates.

4.1 Current gate

economy.ts's computeRates currently treats heat as a binary on/off — a consumer with requiresHeat: true and hasHeat: false contributes 0 to output rates. The Phase 3 change keeps the boolean for the deficit-to-zero case (consumer hits below MIN_HEAT_FACTOR) and adds the multiplicative throttle for the partial-deficit case.

// economy.ts — inside computeRates, where the existing hasHeat gate sits.
// BEFORE:
if (def.requiresHeat && !heatAssignments.hasHeat.get(b.id)) {
  recipeRateFactor = 0;
}

// AFTER:
if (def.requiresHeat) {
  const factor = heatAssignments.heatThrottleFactor.get(b.id) ?? 0;
  if (factor < MIN_HEAT_FACTOR) {
    recipeRateFactor = 0;  // brownout — consumer cannot operate
  } else {
    recipeRateFactor *= factor;  // throttle the recipe rate
  }
}

4.2 The boolean hasHeat stays

The existing inspector UI reads hasHeat to render the "no heat source" tooltip. The boolean stays as a derived value: hasHeat === (heatThrottleFactor >= MIN_HEAT_FACTOR). The resolver derives it from the throttle map after the proportional-budget loop completes.

Drag retable — cubic invariant

Rev-16 §6.2 SHIP_STATS and HELICOPTER_STATS tables replace the existing values. Per-tier maxKits, failureRate, weatherMultiplier stay unchanged (those are progression knobs, not drag).

5.3 Verification arithmetic

Ship T1: 0.25² × 0.50 = 0.03125. Ship T4: 1.00² × 0.0313 = 0.0313 ≈ 0.03125 ✓. Heli T1: 0.55² × 0.40 = 0.121. Heli T4: 1.85² × 0.0354 = 0.1212 ✓. Fuel/sec scaling T4/T1: ship (1.00/0.0313)/(0.25/0.5) = 32 / 0.5 = 64 ✓; heli (1.85/0.0354)/(0.55/0.4) = 52.3 / 1.375 = 38 ✓.

5.4 maxKits / weatherMultiplier deltas

Per rev-16 §6.2: Ship T4 maxKits bumps 2 → 3 (the only progression change beyond drag); heli weather multiplier T2 0.9 → 1.1 stays (heli is weather-vulnerable; rev-16 adjusts but keeps the spirit). The values above are verbatim from rev-16 §6.2 tables.

06

Solar — deferred to Phase 6

Rev-16 §7.1 spec'd a sinusoidal 22% CF curve. The mid-stream user directive reroutes this work to Phase 6 via real-astronomy SunCalc (lat/lon-driven). Phase 3 leaves daynight.ts alone.

Why defer
Phase 6's spec replaces the trapezoidal curve directly with the real-astronomy curve (latitude-dependent day length, real sunrise/sunset, polar regions). Shipping rev-16 §7.1's sinusoidal as a Phase 3 intermediate would mean: write code + tests for sin-scale → re-verify everything → throw away in Phase 6. Net wasted: ~3 h of subagent + ~1 h of review per the original §7.1 scope. Skipping the intermediate saves both.

The Phase 1 playerLat / playerLon fields are in the schema and persisted as null. Phase 3 does not touch them; Phase 6's map picker provides the first non-null values.

Existing trapezoidal preserved. solarMultiplier at daynight.ts:83 is unchanged through Phase 3. Phase 6's commit replaces the body wholesale.

08

Test plan

8.1 Heat proportional throttle

Three fixtures: (a) plasma_heater (184 kW) + 3 coke_ovens (60 kW each = 180 kW demand) → throttle 1.0 for all three; (b) plasma_heater + 4 coke_ovens (240 kW demand) → throttle 184/240 = 0.767 for all four; (c) plasma_heater + 5 coke_ovens (300 kW demand) → throttle 0.61 for all five.

8.2 Heat boolean fallback

A heat source without thermalKW (e.g. a hypothetical un-tabled fantasy building) serving 2 consumers without heatDemandKW still produces hasHeat: true for both, throttle 1.0. Pre-Phase-3 boolean behaviour preserved.

8.3 Brownout at MIN_HEAT_FACTOR

fusion_core + 1 blast_furnace (1 GW vs 3 kW demand) → throttle 1.0 (capped). coal_furnace + 1 blast_furnace (830 kW vs 3000 kW) → throttle 0.277 → above the 0.1 threshold; consumer runs at 27.7% rate. coal_furnace + 5 blast_furnaces → throttle 0.055 → below threshold; all five brownout.

8.4 Drag cubic invariant

for tier in 1..4: expect(speed² × tilesPerFuel ≈ invariant).toBeCloseTo(0.03125, 3) for ships, same for helis at 0.121. Catches future tier-data drift.

8.5 Existing-suite continuity

npm test stays green. Settlement tests that asserted specific fuel-per-trip values will fail and get updated to the new tables — about ~6-12 such tests per grep.

8.6 Save-replay sanity

scripts/profile-economy.ts /tmp/robot-islands-save.json — runs without NaN, with sensible inventory deltas. Vehicles with existing in-flight trips finish per the new fuel curve.

Non-goals

Solar curve replacement
Phase 6 replaces the trapezoidal curve with real astronomy (SunCalc + lat/lon). Phase 3 leaves daynight.ts alone.
CO₂ → weather coupling
Phase 6 reads state.co2Kg + applies storm-probability thresholds.
Power-output retune (coal_gen, biomass_plant)
Per rev-16 §10.3, coal_gen should output 50 unit (= 5 MW) at the current 2-s burn rate vs the existing 100 unit. Phase 3 doesn't touch power.produces — it's a separate balance call that lives in Phase 6's full-suite retune.
Per-building thermalKW backfill beyond the §5.3 table
Fantasy heat sources (e.g. neutronium-something) don't have rev-16 §5.3 entries. Phase 3 leaves them on the boolean fallback. Later phases may populate them as the corresponding fantasy chains land.
UI surfacing of heatThrottleFactor
The inspector UI surface for "consumer is at 60% throttle" lands separately — the rev-16 §5 algorithm produces the data, but the inspector tooltip rewrite is out of scope for this phase. Existing "hasHeat: false" tooltip remains for full-brownout cases.
