Remove redundant manufacturing stubs & sync SPEC §8.3

Context A building-catalog audit flagged 8 buildings as broken. Six verification agents re-traced every claim: the 4 storage buildings are fully functional (audit error, corrected in audit v4), and terrain_modifier is a genuine unbuilt mechanic (deferred). The remaining 3 — fabricator, precision_lab, singularity_forge — are not "missing recipes": they are coarse SPEC §8.3 catalog rows that the implementation already replaced with a finer-grained taxonomy. This spec removes them and brings §8.3 in line with the code.

Decision
Delete the 3 stub defs; rewrite SPEC §8.3 to enumerate the implemented manufacturing taxonomy.
Why remove rather than wire. Every output these buildings could produce is already produced by a dedicated building — wiring a recipe into fabricator would create a literal duplicate of motor_assembly. The buildings are spec-vs-implementation drift artifacts, not feature gaps. The cleanest, evidence-aligned outcome is deletion plus a SPEC correction so the locked spec stops naming buildings the code doesn't have.
What this deliberately does not do. It does not touch terrain_modifier (a real unbuilt mechanic — its own design cycle), does not alter the 4 storage buildings (already functional), and does not attempt to fix the parallel drift in SPEC §8.1 / §8.2 (also stale — out of scope, noted in §07).
01

Background

How three "unimplemented buildings" turned out to be three buildings the codebase already replaced.

The recipe model is one building → one recipe: resolveRecipe() returns RECIPES[def.id], keyed by the building's own defId. A manufacturing building with no matching RECIPES entry is skipped entirely by the economy tick (economy.ts:800, if (!recipe) continue;) — it produces nothing, draws no power, gates nothing.

SPEC §8.3 lists Fabricator ("advanced components, motors, actuators"), Precision Lab ("circuit boards, computing modules") and Singularity Forge ("T4 endgame artifacts") as coarse, theme-named rows. The implementation never used them. Instead it built a fine-grained taxonomy — one building per output — and assigned the recipes there. The coarse rows were left in BUILDING_DEFS as placeholders ("recipe assignment lands in a later balance pass") and never reconciled. They are placeable (the catalog UI iterates every BUILDING_DEFS entry) but the catalog card renders "— no recipe" and the building does nothing once placed.

02

The coarse rows are already covered

Each SPEC §8.3 coarse row maps onto buildings that already exist, already have recipes, and already produce the goods. There is no production gap.

SPEC §8.3 coarse row (stub)
  • Fabricator — T3 "motors, actuators, advanced components"
  • Precision Lab — T3 "circuit boards, computing modules"
  • Singularity Forge — T4 "endgame artifacts"
What the implementation actually built
  • motor_assembly, generator_lab, pump_assembly, hydraulic_assembly, pneumatic_assembly — all T3, all with recipes
  • circuit_assembler, processor_fab, compute_module_fab (displayName: "Computing Module Fabricator"), memory_lab
  • SPEC §7.11's five artifacts — ai_core, antimatter_capsule, exotic_alloy, time_crystal, self_replication_module — each already has a producer

The tell: one of the replacement buildings is literally named "Computing Module Fabricator". "Fabricator" survived as a word inside the fine-grained taxonomy; only the coarse catalog row was orphaned.

Verification

grep -rn 'fabricator\|precision_lab\|singularity_forge' src/ — only legitimate substrings remain (e.g. compute_module_fab); zero references to the removed defIds.
SPEC §8.3 cross-check — every defId in the new table resolves in BUILDING_DEFS; no removed stub reappears; §1272 names a building that exists.
06

Risks

Risk
Sev
Mitigation
A persisted save with one of the 3 buildings placed would hold a PlacedBuilding whose BUILDING_DEFS lookup is now undefined, risking a crash in render/inspect code.
LOW
Accepted, not handled. These buildings were never functional, so a save containing one is implausible; the project is pre-release with no migration infrastructure. Noted here so the decision is explicit, not overlooked. Flag for a guard if real saves exist.
SPEC.md is the "locked specification" (AGENTS.md). Editing it risks drift from its hypothesize-prove provenance.
LOW
The edit makes the spec match shipped reality — it removes drift rather than introducing it. Scoped to §8.3 + the §1272 sentence; §8.3 prose intro and all other sections untouched.
07

Out of scope

Considered and deliberately deferred — named so the reader knows they were not missed.

terrain_modifier — its own design cycle
A genuine unbuilt mechanic (tile conversion + a UI action), not a duplicate. Far larger than this task; deferred per the scoping decision. The def stays in building-defs.ts as a placeholder until that cycle runs.
SPEC §8.1 / §8.2 drift
The Extraction and Smelting/Refining catalog tables are stale against the code in the same way §8.3 was. Fixing them is a separate effort; this spec touches §8.3 only.
The 4 storage buildings
silo, tank, cold_storage, component_warehouse — verified fully functional (audit v4 corrected the v3 "orphaned" error). No change needed.
