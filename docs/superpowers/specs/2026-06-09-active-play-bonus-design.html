Active-Play Production Bonus — a global recipe-rate buff that grows with focused play

Context Trading (SPEC §9.8) already rewards presence: trade cooldowns burn down only while the tab is visible and focused. This spec adds a second active-play reward with a different shape — every focused minute adds +0.1% to every recipe rate on every island, and the bonus leaks away at 3× that rate whenever the player is gone. Brainstormed and approved with the user on 2026-06-09; approach A (world-level accumulator, frame-sampled multiplier) selected.

Decision
A world-level activeBonusMs accumulator, ticked by the same online signal as trading, sampled per frame as a constant multiplier in RatesContext.
The bonus drifts 0.1% per minute, so treating it as constant within one frame (or one offline-catchup call) introduces imperceptible error and leaves §15.3's constant-rate piecewise integration untouched. Exact time-varying integration (approach B) was rejected as invisible precision at real complexity cost; an online-time module refactor (approach C) was rejected as a refactor of working trade plumbing we don't need yet.
Deliberately not done: no cap (user decision — decay is the only counterweight), and no anti-idle detection (focused-but-idle farming is the same accepted limit as trade's online-time).
01

The rule

One unified accrual/decay law covers every focus-loss mode.

Focused frame-dt accrues (clamped); every other wall-clock millisecond decays the balance at 3×. Per ticker frame, with online = document.visibilityState === 'visible' && document.hasFocus() (the boolean main.ts already computes for trading) and frameDt = wall-clock ms since the last frame:

accrued       = online ? min(frameDt, ONLINE_DT_CAP_MS) : 0   // cap = 3000ms, from trade.ts
activeBonusMs = max(0, activeBonusMs + accrued − 3 × (frameDt − accrued))

bonusFraction = activeBonusMs / 60_000 × 0.001                // +0.1% per focused minute
activeBonusMul = 1 + bonusFraction                            // no cap
Focus-loss modeWhat happens mechanically
Blurred but visiblerAF keeps firing; each frame's full dt decays at 3×.
Hidden tab / minimizedrAF stops; the gap arrives as one large frameDt on the refocus frame — accrual clamps to 3 s, the remainder decays.
Game closedAt load, the same formula runs with online = false and frameDt = now − snapshot.savedAt, before offline catch-up advances islands.

Risks & balance

Risk
Sev
Mitigation
Uncapped growth: a 10 h focused marathon yields +60% on every recipe.
MED
User-accepted. Decay (−18%/h away, including closed) makes it effectively a same-session mechanic; overnight ⇒ 0.
Focused-but-idle farming (window focused, player absent).
LOW
Same accepted limit as trade online-time — "covered but focused" isn't JS-detectable.
Frame-sampled multiplier vs. exact integration drift.
LOW
Drift is 0.1%/min; within-frame error ~0.0017%/s. Invisible; preserves §15.3 constant-rate invariant.
05

Out of scope

Shared online-time module (approach C)
Extracting trade's visible+focused/online-dt plumbing into a common module. Revisit if a third active-play mechanic appears.
Anti-idle / input-based activity detection
Out of scope; focus is the agreed proxy for presence, consistent with trading.
