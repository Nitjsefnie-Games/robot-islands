Pseudocode (per spec §10)

// economy.ts — inside the segment integrator's per-building accrual loop.
// The existing loop iterates over each producing building's per-cycle
// output rates; we hook CO₂ here.

for (const b of producingBuildings) {
  const recipe = resolveRecipe(b, state, nowMs);
  if (!recipe) continue;
  const rates = perBuildingOutputRates[b.id];
  if (!rates) continue;

  for (const [outputId, rate] of Object.entries(rates)) {
    // …existing inventory accrual: state.inv[outputId] += rate * segDur…
    if (outputId === 'co2') {
      state.co2Kg += rate * segmentDurationSec;
    }
  }

  // Exogenous-flow CO₂: recipes whose fuel burn produces CO₂ outside
  // the recipe's input/output ledger. Magnitude = per-cycle constant
  // × cycles-this-segment, where cycles = primary-output-rate / primary-yield.
  if (recipe.exogenousFlow === 'fuel-combustion-CO₂' && recipe.exogenousFlowKg) {
    const primary = primaryOutput(b, recipe);  // e.g. 'quicklime' for limekiln
    const primaryRate = rates[primary] ?? 0;
    const primaryPerCycle = recipe.outputs[primary] ?? 1;
    const cyclesThisSegment = (primaryRate * segmentDurationSec) / primaryPerCycle;
    state.co2Kg += recipe.exogenousFlowKg * cyclesThisSegment;
  }
}

Why not just use the primary recipe rate directly?
The cleanest expression of "how many cycles this segment" is recipeRate × segDur, where recipeRate is the rate of cycles, not the rate of any specific output. Pseudocode — per spec §11

// src/mass-balance.test.ts
import { describe, it, expect } from 'vitest';
import {
  RECIPES, RESOURCE_META, RECIPE_SPECULATIVE,
  type RecipeId, type ResourceId,
} from './recipes';

describe('mass-balance auditor — every recipe in RECIPES', () => {
  for (const [recipeId, recipe] of Object.entries(RECIPES)) {
    if (!recipe) continue;

    it(`${recipeId} mass-balances`, () => {
      // Speculative recipes (fantasy chemistry) are exempt.
      if (RECIPE_SPECULATIVE[recipeId as RecipeId]) return;

      const sumMass = (m: Record<string, number | undefined>) =>
        Object.entries(m).reduce(
          (acc, [r, n]) =>
            acc + (n ?? 0) * RESOURCE_META[r as ResourceId].massPerUnitKg,
          0,
        );

      const inputMass  = sumMass(recipe.inputs);
      const outputMass = sumMass(recipe.outputs);
      const delta = outputMass - inputMass;

      // exogenousFlow declares a legitimate Δ. Auditor accepts ±5% input mass + 0.5 kg.
      if (recipe.exogenousFlow) {
        expect(
          Math.abs(delta),
          `${recipeId} declared exogenousFlow=${recipe.exogenousFlow} but |Δ|=${Math.abs(delta)} exceeds tolerance`,
        ).toBeLessThanOrEqual(inputMass * 0.05 + 0.5);
        return;
      }

      // No exemption — exact mass balance (float tolerance).
      expect(
        Math.abs(delta),
        `${recipeId} Δ = ${delta.toFixed(4)} (inputMass=${inputMass}, outputMass=${outputMass})`,
      ).toBeLessThan(0.001);
    });
  }
});

Implementation notes

Practical context for the implementer that didn't rise to the level of a spec requirement.

  • Commit cadence. 11 commits across ~24 h budget = ~2 h/commit average. Phase 7 (cohort + renames) is the largest single commit (~3 h); Phase 0 + Phase 11 are small (no source + measurement-only respectively).
  • Order matters. Phase 1 (resources) and Phase 2 (Recipe interface) MUST land before Phase 3+ — they're the vocabulary every later commit consumes. Phase 8 (CO₂ hook) MUST land before Phase 11 (replay would otherwise leave state.co2Kg at 0). Phase 9 (auditor) MUST land before Phase 10 (sweep) — the auditor reveals which fixtures are stale.
  • Read Phase 1 source first. Several fields (state.co2Kg, RESOURCE_META, RECIPE_SPECULATIVE) are Phase 1 introductions. Phase 0 grep confirms they exist; the implementer should also read the Phase 1 commit's diff to understand the shapes before consuming them.
  • cycleSec is preserved. Rev-16 §9.1 explicitly keeps the existing round-number cycleSec set ("for player legibility"). Every recipe rewrite preserves its current cycleSec — only inputs and outputs change.
  • Multi-output recipes already exist. The integrator handles them today (steel_mill produces 4 outputs in some current rewrites). The new multi-output recipes (coke_oven → 5 outputs, naphtha_cracker → 5 outputs) reuse the same mechanism — no integrator change for output multiplicity, only the new CO₂ side-effect in Phase 8.
  • The biogenic flag is forward-looking. Phase 2 sets biogenic: true on charcoal_kiln (and optionally on biomass_plant if it produces CO₂) but does NOT apply a discount. Phase 6 weather coupling will read it.
  • Don't add unjustified exogenousFlow tags. Each tag corresponds to a real-world process per spec §9.2. If a recipe fails the auditor, the fix is either (a) correct the stoichiometry, (b) verify the recipe maps to a rev-16 §3.8 worked example with a tag, or (c) escalate. Adding a tag to silence an auditor failure is the regression vector this whole rework is designed to prevent.
  • Persistence stays at v15. Phase 1 bumped the schema; Phase 2 ships zero schema changes. SUPPORTED_LOAD_VERSIONS stays at [15] (plus whatever migration paths Phase 1 added).
  • Subagent brief contract. The lead session executes Phase 2 via kimi -p <phase-brief> per commit (typically). Each phase brief includes: spec section reference, file:line citations from the plan, the verify gate, and the commit message template. Phase 9 + Phase 11 binding gates must NOT be silenced if they fail; the subagent reports back to the lead instead of force-passing.
  • The 25 worked examples are gold. Rev-16 foundation §3.8 (~line 1104+ of the foundation HTML) contains the canonical Δ for every audited recipe. When the auditor reports a Δ, the first cross-check is the §3.8 row; if they match, the recipe is correct (whether Δ is 0 or, e.g., −2 with a tag); if not, the recipe needs a fix.
