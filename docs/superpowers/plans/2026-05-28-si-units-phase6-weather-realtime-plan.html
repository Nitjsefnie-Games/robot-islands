2.1 New solarMultiplier

// daynight.ts — replace function body at line 83.
import SunCalc from 'suncalc';

/** Real-astronomy solar insolation factor at the player's geographic
 *  location and the current wall-clock time. Returns 0 below horizon,
 *  sin(altitude) ∈ [0, 1] above. Drives `computeRates`'s solar power
 *  multiplier per SPEC §15.3 the same way the trapezoidal curve did.
 *
 *  When either lat or lon is null (no picker run yet), returns 0 —
 *  solar buildings produce nothing until the player picks a location.
 *  main.ts blocks bootstrap on the picker so this only fires for
 *  fixture / test paths that explicitly pass null. */
export function solarMultiplier(
  nowMs: number,
  lat: number | null,
  lon: number | null,
): number {
  if (lat == null || lon == null) return 0;
  const pos = SunCalc.getPosition(new Date(nowMs), lat, lon);
  if (pos.altitude <= 0) return 0;
  return Math.sin(pos.altitude);
}

2.2 Uniform-grid nextSolarBoundaryMs

The old function dispatches on quadrant — Day/Night flat segments fall through to nextPhaseBoundaryMs, Dawn/Dusk sub-segment at SOLAR_RAMP_SEGMENTS = 8 ticks. The new astronomy curve is continuously varying over the full 24h, so the right replacement is a uniform-grid sub-segment across the whole day:

// daynight.ts — replace function body at line 131.
export function nextSolarBoundaryMs(nowMs: number): number | null {
  const totalSegmentsPerDay = SOLAR_RAMP_SEGMENTS * 4;  // 32 per day
  const segmentMs = DAY_DURATION_MS / totalSegmentsPerDay;
  const nowSegment = Math.floor(nowMs / segmentMs);
  return (nowSegment + 1) * segmentMs;
}

Why uniform vs astronomy-derived boundaries. Tying the boundary integrator to SunCalc's sunrise/sunset times would chain integrator precision to library behaviour and complicate the segment math (sunrise/sunset move continuously with date and latitude). Uniform-grid keeps the integrator self-contained — worst-case integration error is bounded by (slope × segment_width) / 2 ≈ segment_width × 0.5 which at 32 segments/day equals 45 min × 0.5 = 22.5 min of sin drift, well within the §15.3 noise floor.

Note on SOLAR_RAMP_SEGMENTS: The constant at daynight.ts:73 retains the name for compatibility with the integrator's tag. The comment above it should be reworded to "sub-segments per 6h quadrant of the astronomy curve" (still 8; total 32/day). Update the JSDoc.

2.3 Existing exports retained

  • dayPhase, dayPhaseName, nextPhaseBoundaryMs, DAY_DURATION_MS, QUADRANT_MS, DayPhase, SOLAR_RAMP_SEGMENTS — all unchanged. The phase-of-day abstraction is used by weather.ts (severe-storm dawn/night boost) and the maintenance scheduler; both want gameplay-rhythm phases, not astronomy.

4.3 Equirectangular conversion

The SVG uses viewBox="0 0 360 180": every viewBox unit = 1 degree. Click coordinates relative to the SVG bounding rect are converted with:

const rect = svg.getBoundingClientRect();
const xPx = e.clientX - rect.left;
const yPx = e.clientY - rect.top;
// viewBox 0..360 = lon -180..180; 0..180 = lat 90..-90.
const lon = (xPx / rect.width) * 360 - 180;
const lat = 90 - (yPx / rect.height) * 180;

Range guarantee. Clicks land inside the rect (the SVG fills the modal at fixed aspect); lat ∈ [-90, 90], lon ∈ [-180, 180] mathematically. Clamp defensively in case of sub-pixel rounding at corners:

const clampedLat = Math.max(-90, Math.min(90, lat));
const clampedLon = Math.max(-180, Math.min(180, lon));

6.1 CO₂ multiplier function

// weather.ts — module-level helpers.
/** §7.3: CO₂-band multiplier on storm-weight summing.
 *  Bands: <100kg = 1.0; <10t = 1.1; <100t = 1.3; ≥100t = 1.6 (crisis). */
export function co2WeatherMultiplier(totalCo2Kg: number): number {
  if (totalCo2Kg < 100) return 1.0;
  if (totalCo2Kg < 10_000) return 1.1;
  if (totalCo2Kg < 100_000) return 1.3;
  return 1.6;
}

export function sumIslandCo2(world: WorldState): number {
  let sum = 0;
  for (const [, state] of world.islandStates) {
    sum += state.co2Kg ?? 0;
  }
  return sum;
}

6.3 Heatwave event type

The heatwave is a new WeatherState? Or a parallel event flag? Per spec §5.3 it's a "new weather event type" that fires on a per-day roll independent of the storm-tier state — best modeled as a separate flag on the world (or per-island) that the renderer + economy read alongside the current WeatherCell:

// weather.ts — new export.
export interface HeatwaveState {
  /** True for the full game-day this event is active. */
  readonly active: boolean;
  /** Wall-clock ms when the heatwave started. */
  readonly sinceMs: number;
  /** Wall-clock ms when the heatwave ends (sinceMs + DAY_DURATION_MS). */
  readonly untilMs: number;
}

export function rollHeatwave(
  seed: string,
  dayIndex: number,
  totalCo2Kg: number,
): HeatwaveState | null {
  if (totalCo2Kg < 10_000) return null;
  const rng = makeSeededRng(`${seed}_heatwave_${dayIndex}`);
  if (rng() >= 0.05) return null;
  const sinceMs = dayIndex * DAY_DURATION_MS;
  return { active: true, sinceMs, untilMs: sinceMs + DAY_DURATION_MS };
}

Deterministic by seed + day. Same idiom as weather() — RNG keyed by save seed + day index — so the heatwave at a given (CO₂, day) is replayable across save/load cycles. The 5% per-day probability fires only when totalCo2Kg ≥ 10_000 per spec §5.3.

6.4 Effect application

  • Crop yield (×0.8). Apply at the recipe-resolution layer for crop-bearing recipes (plant_a_tree, orchard, whatever the catalog names them — grep building-defs.ts for crop-like producers).

Both consumers are inside economy.ts. The heatwave roll itself lives in weather.ts; the consumer imports the helper. Avoid stashing heatwave state on the world snapshot — deterministic by seed+day means it's re-derivable, and the snapshot stays lean.

Implementation notes

Useful context that didn't rise to the level of a spec requirement.

  • Phase 2 + Phase 3 are atomic. The red-build commit in Phase 2 is intentional but must be paired with Phase 3 immediately. Don't push between them — the subagent-driven-development pattern on master means no branch isolation; an intermediate red commit is a hazard if a parallel session does anything before Phase 3 lands.
  • Map-picker assets are static. The Natural Earth SVG is public-domain and committed inline; no build-time fetch. The implementer downloads + prunes the SVG once before writing Phase 4.
  • Weather signature change is opt-in. The new totalCo2Kg: number = 0 parameter defaults to behaviour-preserving 1.0× — so existing test fixtures continue to work without modification, and only the new tests + runtime threading exercise the CO₂ path.
  • Heatwave state is re-derivable. The spec calls it a "new weather event" but the implementation keeps it stateless — RNG keyed by seed + dayIndex + CO₂-threshold means any call to rollHeatwave(seed, dayIdx, kg) returns the same answer. Don't persist to the world snapshot.
  • Bootstrap mutation idiom. Phase 5's picker → save → reassign pattern needs to match the existing main.ts idiom. Re-read main.ts ~1620 (where the periodic saveWorld fires) before committing — if the snapshot is mutable in places, use that path; if it's structurally replaced, use that. Don't invent a third pattern.
  • SunCalc's altitude is in radians. Spec §02 is explicit; tests should not accidentally pass degrees. The new solarMultiplier body uses Math.sin(pos.altitude) which is radians-correct.
  • UI consideration — first-session warning text. The picker modal includes a one-line tip about high-latitude solar variability. This is the player's only warning before the geographic constraint becomes load-bearing on their progression; the text matters.
  • Test extraction discipline. Two helpers (clickToLatLon, shouldShowPicker) get extracted as exports specifically so they're tested. Keep them lean — they're meant to be pure math + a predicate, nothing more.
  • Scope discipline on the climate-crisis chain. The spec's language at §5.4 is generative ("a sequence of escalating storms + heatwaves + drought events"). Phase 6 picks the minimal interpretation (the 1.6× multiplier IS the chain) and documents the deferral. Don't over-design without product clarity.
  • The Phase 0 baseline isn't a wall-clock perf gate. Phase 6 shouldn't measurably change tick performance — one SunCalc call per solar segment is sub-microsecond. The Phase 8 live-save replay captures the numbers as a regression check, not a "must improve by X%" target.
  • Don't restart robot-islands-dev.service for code changes. Per AGENTS.md guidance — restart only on vite.config.ts or package.json deps changes. Phase 1's package.json change IS a restart trigger; subsequent phases are npm run build + browser reload only.
