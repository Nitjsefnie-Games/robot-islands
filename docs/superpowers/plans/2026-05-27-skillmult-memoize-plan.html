Phase 2 — Thread baseMult through hot-path callers

The actual win lands here. Six call sites in economy.ts opt into reading from ctx?.baseMult instead of recomputing.

Critical: do NOT touch lines 733 and 1644

economy.ts:733 and economy.ts:1644 both do const skillMul = effectiveSkillMultipliers(state); layerConditionalBonuses(skillMul, ...); — they need a fresh, mutable copy because layerConditionalBonuses mutates in place (economy.ts:588-613). Reusing baseMult here would corrupt the cache and pollute every subsequent cap() read with the per-segment conditional bonus factor. Keep both lines exactly as they are.

Code-review checklist
For every modified call site:
  • Does it pass the base (unlayered) multiplier? — not a per-segment skillMul that has been through layerConditionalBonuses.
  • Is the optional ? preserved on the new arg? — legacy callers (UI / orbital / drones) must keep working.
  • If the function receives ctx, it reads ctx?.baseMult; otherwise the cached value is passed positionally.
Pseudocode — cap() body
export function cap(
  state: IslandState,
  r: ResourceId,
  override?: Record<ResourceId, number>,
  opts?: { ignoreGrace?: boolean },
  passedMult?: SkillMultipliers,
): number {
  const nominal = override?.[r] ?? state.storageCaps[r] ?? 0;
  if (nominal === 0) return 0;
  const mult = passedMult ?? effectiveSkillMultipliers(state);
  const skillMul = mult.storageCap;
  const cat = RESOURCE_STORAGE_CATEGORY[r];
  const catMul = cat ? mult.storageCategoryCap[cat] ?? 1 : 1;
  const computedCap = nominal * skillMul * catMul;
  if (opts?.ignoreGrace) return computedCap;
  const grace = state.starterInventoryGrace[r] ?? 0;
  return Math.max(computedCap, grace);
}

Phase 3 — Invalidation-correctness test

Pin the load-bearing invariant: when unlockedNodes changes between two advanceIsland calls, the second call sees the new multiplier. Belt + suspenders against future "let's promote baseMult to a per-state cache" drift.

Edit: src/economy.test.ts

Test 1 — invalidation across advanceIsland calls
describe('effectiveSkillMultipliers memoization', () => {
  it('picks up new unlocked nodes between advanceIsland calls', () => {
    const state = makeBaselineIsland({ level: 10 });
    const storageNode = NODE_CATALOG.find((n) =>
      n.effect.kind === 'storageCapMul')!;

    // Tick once without the node. Cap reflects the base.
    advanceIsland(state, state.lastTick + 1000);
    const baseCap = cap(state, 'iron_ore');

    // Unlock the storage node; tick again.
    state.unlockedNodes.add(storageNode.id);
    advanceIsland(state, state.lastTick + 1000);
    const boostedCap = cap(state, 'iron_ore');

    // Second cap MUST reflect the new multiplier — no stale cache.
    expect(boostedCap).toBeGreaterThan(baseCap);
    expect(boostedCap / baseCap).toBeCloseTo(1 + storageNode.magnitude, 3);
  });
});
Test 2 — conditional bonuses do not pollute baseMult
  it('cap() reads the unlayered base, not the conditional-layered mult', () => {
    const state = makeBaselineIsland({
      level: 30,
      conditionalBonusUnlocked: true,   // helper unlocks a storage cb node
    });

    // Tick once — advanceIsland computes baseMult AND computeRates
    // layers conditional bonuses on its own copy.
    advanceIsland(state, state.lastTick + 1000);

    // External UI-style cap call (no ctx, no mult): should see the
    // base storageCap, not the conditional-bonus-layered storageCap.
    const baseMult = effectiveSkillMultipliers(state);
    const expectedCap = state.storageCaps.iron_ore * baseMult.storageCap;
    expect(cap(state, 'iron_ore')).toBeCloseTo(expectedCap, 3);
  });
Test 3 (optional) — call-count assertion via spy
  it('computes effectiveSkillMultipliers at most a small bounded number of times per tick', () => {
    const state = makeBaselineIsland({ level: 25, buildings: 50 });
    const spy = vi.spyOn(skilltreeModule, 'effectiveSkillMultipliers');
    advanceIsland(state, state.lastTick + 1000);
    // Bound: top-of-advance + per-segment computeRates + per-segment
    // battery block. For a 1-segment frame: 3. Pin a generous upper
    // bound so unrelated future reads don't trip the test.
    expect(spy.mock.calls.length).toBeLessThanOrEqual(8);
    spy.mockRestore();
  });

Phase 5 (optional) — Perf-regression test

Trip-wire for future regressions. Generous wall-clock budget so it doesn't flake on CI; specific enough that the spike-back from accidentally re-adding a per-cap recompute would breach it.

Edit: src/economy.test.ts (append at bottom)

describe('advanceIsland perf-regression gate', () => {
  it('completes one frame on a 50-building L25 island in <5ms', () => {
    const state = makeSyntheticHomeIsland({
      level: 25,
      buildings: 50,
      inventoryKeys: 200,
    });

    // Warm-up — first call pays catalog-build cost.
    advanceIsland(state, state.lastTick + 16);
    const warm = performance.now();
    advanceIsland(state, state.lastTick + 16);
    const dt = performance.now() - warm;
    expect(dt).toBeLessThan(5);  // generous 3× projected mean
  });
});

Why optional. CI timing tests are flaky in containers under load. If the project already has a performance-fixture convention, follow it (search src/*.test.ts for perf-regression / performance.now). If it doesn't, treat this as experimental and skip in CI with .skip; run locally when investigating regressions.
