2.1 Calibration policy — anchors locked (rev-4)

Rev-4 anchor 1: 1 unit coal = 1 kg (resolved from §10.2 picker, see §10 for reasoning chain). All other mineral / ore / refined-metal calibrations cascade from there at 1 unit = 1 kg.

Rev-4 anchor 2 (rev-10 derivation corrected): 1 unit power = 100 kW (= 0.1 MW) of nominal electrical output (resolved from §10.3 picker). rev-10 corrected: the rev-4 derivation cited "~80 % efficient combined-cycle conversion" — this was wrong on two counts: (a) 80 % exceeds the Carnot ceiling for any coal-fired Rankine cycle (~47 % ultrasupercritical USC, ~50 % IGCC); (b) "combined-cycle" is gas-turbine terminology and doesn't apply to coal. Correct derivation: coal_gen's 1 coal/2 s × 25 MJ/kg = 12.5 MW thermal at the input boundary; at realistic 40 % efficiency that's 5 MW electrical out = 50 unit per coal_gen. To preserve the 100-unit catalog claim, Phase 2 has two options: (A) reduce output to 50 unit and keep 1 coal/2 s, OR (B) double the burn rate to 1 coal/s (= 25 MW thermal × 40 % = 10 MW = 100 unit). Recommended path: Option A (50 unit), since it preserves the 2-second cycle driving smelter timing while staying internally honest. verified 40-45 % efficiency target per US DOE FECM "Small-Scale Modular Coal-Fired Plants of the Future" + PowerMag "DOE Set to Support Small Modular Coal Units" [WebSearch from rev-4: "small modular coal power plant 10 MW typical capacity efficiency"]. A 5-10 MW single-tile coal_gen sits at the bottom of the 50-350 MW US DOE FECM modular-coal class but is dimensionally consistent.

Rev-4 principle: no human inefficiency. Robots run 24/7/365. Use mechanical / chemical / physical throughput limits, not industry-annualised tonnages. Real-world rates that include shift changes, maintenance windows, weather diversions, weekly inspections, regulatory downtime — all stripped. The 1 kg-coal anchor is verified by this principle: a small continuous mining-cutter at 5-10 t/min stripped to per-m² gives ~2-3 kg/s/m²; the game's mine at 0.9 unit/s on a 4 m² footprint = ~0.22 kg/s/m², well within mechanical max, so 1 unit = 1 kg is internally conservative. verified continuous-miner throughput per Joy 14CM + Sandvik MB770 datasheets (5-15 t/min sustained cutting rate at 250-500 kW machine power).

Three SI dimensions cover the catalog: kg for solids and gases; L for fluids that are dispensed by volume (saltwater, crude_oil); kWh for energy carriers (power buffers, casimir_energy, plasma_charge). The default rule:

  • Bulk minerals + ores (T0 raw): 1 unit = 1 kg. Anchor-locked at coal; all other ores match.
  • Fluids billed by volume (saltwater, fresh_water, crude_oil, natural_gas): 1 unit = 1 L. Saltwater density 1.03 kg/L (game rounds to 1.0; < 3 % error per CRC Handbook of Chemistry & Physics 95e seawater density table).
  • Refined metals + ingots (T1): 1 unit = 1 kg. Mass-conservation through the smelting chain becomes trivial to check (10 iron_ore + 3 coal → 6 iron_ingot + 2 slag + 5 CO kg balances at 13 = 13, see §3.1).
  • Components (T2+ gear, bolt, microchip): 1 unit = characteristic mass of one part. rev-10 contradiction reconciled: §2.1 and §2.2 had drifted apart (§2.1 said "1 bolt = 80 g, 1 microchip = 0.1 kg"; §2.2 said "bolt = 0.05 kg, microchip = 1 kg adjusted"). Canonical values for rev-10 (matching §2.2 table at line 496): bolt = 0.05 kg (single small fastener, ~M12 industrial machine bolt mass per ISO 4014 weight tables — slightly lighter than M16's 80 g; the 50 g number was the §2.2 choice and is now canonical), gear = 2 kg (industrial spur ~150 mm pitch dia, AGMA 9005), microchip = 0.1 kg (silicon-die + IHS per Intel Xeon datasheet — the §2.2 "1 kg adjusted-for-player-friendliness" was an over-write that mis-anchored the audit; §13.4 vault-capacity arithmetic is recomputed against 0.1 kg). §13 storage-cap arithmetic for "20 microchips = 2 kg" stays correct (20 × 0.1 = 2). Cross-reference: §14 self-audit matrix.
  • Power buffer resources (battery, capacitor, flywheel, casimir_energy): 1 unit = 1 kWh stored. Aligns with the existing BATTERY_CAPACITY_WS table.
  • Fantasy resources (ai_core, exotic_alloy, reality_anchor, plasma_charge): 1 unit = nominal "one assembly"; flagged speculative: true, no mass-balance audit.
  • Skill crystals (60 entries, mining_crystal_t1..oceanography_crystal_t3): 1 unit = one crystal; abstract artifact, no SI quantity.

§3.5

Orphan-output tagging schema

New design principle (rev 2, 2026-05-27): Real chemistry produces byproducts. Not every byproduct has a consumer today. Instead of erasing the orphan to keep recipes "clean", we ship the orphan with an explicit terminal tag so future content has hooks waiting, and the v3 audit's F12 dead-resource test can distinguish intentional orphans from accidental ones (mercury).

3.5.1 The terminal field — three values

Every produced resource (and, by extension, every recipe output) carries one of:

  • 'consumed' — has at least one downstream recipe input today. Default for the active chain.
  • 'expansion-hook:<note>' — no consumer today, but its existence is intentional. The note string documents what it would unlock when consumers ship. The auditor must accept this resource as legitimately produced.
  • 'gameplay-sink' — drained via a mechanism other than a recipe (e.g. skill-tree crystal binding, vehicle fuel-on-arrival, XP scoring). No recipe-graph consumer is expected.

3.5.2 Where the tag lives

Two schema options to pick in §10.9:

  • Per-resource (in recipes.ts ResourceMeta): simpler — one tag per resource, applies to every recipe that produces it.
  • Per-recipe-output: richer — same resource can be 'consumed' from one source and 'expansion-hook' from another (e.g. H₂ from electrolyzer is consumed, H₂ from chlor-alkali is expansion). Costs ~2× the schema verbosity.

The recommendation is per-resource: simpler audit logic, easy to upgrade to per-output later if a real conflict appears. The H₂ case is fine as 'consumed' (existing electrolyzer consumer keeps it active).

3.9 T2 mechanical-components cohort — mass-balanced under §2.2 anchors NEW · rev-14

Cohort scope. Seven T2 per-piece manufactured components (steel_beam, heavy_cable, sheet_metal, pipe, wire, spring, bearing) share a single broken pattern in the pre-rev-14 catalog: each carries a trivial 1 steel → N component recipe that, under §2.2's industrial unit-mass anchors, is mass-imbalanced by orders of magnitude. For thick products (steel_beam = 50 kg/unit, heavy_cable = 8 kg, sheet_metal = 5 kg, pipe = 4 kg), 1 steel → 2 component is mass-positive 100× / 16× / 10× / 8×. For thin products (wire = 0.5 kg, spring = 0.2 kg), 1 steel → N component is mass-negative when N is too small, and the lost mass goes nowhere in the integrator.

Rev-14 fix direction (user-chosen): keep §2.2's realistic-industrial unit masses (50 kg/beam, 8 kg/cable, etc.) and rebalance every cohort recipe under the 1-steel = 1-kg anchor. Hot rolling has two real mass losses (mill scale at the hot surface oxidises to FeO/Fe₂O₃/Fe₃O₄ flakes; crop / trim at the ingot ends), totaling 3-8 % of input mass. Per the WorldSteel "Steel industry by-products fact sheet" + Degarmo/Black/Kohser Materials and Processes in Manufacturing 11e ch. 19 (rolling-mill yields 90-97 %, worse for thinner products), rev-14 picks 95 % yield for thick products (beam, cable, sheet, pipe; few passes, low scale) and 92 % yield for thin products (wire, spring; many passes, more scale). Bearing is an assembly (no rolling-yield loss; ~90 % steel + 10 % lubricant by mass). The 3-8 % mass gap routes to a new mill_scale resource (§2.2 hot-rolling byproducts row + §3.5.5 expansion-hook entry).

Tool used: scripts/recalc-rolling-mill.py (committed at 87f9620) derives each integer recipe from the unit-mass + yield inputs. §5

Heat-flux engine respec

Current heat engine (heat.ts:1-7): N:1 adjacency gate, one source serves unlimited consumers, no flux metering. v3 F16 (severity MEDIUM): a 200 W Plasma Heater serves N×60 W Coke Ovens at zero incremental cost — energy violation by design.

5.1 Proposal — thermal-kW budget on top of adjacency

Adjacency stays as the placement gate (you still need a heat source next door). New layer: each source declares thermalKW, each consumer declares heatDemandKW. When the same source serves multiple consumers, the engine sums demand vs supply per source and proportionally throttles on deficit. A 200 W Plasma Heater (180 kW thermal at 90 % efficiency) can fully drive 3 Coke Ovens at 60 kW each, partially drive 4 (45 kW each), and rate-throttle 5+.

5.2 Pseudocode (TypeScript-style) — before / after

BEFORE — heat.ts:124-205 (current)
// resolveHeatAssignments — adjacency only
for (const consumer of sortedConsumers) {
  const border = borderTiles(footprintKeySet(consumer));
  // any free source adjacent?
  if (anyOf(freeSources, sourceTouchesBorder(s, border))) {
    hasHeat.set(consumer.id, true);
    continue;
  }
  // any coal source adjacent?
  const coal = firstOf(sortedCoal, sourceTouchesBorder);
  if (coal) {
    hasHeat.set(consumer.id, true);
    coalConsumers.set(coal.id, (coalConsumers.get(coal.id) ?? 0) + 1);
  } else {
    hasHeat.set(consumer.id, false);
  }
}
AFTER — thermal-kW metered
// Phase 1: discover adjacency edges (unchanged)
const edges = buildAdjacencyEdges(sources, consumers);

// Phase 2: per-source budgets
for (const src of sources) {
  const adjConsumers = edges.consumersOf(src);
  const supply  = src.def.heatSource.thermalKW;
  const demand  = sum(adjConsumers, c => c.def.heatDemandKW);
  const ratio   = Math.min(1, supply / Math.max(demand, 0.001));
  for (const c of adjConsumers) {
    // Multiple sources? Pick max ratio (best feed)
    const prev = heatFactor.get(c.id) ?? 0;
    heatFactor.set(c.id, Math.max(prev, ratio));
  }
}

// Phase 3: economy gate downstream
for (const c of consumers) {
  const r = heatFactor.get(c.id) ?? 0;
  hasHeat.set(c.id, r >= MIN_HEAT_FACTOR);  // e.g. 0.1
  heatThrottleFactor.set(c.id, r);          // computeRates multiplies
}

6.1 Resolution — Option A (constant work) rev-10 CORRECTED for cubic

corrected Rev-9 and earlier encoded the wrong drag law. The pre-rev-10 table held speed × tilesPerFuel = const, which makes fuel-per-second scale as v², NOT v³. Across T1→T4 (0.25 → 1.00 t/s for ships) the pre-rev-10 numbers gave fuel/sec ratios (1, 2.56, 5.76, 16) — the v² curve. Real cruise drag is P ∝ v³ (doubling speed → 8× engine power, verified per Geography of Transport Systems "doubling a ship's speed requires eight times more engine power"), which demands fuel/sec ratios (1, 4.1, 13.8, 64) across T1→T4.

Correct invariant: speed² × tilesPerFuel = const. Then fuel-per-second = speed × (1/tilesPerFuel) = speed × (speed²/c) = speed³/c — cubic, matching real drag. Rev-10 rederives the SHIP_STATS / HELICOPTER_STATS tables under this correct rule.

7.1 Sinusoidal solar curve (rev-4: theoretical clear-sky, no human inefficiency)

Current curve (daynight.ts:83-97) is trapezoidal: dawn 0→1 linear 6 h, day 1 for 6 h, dusk 1→0 linear 6 h, night 0 for 6 h. Integral over 24 h = 12 effective hours = 50 % capacity factor. Real-world reported CF: 21-34 % per verified NREL ATB 2024 utility-scale PV capacity factor by resource category [WebSearch: "solar photovoltaic capacity factor real world annual average NREL"]. BUT that includes cloud cover, dust, snow, soiling, panel degradation, inverter losses, scheduled maintenance — all human/environmental inefficiencies.

Rev-6 resolution: user picked sin-scale in §10.7 — amplitude-scaled sinusoidal to 22 % annual-average CF. This is a deliberate exception to the rev-4 no-human-inefficiency principle for solar specifically: the player-facing nameplate W stays interpretable as "real-world expected W", baking in cloud / dust / soiling / maintenance / degradation losses to match NREL ATB 2024 annual-average data. Half-sine with amplitude 0.69× nameplate peaks at solar noon, integrating to 0.22 of the 24-h day.

Implementation: half-sine over the 12 daylight hours, zero overnight, scaled by amplitude factor 0.69 so the 24-h integral lands at 0.22 of nameplate-W × 24 h.

export const SOLAR_AMPLITUDE = 0.69;  // targets 22 % annual-avg CF (rev-6)

export function solarMultiplier(nowMs: number): number {
  const p = dayPhase(nowMs);
  // Sun above horizon during phase ∈ [0.125, 0.625] — 12 h day, 12 h night.
  if (p < 0.125 || p >= 0.625) return 0;
  // Map [0.125, 0.625] → [0, π] for the half-sine.
  const t = (p - 0.125) / 0.5;
  return SOLAR_AMPLITUDE * Math.sin(t * Math.PI);  // peaks 0.69 at solar noon
}
// 24-h integral verification:
//   ∫₀^½ 0.69·sin(2πt) dt = 0.69 · (1/π) ≈ 0.69 · 0.318 = 0.22 day-fraction
//   = 22 % capacity factor. Matches NREL ATB 2024 utility-scale PV CF range
//   (21-34 %) at the lower-typical end. WebSearch verified.

13.2 Architectural choice — per-resource base cap with building scale

Rev-7 proposed schema:

  • Each resource gets a baseCap field (kg, defaults from category). dry_goods default 100 kg; liquid_gas 100 kg; temp_sensitive 50 kg; components 20 kg (small parts, by count weight); rare 1 kg (precious materials are inherently scarce); fantasy sub-calibrated resources (helium_3, antimatter_propellant) get explicit per-resource baseCap reflecting their sub-anchor.
  • Each storage building declares a capAdd in kg per category, calibrated against real-industrial analogs (§13.3 table).
  • Architecture: additive (linear), not multiplicative. Effective cap = baseCap + Σ buildings.capAdd[category]. Reasoning: multiplicative compounds too fast (10 silos = 100^10 = absurd); diminishing returns better modeled by gameplay scarcity of buildable tiles than by mathematical decay. Linear additive matches the current code shape, so the migration is data-only (no algorithm change in aggregateStorageCaps).
  • What changes: per-resource baseCap replaces the global BASELINE_STORAGE_CAP = 2000. Per-building capAdd values are recalibrated to real-industrial scales. Algorithm stays the same.
