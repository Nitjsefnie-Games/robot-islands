Phase 3 — Drain integrator hook

The runtime payoff. Append the drain block to advanceIsland's segment loop, immediately after Phase 2 of the rework's accrual hook. Reads def.co2CaptureKgPerCycle, applies the adjacency gate when set, scales by segment duration and recipe cycle, clamps at zero.

Files

  • src/economy.ts — segment integrator inside advanceIsland, after the existing CO₂ accrual hook (approx line ~1600 per the rework's Phase 2 plan).

Reuse — adjacency helper

The 4-neighbor matching logic is already implemented in adjacency.ts:93 as the private neighborMatches + the public computeBuffStack path. Phase 5 doesn't reuse those directly — it needs a much lighter "does ANY neighbor's defId appear in the gate list?" check. Inline that as a small local helper inside economy.ts (or factor into a new hasAnyNeighborWithDefId export in adjacency.ts — implementer choice; the former is smaller, the latter is symmetric with existing factoring).

Pseudocode — drain block

// economy.ts — append to the segment integrator, after Phase 2's accrual.
// `segmentDurationSec` is the integrated time slice; `b` is a placed
// building; `def` is the matching BuildingDef.

for (const b of state.buildings) {
  const def = BUILDING_DEFS[b.defId];
  const capture = def.co2CaptureKgPerCycle ?? 0;
  if (capture <= 0) continue;

  // Adjacency gate — present on exhaust_scrubber only in Phase 5.
  if (def.co2CaptureAdjacency) {
    const has = hasNeighborWithAnyDefId(b, state.buildings, def.co2CaptureAdjacency);
    if (!has) continue;
  }

  // Cycles this segment — derived from recipe cycleSec, scaled by the
  // running factor (heat throttle, brownout, etc.). resolveRecipe is
  // the same path the rate integrator uses; running factor mirrors the
  // production scaling so a half-throttled scrubber drains at half rate.
  const recipe = resolveRecipe(def, b, terrainAt, ctx?.inventory ?? state.inventory);
  if (!recipe) continue;  // recipe-less building somehow → skip
  const factor = runningFactorFor(b, state, recipe);
  const cyclesThisSegment = (segmentDurationSec / recipe.cycleSec) * factor;

  const drainKg = capture * cyclesThisSegment;
  state.co2Kg = Math.max(0, state.co2Kg - drainKg);
}

Helper — hasNeighborWithAnyDefId

Local to economy.ts (or co-located with the rest of the adjacency primitives if the implementer prefers factoring). Walks the focal's 4-neighbor border via the existing borderTiles / footprintKeySet primitives from adjacency.ts; returns true on first matching neighbor.

function hasNeighborWithAnyDefId(
  focal: PlacedBuilding,
  buildings: ReadonlyArray<PlacedBuilding>,
  defIds: ReadonlyArray<BuildingDefId>,
): boolean {
  const fp = footprintKeySet(focal, BUILDING_DEFS);
  const border = borderTiles(fp);
  const wanted = new Set(defIds);
  for (const other of buildings) {
    if (other.id === focal.id) continue;
    if (!wanted.has(other.defId)) continue;
    if (touchesBorder(other, border, BUILDING_DEFS)) return true;
  }
  return false;
}
Load-bearing detail — zero clamp
The Math.max(0, state.co2Kg - drainKg) clamp is essential. Phase 6 (Weather) reads state.co2Kg for storm-probability thresholds. If the integrator allowed negative values, a heavily-scrubbed island would have negative CO₂ — when summed for world-total emissions, Phase 6's weather logic would behave erratically (negative summands cancelling other islands' emissions). The clamp is the correctness contract.
Order of operations
Per spec §04 closing paragraph: accrual fires first, drain second within each segment. This matches real industrial behaviour — a scrubber captures emissions after the gen produces them, not before. Don't reorder.

Implementation notes

Useful context for the implementer that didn't rise to the level of a spec requirement.

  • plant_a_tree biome gate. The requiredBiomes: ['forest'] field plugs into the existing biome-gate infrastructure (same path used by Pyroforge's volcanic gate at building-defs.ts:1686). No new code; just data. Test 7 has to thread a forest terrainAt through makeIsland for the recipe to fire.
  • Recipe "running" semantics for no-op recipes. The integrator already handles maintenance recipes with empty outputs (recipes that consume but produce nothing). The no-op recipe (empty inputs AND empty outputs) is an extension of that pattern.
