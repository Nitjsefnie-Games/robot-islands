1.4 Append proportional-budget loop

Insert at the tail of resolveHeatAssignments, AFTER the existing priority A/B/C loop and BEFORE the final return. The loop reads assignedSource (already populated by priority B for coal sources and priority A for free sources) to know which consumers were attached to which source.

// heat.ts — append before the `return` at line 205.
const heatThrottleFactor = new Map<string, number>();

// Group consumers by their assigned source. Consumers with no source
// (priority C — hasHeat=false) get no throttle entry; downstream
// `?? 0` reads as below-threshold and the existing baseRate=0 branch
// in computeRates handles them.
const consumersBySource = new Map<string, PlacedBuilding[]>();
for (const consumer of sortedConsumers) {
  const srcId = assignedSource.get(consumer.id);
  if (!srcId) continue;
  const list = consumersBySource.get(srcId) ?? [];
  list.push(consumer);
  consumersBySource.set(srcId, list);
}

// For each source: sum demand, compute ratio, propagate per consumer.
for (const [srcId, srcConsumers] of consumersBySource) {
  const src = buildings.find((b) => b.id === srcId);
  if (!src) continue;  // defensive — assignedSource was populated from this list
  const srcDef = BUILDING_DEFS[src.defId];
  const supply = srcDef.heatSource?.thermalKW;

  if (supply == null) {
    // Backward-compat: source has no thermalKW → boolean N:1 behaviour.
    for (const c of srcConsumers) heatThrottleFactor.set(c.id, 1);
    continue;
  }

  const demand = srcConsumers.reduce((acc, c) => {
    const d = BUILDING_DEFS[c.defId].heatDemandKW ?? 0;
    return acc + d;
  }, 0);

  if (demand <= 0) {
    // No measured demand from any consumer (untabled) → all run at 1.0.
    for (const c of srcConsumers) heatThrottleFactor.set(c.id, 1);
    continue;
  }

  const ratio = Math.min(1, supply / demand);
  for (const c of srcConsumers) {
    // Multiple sources potentially feeding same consumer: take max ratio.
    // Today's heat.ts assigns 1:1 (single source per consumer), but the
    // max-aware update is cheap and future-proofs the loop.
    const prev = heatThrottleFactor.get(c.id) ?? 0;
    heatThrottleFactor.set(c.id, Math.max(prev, ratio));
  }
}

// Return statement extended.
return { hasHeat, coalConsumersByFurnace, assignedSource, heatThrottleFactor };

Implementation notes

Context for the implementer that didn't rise to the level of a spec requirement.

  • Order matters but isn't load-bearing. Phase 1 (resolver) + Phase 2 (data backfill) can land in either order — neither has runtime effect because computeRates doesn't read the new field until Phase 3. Phase 3 must land AFTER both; Phase 4 (drag retable) is independent and can land at any point.
  • Phase 1's geothermalActive short-circuit is easy to miss. The current resolver has TWO early-return paths (geothermalActive at line 152, empty-consumers at line 159). Both must populate the new heatThrottleFactor field (empty Map is fine for the second). The strict-mode build will catch a missing field at the return site.
  • The throttle's "max over multiple sources" clause. Today's heat.ts assigns each consumer to exactly one source (priority A then B). The max-aware update in the proportional-budget loop (Math.max(prev, ratio)) is defensive — a future change that assigns a consumer to multiple sources gets the best supply without further code. Cost: zero. Benefit: insurance.
  • Where to multiply the throttle in pass-4. The spec §04 pseudocode shows recipeRateFactor *= factor. Find the equivalent local in economy.ts's actual computeRates implementation — it may be named differently (the spec is sketching, not citing). The end-to-end test in Phase 5 §07 catches a wrong placement: if the throttle is multiplied into baseRate vs effectiveMul when the other multipliers are also present, the magnitude can drift in subtle ways. The integration test asserts the FINAL rate.
  • Settlement.ts has tuningFor, not direct table reads. Per settlement.ts:155, every consumer reads SHIP_STATS / HELICOPTER_STATS via the tuningFor(kind, tier) helper. Phase 4 changes ONLY the literals at lines 119 / 126; tuningFor is unchanged. No call-site edits needed.
  • VehicleStats interface stays identical. The shape at settlement.ts:111-117 doesn't change — only the numbers in the records do. tsc-b drift is impossible.
  • Vehicles in flight at retable time. SettlementVehicle has readonly speed and readonly fuelLoaded at dispatch time. After Phase 4 commits, an in-flight vehicle uses its committed speed (the old, slower value) and committed fuel-burn rate. The NEW table only governs the NEXT dispatch. No retroactive harm; no migration.
  • The "smelter exemption" needs a deliberate decision. Plan §03.3 stages this as "leave smelter untouched"; the spec §03.2 table lists 50 kW. If the user disagrees and wants smelter in, the change is one line + a fixture-sweep across smelter-using tests. Surface this as a question in the PR description.
