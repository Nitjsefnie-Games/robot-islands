Remove redundant manufacturing stubs & sync SPEC §8.3 — Implementation Plan

Plan summary
Goal: Delete the 3 redundant catalog stubs fabricator, precision_lab, singularity_forge and rewrite SPEC §8.3 to enumerate the implemented manufacturing taxonomy.

Tech Stack: TypeScript (strict, noUnusedLocals), Vite 5, vitest. Markdown for SPEC.md.

Task 2: Rewrite SPEC §8.3 & fix §1272

Markdown-only — no build impact. SPEC.md is the project's locked specification; keep the change scoped to the §8.3 table and the one §1272 sentence. The §8.3 prose heading/intro and every other section stay untouched.

Files
  • Modify: SPEC.md — §8.3 table (line ~810), §1272 (line ~1272)
Step 1: Derive the manufacturing roster from source

Do not trust a hand-copied list. Enumerate the authoritative set from src/building-defs.ts — every BUILDING_DEFS entry whose category is 'manufacturing' (the 3 stubs are already removed by Task 1, so they will not appear). Read the file and collect every def block, recording id, displayName, tier, footprint for those with category: 'manufacturing'; group by tier. This yields 68 buildings across T1–T6.

The expected counts, as a checksum: T1 = 6, T2 = 19, T3 = 15, T4 = 6, T5 = 14, T6 = 8 — total 68. If your enumeration disagrees, reconcile before writing the table. The full roster (defId · display name · footprint) is in the design spec §04 — use it as the cross-check, not the source.

Step 2: Build each row's Notes from its recipe

The §8.3 table columns are Building | Footprint | Tier | Notes. For each building, the Notes cell is a short phrase naming its primary output, taken from RECIPES[defId].outputs in src/recipes.ts. Worked examples:

  • motor_assembly → recipe outputs electric_motor → Notes: "Electric motors"
  • circuit_assembler… is category: 'electronics', so it does not appear in §8.3 — only 'manufacturing' category buildings do.
  • pump_assembly → outputs pump → Notes: "Pumps"
  • kit_assembler_refined → outputs foundation_kit_refined → Notes: "Refined foundation kits"

If a manufacturing building has no RECIPES entry, write Notes as "—" rather than inventing one, and note it in the commit message — but after Task 1 the only recipe-less manufacturing defs should be gone, so flag any survivor as a finding.

Step 3: Replace the §8.3 table

In SPEC.md, §8.3 (heading ### 8.3 Manufacturing, ~line 810) currently has a 6-row table. Replace the table body — keep the heading and the |Building|Footprint|Tier|Notes| header row — with the 68-row roster from Steps 1–2, grouped in tier order (T1 first). Notes & verification

Strict TypeScript is the safety net. tsconfig.json has strict + noUnusedLocals + noUncheckedIndexedAccess. Any reference to a removed defId — including BUILDING_DEFS.fabricator property access — fails npm run build. Task 1 cannot be silently incomplete.
No UI / live check needed. The 3 buildings had no recipe and no effect code; removing them changes no rendered behaviour. The build + test suite fully cover this change — there is nothing to screenshot.
Accepted risk (LOW). A pre-existing save with one of the 3 buildings placed would, after removal, hold a PlacedBuilding whose def lookup is undefined. Not handled — these buildings were never functional, the project is pre-release, no migration infrastructure exists. Noted, not mitigated.
Out of scope. terrain_modifier (a genuine unbuilt mechanic) and the parallel drift in SPEC §8.1 / §8.2 are deliberately untouched — see design spec §07.
