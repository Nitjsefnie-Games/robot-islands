Implementation plan · robot-islands · skilltree rebalance · 2026-05-25

Skill-tree rebalance — cap-per-pool via calibrated magnitudes

For agentic workers REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development. Fresh kimi subagent per task + two-stage review (spec compliance reviewer → code quality reviewer). Trailer literal on commits: Co-Authored-By: Kimi K2.6 <noreply@kimi.com>.

Goal: reshape the skill catalog so every multiplier-style pool caps at ×10 (or ÷10 / +N) via calibrated per-node magnitudes — no runtime clamp. Eliminates the ×10⁹ Mine runaway observed on the user's L128 save while preserving the catalog topology (sub-paths, edges, costs).
Architecture: wire 4 new SkillEffect kinds + matching SkillMultipliers fields end-to-end (engine wiring); add 6 new filler archetypes (4 new yield-bonus chains + 2 cliff-fill chains for N=1 effects); re-magnitude every hand-curated catalog entry per the spec §03 table; re-tune existing archetype baseMag values; ship a v13 → v14 migration that resets level/xp/skillTree.{unlockedNodes, unspentSP, totalSpent} per island while preserving buildings + inventory; lastly land the magnitude calculator as a permanent dev script + a vitest invariant that asserts per-effect product ≈ pool target (so future catalog additions fail CI until magnitudes are re-derived).
Tech stack: Vite 5 + TypeScript strict + PixiJS 8 + vitest. strict + noUncheckedIndexedAccess + noUnusedLocals.
Source of truth: design spec at docs/superpowers/specs/2026-05-25-skilltree-rebalance-design.html (v1, locked 2026-05-25). All magnitudes in this plan are reproduced from spec §03 + §04; if a number disagrees, the spec wins.
Locked design choices (do NOT deviate):
  • Caps per pool: shared pools (Mine/Logger/Drill/Aqua/Patronage/T5-extractor) joint ×10, split 50/50 between recipeRateMul:extraction and the per-building yield bonus → each side ×√10. Singleton multipliers ×10 each. Singleton divisors ÷10 each. xpGainMul tightened to ×3. parallelBuildCapAdd +4 total. launchSuccessAdditive +0.90 total.
  • Per-node magnitudes are computed via m = C^(w/N) − 1 where C = pool cap, w = effect's share of the pool budget, N = catalog count. NO runtime clamp anywhere.
  • 4 new SkillEffect kinds: drillYieldBonusMul, aquacultureYieldBonusMul, patronageYieldBonusMul, t5ExtractorYieldBonusMul. Mirror the existing mineYieldBonusMul / loggerYieldBonusMul shape exactly (no category / no extra fields).
  • 6 new archetype entries: 4 yield-bonus chains (one per new kind) + 2 cliff-fill chains (manufacturing in storage sub-path, batteryCapacity in power_systems sub-path). All count: 8.
  • Migration v13 → v14: reset per-island level=1, xp=0, skillTree.unlockedNodes=[], skillTree.unspentSP=0, skillTree.totalSpent=0. Preserve buildings, inventory, drones, routes, satellites, all other state. STORAGE_KEY bumps to 'robot-islands:save:v14'.
  • Cost curve + SP grant unchanged. No edits to costForDepth, skillPointsForLevelUp, cumulativeSkillPointsForLevel. The cap absorbs the runaway naturally.
  • FAMILY-set membership: only include building-defIDs that actually exist in src/building-defs.ts. Verify by grep at the time of implementation. The spec lists candidate IDs; do NOT add IDs that don't resolve. An empty FAMILY set is acceptable (the multiplier folds but no building reads it yet).

// scripts/skilltree-magnitudes.ts — dev script. Computes per-effect-kind

import { FULL_CATALOG } from '../src/skilltree-catalog.js';
import type { SkillEffect } from '../src/skilltree.js';

function effectKey(e: SkillEffect): string {
  if (e.kind === 'recipeRateMul') return `recipeRateMul:${e.category}`;
  if (e.kind === 'storageCategoryCapMul') return `storageCategoryCapMul:${e.category}`;
  if (e.kind === 'xpGainMul') return `xpGainMul:${e.category ?? '(global)'}`;

export function computeProducts(): Map<string, number> {
  const products = new Map<string, number>();
  for (const n of FULL_CATALOG) {
    if (!('magnitude' in n)) continue;
    const key = effectKey(n.effect);
    if (!(key in POOL_TARGETS)) continue;

  return products;
