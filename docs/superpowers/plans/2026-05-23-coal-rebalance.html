Coal loop rebalance — Implementation Plan

Plan summary
Goal: Make the mine_on_coal + coal_gen pair self-sustain at 1:1 with comfortable headroom on both fuel and power. Move mine_on_coal from 1 coal / 17 s to 9 coal / 10 s (mirroring commit 9cda2aa's Heavy-Logger pattern), and drop BUILDING_DEFS.mine.power.consumes from 40 W → 25 W. Result at 1:1: +0.4 coal/s · +75 W.
Architecture: Single-task numeric balance change. Two source lines (src/recipes.ts:1077-1080, src/building-defs.ts:612) plus a swept update to closed-form power and coal-output assertions in economy.test.ts and routes.test.ts. Pure layer only — no PixiJS, no rendering changes.
Tech Stack: TypeScript (strict, noUncheckedIndexedAccess, noUnusedLocals), Vite 5, PixiJS 8, vitest.
Spec: docs/superpowers/specs/2026-05-23-coal-rebalance-design.html (v1) · Audit: docs/superpowers/audits/2026-05-23-coal-loop-balance.html · Reference commit: 9cda2aa (Heavy Logger / Biomass Plant)
User picks (locked): tune_direction = boost_mine, coal_yield_per_cycle = 9_per_10s, mine_power_draw = 25W.
For agentic workers: REQUIRED SUB-SKILL: use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (- [ ]) syntax for tracking. Mechanical edit-test-fix loop — well-suited to a Kimi K2.6 subagent brief.

Notes

Spillover risk — iron-mine power cost drops too. BUILDING_DEFS.mine backs both mine_on_ore and mine_on_coal via runtime resolveRecipe. Dropping power.consumes from 40 → 25 also drops the iron mine's draw, which the user pick (mine_power_draw=25W) implicitly accepts. Downstream effects: smelter / workshop / bolt chain become cheaper to electrify by 37.5% per mine; any balance assumption pinned on the 40 W iron-mine line now lives on 25 W. The routes.test.ts sweep in Task §02 step 4 is the visible surface of this spillover — every computeCableNetworkBalance fixture's consumedTotal and requiredTransmission rebases onto the new constant. If a future test or balance audit treats 40 W as the canonical iron-mine number, this is the entry point to chase.
Skill-tree mineYieldBonusMul interaction. src/skilltree.ts:521 defines a depth-2 mining node whose effect (mineYieldBonusMul) multiplies every mine recipe rate, not just mine_on_ore — the lookup path has no recipe-id discrimination. With the post-rebalance base of 0.9 coal/s and a maxed mining sub-path (~1.3× typical, or +33% for two stacked nodes per SPEC.md), the effective coal rate climbs to roughly 0.9 × 1.3 ≈ 1.17 coal/s (net +0.67 coal/s after the 0.5 gen burn). The audit's headline +0.4 number is base, pre-skill-tree — expected and not a bug.
Downstream coal consumers still in budget. Smelter burns coal at 1 / 27 s ≈ 0.037 /s; Workshop at 1 / 33 s ≈ 0.030 /s. The +0.4 coal/s base headroom (after the generator's 0.5 /s burn) supports up to ~10 smelters or ~13 workshops per coal pair before going coal-negative — comfortable margin. Coal is meant to be both fuel and process input (see audit §6); the rebalance was sized for that dual role.
Tests that look like they should break but don't. economy.test.ts line 569's mineHeavyCatalog() spreads BUILDING_DEFS.mine then overrides power: { consumes: 80 }. The override fully replaces the base — the 25 W change doesn't leak through. Lines 694 (power.consumed).toBe(140)) and 695 (factor === 100/140) keep working unchanged. Similarly economy.test.ts tests in §5.1 that pin workshop draw (60 W) are unaffected because the workshop def is untouched.
Comments mentioning 40 W are descriptive, not load-bearing. economy.test.ts has ~13 inline comments mentioning "40 W" (e.g. "Mine 40 W consumer", "40 W zeroed; workshop 60 W remains") — these don't drive test outcome but lying around stale will confuse future readers. Step 3 above flags the high-value ones to refresh; mass-replacing every comment is not required.
Pure layer only — no PixiJS, no rendering touches. Changes confined to recipes.ts, building-defs.ts, and two test files. The HUD pulls power values dynamically from the def catalog and recipe table — no hard-coded 40 W or 17 s anywhere in the render layer to chase. Run npm run build as a strict-mode sanity pass; the live preview tab needs a manual reload to pick up the new bundle.
