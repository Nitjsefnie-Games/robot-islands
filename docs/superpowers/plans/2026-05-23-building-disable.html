Per-building enable/disable toggle — Implementation Plan

Plan summary
Goal: Add a per-building disabled?: boolean flag toggled from the inspector. A disabled building is functionally non-existent — no power flow, no recipe in/out, no gate provisioning, no maintenance accrual, no drone/satellite/route dispatch from it. Toggling is free, instant, lossless; disabling does trigger route drain (one-way) per the picks.
Architecture: One chokepoint at economy.ts:617 covers power, recipes, gates, heat, and buff adjacency in a single line. A second site at economy.ts:1627 freezes operatingMs. A shared hasOperationalBuilding(buildings, defId) helper replaces ~25 scattered provider-scan call sites so disable, invalid, and operational checks all stay in lock-step. Render-side cue (0.40 alpha + dashed red outline) lives in building-alerts-overlay.ts, mirroring the construction-tint precedent. New inspector button in inspector-ui.ts matches the convertBtn styling stamp; DOM-disabled while the target is under construction.
Tech Stack: TypeScript (strict, noUncheckedIndexedAccess, noUnusedLocals, noUnusedParameters), Vite 5, PixiJS 8, vitest.
Spec: docs/superpowers/specs/2026-05-23-building-disable-design.html (research §01, change inventory §04, verification §05). Step 2: Implement hasOperationalBuilding in src/buildings.ts

Append to src/buildings.ts (after the PlacedBuilding interface declaration, before renderBuildings). The helper bakes in the union of the three runtime filters and the future disabled filter so all call sites share one predicate.

/** Returns true iff at least one placed building of `defId` in `buildings`
 *  is operational: not invalid, not still under construction, not
 *  player-disabled. Accepts both `state.buildings` and `spec.buildings`
 *  (the arrays are aliased per `world.ts:1036`'s
 *  `makeInitialIslandState`).
 *
 *  Consolidates the ~25 scattered `buildings.some(b => b.defId === '…')`
 *  provider-scan call sites so adding a new "is this thing actually
 *  available?" filter (e.g. §NEW disabled toggle) is a single-line edit
 *  here instead of a fan-out across the tree. Pure. */
export function hasOperationalBuilding(
  buildings: ReadonlyArray<PlacedBuilding>,
  defId: BuildingDefId,
): boolean {
  for (const b of buildings) {
    if (b.defId !== defId) continue;
    if (b.invalid === true) continue;
    if ((b.constructionRemainingMs ?? 0) > 0) continue;
    if (b.disabled === true) continue;
    return true;
  }
  return false;
}

The b.disabled === true branch references a field that doesn't yet exist on PlacedBuilding in this commit — that's fine: TypeScript structural typing accepts any property access; b.disabled returns undefined for all current placements, and undefined === true is false. Task 2 adds the field officially. (If tsc objects with strict, fall back to ((b as unknown) as { disabled?: boolean }).disabled === true until Task 2 lands.)

state.buildings === spec.buildings — array aliasing. makeInitialIslandState at world.ts:1036 assigns the spec's buildings array directly to the new state (no clone). Mutating a building through state.buildings[i] is the same as mutating spec.buildings[i]. The helper takes ReadonlyArray<PlacedBuilding> so both call patterns hit the same predicate. Inspector mutation via target.building.disabled = !target.building.disabled is therefore visible to all of economy.ts, antenna.ts (which iterates spec.buildings), and persistence (which serializes s.buildings.map(b => ({...b, ...}))) — no extra propagation step.
complete_then_block interpretation. In-flight drones / satellites / route cargo do NOT re-check their source building's disabled state mid-flight — they reference the source by id in their own structures (route.inFlight, drone expectedReturnTime, satellite state). Their completion paths land cargo / return drones / deliver satellite outputs normally. Only the dispatch path (which calls hasOperationalBuilding through Task 1) is blocked while the owner is disabled. The result: a player who disables a Drone Pad with three drones in flight watches the drones return and idle on the pad; new launches are blocked until re-enable. This mirrors p_routes_pending_inbound=deliver_anyway exactly — both pickers agreed that pre-dispatched work completes.
The route-drain handshake — the one place "lossless" breaks. Per p_routes_disabled_source=route_drains_and_removes, disabling a route-owner triggers drainRoutesForBuilding(world, building.id) — the same call demolish makes. Routes wind down (in-flight cargo lands, then the route is pruned by tickRoutes). Re-enabling the building does NOT restore the routes. This is the single asymmetry in an otherwise-symmetric toggle. The button's title tooltip surfaces this explicitly ("Drains routes owned by this building. Free; reversible — but drained routes are not restored.") to avoid the player learning by surprise.
Construction guard is UI-only (defence-in-depth). Per p_constructed_disable=no_finish_first: the engine never wraps tickConstruction in a disabled-guard, because the UI prevents the toggle while construction is in progress (DOM disabled attribute + early-return in both the button click handler and the action handler). Construction proceeds normally on any disabled-but-under-construction building (which only arises through hand-edited saves); the disabled bit kicks in once constructionRemainingMs reaches 0 and the building stops short-circuiting through the construction branch. The maintenance-loop if (b.disabled) continue; at economy.ts:~1640 is unreachable through the UI flow but is correct defence — a hand-edited save with both bits set would otherwise accrue operatingMs as soon as construction completes, before the player notices.
Verification suite — exact test names from spec §05. The four describe strings in Task 3 step 1 ("disabled building contributes 0 to power balance", "disabled building does not accrue operatingMs", "disabled provider fails downstream gates", "disable is lossless across save round-trip") are lifted verbatim from the spec §05 verification table. Use the same strings so future-you can grep -r "disabled building contributes" src/ to find both the spec entry and the test. The "construction freezes while disabled" row in the spec §05 belongs to the alternative pick (yes_pauses_construction) and is intentionally absent here — its slot is taken by an inspector-button test that asserts the button is DOM-disabled while constructing.
Persistence — no schema bump. persistence.ts:329's s.buildings.map(b => ({ ...b, ...timestamp-shifts })) shallow-copies every field of PlacedBuilding by spread. Adding disabled?: boolean rides through automatically; legacy saves load with the field absent (treated as undefined, which the helpers compare against === true so undefined and false are equivalent).
